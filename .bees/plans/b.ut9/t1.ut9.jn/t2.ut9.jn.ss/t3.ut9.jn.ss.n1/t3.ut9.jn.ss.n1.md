## Context

`/quo-doc-writer-review` Step 6 ("Output Work Items", `skills/quo-doc-writer-review/SKILL.md`, lines ~110–137) currently emits findings as a plain numbered list with NO severity tag and NO fix-path/depth structure. Line 114 explicitly states: "Findings here are not severity-tagged the way `/quo-spec-review`'s are, so the trailer collapses to two shapes...". This Subtask is the first introduction of severity tagging to this skill (SR-1.1) and layers the orthogonal per-fix-path depth dimension and fix-path enumeration on top (SR-1.2). It is the Phase-1 emission extension symmetric to the `/quo-engineer-review` Step 4 extension; the depth tags become *emitted* here but are not yet *consumed* by any orchestrator routing rule, so this ships standalone value with zero new user gates.

The authoritative data model is the Spec Bee SDD child `t1.tip.ux` section "Finding shape with two-dimensional tagging" — NOT the cumulative `docs/sdd.md`. That model defines:
- Finding line: `` <n>. `<severity>` <one or more fix-path lines, each with its own depth tag> — <description> `` where `<severity>` is one of `blocker` / `suggestion` / `nit`.
- Each fix-path line: `(<letter>) [depth:<trivial-tweak|refactor-locally|re-architect>] <description of that fix path>`.
- Single-fix-path findings remain valid (one `(a)` block); the shape is uniform whether 1 or N paths are enumerated, which simplifies the orchestrator's parser.

The finding emission shape adopted here MUST be IDENTICAL to the shape defined in the sibling emission Epics for `/quo-engineer-review`, `/quo-test-writer-review`, and `/quo-spec-review`, so the Phase-2 orchestrator parser is uniform across all four review skills. Keep the severity vocabulary (`blocker` / `suggestion` / `nit`) and depth vocabulary (`trivial-tweak` / `refactor-locally` / `re-architect`) spelled exactly as in the SDD.

This is a skill-prose change in the quorum repo. The three design rules in `CLAUDE.md` (language-agnostic, OS-paired snippets, project-neutral prose), the scratch-file convention, and `## Review criteria for skill changes` apply. This change is pure prose with no new shell snippets, so the OS-pairing rule is not triggered by new commands; do not introduce any.
