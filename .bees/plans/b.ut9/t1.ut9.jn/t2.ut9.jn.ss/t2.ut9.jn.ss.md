## Context

`/quo-doc-writer-review` currently emits findings as a plain numbered list with **no severity tag** and **no fix-path/depth structure** (Step 6 "Output Work Items", `skills/quo-doc-writer-review/SKILL.md` lines 110–137; line 114 explicitly states findings are *not* severity-tagged). This Task is the **first introduction of severity tagging** to this skill (SR-1.1) and adds the orthogonal per-fix-path depth dimension and fix-path enumeration on top. It is symmetric to the `/quo-engineer-review` extension (Epic `t1.ut9.9c`).

The two-dimensional finding emission (severity + per-fix-path depth + enumerated fix paths) is the foundational Phase-1 signal that the Phase-2 orchestrator routing table (sibling Epic `t1.ut9.no`) will consume. Depth tags become *emitted* here but are not yet *consumed* by any routing rule, so this Task ships standalone value with zero new user gates and zero `b.wii` exposure.

## What Needs to Change

- In `skills/quo-doc-writer-review/SKILL.md` **Step 6 (Output Work Items)**, document the extended finding emission shape: every finding carries **exactly one severity tag** (`blocker` / `suggestion` / `nit`) AND **one or more enumerated fix paths**, each fix path carrying **its own depth tag** (`trivial-tweak` / `refactor-locally` / `re-architect`), following the SDD's "Finding shape with two-dimensional tagging" data model (authoritative source: Spec Bee SDD child `t1.tip.ux`, NOT the cumulative `docs/sdd.md`):
  - finding line: `` <n>. `<severity>` <one or more fix-path lines> — <description> ``
  - fix-path line: `(<letter>) [depth:<trivial-tweak|refactor-locally|re-architect>] <description of that fix path>`
- Replace the prose at current line 114 (which states findings are NOT severity-tagged) with prose introducing the severity dimension and the depth/fix-path dimension.
- Provide worked examples in the prose: **at least one per depth bucket** (`trivial-tweak`, `refactor-locally`, `re-architect`) per SR-1.2, and at least one example showing **single-path** emission and one showing **multi-path** emission per the Epic's Scope.
- Update the existing Shape 1 worked example (current lines 118–126) so its findings render in the new two-dimensional shape.
- **Preserve the two existing trailer shapes verbatim** — the new emission shape sits *inside* the work-item list; the findings-present / clean trailers (current Shape 1 trailer paragraph and Shape 2 block at lines 128–136) are unchanged in form. The counter-anchor clause and the two-step `TaskCreate` → `AskUserQuestion` gate prose stay intact.

## Why

Severity describes *how important fixing-at-all is*; depth describes *what fixing costs*. The two dimensions are orthogonal, and the orchestrator's Phase-2 routing decision keys on depth (and the multi-path flag), not severity. Emitting this shape from `/quo-doc-writer-review` is one of the four in-flow review skills' Phase-1 emission extensions; the routing table that consumes it is unimplementable without uniform emission across all four.

## Success Criteria

- The Step 6 output-shape section documents severity tags AND per-fix-path depth tags AND enumerated fix paths.
- At least one worked example per depth bucket renders in the skill prose; single-path and multi-path emission are both demonstrated.
- The two existing trailer shapes (findings-present / clean) are preserved verbatim.
- A live run of `/quo-doc-writer-review` against any doc diff produces findings in the documented shape.
- The change satisfies this repo's design rules (language-agnostic, OS-paired snippets where any are added, project-neutral prose) — though this change is pure prose with no new shell snippets.
