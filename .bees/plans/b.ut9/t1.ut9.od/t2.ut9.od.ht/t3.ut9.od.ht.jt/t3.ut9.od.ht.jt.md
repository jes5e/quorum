## Context

This Subtask implements the user-facing read of the session-scoped compromise tracker (PRD FR-5 / SDD SR-3.4 / SR-5.3 / SR-6.5). It adds an "Accepted compromises" section to the end-of-run summary in BOTH orchestrator skills. The section READS the tracker file that sibling Epic `t1.ut9.po` writes (via the `Read` tool, against the documented path convention) and renders its accepted-compromise entries under a labeled heading. This Subtask is a READER of the tracker only — it does NOT write tracker entries (that is `po`'s job), define the routing table/gates (`no`), or implement the post-completion review's own tracker consumption (`pw`, a separate, independent consumer).

The tracker file convention and the entry shape are SDD-locked, so this work is stable regardless of `po`'s implementation timing:
- **Path convention (SR-3.1, defined by `po`):** a single markdown file at `<tempdir>/.quorum/compromises-YYYYMMDD-HHMM-<short-suffix>.md`, where `YYYYMMDD-HHMM` is a UTC timestamp generated once at the start of the run and `<short-suffix>` is a short collision-resistant string. The run's tracker path is therefore known at summary time (generated once at run start per SR-3.1).
- **Entry shape (SDD `t1.tip.ux` Data models, "Compromise tracker entry shape"):** each accepted compromise is one markdown section keyed `## Compromise <n>` with five fields — `Finding (verbatim)`, `Fix paths surfaced by reviewer`, `Decision`, `Rationale`, `Follow-up Issue` (ticket ID or `none`).

The "Accepted compromises" surface is near-identical prose across both orchestrators; divergence is a defect (this is why both files are edited in a single Subtask — authoring the prose once and pasting it into the analogous surface in each file eliminates the divergence risk that splitting per-file would create). This is a skill-prose change in the quorum repo and must honor CLAUDE.md's three design rules, the scratch-file convention, and `## Review criteria for skill changes`.

## What Needs to Change

Add the "Accepted compromises" end-of-run summary section to both files, using near-identical prose:

1. **`skills/quo-fix-issue/SKILL.md` — Section 7 per-Issue summary (the `## Issue [x] of [total] done:` markdown block, ~line 479-487).** Extend this rendered summary block with a new `**Accepted compromises**` surface. Because the tracker is session-scoped and accumulates across the whole run/batch (SR-3.3), the section reflects the tracker file's current contents at the moment the summary renders.

2. **`skills/quo-execute/SKILL.md` — the run-end summary surface.** Place the section at the analogous run-end summary surface, keeping the placement parallel to `/quo-fix-issue` so the prose stays near-identical. The rendered summary artifact in `/quo-execute` is Section 9's `## Bee Execution Complete:` markdown block (~line 608-619); add the `**Accepted compromises**` surface there. Cross-reference it from Section 7's "Final Output" run-end review surface (~line 570) if a pointer aids the reader, but render the section in the Section 9 summary block so it sits alongside the other rendered run-end summary fields, mirroring where `/quo-fix-issue` renders it inside its per-Issue close-out summary block.

3. **Rendering logic (SR-5.3).** In both files, the new prose instructs the orchestrator to: read the run's tracker file at the documented path (`<tempdir>/.quorum/compromises-YYYYMMDD-HHMM-<short-suffix>.md`, the path generated once at run start per Epic `po`'s convention) via the `Read` tool; iterate its `## Compromise <n>` entries; and render ONE bullet per entry surfacing exactly the four user-facing fields — the **finding** (the entry's `Finding (verbatim)`), the **chosen path** (the entry's `Decision`), the **rationale** (the entry's `Rationale`), and the **follow-up Issue ID** (the entry's `Follow-up Issue`, or `none`). Do NOT surface the fifth field (`Fix paths surfaced by reviewer`) — SR-5.3 surfaces the chosen path, not the full menu.

4. **Empty / absent case (SR-3.4, FR-5).** When the tracker is empty (no compromises accepted during the run) OR the tracker file does not exist for the current run, the section is OMITTED ENTIRELY — do NOT render an empty heading, an `N/A`, or a "no compromises" placeholder. Reading a nonexistent tracker path is the expected common-case state (most runs accept zero compromises), so the prose must treat "file absent" and "file present but no `## Compromise` entries" identically: omit the section.

5. **Volume edge case (SR-6.5).** When the tracker accumulates more than ~10 entries, surface them ALL in full (no truncation), preceded by a short prologue noting the volume. The prose must explicitly forbid truncation — the tracker exists to preserve exactly this signal.

**Mechanism / design-rule constraints.** Prefer the `Read` tool / no-shell approach for reading the tracker (the path is known, so no shell is required to locate or test it). If any new shell snippet is nonetheless introduced (e.g., to test tracker-file existence), it MUST ship in paired POSIX bash + Windows PowerShell form per design rule #2 — but the no-shell `Read` path is preferred and should make a snippet unnecessary. The change is otherwise pure prose: language-agnostic and project-neutral (no hardcoded language commands, file extensions, or manifest names), with NO scratch-file writes (the tracker is read here, not written — the write side is `po`'s lane) and NO `rm` / `Remove-Item` instructions.
