## Context

`/quo-test-writer-review` currently emits findings as a plain numbered list with no severity tag and no fix-path/depth structure (Step 4 "Generate Work Item List", `skills/quo-test-writer-review/SKILL.md` lines ~150–178). Line 154 explicitly states findings are NOT severity-tagged. This Subtask is the first introduction of severity tagging to this skill (SR-1.1) and adds the orthogonal per-fix-path depth dimension and fix-path enumeration on top. It is symmetric to the parallel `/quo-engineer-review`, `/quo-doc-writer-review`, and `/quo-spec-review` emission extensions; the emitted shape MUST be byte-identical across all four so the Phase-2 orchestrator routing parser is uniform. The authoritative finding-shape grammar is fixed by the parent Task body (PM-cross-checked against Spec Bee SDD child `t1.tip.ux`, NOT the cumulative `docs/sdd.md`):
  - finding line: `` <n>. `<severity>` <one or more fix-path lines> — <description> `` where `<severity>` is one of `blocker` / `suggestion` / `nit`
  - fix-path line: `(<letter>) [depth:<trivial-tweak|refactor-locally|re-architect>] <description of that fix path>`

Research note: no `depth:` / `trivial-tweak` / `refactor-locally` / `re-architect` vocabulary exists anywhere in the repo yet (the sibling emission Epics `9c` / `jn` / `29` have not landed), so this Subtask defines the shape fresh from the Task-body grammar above — do NOT mirror `/quo-spec-review`'s older `[blocker]` bracket-severity form (that skill's lines 208–227); the backtick severity form `` `<severity>` `` plus bracketed `[depth:<...>]` is the authoritative cross-Epic contract here.

This is a skill-prose change in the quorum repo. The three design rules (language-agnostic commands, OS-paired shell snippets, project-neutral prose), the scratch-file convention, and CLAUDE.md `## Review criteria for skill changes` all apply — though this change is pure prose with no new shell snippets, helper scripts, or scratch-file writes, so only the project-neutral-prose rule is materially engaged.

Depth tags become *emitted* here but are not yet *consumed* by any routing rule, so this ships standalone value with zero new user gates. Do not add any routing/consumption prose for the depth tag.
