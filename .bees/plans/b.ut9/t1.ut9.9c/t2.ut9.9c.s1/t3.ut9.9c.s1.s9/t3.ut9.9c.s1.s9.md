## Context

`/quo-engineer-review` Step 4 ("Generate Work Item List") in `skills/quo-engineer-review/SKILL.md` (lines ~148–175) currently emits findings as a plain numbered list with NO severity tag and NO fix-path/depth structure. Lines 152–153 explicitly state findings are *not* severity-tagged the way `/quo-spec-review`'s are, and the Shape 1 worked example (lines 156–165) renders findings as bare `<n>. <description>` lines.

This Subtask is the first introduction of severity tagging to this skill and layers an orthogonal per-fix-path depth dimension plus fix-path enumeration on top. The resulting two-dimensional shape (severity + per-fix-path depth + enumerated fix paths) is the foundational Phase-1 signal that a sibling Phase-2 orchestrator routing-table Epic (`t1.ut9.no`) will later consume; depth tags become *emitted* here but are not yet *consumed* by any routing rule, so this ships standalone value with zero new user gates.

The finding emission shape defined here is a cross-Epic contract — three sibling review skills (`t1.ut9.jn`, `t1.ut9.q2`, `t1.ut9.29`) and a routing-table Epic (`t1.ut9.no`) must adopt the identical rendering — so the tag vocabulary and line shapes must be implemented exactly as specified below (those sibling changes are out of scope here; only `skills/quo-engineer-review/SKILL.md` is in scope for this Subtask). The authoritative data model is the Spec Bee SDD child `t1.tip.ux` "Finding shape with two-dimensional tagging" section (NOT the cumulative `docs/sdd.md`, which does not yet describe this feature); the exact shapes are reproduced inline below so no cross-reference is required.

This is a skill-prose change in the quorum repo. The repo's CLAUDE.md three design rules (language-agnostic commands, OS-paired shell snippets, project-neutral prose), the scratch-file convention, and the `## Review criteria for skill changes` section all apply. This change is pure prose with no new shell snippets, so the OS-pairing and scratch-file rules are satisfied vacuously; the language-agnostic and project-neutral rules still bind the prose.

## What Needs to Change

In `skills/quo-engineer-review/SKILL.md` Step 4 (Generate Work Item List):

1. **Replace the prose at current lines 152–153** (which states findings are NOT severity-tagged) with prose introducing two orthogonal dimensions:
   - A **severity** dimension: every finding carries exactly one severity tag — `blocker` / `suggestion` / `nit`.
   - A **depth** dimension carried per fix path: every finding enumerates one or more fix paths, and each fix path carries its own depth tag — `trivial-tweak` / `refactor-locally` / `re-architect`.
   - Briefly state the orthogonality (severity describes how important fixing-at-all is; depth describes what fixing costs) so the reader understands why both dimensions are emitted.

2. **Document the exact line shapes** the finding emission must follow:
   - finding line: `` <n>. `<severity>` <one or more fix-path lines> — <description> `` (severity tag backticked, matching the `/quo-spec-review` convention referenced in the existing prose)
   - fix-path line: `(<letter>) [depth:<trivial-tweak|refactor-locally|re-architect>] <description of that fix path>`

3. **Provide worked examples in the prose** covering:
   - At least one example per depth bucket (`trivial-tweak`, `refactor-locally`, `re-architect`).
   - At least one finding showing single-path emission (one fix-path line) and at least one showing multi-path emission (two or more fix-path lines under a single finding).

4. **Update the existing Shape 1 worked example** (current lines 156–165, the fenced ```markdown` block under `## Code Review Work items`) so its four findings render in the new two-dimensional shape — each finding gets a backticked severity tag and one or more `(letter) [depth:...]` fix-path lines. The existing example findings (SQL injection, input validation, refactor, dead-code removal) are good carriers for the different severity/depth combinations; reuse them and tag them sensibly (e.g., the dead-code removal as a `trivial-tweak`, the 60-line-function refactor as `refactor-locally`, etc.), and use at least one of them to demonstrate multi-path emission.

5. **Preserve verbatim** the two existing trailer shapes that sit at the END of the Step 4 examples — the findings-present trailer text (the `**Your next tool use MUST address these findings now.** ...` paragraph through the counter-anchor clause, currently inside the Shape 1 block) and the Shape 2 / no-findings block (currently lines 167–175). The new emission shape sits *inside* the work-item list ABOVE the trailer; the trailer paragraphs themselves, the two-step `TaskCreate` → `AskUserQuestion` gate prose, and the counter-anchor clause (`Do not yield with this text as your assistant response — ...`) are unchanged in wording. Only the numbered work-item lines above the trailer change form.
