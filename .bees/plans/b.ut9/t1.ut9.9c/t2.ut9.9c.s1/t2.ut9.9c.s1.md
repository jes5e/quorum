## Context

`/quo-engineer-review` currently emits findings as a plain numbered list with **no severity tag** and **no fix-path/depth structure** (Step 4 "Generate Work Item List", `skills/quo-engineer-review/SKILL.md` lines 148–175; lines 152–153 explicitly state findings are *not* severity-tagged). This Task is the **first introduction of severity tagging** to this skill (SR-1.1) and adds the orthogonal per-fix-path depth dimension and fix-path enumeration on top.

The two-dimensional finding emission (severity + per-fix-path depth + enumerated fix paths) is the foundational Phase-1 signal that the Phase-2 orchestrator routing table (sibling Epic `t1.ut9.no`) will consume. Per the SDD, depth tags become *emitted* in Phase 1 but are not yet *consumed* by any routing rule, so this Task ships standalone value (severity tags emitted for the first time) with zero new user gates and zero `b.wii` exposure.

## What Needs to Change

- In `skills/quo-engineer-review/SKILL.md` **Step 4 (Generate Work Item List)**, document the extended finding emission shape: every finding carries **exactly one severity tag** (`blocker` / `suggestion` / `nit`) AND **one or more enumerated fix paths**, each fix path carrying **its own depth tag** (`trivial-tweak` / `refactor-locally` / `re-architect`), following the SDD's "Finding shape with two-dimensional tagging" data model:
  - finding line: `` <n>. `<severity>` <one or more fix-path lines> — <description> ``
  - fix-path line: `(<letter>) [depth:<trivial-tweak|refactor-locally|re-architect>] <description of that fix path>`
- Replace the prose at current lines 152–153 (which states findings are NOT severity-tagged) with prose introducing the severity dimension and the depth/fix-path dimension.
- Provide worked examples in the prose: **at least one per depth bucket** (`trivial-tweak`, `refactor-locally`, `re-architect`) per SR-1.2, and at least one example showing **single-path** emission and one showing **multi-path** emission per the Epic's Scope.
- Update the existing Shape 1 worked example (current lines 156–165) so its findings render in the new two-dimensional shape.
- **Preserve the two existing trailer shapes verbatim** — the new emission shape sits *inside* the work-item list; the findings-present / clean trailers at the end of Step 4 (current lines 154–165 trailer text and 167–175) are unchanged in form. The counter-anchor clause and the two-step `TaskCreate` → `AskUserQuestion` gate prose stay intact.

## Why

Severity describes *how important fixing-at-all is*; depth describes *what fixing costs*. The two dimensions are orthogonal, and the orchestrator's Phase-2 routing decision keys on depth (and the multi-path flag), not severity. Emitting this shape from `/quo-engineer-review` is the first of the four in-flow review skills' Phase-1 emission extensions; the routing table that consumes it is unimplementable without it.

## Success Criteria

- The Step 4 output-shape section documents severity tags AND per-fix-path depth tags AND enumerated fix paths.
- At least one worked example per depth bucket renders in the skill prose; single-path and multi-path emission are both demonstrated.
- The two existing trailer shapes (findings-present / clean) are preserved verbatim.
- A live run of `/quo-engineer-review` against any diff produces findings in the documented shape.
- The change satisfies this repo's design rules (language-agnostic, OS-paired snippets where any are added, project-neutral prose) — though this change is pure prose with no new shell snippets.
