## What Needs to Change

### Part A — Post-completion review reframe (SR-4.1 / SR-4.2 / SR-4.3 / SR-4.4 / SR-5.4)

Rewrite the fresh-reviewer dispatch prompt in both `quo-fix-issue` Section 8 and `quo-execute` Section 6 (lockstep, near-identical) so the reviewer's instructed structure is:

1. **Consume the compromise tracker as input, passed as a FILE PATH (SR-4.3)** — the orchestrator passes the tracker file path in the dispatch prompt (NOT inlined contents); the dispatched general-purpose Agent reads it via its `Read` tool. **Missing-tracker handling (SR-6.3):** if the tracker file is missing/unreadable, the review PROCEEDS but flags the missing tracker explicitly in its output, so the dropped hand-off is visible.
2. **Challenge each tracked compromise on its merits** — push back when the chosen path is not defensible. Includes the **SR-6.6 check**: the tracker records what the user asked for; the diff is what shipped — flag any mismatch (e.g., user picked a deep fix path but the implementer's diff implemented the cheap path).
3. **Depth-plausibility check (SR-4.5)** — for each auto-routed `(1, refactor-locally)` tracker entry, explicitly evaluate whether the routed fix's depth was plausible. When the reviewer judges the depth was misjudged and the true depth is `re-architect`, emit a `[compromise-challenge]` finding naming the misclassification.
4. **Fix-path enumeration plausibility check (SR-4.6)** — for every finding the in-flow reviewer surfaced (regardless of severity/depth/path-count), evaluate whether the in-flow reviewer should have enumerated an additional plausible fix path. When the reviewer judges under-enumeration (one path surfaced + auto-dispatched under SR-2.4's single-path branch, but a second plausible path warrants a user gate), emit a `[compromise-challenge]` finding naming the under-enumeration. (SR-4.5 catches misjudged depth on a surfaced path; SR-4.6 catches a plausible path not surfaced at all — complementary.)
5. **Holistic solution-quality judgment** beyond the logged compromises.
6. **Discrete-defect sweep** (the prompt's current scope — preserve the existing self-contained, fresh-eyes, no-lane-scoping, stay-on-diff framing and the existing anti-pattern callouts forbidding re-invoking the lane-scoped review skills).

- **Finding tagging (FR-6 / SR-4.4):** every emitted finding carries one of `[compromise-challenge]` / `[design]` / `[defect]`, PLUS the severity tag + per-fix-path depth tag + enumerated fix paths from the Phase-1 emission contract (the depth tag is informative here — the post-completion sweep is the final pre-merge gate, so any finding it emits is candidate to gate regardless of depth).
- **Orchestrator-side preamble (SR-5.4):** when the post-completion reviewer returns one or more `[compromise-challenge]` findings, the orchestrator's preamble (prose rendered before presenting the findings) names this explicitly so the user reads the section with the right framing. Mirror the verdict-keyed preamble pattern in `skills/quo-plan/SKILL.md` Step 5e.

### Part B — Recovery gates (SR-6.7 + the SR-4.6 analog gate)

**SR-6.7 depth-misclassification recovery gate.** When the post-completion reviewer emits a `[compromise-challenge]` finding flagging an auto-routed `(1, refactor-locally)` decision as misjudged (per the SR-4.5 check), the orchestrator fires a three-choice `AskUserQuestion` (via the two-step `TaskCreate` → `AskUserQuestion` contract) in BOTH skills:
- `File follow-up Issue to revisit the depth decision` — dispatch `/quo-file-issue` via the Skill tool capturing the depth-mismatch finding + the original compromise-tracker entry as context; the **original Trigger C tracker entry's `Follow-up Issue` field is updated in place** with the new Issue ID (NO new tracker entry — per Epic `po`'s Trigger D note).
- `Accept the misjudgment and proceed` — triggers Epic `po`'s **Trigger D** (a NEW tracker entry appended with `Decision: User overrode auto-route after post-completion challenge` and the post-completion reviewer's challenge text as Rationale).
- `Pause to discuss` — stop the post-completion flow at this gate, surface a prose discussion; the user can re-issue any of the three choices afterward.

The gate **fires per misjudged auto-routed finding** (once per such `[compromise-challenge]` finding — three findings ⇒ three gate firings, in emission order), NOT aggregated.

**SR-4.6 under-enumeration recovery gate (AC 4a).** When the reviewer emits a `[compromise-challenge]` finding flagging under-enumeration (per the SR-4.6 check), the orchestrator fires the **same three-choice gate shape** with relabeled choices: `File follow-up Issue to surface the missing path` / `Accept the under-enumeration and proceed` / `Pause to discuss`, with the same per-branch behavior shape (file follow-up Issue / append explicit-override tracker entry / pause-and-resume).

## Why

The post-completion review fires too late to PREVENT a compromise from shipping (Epic `no`'s gates do that earlier) — but it is the necessary backstop that ensures compromises which DO legitimately ship get challenged rather than baked silently into the new baseline. Leading with "challenge accepted compromises" (consuming the tracker) reframes the review from defect-enumeration to holistic-challenge. The SR-4.5/SR-4.6 plausibility checks + the recovery gates close the auto-route audit loop: when reviewer mis-classification slips a depth or under-enumerates a path past the routing-table gate, the post-completion challenge catches it and gives the user a remediation path.
