### Part A — Post-completion review reframe (both files, lockstep)

Rewrite the fenced fresh-reviewer dispatch prompt in both `quo-fix-issue` Section 8 and `quo-execute` Section 6 so the reviewer's instructed structure becomes an ordered set of phases. The reframe **WRAPS** the existing fresh-eyes defect sweep as its FINAL phase — it does NOT replace it. Preserve verbatim (modulo the per-file substitutions noted in Context) the existing framing: self-contained prompt, fresh-eyes / no-context-of-how-the-work-was-done, `git diff <pre-session-sha>..HEAD` (or `<pre-bee-sha>`) computed by the agent itself, the skill-repo markdown-is-source-code note, the contract-key-rename prohibition, `One generalist pass covers code AND docs AND tests — do not lane-scope.`, `Do NOT do a general repo audit. The reframed prompt instructs the reviewer to perform these phases in order:

1. **Consume the compromise tracker, passed as a FILE PATH (SR-4.3).** The orchestrator passes the tracker file path in the dispatch prompt (a `<compromise-tracker-path>` substitution token in the fenced skeleton, alongside the existing SHA / ID-list tokens) — the path, NOT the inlined contents. The dispatched general-purpose Agent reads the file via its own `Read` tool. **Missing-tracker handling (SR-6.3):** if the tracker file is missing or unreadable, the review PROCEEDS but flags the missing tracker explicitly in its output, so the dropped hand-off is visible rather than silent.
2. **Challenge each tracked compromise on its merits** — push back when the chosen path is not defensible. Include the **SR-6.6 check**: the tracker records what the user asked for; the diff is what shipped — flag any mismatch (e.g., the user picked a deep fix path but the implementer's diff implemented the cheap path).
3. **Depth-plausibility check (SR-4.5)** — for each auto-routed `(1, refactor-locally)` tracker entry, explicitly evaluate whether the routed fix's depth was plausible. When the reviewer judges the depth was misjudged and the true depth is `re-architect`, emit a `[compromise-challenge]` finding naming the misclassification.
4. **Fix-path-enumeration plausibility check (SR-4.6)** — for every finding the in-flow reviewer surfaced (regardless of severity/depth/path-count), evaluate whether the in-flow reviewer should have enumerated an additional plausible fix path. When the reviewer judges under-enumeration (one path surfaced + auto-dispatched under SR-2.4's single-path branch, but a second plausible path warrants a user gate), emit a `[compromise-challenge]` finding naming the under-enumeration. (SR-4.5 catches misjudged depth on a surfaced path; SR-4.6 catches a plausible path not surfaced at all — complementary; state this complementarity in-prompt.)
5. **Holistic solution-quality judgment** beyond the logged compromises.
6. **Discrete-defect sweep** — the prompt's existing scope, preserved as the final phase (see preserve-verbatim list above).

**Finding tagging (FR-6 / SR-4.4).** Update the prompt's return-format instruction so every emitted finding carries one of `[compromise-challenge]` / `[design]` / `[defect]`, PLUS the severity tag + per-fix-path depth tag + enumerated fix paths from the Phase-1 emission contract. Note in-prompt that the depth tag is informative here (the post-completion sweep is the final pre-merge gate, so any finding it emits is a gate candidate regardless of depth). Preserve the `file:line` + severity (`blocker` / `suggestion` / `nit`) + `no issues found` clean-exit shape; the new tags are additive to it.

**Orchestrator-side preamble (SR-5.4).** Where the orchestrator synthesizes and presents the fresh reviewer's findings (the synthesize-before-presenting step that precedes the findings gate — `quo-fix-issue` Section 8 step 3 / `quo-execute` Section 6 step 3), add prose: when the post-completion reviewer returns one or more `[compromise-challenge]` findings, the orchestrator's preamble (rendered before presenting the findings) names this explicitly so the user reads the section with the right framing.
