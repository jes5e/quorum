Reframe the post-completion review dispatch prompts in `skills/quo-fix-issue/SKILL.md` Section 8 AND `skills/quo-execute/SKILL.md` Section 6 to lead with a "challenge accepted compromises" phase consuming the compromise tracker, then holistic judgment, then defect sweep. Additionally, wire in the depth-plausibility check (SDD SR-4.5) and the auto-route depth-misclassification recovery gate (SDD SR-6.7). Per PRD FR-6 and SDD SR-4 + SR-6.7.

## Scope

### Part A — Post-completion review reframe (the original Epic 10 scope)

- Rewrite the dispatch prompt at `skills/quo-fix-issue/SKILL.md` Section 8 lines 517–530 so its new structure is:
  1. Consume the compromise tracker as input (passed in as a file path per SDD SR-4.3 — dispatched general-purpose Agent reads via `Read` tool).
  2. Challenge each tracked compromise on its merits; push back when the chosen path is not defensible.
  3. **Depth-plausibility check (per SR-4.5):** for each auto-routed `(1, refactor-locally)` tracker entry, explicitly evaluate whether the routed fix's depth was plausible. When the reviewer judges the depth was misjudged and the true depth is `re-architect`, the reviewer emits a `[compromise-challenge]` finding naming the misclassification.
  4. Judge solution quality holistically beyond logged compromises.
  5. Sweep for discrete defects (the current scope of the prompt).
- Rewrite the analogous dispatch prompt at `skills/quo-execute/SKILL.md` Section 6 lines 449–483 with the same structure (lockstep with Section 8 per reuse convention).
- Both rewritten prompts require findings tagged as one of `[compromise-challenge]` / `[design]` / `[defect]`, plus the severity + per-fix-path depth + enumerated fix paths from Epic 1–5's emission contract.
- Orchestrator-side preamble: when the post-completion reviewer returns findings tagged `[compromise-challenge]`, the orchestrator's preamble (prose rendered before presenting the findings) names this explicitly. Mirror the verdict-keyed preamble pattern in `skills/quo-plan/SKILL.md` Step 5e.
- Tracker hand-off: pass the tracker file path in the dispatch prompt (not inlined contents). Per SDD SR-4.3 and the corresponding `## Decisions` entry.

### Part B — Auto-route depth-misclassification recovery gate (SDD SR-6.7)

When the post-completion reviewer emits a `[compromise-challenge]` finding that flags an auto-routed `(1, refactor-locally)` decision as misjudged (per the SR-4.5 plausibility check from Part A), the orchestrator MUST fire a three-choice `AskUserQuestion` gate in BOTH `/quo-fix-issue` and `/quo-execute`:

- `File follow-up Issue to revisit the depth decision` — orchestrator dispatches `/quo-file-issue` via the Skill tool capturing the depth-mismatch finding with the original compromise-tracker entry as context. The original Trigger C tracker entry's `Follow-up Issue` field is updated (in place) to reference the new Issue ID. Per Epic 8 Trigger D's note, no new tracker entry is appended for this branch.
- `Accept the misjudgment and proceed` — orchestrator triggers Epic 8 Trigger D (a new tracker entry is appended with `Decision: User overrode auto-route after post-completion challenge` and the post-completion reviewer's challenge text as the Rationale field).
- `Pause to discuss` — orchestrator stops the post-completion flow at this gate and surfaces a prose discussion with the user. The user can re-issue any of the three choices after the discussion completes.

The gate fires per misjudged auto-routed finding (i.e., once per `[compromise-challenge]` finding flagging a depth misclassification — if the post-completion review surfaces three such findings, the gate fires three times, once per finding). The order is the order the reviewer emits the findings.

## Acceptance criteria

1. Both dispatch prompts (`skills/quo-fix-issue/SKILL.md` Section 8 and `skills/quo-execute/SKILL.md` Section 6) lead with the "challenge accepted compromises" phase consuming the tracker.
2. Both dispatch prompts require findings tagged as `[compromise-challenge]` / `[design]` / `[defect]`.
3. Both prompts pass the tracker as a file path (not inlined content).
4. Both dispatch prompts explicitly require the SR-4.5 depth-plausibility check on auto-routed `(1, refactor-locally)` findings, naming the `[compromise-challenge]` finding shape the reviewer emits when a depth misclassification is detected.
5. The orchestrator-side preamble pattern names `[compromise-challenge]` findings explicitly when present.
6. Both orchestrators' SKILL.md prose documents the SR-6.7 three-choice recovery gate (`File follow-up Issue` / `Accept the misjudgment` / `Pause to discuss`) and names the per-branch behavior (update existing tracker entry / Trigger D append / pause-and-resume).
7. The gate fires per misjudged auto-routed finding (not aggregated across multiple findings into a single gate).
8. A live post-completion run that surfaces ≥1 `[compromise-challenge]` finding flagging an auto-route misclassification triggers the SR-6.7 gate; the user's choice produces the documented downstream behavior in each branch.
9. The b.bjp walk-through validation acceptance criterion from the PRD: replaying b.bjp under the new rules, both surviving issues would have fired user gates at Epic 7's routing stage (this Epic's value is validating that compromises that DO ship — legitimately — get challenged by the post-completion review, AND that auto-route misjudgments get a recovery path).
