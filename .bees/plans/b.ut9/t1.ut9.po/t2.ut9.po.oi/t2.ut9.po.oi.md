## What Needs to Change

Add the compromise-tracker mechanism to both orchestrator SKILL.md files (near-identical prose), comprising:

**1. Tracker file convention (SR-3.1).** Canonical filename: `compromises-YYYYMMDD-HHMM-<short-suffix>.md` under `<tempdir>/.quorum/` (`/tmp/.quorum/` on POSIX, `%TEMP%\.quorum` on Windows), where `YYYYMMDD-HHMM` is a UTC timestamp generated once at the start of the run and `<short-suffix>` is a short collision-resistant random string. The timestamp prefix makes tracker files debuggably identifiable across runs accumulated in `<tempdir>/.quorum/`. Include the paired POSIX + PowerShell create-if-absent snippets (`mkdir -p /tmp/.quorum` / `New-Item -ItemType Directory -Force -Path "$env:TEMP\.quorum"`).

**2. Four append triggers (SR-3.2), each with its precise moment-of-write** (verbatim from the Epic scope):
- **Trigger A — user picks `Defer to follow-up Issue` at the scope-bounding gate** (the gate Epic `no` added per SR-5.2). Write moment: immediately after `/quo-file-issue` returns successfully with the new Issue ticket ID, before the orchestrator continues with the soft-fix dispatch. Captures the follow-up Issue ID in the entry.
- **Trigger B — user picks `Accept the limitation` at the scope-bounding gate.** Write moment: immediately after the `AskUserQuestion` returns the user's choice, before the orchestrator continues without a fix. Captures `none` in the follow-up Issue field.
- **Trigger C — orchestrator auto-routes a single-path `(1, refactor-locally)` finding** per Epic `no`'s routing table. Write moment: **at the moment of implementer dispatch** (NOT at any gate-close — there is no gate for this branch). The tracker write happens in the same logical action as the auto-dispatch step. Captures `Auto-routed (a) per single-path refactor-locally rule` as the Decision and the routing-table rule wording as the Rationale.
- **Trigger D (GENERALIZED) — user overrides an auto-route decision after the post-completion review challenges it**, at EITHER the SR-6.7 depth-misjudgment recovery gate OR the SR-4.6 under-enumeration analog recovery gate. Both gates are **owned by Epic `t1.ut9.pw`**; THIS Epic owns the tracker-write logic that fires from them. Trigger D is the single owner of all post-completion-override tracker writes (closing the cross-Epic gap where the SR-4.6 under-enumeration accept-branch write would otherwise fall between `po` and `pw`).
  - **SR-6.7 (depth misjudgment):** `Accept the misjudgment and proceed` → append a NEW entry with `Decision: User overrode auto-route after post-completion challenge (depth misjudgment)` and the reviewer's challenge text as Rationale. `File follow-up Issue to revisit the depth decision` → NO new entry; the ORIGINAL Trigger C entry's `Follow-up Issue` field is updated in place. `Pause to discuss` → defer the write.
  - **SR-4.6 (under-enumeration):** `Accept the under-enumeration and proceed` → append a NEW entry with `Decision: User accepted under-enumeration after post-completion challenge` and the reviewer's under-enumeration challenge text as Rationale. `File follow-up Issue to surface the missing path` → capture the new Issue ID (in-place update when an originating Trigger C entry exists, else append a new entry). `Pause to discuss` → defer the write.
  Write moment in all append cases: immediately after the user picks the accept choice, before the post-completion flow continues. The Decision enum therefore carries two post-completion-override values (depth-misjudgment and under-enumeration).

**3. Tracker entry shape (SR-3.2 / Data models — five canonical fields):**
- Finding verbatim (severity tag + fix paths with depth tags + description).
- Decision (user-picked path / `Defer to follow-up Issue` / `Accept the limitation` / auto-route rule / explicit override).
- One-line rationale (user's stated reason, or the auto-route rule that fired).
- Follow-up Issue ID (or `none`).
- (The `## Compromise <n>` section heading with a 1-based run-scoped counter, per the Data models shape.)

Document the entry shape in both SKILL.md files (or reference the SDD `t1.tip.ux` Data models section explicitly).

**4. Persistence (SR-3.3).** The tracker is a file-system artifact — it persists across orchestrator-yield events within a single run inherently. A new `<session-suffix>` (and timestamp) is generated at the start of each run, so previous-run tracker files remain visible but are never appended to. Never delete the tracker file (scratch-file convention).

## Why

When a compromise is legitimately accepted (Defer / Accept / refactor-locally auto-route / post-completion override), it must remain visible and challengeable rather than baked silently into the new baseline. The tracker is the durable session-scoped record that the end-of-run summary (Epic `od`) surfaces and the post-completion review (Epic `pw`) consumes as input. The auto-route Trigger C is what makes the tracker materialize even when no user gate fires (so non-trivial `refactor-locally` decisions stay auditable).
