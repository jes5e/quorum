Add the session-scoped compromise tracker mechanism to BOTH `skills/quo-fix-issue/SKILL.md` AND `skills/quo-execute/SKILL.md`. The tracker is a markdown file under `<tempdir>/.quorum/compromises-<session-suffix>.md` that accumulates one entry per accepted compromise. Per PRD FR-4 and SDD SR-3.

## Scope

- Define the tracker file convention (`compromises-<session-suffix>.md` under `<tempdir>/.quorum/`).
- Add tracker-write logic to both orchestrators at **four** trigger points (per SDD SR-3.2). Each trigger has a precisely-defined moment-of-write so future implementers know exactly when the append happens:
  - **Trigger A — User picks `Defer to follow-up Issue` at a scope-bounding gate (Epic 7's gate per SR-5.2).** Write moment: immediately after `/quo-file-issue` returns successfully with the new Issue ticket ID, before the orchestrator continues with the soft-fix dispatch. Captures the follow-up Issue ID in the entry.
  - **Trigger B — User picks `Accept the limitation` at a scope-bounding gate (Epic 7's gate per SR-5.2).** Write moment: immediately after the `AskUserQuestion` returns the user's choice, before the orchestrator continues without a fix. Captures `none` in the follow-up Issue field.
  - **Trigger C — Orchestrator auto-routes a single-path `(1, refactor-locally)` finding per Epic 7's routing table.** Write moment: **at the moment of implementer dispatch** (not at any gate-close, because there is no gate for this branch). The orchestrator writes the tracker entry as part of the auto-dispatch step, in the same logical action as the dispatch itself. Captures `Auto-routed (a) per single-path refactor-locally rule` as the Decision field and the routing-table rule wording as the Rationale field.
  - **Trigger D — User overrides an auto-route decision after the post-completion review challenges it** (per SDD SR-6.7's recovery gate, owned by Epic 10). Write moment: immediately after the user picks `Accept the misjudgment and proceed` at the SR-6.7 three-choice gate, before the post-completion flow continues. Captures `User overrode auto-route after post-completion challenge` as the Decision field and the post-completion reviewer's challenge text as the Rationale field. Note: when the user picks `File follow-up Issue to revisit the depth decision` at the SR-6.7 gate instead, the ORIGINAL Trigger C tracker entry (already written at auto-dispatch time) is amended with the new follow-up Issue ID — no new tracker entry is appended for the `File follow-up Issue` path, just the original entry's `Follow-up Issue` field is updated. Trigger D only fires on the `Accept the misjudgment` path.
- Entry shape per SDD `## Codebase exploration findings` → "Data models" → "Compromise tracker entry shape":
  - Finding verbatim (severity + fix paths with depth tags + description).
  - Decision (user-picked path / Defer / Accept / auto-route rule / explicit override).
  - One-line rationale.
  - Follow-up Issue ID (or `none`).
- Tracker persistence: file system, persists across orchestrator-yield events within a single run; new `<session-suffix>` per run (no cross-run accumulation).
- Cross-platform compliance: paired POSIX bash + Windows PowerShell snippets for `mkdir -p` / `New-Item -ItemType Directory -Force` create-if-absent.

## Acceptance criteria

1. Both `skills/quo-fix-issue/SKILL.md` and `skills/quo-execute/SKILL.md` document the tracker file path convention.
2. Both orchestrators' SKILL.md prose names the **four** append triggers (Defer / Accept / auto-route refactor-locally / explicit override after post-completion challenge) with their precisely-defined moment-of-write per the Scope section above.
3. The tracker entry shape is documented in both SKILL.md files (or referenced via the SDD) with the five canonical fields.
4. A live `/quo-fix-issue` or `/quo-execute` run that triggers at least one append produces a tracker file at the documented path with at least one entry following the canonical shape.
5. Paired POSIX + PowerShell snippets are present for any new shell commands in the tracker-write path.
6. The auto-route Trigger C write fires at implementer dispatch time (not deferred to gate-close, since there is no gate for this branch). Verifiable in the SKILL.md prose by checking that the tracker-write call appears in the same logical block as the auto-dispatch step, not in a separate post-gate block.
7. Trigger D fires only on the `Accept the misjudgment` branch of SR-6.7's three-choice gate. The `File follow-up Issue` branch updates the original Trigger C entry rather than appending a new entry; the `Pause to discuss` branch defers any tracker write until the discussion resolves and the user picks again.
