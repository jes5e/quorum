## What Needs to Change

Author the new **"Orchestrator discipline: routing review findings"** section ONCE, then place a near-identical copy in each of the two files at the insertion point named in Context. The single permitted byte-level divergence between the two copies is the Cancel-semantics wording in part (d). The section comprises the following parts, all carried in both copies:

**(a) Deterministic routing table (FR-2 / SR-2.3 / SR-2.4).** A two-column-keyed table mapping the reviewer-emitted `(num-paths, max-depth-across-paths)` tuple to exactly one routing decision (mechanically followable — no tuple maps to two decisions):

```
| Number of fix paths | Maximum depth across paths | Orchestrator routing                         |
|---------------------|----------------------------|----------------------------------------------|
| 1                   | trivial-tweak              | Auto-dispatch the implementer with the fix   |
| 1                   | refactor-locally           | Auto-dispatch the implementer with the fix   |
| 1                   | re-architect               | User gate before dispatching                 |
| > 1                 | (any)                      | User gate before dispatching                 |
```

State explicitly that the orchestrator reads the reviewer's emitted tags and consults the table; it does NOT invent its own classification.

**(b) Anti-pattern callout (FR-3 / SR-2.5).** A callout block titled exactly **"ANTI-PATTERN — do not write this:"** forbidding scope-bounding directives inside R2/R3 dispatch prompts. It MUST name the specific forbidden phrasings verbatim, including at minimum: `"out of scope for this issue"`, `"out of scope for <id>"`, `"prefer option (a)"` / `"prefer (a)"`, `"Do NOT add X — out of scope"`. It directs the orchestrator to surface scope-bounding decisions via the scope-bounding gate in part (c) instead of inlining them. The forbidden phrasings must appear ONLY inside this callout's quoted-forbidden-phrasings list and nowhere else in either file (see Acceptance Criteria).

**(c) Scope-bounding gate prose (FR-3 / SR-5.2).** The behavior that REPLACES the forbidden directives: when the orchestrator would otherwise scope-bound a finding, it fires an `AskUserQuestion` via the two-step `TaskCreate` → `AskUserQuestion` contract with exactly three finite choices — `Fix properly now` / `Defer to follow-up Issue` / `Accept the limitation`. The question text includes the finding verbatim plus a one-line context line for why the gate is firing. **No `Cancel` option at this gate** — scope-bounding is a per-finding decision and `Cancel` would be ambiguous; the user retains `Ctrl-C` for run-level abort. State that `Defer to follow-up Issue` files a follow-up Issue via `/quo-file-issue`, and that `Accept the limitation` records the limitation (note that the compromise tracker which consumes it is Epic `po`; this Epic only emits the gate — do not implement the tracker here).

**(d) Routing-decision gate prose (FR-2 / SR-5.1).** When the routing table yields "User gate before dispatching" (single-path `re-architect`, or any multi-path finding), fire an `AskUserQuestion` via the two-step contract whose choices are: one choice per reviewer-surfaced fix path (each choice's description includes that path's depth tag), PLUS a `Defer to follow-up Issue` option, PLUS a `Cancel` option. The `(Recommended)` marker goes on the path the reviewer flagged as preferred when the reviewer surfaced a preference; otherwise no path is marked Recommended. The question text includes the finding verbatim. **This is the ONE intentional per-skill divergence:** in `/quo-fix-issue`, `Cancel` aborts the **current Issue's fix run** (proceed to the next Issue in batch/`all` mode, or end cleanly if none remain); in `/quo-execute`, `Cancel` aborts the **current Task's execution** (proceed to the next Task, or end if none remain). In both, `Cancel` does NOT abort the whole run (that remains `Ctrl-C`). Everything else in part (d) is identical between the two copies.

**(e) Backwards-compatibility shim (FR-7 / SR-2.6).** A finding emitted without a depth tag (legacy reviewer emission during rollout, or a hand-authored finding from a future call site) MUST be treated by the routing table as if it carried `re-architect` depth — i.e., it routes to the user gate. State that the shim errs toward user input when uncertain.

**(f) Edge-case handling (SR-6.1 / SR-6.2 / SR-6.4).** Include all three: (i) **malformed tags (SR-6.1)** — when a severity tag is not exactly `blocker`/`suggestion`/`nit`, or a depth tag is not exactly `trivial-tweak`/`refactor-locally`/`re-architect`, the orchestrator treats the finding as `re-architect` depth (per the shim) AND surfaces the parse failure to the user so the reviewer emission can be corrected; (ii) **routing ambiguity (SR-6.4)** — if the routing table somehow returns more than one decision (impossible by construction), default to the user gate and surface the ambiguity; (iii) **`/quo-file-issue` failure at the Defer gate (SR-6.2)** — when the user picks `Defer to follow-up Issue` (at either the scope-bounding gate (c) or the routing-decision gate (d)) but the inline `/quo-file-issue` dispatch fails or the user cancels at one of its gates, the orchestrator MUST NOT silently ship the soft fix (or no fix). Instead it surfaces the failure to the user and re-prompts with the same gate's choices, so the user can re-attempt the defer, pick `Accept the limitation` / a specific fix path explicitly, or (at the routing gate) Cancel.

For the two-step gate prose in parts (c) and (d), match the house pattern verbatim: first `TaskCreate` a `gate-askuserquestion-<short-suffix>` TaskList task naming the gate (per each file's Section 4 / Section 3 TaskList naming convention's gate-task entry, with a distinct per-fire suffix), then call `AskUserQuestion` in the same turn; cite `docs/doc-writing-guide.md` `## The two-step TaskCreate → prescribed-tool contract`; and instruct the orchestrator not to produce a text response describing the gate. Mark the `gate-*` task `completed` once the user's answer is consumed and the routing branch is entered. No new shell snippets are expected (the gates are `AskUserQuestion`-based) — if any are added, they must ship in paired POSIX + PowerShell form.

Pre-implementation grep during planning found none; if any pre-existing instance is found, rewrite it to use the new gate prose and note in the commit message that this was verified (the common case being "no pre-existing instances found").
