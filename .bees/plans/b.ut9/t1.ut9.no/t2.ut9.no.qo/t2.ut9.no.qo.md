## Context

This Epic adds a new **"Orchestrator discipline: routing review findings"** section to BOTH orchestrator skills — `skills/quo-fix-issue/SKILL.md` (immediately before its Section 5 "Review Loop", currently ~line 411) AND `skills/quo-execute/SKILL.md` (immediately before its Section 5 "Final Bee-level Code, Doc and Eng reviews", currently ~line 416). Per the project's reuse convention and SDD SR-2.2, **the two sections must be near-identical prose; divergence between them is a defect, not an intentional difference.**

This is Phase 2's first Epic. It is the **consumer** of the depth-tagged finding emission that Phase-1 Epics (`9c` / `jn` / `q2` / `29` / `53`) added. The backwards-compat shim (FR-7) decouples it: it can land standalone because findings without a depth tag are treated as `re-architect`. Per the SDD/PRD `## Risk / Known Dependencies`, the new user gates this section adds inherit the `b.wii` adherence fragility — that is an execution-time risk acknowledged at `/quo-execute` time, not a reason to alter this breakdown.

This is a skill-prose change in the quorum repo: honor CLAUDE.md's three design rules, the scratch-file convention, and `## Review criteria for skill changes`. Any new gate prose MUST use the two-step `TaskCreate` → `AskUserQuestion` contract (per `docs/doc-writing-guide.md`) and respect CLAUDE.md `## AskUserQuestion usage` (multi-choice only; no fake free-text options).

## What Needs to Change

Add the **"Orchestrator discipline: routing review findings"** section (titled exactly that) to both SKILL.md files, positioned immediately before each one's existing Section 5 review-loop content. Both copies carry the same prose, comprising:

**(a) Deterministic routing table (FR-2 / SR-2.3 / SR-2.4).** Two columns mapping the reviewer-emitted `(num-paths, max-depth-across-paths)` tuple to a routing decision; mechanically followable (exactly one decision per tuple):

```
| Number of fix paths | Maximum depth across paths | Orchestrator routing                         |
|---------------------|----------------------------|----------------------------------------------|
| 1                   | trivial-tweak              | Auto-dispatch the implementer with the fix   |
| 1                   | refactor-locally           | Auto-dispatch the implementer with the fix   |
| 1                   | re-architect               | User gate before dispatching                 |
| > 1                 | (any)                      | User gate before dispatching                 |
```

The orchestrator reads the reviewer's emitted tags and consults the table; it does NOT invent its own classification.

**(b) Anti-pattern callout (FR-3 / SR-2.5).** A callout block titled exactly **"ANTI-PATTERN — do not write this:"** forbidding scope-bounding directives inside R2/R3 dispatch prompts. It MUST name the specific forbidden phrasings verbatim, including: `"out of scope for this issue"`, `"out of scope for <id>"`, `"prefer option (a)"` / `"prefer (a)"`, `"Do NOT add X — out of scope"`. It directs the orchestrator to surface scope-bounding decisions via the scope-bounding gate (c) instead of inlining them.

**(c) Scope-bounding gate prose (FR-3 / SR-5.2).** The behavior that REPLACES the forbidden directives: when the orchestrator would otherwise scope-bound a finding, it fires an `AskUserQuestion` (via the two-step `TaskCreate` → `AskUserQuestion` contract) with exactly three finite choices — `Fix properly now` / `Defer to follow-up Issue` / `Accept the limitation`. The question text includes the finding verbatim plus one-line context for why the gate is firing. **No `Cancel` option at this gate** (scope-bounding is a per-finding decision; `Cancel` would be ambiguous — the user retains `Ctrl-C` for run-level abort). The `Defer to follow-up Issue` choice files a follow-up Issue via `/quo-file-issue`; `Accept the limitation` records the limitation (the compromise tracker that consumes it is Epic `po` — this Epic only emits the gate, do not implement the tracker here).

**(d) Routing-decision gate prose (FR-2 / SR-5.1).** When the routing table yields "User gate before dispatching" (single-path `re-architect`, or any multi-path finding), fire an `AskUserQuestion` (via the two-step contract) whose choices are: one per reviewer-surfaced fix path (each choice's description includes that path's depth tag) PLUS a `Defer to follow-up Issue` option PLUS a `Cancel` option. The `(Recommended)` marker goes on the path the reviewer flagged as preferred, when the reviewer surfaced a preference; otherwise no path is marked Recommended. The question text includes the finding verbatim. **Cancel semantics differ per skill:** in `/quo-fix-issue`, `Cancel` aborts the **current Issue's fix run** (proceed to the next Issue in batch/`all` mode, or end cleanly if none remain); in `/quo-execute`, `Cancel` aborts the **current Task's execution** (proceed to the next Task, or end if none remain). `Cancel` does NOT abort the whole run (that remains `Ctrl-C`). This is the one intentional per-skill wording difference (Issue vs Task); everything else in the two sections is identical.

**(e) Backwards-compatibility shim (FR-7 / SR-2.6).** A finding emitted without a depth tag (legacy reviewer emission during rollout, or a hand-authored finding from a future call site) MUST be treated by the routing table as if `re-architect` depth — i.e., it routes to the user gate. The shim errs toward user input when uncertain.

**(f) Edge-case handling (SR-6.1, SR-6.2, SR-6.4).** Include: (i) **malformed tags (SR-6.1)** — when a severity tag is not exactly `blocker`/`suggestion`/`nit`, or a depth tag is not exactly `trivial-tweak`/`refactor-locally`/`re-architect`, the orchestrator treats the finding as `re-architect` depth (per the shim) AND surfaces the parse failure to the user so the reviewer emission can be corrected; (ii) **routing ambiguity (SR-6.4)** — if the routing table somehow returns more than one decision (impossible by construction), default to the user gate and surface the ambiguity; (iii) **Defer-gate `/quo-file-issue` failure (SR-6.2)** — when the user picks `Defer to follow-up Issue` (at either gate) but `/quo-file-issue` fails or is cancelled, the orchestrator MUST NOT silently ship the soft fix; it surfaces the failure and re-prompts with the same gate's choices. SR-6.2's home is this Epic because the `Defer to follow-up Issue` choice originates in this Epic's two gates (it is owned by neither the tracker Epic `po`, the summary Epic `od`, nor the post-completion Epic `pw`).

## Why

The root cause this feature addresses: the orchestrator has had unilateral authority to pre-narrow fix paths and scope-bound findings inside dispatch prompts, letting soft fixes ship without user input. The depth dimension gives a deterministic basis for when to gate the user; the anti-pattern callout removes the inline scope-bounding habit; the two gates make compromise decisions user-owned. This section is the structural heart of the feature.

## Success Criteria

1. Both SKILL.md files contain a section titled exactly "Orchestrator discipline: routing review findings".
2. Both sections contain the deterministic routing table; mechanically followable (exactly one decision per `(num-paths, max-depth)` tuple).
3. Both sections contain the anti-pattern callout with the verbatim forbidden phrasings.
4. Both sections contain the scope-bounding gate prose (SR-5.2): `Fix properly now` / `Defer to follow-up Issue` / `Accept the limitation`, NO Cancel.
5. Both sections contain the routing-decision gate prose (SR-5.1): one choice per fix path + `Defer to follow-up Issue` + `Cancel`, Recommended marker on the reviewer-preferred path, per-skill Cancel semantics (Issue vs Task).
6. Both sections contain the backwards-compat shim wording (missing-depth → `re-architect` → user gate) and the malformed-tag / routing-ambiguity edge-case handling.
7. **A `grep` for the forbidden anti-pattern phrasings (case-insensitive) against BOTH affected SKILL.md files returns hits ONLY inside the anti-pattern callout's quoted-forbidden-phrasings list — nowhere else.** Pre-implementation grep (done during b.tip planning) confirmed neither file currently contains the forbidden phrasings; if any pre-existing dispatch-prompt example is found to contain one at implementation time, rewrite it to use the new gate prose, and note in the commit message that this was verified (the common case being "no pre-existing instances found").
8. The two orchestrators' "Orchestrator discipline" sections are near-identical; the ONLY permitted difference is the routing-gate `Cancel` semantics wording (per-Issue in `/quo-fix-issue` vs per-Task in `/quo-execute`).
9. All new gate prose uses the two-step `TaskCreate` → `AskUserQuestion` contract and is multi-choice only (no fake free-text options). The change is pure prose: language-agnostic, project-neutral; any new shell snippets ship in paired POSIX + PowerShell form (none are expected — the gates are `AskUserQuestion`-based, not shell-based).
