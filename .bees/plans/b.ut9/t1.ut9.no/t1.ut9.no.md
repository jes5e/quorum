Add the new "Orchestrator discipline: routing review findings" section to BOTH `skills/quo-fix-issue/SKILL.md` (before Section 5) AND `skills/quo-execute/SKILL.md` (before Section 5). Per the project's reuse convention, both orchestrators land the same prose in lockstep; divergence between them is a defect, not an intentional difference.

The section contains:

(a) The deterministic routing table per PRD FR-2.

(b) The anti-pattern callout per PRD FR-3 and SR-2.5 forbidding scope-bounding directives in R2/R3 dispatch prompts.

(c) **The scope-bounding gate prose** per PRD FR-3 and SR-5.2 — the three-choice `AskUserQuestion` shape (`Fix properly now` / `Defer to follow-up Issue` / `Accept the limitation`) that REPLACES the anti-pattern's forbidden directives. The anti-pattern callout names what NOT to do; this gate prose is what to do instead. Without this gate prose, the design's central promise — "scope-bounding becomes a user gate, never an inline directive" (PRD G2) — has no implementer.

(d) The backwards-compatibility shim per PRD FR-7 (missing-depth findings treated as `re-architect`).

## Scope

- Add the new section (titled exactly "Orchestrator discipline: routing review findings") to `skills/quo-fix-issue/SKILL.md`.
- Add the same titled section to `skills/quo-execute/SKILL.md`.
- Both sections share the same routing table:
  | Number of fix paths | Maximum depth across paths | Orchestrator routing                         |
  |---------------------|---------------------------|---------------------------------------------|
  | 1                   | trivial-tweak             | Auto-dispatch the implementer with the fix  |
  | 1                   | refactor-locally          | Auto-dispatch the implementer with the fix  |
  | 1                   | re-architect              | User gate before dispatching                |
  | > 1                 | (any)                     | User gate before dispatching                |
- Both sections share the same anti-pattern callout (titled "ANTI-PATTERN — do not write this:") forbidding `"out of scope for this issue"`, `"out of scope for <id>"`, `"prefer option (a)"`, `"Do NOT add X — out of scope"`, etc. **Pre-implementation grep verification (done during the b.tip planning phase) confirmed neither `skills/quo-fix-issue/SKILL.md` nor `skills/quo-execute/SKILL.md` currently contains the forbidden phrasings — the new callout is purely preventive going forward.** The Epic still includes the grep verification step as an acceptance criterion in case future SKILL.md additions accidentally introduce a forbidden phrasing before this Epic lands; if any new forbidden-phrasing instances are found at implementation time, they must be rewritten to use the scope-bounding gate per (c) below before this Epic ships.
- **Both sections include the scope-bounding gate prose** per SR-5.2:
  - The orchestrator's behavior when it would otherwise write a scope-bounding directive: instead fire an `AskUserQuestion` with the three finite choices `Fix properly now`, `Defer to follow-up Issue`, `Accept the limitation`.
  - The question text includes the finding verbatim and one-line context for why the gate is firing.
  - Cancel semantics: explicitly NOT one of the three choices — scope-bounding decisions are gate-level, not Issue/Bee-run-level. (For Issue/Bee-run-level Cancel, see SR-5.1's routing gate; that's a different gate shape.)
- Both sections share the same backwards-compat shim: findings emitted without a depth tag are treated as if `re-architect` depth.
- Both sections also include the routing-decision gate prose per SR-5.1 (the multi-path / single-path-re-architect gate), with its own choices: one per reviewer-surfaced fix path, plus `Defer to follow-up Issue`, plus `Cancel`. Cancel semantics here are defined by the PRD revision concurrent with this Epic (see PRD `## UI/UX Requirements` for the Cancel definition).
- The new section is positioned immediately before each orchestrator's existing Section 5 review-loop content.

## Acceptance criteria

1. Both SKILL.md files contain a section titled exactly "Orchestrator discipline: routing review findings".
2. Both sections contain the deterministic routing table; mechanically followable.
3. Both sections contain the anti-pattern callout with the verbatim forbidden phrasings.
4. Both sections contain the scope-bounding gate prose per SR-5.2 — the `Fix properly now` / `Defer to follow-up Issue` / `Accept the limitation` three-choice `AskUserQuestion` shape.
5. Both sections contain the routing-decision gate prose per SR-5.1 — one choice per fix path plus `Defer to follow-up Issue` plus `Cancel`, with the Recommended marker on the reviewer-preferred path.
6. Both sections contain the backwards-compat shim wording.
7. **A `grep` for the anti-pattern phrasings against the affected SKILL.md files (BOTH the new "Orchestrator discipline" section AND any pre-existing dispatch-prompt examples/templates elsewhere in the same files) returns hits ONLY inside the anti-pattern callout's quoted-forbidden-phrasings list, not in any other location.** Any existing dispatch-prompt example that contains a forbidden phrasing is rewritten by this Epic to use the new gate prose; if no such examples exist today, document that explicitly in the commit message so future reviewers see this was verified.
8. The two orchestrators' "Orchestrator discipline" sections are near-identical in prose; divergence is a defect.

But the shim means this Epic can land standalone if needed.
