# Epic C — Layer 2.5: backend-equivalence harness + integration

**Status at completion.** Backend-equivalence harness verifies the dispatcher behaves identically on bees and beads. A single `make test` runs all three layers. CLAUDE.md `## Test Commands` documents the entrypoint. README Contributing section names the strategy. CI runs `make test` on every push and PR.

**Dependencies.** Blocked by Epic A (uses pytest infrastructure) and Epic B (linter must already be wired to the make target).

## Description

Build the backend-equivalence harness and wire all three test layers to a unified `make test` entrypoint with full contributor-facing documentation and CI integration.

The harness lives at `tests/equivalence/test_dispatcher_equivalence.py` (a new top-level `tests/` directory; cross-cutting harnesses don't belong inside a single skill's `scripts/`). Each test:

1. Spins up two temp directories — one initialized with `ticket_backend.py setup-spaces --backend bees`, the other with `--backend beads`.
2. Runs the same sequence of dispatcher verb calls against each (e.g., create Plan Bee → create Epic → set up_dependencies → query unblocked tickets → show ticket).
3. Captures resulting state via the dispatcher's read verbs (`query`, `show`).
4. Normalizes responses — strips `id` (different formats per backend), `created_at` (timestamps), `guid`, and any other backend-specific noise — and asserts deep equality on the remaining structure.

Use a shared normalization helper so each test is concise. The harness skips gracefully (`pytest.skip`) when either CLI is missing, naming the missing one in the message. Tests are pytest-parametrized over a small canonical set of dispatcher operation sequences (cover every verb at least once).

Wire the unified entrypoint via a `Makefile` at the repo root with `test`, `test-helpers`, `test-lint`, and `test-equivalence` targets. The `test` target depends on all three. Add a `tools/run_tests.py` script for Windows users without `make` — invokes the same three commands via subprocess and exits non-zero on any failure.

Add a `## Test Commands` section to CLAUDE.md (contributor-facing only; skills don't read it) listing the four entrypoints. Add a short paragraph to the README's Contributing section naming the three layers and pointing at `make test` and CLAUDE.md `## Test Commands`.

Add a `.github/workflows/test.yml` that runs `make test` on every push and PR. Both backend CLIs (bees via pipx, bd via the install path beads documents) must be installed in the workflow image so Layer 2.5 doesn't skip. The workflow file lives in the repo so contributors can see what's gating their PR.

## Acceptance criteria

1. **Demo: harness passes locally.** Agent runs `pytest tests/equivalence/` from the repo root with both backends installed; output ends with all tests passing.
2. **Demo: harness skips on missing backend.** Agent uninstalls one backend (or sets PATH to hide it), runs `pytest tests/equivalence/`, and confirms each test SKIPS (not FAILS) with a clear message naming the missing CLI.
3. **Demo: full make test.** Agent runs `make test` from the repo root and the output shows all three layers running in sequence (lint → helpers → equivalence) with all passing. Exit code 0.
4. **Demo: Windows fallback.** Agent runs `python tools/run_tests.py` (which works cross-platform) and confirms equivalent output to `make test`.
5. **Demo: CI green.** Agent confirms the GitHub Actions workflow runs `make test` on every PR (visible in `.github/workflows/test.yml`) and a test PR passes the workflow.
6. **Demo: docs in place.** CLAUDE.md `## Test Commands` lists all four entrypoints; README Contributing section has the new paragraph naming the three layers.
7. **Demo: deliberate divergence catches.** Agent introduces a deliberate behavior divergence between the bees and beads adapter paths in `ticket_backend.py` (e.g., a different default), runs `pytest tests/equivalence/`, and confirms the harness reports the divergence with a clear failure message.
