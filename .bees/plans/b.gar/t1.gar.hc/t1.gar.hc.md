# Epic B — Layer 2: structural SKILL.md linter

**Status at completion.** A `tools/lint_skills.py` script enforces the project's design rules across all 11 SKILL.md files, exits 0 on the current repo, and exits 1 with clear error messages on violations.

## Description

Write a Python 3 (no third-party deps) script at `tools/lint_skills.py` that walks every `skills/*/SKILL.md` file and asserts the design rules baked into CLAUDE.md. The linter is the automated enforcement of the three contributing principles plus the project-specific bees etiquette rules.

Rules to enforce:

- **Frontmatter check.** Every SKILL.md has YAML frontmatter with `name` and `description` keys.
- **OS-pair check.** Every fenced code block tagged `bash` or labeled "POSIX" has a sibling block tagged `powershell` or labeled "Windows" within the same H3-or-deeper section, and vice versa. Tolerate fence labels in both styles (`bash` / `# POSIX (bash / zsh):` / `powershell` / `# Windows (PowerShell):`).
- **No hardcoded user paths.** No path starting with `/Users/`, `/home/`, or `C:\Users\` anywhere in skill prose.
- **No raw stack-specific commands** (`cargo`, `npm`, `pip`, `pipx`) outside sections explicitly marked as install instructions (e.g., headers containing "Install" or "Prerequisites").
- **Bundled-script reference check.** All references to bundled helper scripts use the `<bees-setup-base-dir>` (or equivalent skill base-dir) literal placeholder, never an absolute path.
- **bees/bd subcommand allow-list.** All `bees ...` and `bd ...` subcommands in skill prose match an allow-list (catches typos and out-of-vocabulary subcommands). The allow-list lives at the top of `lint_skills.py` for easy maintenance.
- **Backend-conditional pair check.** Every "if backend=bees" block has an "if backend=beads" sibling within the same section (and vice versa) — applies only where backend-conditional blocks exist; non-conditional sections are fine.

Output is human-readable: one line per violation, format `<file>:<line>: <rule>: <message>`. Exits 0 on success, 1 on any violation.

A `Makefile` `test-lint` target runs `python tools/lint_skills.py` (top-level `make test` lands in Epic C).

## Scope

- Create `tools/` directory at the repo root (if not already present).
- Write `tools/lint_skills.py` with the seven rules listed above.
- Implement a tiny minimal YAML frontmatter parser to keep zero third-party deps (or use `pyyaml` only if it's already declared a runtime dep elsewhere; prefer the former).
- Add a self-test fixture: a tiny `tools/test_data/` directory with one valid skill (passes) and one violating skill per rule, plus a pytest file `tools/test_lint_skills.py` asserting each rule fires on its negative fixture and doesn't on the positive.
- Add a `Makefile` `test-lint` target that runs `python tools/lint_skills.py`.
- Verify `python tools/lint_skills.py` exits 0 on the current `skills/` directory.

## Acceptance criteria

1. **Demo: clean repo passes.** Agent runs `python tools/lint_skills.py` from the repo root and the script exits 0 with no output (or with a one-line success message).
2. **Demo: deliberate violations fail.** Agent applies seven temporary edits — one per rule — to a copy of a skill, runs the linter, confirms it exits 1 and prints exactly one violation per rule with the right `<file>:<line>: <rule>: <message>` format.
3. **Demo: self-test passes.** Agent runs `pytest tools/test_lint_skills.py` and all per-rule tests pass.
4. **Demo: existing skills clean.** Linter exits 0 on the current repo (post-b.9xr completion), confirming no hidden violations exist before this Epic merges.

## Out of scope

- Linting CLAUDE.md, README.md, or docs/ (only `skills/*/SKILL.md` is in scope).
- Auto-fix behavior — the linter reports only.
- Performance optimization (the repo has 11 skills; even a naive walk is fine).
- Stylistic checks (line length, prose quality, etc.) — focus is on contract violations only.
- Layer 1 helper tests (Epic A).
- Layer 2.5 equivalence harness (Epic C).
- Top-level `make test` target (Epic C).
- CI workflow (Epic C).
