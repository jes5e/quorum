## Why

The `bees`-resolver pattern adoption (parent Bee body Decisions section: "Adopt apiary's Spec Bee + reference_materials pattern") preserves the polymorphism the bees CLI already supports for `reference_materials` — file paths, ticket IDs via the `bees` resolver, and future external-system resolvers (GitHub issues, Linear, etc.) drop into the same field with no skill changes.

Dropping the Scoped-marker emission removes the cumulative-doc-coupling overhead the parent Bee body Problem 1 calls out: the Scoped-marker machinery existed precisely so downstream skills could find the right `### Feature:` subsection in shared cumulative docs. With `/bees-plan` no longer co-mingling content into shared docs, the marker is unnecessary on this path. (`/bees-plan-from-specs --feature` still uses it for hand-authored cumulative docs.)

Body simplification mirrors apiary's Plan Bee shape and matches the parent Bee `b.31f`'s own body (which was authored in this style as a self-bootstrap example).
