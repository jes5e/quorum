## Context

Epic `t1.31f.y2` (parent Bee `b.31f`) redesigns `/quo-plan` to drop its plan-time doc-mutation behavior and instead delegate spec authoring to the new `/quo-write-prd` (sibling Epic `t1.31f.by`) and `/quo-write-sdd` (sibling Epic `t1.31f.5u`) skills. This Task delivers the new Step 4 of `/quo-plan`'s workflow: it strips the existing doc-mutation logic (which today writes to `docs/prd.md`, `docs/sdd.md`, and the README at plan time) and replaces it with the delegated-spec-authoring pattern (create a Spec Bee in the Specs hive, then invoke `/quo-write-prd` and `/quo-write-sdd` via the Skill tool, then transition the Spec Bee to `ready`).

Sibling Tasks under this Epic:
- **Task 2** of this Epic (`t2.31f.y2.<id>`) — mid-conversation context awareness for Steps 1/2 of `/quo-plan`.
- **Task 3** of this Epic — Plan Bee creation update (`bees`-resolver `reference_materials`), drop Scoped-marker emission, simplify Plan Bee body, verify Step 7 commit logic.

This Task focuses solely on the Step 4 redesign — the doc-mutation strip + the new delegation flow + the Spec Bee lifecycle.

## What Needs to Change

`skills/quo-plan/SKILL.md` is the file modified by this Task. The redesign covers Step 4 of the existing `/quo-plan` workflow:

1. **Strip Step 4 entirely.** Remove the existing "Determine Doc Updates" workflow step. This includes:
   - The "categorize the feature" table that classifies the feature as Product / Architecture-reliability / Deployment-infra / CI-testing / Internal-refactor.
   - The PRD/SDD draft-update generation (the prose that writes `### Feature: <title>` subsections to cumulative project docs).
   - The README-update branch (which was Step 4a's "Always check the README"). Note: this branch logically moves to Epic 6 (`t1.31f.6w`)'s doc-writer post-impl flow — it is NOT in this Task's scope to rewrite the README-update logic, only to remove it from `/quo-plan`'s Step 4.
   - The Step 4 user-approval gate that confirms the proposed doc updates.

2. **Add new Step 4: Delegated spec authoring.** The replacement Step 4 must:
   - **Create the Spec Bee.** Use `bees create-ticket --hive specs --ticket-type bee --title <feature-title> --status drafted` to create the Spec Bee. The title comes from the feature scope distilled in earlier Steps; status starts at `drafted` per the Specs hive contract from sibling Epic `t1.31f.ho`.
   - **Invoke `/quo-write-prd` via the Skill tool.** Pass the Spec Bee ID and the conversation-distilled scope content. The exact handoff shape is the contract sibling Epic `t1.31f.by`'s Subtask `t3.31f.by.7h.xa` defines — read that Subtask body and use the documented input shape exactly. The skill returns the PRD ticket ID and final status.
   - **Invoke `/quo-write-sdd` via the Skill tool.** Same pattern, mirrored shape (sibling Epic `t1.31f.5u`'s Subtask `t3.31f.5u.f3.pt` defines the SDD-side contract; per cross-Task contracts in `.5u`, the shape is intentionally identical to `.by`'s so this Step 4 can dispatch both writers with the same arguments).
   - **Transition the Spec Bee to `ready`.** After both child tickets (PRD, SDD) have transitioned to `ready` (the writer skills handle their own `drafted → ready` transitions), update the Spec Bee from `drafted` to `ready` via `bees update-ticket --ids <spec-bee-id> --status ready`. The Plan Bee's eventual transition to `ready` (in Step 5 of `/quo-plan`, modified by Task 3 of this Epic) gates on this.

3. **Skill-tool invocation pattern.** Use `Skill(skill="quo-write-prd", args="<spec-bee-id>")` (or whatever literal-text shape the Skill tool supports — verify against the Skill tool documentation in CLAUDE.md). The conversation-distilled scope from earlier Steps must be passed in a way the writer skills accept; use the input shape from `t3.31f.by.7h.xa` and `t3.31f.5u.f3.pt` verbatim.

4. **Idempotency consideration.** If `/quo-plan` is re-run on the same feature (e.g., the user wants to refine after thinking more), the writer skills are idempotent against an existing Spec Bee — they update rather than duplicate. So Step 4's Spec Bee creation must check for an existing Spec Bee first (e.g., via `bees execute-freeform-query` against the Specs hive with a title or other identifier) and reuse it if present, rather than creating duplicates. The exact detection heuristic is the implementer's call but must be documented in the prose.

5. **Error handling.** If either writer skill returns an error (e.g., the user cancels the approval gate inside the writer), `/quo-plan` must surface the error and not proceed to subsequent Steps. The Spec Bee remains in `drafted` so the user can re-run.

6. **Skill-prose constraints.** Project-neutral; paired POSIX bash + Windows PowerShell snippets where applicable; single-literal-command Bash etiquette; scratch-file convention if any temp files are needed; no `rm`/`Remove-Item` cleanup.

## Why

Today's `/quo-plan` Step 4 mutates project docs at plan time, which causes three downstream problems documented in the parent Bee body's "Why" section: plan-without-execute pollutes docs; pass-through coordination requires the Scoped-marker machinery; parallel planning becomes incoherent. The redesign moves spec content into ticketed-form (a Spec Bee with PRD and SDD `t1=Doc` children) so plan-without-execute leaves project docs untouched, and parallel planning of multiple unrelated features stays coherent because each feature has its own Spec Bee.

The factored-skill delegation approach (rather than inlining all spec-authoring logic into `/quo-plan`) is documented in the parent Bee body's "Decisions and rejected alternatives" — composability, single-place quality bar, manageable size for `/quo-plan`, mental-model parity with apiary.

## Cross-Task contracts established by this Task

Sibling Task 3 of this Epic (Plan Bee creation) consumes this — the Plan Bee's `reference_materials` points at the Spec Bee, and the Plan Bee's transition to `ready` indirectly gates on the Spec Bee's `ready` state.
