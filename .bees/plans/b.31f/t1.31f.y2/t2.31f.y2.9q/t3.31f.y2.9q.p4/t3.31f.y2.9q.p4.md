This Subtask lands the first piece of the new Step 4: the Spec Bee handling — detect any existing Spec Bee for this feature (idempotent re-runs), reuse it if present, otherwise create a new one in the Specs hive with status `drafted`.

The Spec Bee anchors the PRD and SDD `t1=Doc` children that the next sibling Subtask adds via inline `/bees-write-prd` and `/bees-write-sdd` Skill-tool invocations. Cross-Task contracts consumed:
- **Specs hive contract** (sibling Epic `t1.31f.ho`): hive normalized name `specs`, allowed `bee`-tier creation, status values `drafted` and `ready` only.
- **Idempotency** (parent Task body item 4): re-running `/bees-plan` on the same feature must reuse an existing Spec Bee rather than duplicate. The detection heuristic is the implementer's call but must be documented inline.

## What Needs to Change

Edit `skills/bees-plan/SKILL.md`. Replace the placeholder stub left by the strip Subtask (or insert directly under the `### 4. Author Specs` heading) with the Spec Bee handling prose:

1. **Section intro.** A short paragraph explaining that Step 4 now creates or reuses a Spec Bee in the Specs hive, then invokes the writer skills (forward-references the next sibling Subtask's prose) and finally promotes the Spec Bee to `ready` (forward-references the final sibling Subtask's prose). Note the design rationale at a high level: spec content lives in ticketed form so plan-without-execute leaves project docs untouched and parallel planning of unrelated features stays coherent.

2. **Sub-step: Detect-or-create the Spec Bee.**

   - **Detection heuristic.** Document an idempotency check via `bees execute-freeform-query`: query the Specs hive for `bee`-tier tickets whose title matches the feature title from Step 3. Use a query along these lines (the exact YAML is the implementer's call — pick a shape that yields a clean ID + title list):

     ```yaml
     stages:
       - [type=bee, hive=specs]
     report: [ticket_id, title, ticket_status]
     ```

     Then inspect the returned list and fuzzy-match against the feature title from Step 3 (exact-equal-after-normalization is fine; document inline that the implementer can use a stricter or looser match as long as it does not create duplicates). If a match is found, present the existing Spec Bee to the user via `AskUserQuestion` (finite choices: `Reuse existing Spec Bee` / `Create a new Spec Bee anyway` / `Cancel`) so the user can override the heuristic when they really do want a fresh ticket. If no match is found, proceed to create.

   - **Create branch.** When no existing Spec Bee matches, author the body to a temp file under `<tempdir>/.quorum/` (per the scratch-file convention from CLAUDE.md) and call `bees create-ticket`:

     ```bash
     # POSIX (bash / zsh):
     mkdir -p /tmp/.quorum
     # then write the body to /tmp/.quorum/bees-spec-body-<short-suffix>.md via the Write tool
     bees create-ticket --ticket-type bee --hive specs --status drafted --title "<feature title>" --body-file <path>
     ```

     ```powershell
     # Windows (PowerShell):
     New-Item -ItemType Directory -Force -Path "$env:TEMP\.quorum" | Out-Null
     # then write the body to $env:TEMP\.quorum\bees-spec-body-<short-suffix>.md via the Write tool
     bees create-ticket --ticket-type bee --hive specs --status drafted --title "<feature title>" --body-file <path>
     ```

     Capture the returned Spec Bee ID for use in the Skill-tool invocations sibling Subtask and the Spec Bee promotion sibling Subtask.

   - **Reuse branch.** When the user picks `Reuse existing Spec Bee`, capture the existing Spec Bee ID (from the freeform-query result) and proceed to the Skill-tool invocations. The writer skills are themselves idempotent against existing PRD/SDD child tickets under the Spec Bee (per sibling Epics `t1.31f.by` and `t1.31f.5u`'s contracts), so reusing the Spec Bee cleanly cascades into update-rather-than-duplicate behavior on the children.

3. **Body content.** Document inline that the Spec Bee body is a brief 2-3 sentence summary of the feature scope distilled from Step 3, intentionally light because the substantive content lives in the PRD and SDD `t1=Doc` children. Do NOT dump full PRD/SDD content into the Spec Bee body.

4. **No `--reference-materials` on the Spec Bee.** The Spec Bee's children (PRD and SDD `t1=Doc` tickets) ARE the reference materials. The Spec Bee itself does not need `--reference-materials` set; it is its own anchor for downstream `bees`-resolver lookups.

5. **Idempotency-detection rationale documented inline.** Explicitly call out the err-toward-reuse principle: if the heuristic is ambiguous, prefer presenting the user with the reuse option rather than silently creating a duplicate. A duplicate Spec Bee is cheap to fix manually but easy to miss; a missed-reuse opportunity is silent corruption.

6. **Constraint compliance.** Paired POSIX bash + Windows PowerShell snippets. Single-literal-command Bash etiquette. Scratch-file convention under `<tempdir>/.quorum/` with `mkdir -p` / `New-Item -ItemType Directory -Force` create-if-absent. No `rm` / `Remove-Item` cleanup. Project-neutral prose.
