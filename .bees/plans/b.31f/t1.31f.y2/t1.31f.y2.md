## Outcome

`/bees-plan` redesigned to (a) drop its Step 4 doc-mutation behavior
entirely, (b) delegate spec authoring to `/bees-write-prd` and
`/bees-write-sdd` via the Skill tool, (c) detect mid-conversation
context and skip its own discovery questions when present, and (d) set
`reference_materials` to point at the Spec Bee via the `bees` resolver.

## Scope

1. **Strip Step 4 entirely.** Remove the "Determine Doc Updates"
   workflow step from `skills/bees-plan/SKILL.md`. No more drafting
   PRD/SDD updates, no more "categorize the feature" table, no more
   bootstrapping `docs/prd.md` and `docs/sdd.md` from a single
   feature's scope. The README-update branch (Step 4a's "Always check
   the README") moves to Epic 6's doc-writer post-impl flow.

2. **Mid-conversation context awareness (Step 1/2 redesign).** When
   `/bees-plan` is invoked from a substantive preceding conversation,
   skip the "Before I start researching, is there anything I should
   know?" prompt and the numbered clarifying-questions list. Instead,
   summarize what was already discussed and ask the user once: "Here's
   the scope I distilled from our conversation — is this right?" If
   the user wants to refine, iterate in prose; if approved, proceed.

   When invoked cold (no preceding context, or `/bees-plan
   <description>` with no further conversation), Steps 1/2 run as
   today.

3. **Spec authoring via delegation.** New Step 4 (replacing the
   stripped one): create the Spec Bee in the new Specs hive (status
   `drafted`), then invoke `/bees-write-prd` and `/bees-write-sdd`
   via the Skill tool, passing the Spec Bee ID and the
   conversation-distilled scope. Each writer skill creates its
   respective `t1=Doc` child.

4. **Plan Bee creation with `bees` resolver.** Step 5a creates the
   Plan Bee with
   `--reference-materials '[{"value":"<spec-bee-id>","resolver":"bees"}]'`.
   Drop the existing PRD/SDD-paths variant of `--reference-materials`
   (that was for the file-based flow which moves entirely to
   `/bees-plan-from-specs`).

5. **Spec Bee status transition.** After both write-prd and write-sdd
   have produced their child tickets, transition the Spec Bee from
   `drafted` to `ready`. The Plan Bee's transition to `ready` (after
   Epics are wired) gates on this.

6. **Drop Scoped-marker emission.** Remove the Scoped-marker
   `Scoped to '### Feature: <title>' ...` line that today's
   `/bees-plan` Step 5a emits when invoked against cumulative docs.
   The marker remains valid for `/bees-plan-from-specs --feature` mode
   — but `/bees-plan` no longer co-mingles content into shared docs,
   so it has no markers to emit.

7. **Body of the Plan Bee itself.** With `reference_materials`
   carrying the spec content, the Plan Bee body returns to a brief
   2-3 sentence summary plus the `## Anticipated doc impact` section
   noting which cumulative project docs Epic 6 (doc-writer post-impl)
   will need to update.

8. **Step 7 commit logic.** Already does
   the right thing — stages `docs/` AND the Plans hive's ticket
   tree, dropping `docs/` from the add list when no doc changes
   occurred. With Step 4 stripped, `docs/` will not change at plan
   time, so the existing "drop docs/ from add list" branch fires
   naturally. Verify the prose still reads correctly after Step 4
   is gone.

## Acceptance criteria

- `/bees-plan` invoked end-to-end on a fresh feature creates: Spec Bee
  (in Specs hive) + PRD child + SDD child + Plan Bee (in Plans hive)
  + Epic children. The Plan Bee's `reference_materials` is
  `[{"value":"<spec-bee-id>","resolver":"bees"}]`.
- `docs/prd.md`, `docs/sdd.md`, and the README are unchanged after
  the run. Verifiable via `git status` showing no staged or unstaged
  changes to those paths.
- Running `/bees-plan` twice for two unrelated features produces two
  independent Spec Bee + Plan Bee pairs with no cross-contamination
  in any shared doc file.
- When invoked from a preceding rich conversation, `/bees-plan` does
  not re-ask discovery questions; it distills the conversation into
  the scope statement directly.
- The Spec Bee's `t1=Doc` children include both `PRD` and `SDD`
  titles, each with rich content per Epics 2 and 3's required
  sections.
- The Spec Bee transitions to `ready` after both child tickets are
  written; the Plan Bee transitions to `ready` after Epics are wired.

## Implementation notes

- The Skill tool can invoke other skills inline. Pattern:
  `Skill(skill="bees-write-prd", args="<spec-bee-id>")`.
- Inline invocation must pass the conversation context (the
  distilled scope) so the writer skills don't restart their own
  discovery. The exact handoff format is part of Epic 2/3's design;
  Epic 4 must conform to whatever those skills accept.
- The Plan Bee scope-approval gate (today's Step 3 with
  `AskUserQuestion`) must remain — it's the user-facing checkpoint
  for "is this what you want?". Same for Step 5c's Epic-list approval.
- Update the skill catalog table in README.md if the description
  changes.
