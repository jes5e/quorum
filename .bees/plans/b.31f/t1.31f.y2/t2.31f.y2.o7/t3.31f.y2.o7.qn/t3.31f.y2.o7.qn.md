1. **Insert a mid-conversation detection step before Step 1.** Add a clearly-labeled new step (e.g., `### 0. Detect Mid-Conversation Context` or renumber Steps 1-7 to 1-8 with the new step as Step 1 — skill author's call, but the labeling must make the branching obvious to a reader scanning the SKILL.md). The step branches on whether the prompt window contains substantive prior context about the feature.

2. **Detection heuristic prose.** Mirror `t3.31f.by.7h.xa`'s detection language (same indicative signals, same err-toward-distill principle, same finite-choice approval-gate shape). Read `.xa`'s SKILL.md edits first and align verbatim where the prose makes sense (it describes the same heuristic). Indicative signals to enumerate:
   - Rich back-and-forth in the same session about the feature's scope, rationale, or alternatives.
   - The user's invocation message (or `/bees-plan <description>` argument) contains substantive scope information beyond a one-line title — e.g., a paragraph or more describing what they want, including motivation, constraints, or rejected alternatives.
   - An explicit hint that this is a continuation of a planning discussion (e.g., the user has been iterating on scope with the assistant before invoking the skill).

3. **Document the err-toward-distill principle inline.** State explicitly in the prose that the heuristic should tilt toward distilling rather than restarting when the choice is ambiguous (a wasted distill draft is cheaper than restarting a 30-minute planning conversation). Future maintainers must not tighten the heuristic into a stricter "only fire when X is unambiguously true" gate, which would defeat the design intent. Mirror `.xa`'s exact phrasing of this principle so the three skills stay in lockstep.

4. **Distill branch.** When the heuristic fires:
   - Skip the existing Step 1 "Before I start researching, is there anything I should know?" prompt.
   - Skip the existing Step 2 numbered clarifying-questions list.
   - Distill the prior conversation into a scope summary covering: what the feature does (one paragraph), why (motivation), key decisions made during the conversation, rejected alternatives, and constraints. Format the distillation as a markdown bullet list or short prose block — whichever format reads cleanly when shown to the user.
   - Present the distilled scope to the user via `AskUserQuestion` per CLAUDE.md `## AskUserQuestion usage` (it's multi-choice only). Finite choices: approve / revise / cancel.
   - On approve, proceed directly to Step 3 (existing Plan Bee scope-approval gate). The distilled scope feeds Step 3's What/Why/Acceptance criteria/Out-of-scope shape — no separate Step 3 iteration needed if the user already approved the distillation, but Step 3's `AskUserQuestion` remains as a final user-facing checkpoint.
   - On revise, iterate in prose with the user, then re-present the revised distillation via `AskUserQuestion`.
   - On cancel, exit the skill cleanly.

5. **Distilled-scope shape.** The format of the distilled scope passed forward must be precise enough that sibling Task `t2.31f.y2.9q`'s Skill-tool invocations of `/bees-write-prd` and `/bees-write-sdd` can consume it without further parsing. Mirror the input-shape requirements from `t3.31f.by.7h.xa`'s inline-invocation contract section. At minimum: a markdown block containing the feature title, what/why, acceptance criteria, out-of-scope, decisions, rejected alternatives, and constraints — labeled by section header so the writer skills can extract each section deterministically.
