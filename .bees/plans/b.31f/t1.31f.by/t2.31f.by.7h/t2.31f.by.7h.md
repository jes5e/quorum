## Context

Epic `t1.31f.by` (parent Bee `b.31f`) introduces a new `/bees-write-prd` skill that authors or revises a PRD ticket as a `t1=Doc` child of a Spec Bee. The skill is composable: invokable solo (`/bees-write-prd <spec-bee-id>`) for spec revisions, and invokable inline from the redesigned `/bees-plan` (sibling Epic `t1.31f.y2`) via the Skill tool. This Task delivers the skill itself — its `SKILL.md` prose and any required helper scripts.

The Specs hive (sibling Epic `t1.31f.ho`, broken down in this same Plan Bee) is the storage substrate this skill writes into. The contract values fixed by `.ho`'s Subtasks are consumed by this Task verbatim:

- Hive normalized name: `specs` (lowercase) when targeting `bees create-ticket --hive specs`.
- Tier name for the PRD ticket: `t1` (the only tier in the Specs hive).
- Title for the PRD ticket: exactly `PRD` (case-sensitive — sibling Epic `t1.31f.4u`'s PM two-hop lookup uses exact-match title to differentiate PRD from SDD).
- Status values for the PRD ticket: `drafted` (during authoring) and `ready` (when authoring completes); no other statuses.

Sibling Epic `t1.31f.5u` (the `/bees-write-sdd` skill) is the SDD-side analog of this Task; the two skills should share structural conventions but each owns its own SKILL.md. Sibling Epic `t1.31f.y2` (the `/bees-plan` redesign) is the inline caller that this Task's skill must accept handoff from. ## What Needs to Change

`skills/bees-write-prd/SKILL.md` is the single new file authored by this Task. The skill must implement, in prose Claude Code can follow:

1. **Skill scaffolding.** YAML frontmatter (`name`, `description`) precise enough that Claude Code's skill auto-invocation correctly fires on `/bees-write-prd` invocations. The `description` field must be specific enough to avoid mis-invocation per CLAUDE.md `## When editing skills` guidance — vague descriptions cause mis-invocation. Use apiary's `skills/write-prd/SKILL.md` frontmatter as the starting reference, but adapt the description to mention the Specs hive and the `/bees-write-prd` flow specifically.

2. **Required sections in the PRD ticket body.** The skill must always produce a PRD body containing the following sections in this order. The first ten are inherited from apiary's PRD template; the last two are NEW for this workflow because we lack apiary's loose-capture Idea Bee phase where rationale content lives in apiary's flow:

   - Problem Statement
   - Goals
   - Non-Goals / Out of Scope
   - Functional Requirements
   - Edge Cases and Error Handling
   - Non-Functional Requirements (if applicable)
   - UI/UX Requirements (if applicable)
   - Acceptance Criteria
   - Assumptions
   - Open Questions (if any)
   - **Background and rationale** (NEW)
   - **Decisions and rejected alternatives** (NEW)

   The two NEW sections must always be present in the body. They may be filled with substance or explicitly marked as "none — this PRD has no captured rationale from prior conversation" — but must never be silently omitted. Future cold-session agents (PM, breakdown, engineer) rely on the section's presence to know there is no captured rationale to consult; a missing section is ambiguous between "no rationale" and "author forgot."

3. **Mid-conversation context awareness.** When the skill is invoked from a substantive preceding conversation (e.g., as a delegated sub-skill from `/bees-plan`, or directly mid-discussion), it must distill that conversation into the required sections rather than restart with apiary's discovery questions. Detection heuristic: if the prompt window contains substantive prior context (rich back-and-forth about the feature's scope, rationale, alternatives), skip the "interview the user" step and propose a draft for review instead. The exact phrasing of the detection threshold is the skill author's call, but the skill must err toward distilling rather than restarting when the choice is ambiguous (a wasted draft is cheaper than restarting a 30-minute discovery).

4. **Solo invocation.** `/bees-write-prd <spec-bee-id>` invoked directly by the user against an existing Spec Bee must:
   - Read the Spec Bee via `bees show-ticket --ids <spec-bee-id>`.
   - Enumerate the Spec Bee's children to detect whether a `t1=Doc` child titled `PRD` already exists.
   - If absent: create the PRD child (`bees create-ticket --hive specs --ticket-type t1 --parent <spec-bee-id> --title PRD --body-file <path> --status drafted`). Use the namespaced workflow scratch dir (`<tempdir>/.bees-workflow/`) per CLAUDE.md scratch-file convention; do NOT remove the file after the create.
   - If present: update the existing PRD child via `bees update-ticket --ids <prd-id> --body-file <path>`. Default to full-rewrite semantics for coherence — `bees append-ticket-body` is available for explicit append-only revisions but is not the default.
   - Idempotent: re-running on the same Spec Bee does not create a duplicate PRD ticket.

5. **Inline invocation via Skill tool.** Sibling Epic `t1.31f.y2` (`/bees-plan` redesign) will invoke this skill via the Skill tool, passing the Spec Bee ID it just created and the conversation-distilled scope information. The skill must define a clean handoff contract — what input the inline caller passes, what output the skill produces — so `.y2`'s Skill-tool invocation can succeed without `.y2` having to inspect this skill's internals. Document the contract in the SKILL.md so future maintainers of `.y2` can rely on it.

6. **Quality bar.** Adapt apiary's "what NOT to include" rules and plain-english style guidance from `skills/write-prd/SKILL.md` — reject implementation details (those belong to the SDD), reject vague language ("should be fast"), require measurable acceptance criteria. The exact wording is the skill author's call but the spirit must match.

7. **Skill-prose constraints (this repo's design rules).** The new skill MUST follow the three non-negotiable design rules in this repo's CLAUDE.md:

   - Project-neutral: no hardcoded language-specific commands, file extensions, or manifest filenames.
   - POSIX + Windows PowerShell paired snippets for any shell command shown in prose; no bash-only fallback.
   - No baking-in of this repo's specific paths, ticket IDs, or internal workflow specifics.

   The scratch-file convention (write `--body-file` payloads under `<tempdir>/.bees-workflow/` with `mkdir -p` / `New-Item -ItemType Directory -Force` create-if-absent; never `rm` after) MUST be honored for any temp-file usage in the skill.

8. **Bash etiquette in skill prose.** Per CLAUDE.md `## Bash etiquette`, every Bash tool call shown in the skill must be a single literal command — no compound chains (`&&`, `||`, `;`), no pipes (`|`), no redirects (`>`), no shell variable expansion, no command substitution backticks. For sequential commands, use multiple Bash calls. For multi-step or variable-bearing logic, write a Python helper script.

## Why

Today's `/bees-plan` Step 4 mutates `docs/prd.md` at plan time. The redesign moves PRD content into a `t1=Doc` ticket child of a Spec Bee, decoupling spec authoring from doc mutation. The `/bees-write-prd` skill is the authoring surface that produces that ticket child; without this skill, sibling Epic `t1.31f.y2` (`/bees-plan` redesign) cannot complete its delegated-spec-authoring step.

Factoring out the spec-authoring logic into its own skill (rather than inlining it into `/bees-plan`) is a deliberate decision documented in the parent Bee body's "Decisions and rejected alternatives" section. The reasons: (1) re-authoring a PRD without re-planning is a real use case (revise after learning something during execution); (2) quality-bar prose lives in one place per skill rather than being copy-pasted; (3) `/bees-plan` shrinks substantially and becomes maintainable; (4) mental-model parity with apiary's `/write-prd` + `/write-srd` separation.
