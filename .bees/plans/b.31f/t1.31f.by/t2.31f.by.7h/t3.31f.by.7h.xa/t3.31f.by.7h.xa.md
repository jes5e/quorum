## Context

The sibling Subtask lands the scaffold of `skills/quo-write-prd/SKILL.md` and the full solo-invocation flow against an existing Spec Bee. This Subtask layers on the two remaining behaviors required by the parent Task (`t2.31f.by.7h`):

1. **Mid-conversation context awareness** (Task body item 3). When the skill is invoked from a substantive preceding conversation — either as a delegated sub-skill from sibling Epic `t1.31f.y2`'s redesigned `/quo-plan`, or directly mid-discussion when the user has been planning a feature with the assistant — the skill must distill that conversation into the twelve required PRD sections rather than restart with apiary-style discovery questions. The detection heuristic and the "distill, propose draft for review" branch belong to this Subtask.

2. **Inline-invocation contract via the Skill tool** (Task body item 5). Sibling Epic `t1.31f.y2`'s redesigned `/quo-plan` will invoke this skill via the Skill tool, passing the Spec Bee ID it just created plus the conversation-distilled scope information. This Subtask defines and documents the clean handoff contract — the input shape `.y2` passes, the output shape this skill produces — so `.y2`'s Skill-tool invocation can succeed without `.y2` having to inspect this skill's internals. The contract lives in the SKILL.md so future maintainers of `.y2` can rely on it as a stable API surface.

This Subtask does not modify the solo-invocation flow the sibling Subtask lands; it adds branching logic before discovery (to detect rich prior context) and adds a dedicated section in the SKILL.md documenting the Skill-tool handoff contract. After this Subtask, the parent Task's full acceptance set is met (modulo the Specs hive existing in the target repo, which sibling Epic `t1.31f.ho`'s Subtask provides, and the inline path being exercised end-to-end, which sibling Epic `t1.31f.y2` provides).

## What Needs to Change

Edit `skills/quo-write-prd/SKILL.md` (created by the sibling Subtask) to add two new behaviors. Existing prose stays; new prose layers on top.

1. **Mid-conversation detection branch.** Insert a step at the start of the workflow (before the discovery step the sibling Subtask lands) that branches on whether the prompt window contains substantive prior context about the feature.

   - **Detection heuristic (skill author's call on exact wording).** The heuristic must tilt toward distilling rather than restarting when the choice is ambiguous (per the parent Task body item 3 explicit guidance: "a wasted draft is cheaper than restarting a 30-minute discovery"). Indicative signals to enumerate in the prose: rich back-and-forth in the same session about the feature's scope, rationale, or alternatives; a `Description` argument passed to the skill that itself contains substantive scope information beyond a one-line title; an explicit hint from the Skill-tool caller (sibling `.y2`) that distilled scope is being passed in.
   - **Distill branch.** When the heuristic fires, skip the apiary-style discovery questions the sibling Subtask lands. Instead: read the prior context, distill it into the twelve required PRD sections (especially `## Background and rationale` and `## Decisions and rejected alternatives`, which are precisely where prior-conversation richness should land), and present the draft to the user for review before the create-or-update branch fires. Use `AskUserQuestion` for the approval gate (finite choices: approve / revise / cancel) per CLAUDE.md `## AskUserQuestion usage`.
   - **Restart branch.** When the heuristic does NOT fire (cold solo invocation), proceed with the sibling Subtask's discovery flow unchanged.
   - **Document the err-toward-distill principle inline** so future maintainers don't tighten the heuristic into a stricter "only fire when X is unambiguously true" gate, which would defeat the design intent.

2. **Inline-invocation contract section.** Replace the placeholder the sibling Subtask left in the Overview's "two invocation paths" listing with a dedicated `## Inline invocation via the Skill tool` section (or equivalently-titled subsection within Workflow) that documents the contract end-to-end:

   - **Input shape (caller → this skill).** Document what sibling Epic `t1.31f.y2`'s `/quo-plan` passes when it invokes this skill via the Skill tool. At minimum: the Spec Bee ID (the parent for the PRD `t1=Doc` child) and the distilled scope content from the planning conversation. The exact shape (free-text args string vs structured JSON-in-prose) is the skill author's call, but it must be precise enough that `t1.31f.y2`'s authors can implement the inline call without inspecting this skill's internals. Whatever shape is chosen, document it as a stable contract — future maintainers of `.y2` rely on it.
   - **Output shape (this skill → caller).** Document what this skill produces and reports back when invoked inline. At minimum: the PRD ticket ID it created or updated, plus the final status (`drafted` vs `ready`). This is what `.y2` needs to wire the Plan Bee's `reference_materials` to point at the Spec Bee whose child this PRD now is.
   - **Behavioral guarantees.** Same idempotency, same twelve required sections, same `drafted → ready` lifecycle as solo invocation — the inline path is functionally identical to solo from the Spec Bee's perspective; only the discovery-vs-distillation branch differs (inline always takes the distill branch because `.y2` always passes substantive distilled scope).
   - **Cross-reference to the mid-conversation detection.** Note that inline invocation is the canonical case where the distill branch fires.

3. **No regression to the sibling Subtask's solo flow.** The solo-invocation workflow the sibling Subtask lands continues to work identically when invoked without substantive prior context. The branching added here is a guard, not a replacement.

4. **Continued constraint compliance.** Every new Bash snippet (if any are added) ships as paired POSIX bash + Windows PowerShell variants. Every Bash tool call shown is a single literal command — no compound chains, pipes, redirects, shell-variable expansion, or diagnostic tails. Scratch-file usage stays under `<tempdir>/.quorum/` with create-if-absent. No `rm` / `Remove-Item`. Project-neutral prose (no this-repo-specific Bee IDs or paths).
