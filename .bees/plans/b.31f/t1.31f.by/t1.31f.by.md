## Outcome

A new `/quo-write-prd` skill that authors or revises a PRD ticket as a
`t1=Doc` child of a Spec Bee. Composable: invokable solo for spec
revisions, also invoked inline by the redesigned `/quo-plan` (Epic 4).

## Scope

1. **New skill at `skills/quo-write-prd/SKILL.md`.** Adapt apiary's
   `skills/write-prd/SKILL.md` as the starting point — its required
   sections, quality checklists, "what NOT to include" rules, and
   plain-english style guidance are directly applicable.

2. **Required sections in the PRD ticket body** (extending apiary's set):
   - Problem Statement
   - Goals
   - Non-Goals / Out of Scope
   - Functional Requirements
   - Edge Cases and Error Handling
   - Non-Functional Requirements (if applicable)
   - UI/UX Requirements (if applicable)
   - Acceptance Criteria
   - Assumptions
   - Open Questions (if any)
   - **Background and rationale** (NEW — apiary doesn't have this; we
     do because we don't have apiary's loose-capture Idea Bee phase
     where this content lives in apiary's flow)
   - **Decisions and rejected alternatives** (NEW — same reasoning;
     captures the *why* behind requirements so downstream cold-session
     agents don't re-litigate)

3. **Mid-conversation context awareness.** When invoked from a
   substantive preceding conversation (e.g., as a delegated sub-skill
   from `/quo-plan`, or directly mid-discussion), the skill must
   distill that conversation into the required sections rather than
   restart with apiary's discovery questions. Detection heuristic:
   if the prompt includes substantive prior context, skip the
   "interview the user" step and propose a draft for review instead.

4. **Solo invocation use case.** `/quo-write-prd <spec-bee-id>`
   creates the PRD child if it doesn't exist, OR updates it if it
   does. Idempotent. Useful for revising a PRD after learning
   something during execution.

5. **Inline invocation via Skill tool.** When `/quo-plan` (Epic 4)
   invokes this skill, it passes the Spec Bee ID it just created and
   the conversation-distilled scope information. The skill body must
   be designed to accept that handoff cleanly.

## Acceptance criteria

- `/quo-write-prd <spec-bee-id>` invoked solo against an existing
  Spec Bee creates a `t1=Doc`-titled `PRD` child with all required
  sections present. Verifiable by inspecting the resulting ticket
  body via `bees show-ticket`.
- `/quo-write-prd <spec-bee-id>` invoked again on the same Spec Bee
  updates the existing PRD ticket body rather than creating a duplicate.
- The PRD body ALWAYS includes `Background and rationale` and
  `Decisions and rejected alternatives` sections (filled when there is
  content to capture, omitted only when explicitly empty — never just
  missing).
- When invoked from a preceding rich conversation, the skill does not
  restart apiary-style discovery; it produces a draft based on the
  prior context for review.
- The skill is invokable both directly by the user and via the Skill
  tool from `/quo-plan` (Epic 4 verifies the inline path).

## Implementation notes

- Apiary's `/write-prd` adds the PRD ticket as a `t1` child of the
  Idea Bee with title `PRD` and status `larva`. We use the same
  shape but `status=drafted` (per our hive's status values) and
  parent is a Spec Bee.
- Use `--body-file <path>` per the scratch-file convention; do not
  inline rich bodies via `--body`.
- Update operations: use `bees update-ticket` or
  `bees append-ticket-body` depending on full-rewrite vs
  incremental-append semantics. The skill should default to
  full-rewrite for revisions to keep the body coherent.
- Update the skill catalog table in README.md to include
  `/quo-write-prd`.
