## Context

Sibling Task `t2.31f.g8.os` adds a new optional `## Doc divergence noted` section to the Issue body template authored by `/quo-file-issue`. When `/quo-fix-issue` later runs on such an Issue, its existing doc-handling prose must explicitly tell the consuming agent (Doc Writer in Section 3, and the orchestrator-direct spec-vs-code check in Section 5) that this section, when present, is a load-bearing input — the named file/section plus the documented correction become part of the fix's doc-sync scope.

Without this prose update, the new section's content reaches the Issue body but never triggers any visible behavior change in `/quo-fix-issue`. Today's Doc Writer bullet (Section 3, line 239) is generic — "reviews the Engineer's diff for documentation gaps and updates customer-facing and internal docs as needed" — and Section 5's spec-vs-code check (lines 320-339) reads only the project spec docs (PRD/SDD-equivalent), not the Issue body. Neither path mentions `## Doc divergence noted`, so the new section is silently ignored.

The parent Task body (`t2.31f.g8.g9`) directs this work and frames it as a small prose-only update bounded to `skills/quo-fix-issue/SKILL.md`. Note: the parent Task body's reference to "Section 5 (doc-sync pass via doc-writer dispatch)" is slightly misaligned with the current SKILL.md structure — Section 5 is the spec-vs-code check, not a Doc Writer dispatch site. The actual Doc Writer dispatch is in Section 3. This Subtask covers both touch points.
