## Context

Epic `t1.31f.g8` (parent Bee `b.31f`) redesigns `/bees-file-issue` to (a) drop its Step 4 doc-mutation behavior, (b) capture doc-divergence observations in the Issue body via a new `## Doc divergence noted` section, (c) be mid-conversation aware, and (d) deepen the body template with optional rationale-preserving sections.

This Task delivers the bulk of the redesign by editing `skills/bees-file-issue/SKILL.md`. Sibling Task 2 of this Epic separately handles the small `/bees-fix-issue` consumption update (verifying that `/bees-fix-issue`'s doc-sync pass reads the new `## Doc divergence noted` section and adding a small prose mention if needed).

This Epic is independent of the rest of the Bee (no `up_dependencies` from this Epic to others) — it can land in any order relative to the other Epics. The redesign mirrors the analogous changes happening in sibling Epic `t1.31f.y2` (`/bees-plan` redesign) but is smaller in scope: only Step 4 doc-mutation is being stripped (no spec-Bee-creation flow), and the rationale-preservation surface is via optional Issue-body sections (no delegated authoring).

## What Needs to Change

`skills/bees-file-issue/SKILL.md` is the file modified by this Task. Four logical changes:

1. **Strip Step 4 doc-mutation.** Remove the existing "Check if any project documentation needs updating" step (the prose that today edits `docs/prd.md` / `docs/sdd.md` / README at filing time). The remediation belongs to `/bees-fix-issue`'s doc-writer pass (sibling Epic `t1.31f.6w`'s expanded responsibility), not to filing.

2. **Add `## Doc divergence noted` body section to the Issue template.** When the issue surfaces a doc claim that's wrong (or a doc gap), record it in this new optional section as plain prose pointing at the file/section that's wrong and what's wrong about it. Sibling Epic `t1.31f.6w`'s expanded doc-writer responsibility consumes this section during `/bees-fix-issue`'s doc-sync pass.

3. **Mid-conversation context awareness.** When `/bees-file-issue` is invoked from a substantive preceding conversation, skip the discovery prompt ("What's the issue?") and distill the conversation directly into the Issue body. Detection heuristic: same as sibling Epic `t1.31f.y2`'s `/bees-plan` redesign (rich back-and-forth in same session, substantive scope info in invocation, etc.). Mirror the heuristic prose so the two skills make the same distill-vs-restart decision.

4. **Body template extension — optional rationale sections.** The current Issue body template (`Description / Current behavior / Expected behavior / Impact / Suggested fix`) is shallow and loses analytical depth. Add two optional sections:
   - `## Background and rationale` — populated when the issue came out of substantive analysis. Captures *why* this is a bug, *what root cause was ruled out*, *trade-offs* surfaced during analysis.
   - `## Decisions and rejected alternatives` — populated when the suggested-fix path was chosen over alternatives that were considered. Captures the alternatives and reasoning so `/bees-fix-issue`'s engineer doesn't re-litigate.

   These sections are OPTIONAL. Casual one-line bug reports omit them; analytical issues populate them. The optional-when-empty disambiguation is different from `/bees-write-prd`/`/bees-write-sdd`'s mandatory-always-present rule because Issues come in many sizes (one-line bug reports vs deep analytical sessions); requiring the rationale sections always-present would force noise into casual issues.

## Why

Today's `/bees-file-issue` Step 4 mutates project docs at filing time when the issue surfaces doc divergence — the smaller analog of the `/bees-plan` Step 4 problem (parent Bee body Problem 3). Smaller in scope (only fires for divergence-revealing issues, only edits existing claims rather than adding wholesale sections), but the same conflation of "observation" with "remediation," and the same coupling of doc updates to the wrong skill phase.

The Issue body itself is the spec source for `/bees-fix-issue` (no `reference_materials`, no children, no PRD/SDD pointer). When a thick discussion produces an Issue, the body's shallow template loses the analytical depth that came out of the discussion. The optional rationale sections preserve that depth without forcing it on casual issues.
