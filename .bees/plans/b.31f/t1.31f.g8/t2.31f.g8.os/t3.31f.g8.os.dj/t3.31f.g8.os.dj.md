Parent Task `t2.31f.g8.os` (parent Epic `t1.31f.g8`, parent Bee `b.31f`) redesigns `/quo-file-issue` to drop its filing-time doc-mutation behavior in favor of capturing doc-divergence observations in the Issue body. This Subtask delivers the swap: strip Step 4's doc-edit prose and replace it with prose that adds a `## Doc divergence noted` section to the Issue body when the issue surfaces a doc claim that's wrong (or a doc gap).

Sibling Epic `t1.31f.6w` expands `/quo-fix-issue`'s doc-writer pass to consume the `## Doc divergence noted` section during fix execution — that's where the remediation belongs, not at filing time.

## What Needs to Change

Edit `skills/quo-file-issue/SKILL.md`. Coupled edits:

1. **Strip Step 4 doc-mutation prose.** Remove the existing Step 4 "Check if any project documentation needs updating" body — the prose that today edits `docs/prd.md` / `docs/sdd.md` / README at filing time, and the closing paragraph that says "update them now so they reflect the intended/correct behavior." The remediation belongs to `/quo-fix-issue`'s doc-writer pass, not to filing.

2. **Replace Step 4 with a `## Doc divergence noted` capture step.** Rewrite Step 4 as an observation-only step: when the issue surfaces a doc claim that's wrong (or a doc gap), append a `## Doc divergence noted` section to the Issue body before the bees `create-ticket` call — OR fold the divergence-capture into Step 3's body authoring so the section is included in the initial body file. Skill author's call on whether to keep Step 4 as a separate "divergence sweep" step or merge it into Step 3's body authoring; either shape is fine as long as the resulting Issue body contains the section when divergence is observed and project docs remain untouched.

3. **Add `## Doc divergence noted` to Step 3's body-template documentation.** The Step 3 "Body structure" markdown block today lists `Description / Current behavior / Expected behavior / Impact / Suggested fix`. Add `## Doc divergence noted` to this template as an OPTIONAL section, with a one-line note that it's populated when the issue surfaces a doc claim that's wrong (or a doc gap) and omitted otherwise. Format: plain prose pointing at the file/section that's wrong and what's wrong about it. Use the Documentation-Locations paths from CLAUDE.md (`Internal architecture docs (SDD)`, `Customer-facing docs`, and the PRD-equivalent) as the canonical pointers.

4. **Update Step 5 commit prose.** Today's Step 5 stages `docs/` in the `git add` invocations (POSIX and PowerShell) on the assumption that Step 4 may have edited project docs. With Step 4's doc-mutation stripped, `docs/` should no longer be staged by default. Drop `docs/` from the `git_add_args` / `$addArgs` defaults; the `git add` should stage only the Issues hive path (when in-repo). Update the closing paragraph to remove the doc-changes language. Drop any "and any doc updates" framing in the Step 5 heading.

5. **Update Step 6 report-back.** Today's Step 6 says "Whether any docs were updated (and what changed)." Remove that line; replace with "Whether the issue captured a doc-divergence observation (the `## Doc divergence noted` section in the body) so the user knows `/quo-fix-issue`'s doc-sync pass will consume it."

6. **Preconditions touch-up.** Today's Preconditions says `Step 4 (doc-update check) reads architecture/customer-doc paths from this section by exact key.` Reword to reflect the new role: Step 3's body authoring (or the renamed Step 4) reads architecture/customer-doc paths from the `## Documentation Locations` section so the `## Doc divergence noted` capture can point at the right file. The hard-fail precondition itself stays — CLAUDE.md `## Documentation Locations` is still required — only the rationale sentence changes.
