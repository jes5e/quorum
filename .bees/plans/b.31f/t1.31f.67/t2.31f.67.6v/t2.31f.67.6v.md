## Context

Epic `t1.31f.67` (parent Bee `b.31f`) verifies that `/bees-plan-from-specs` continues to work unchanged for the file-based PRD/SDD path after Epics 1-6 land. Specifically: the file-resolver code path in PM and breakdown two-hop logic must fall through cleanly, the Scoped-marker machinery must remain valid for `--feature` mode, and the skill prose must clearly position itself as the "I have docs on disk already" path while the redesigned `/bees-plan` (sibling Epic `t1.31f.y2`) is the "let's write specs as tickets" path.

This Task delivers the verification + prose-clarification work in two parts: (1) static verification by reading the affected skills' prose (after sibling Epics' implementations land) and confirming the file-resolver path continues to behave correctly, and (2) a small skill-prose update to `skills/bees-plan-from-specs/SKILL.md` Overview clarifying positioning relative to the redesigned `/bees-plan`.

This Task is a regression-protection Epic. The risk being mitigated: Epics 4-6 introduce the `bees`-resolver path; if PM (sibling Epic `t1.31f.4u`'s `agents/pm.md`) or breakdown (sibling Epic `t1.31f.4u`'s `skills/bees-breakdown-epic/SKILL.md`) two-hop logic is implemented in a way that breaks the `file-path` branch, `/bees-plan-from-specs`-created Plan Bees would silently fail or use wrong spec content.

## Why

`/bees-plan-from-specs` continues to be the right entry point for users who have hand-authored or externally-tooled PRD/SDD on disk. The `bees`-resolver path introduced in sibling Epics is the new default for fresh planning via `/bees-plan`, but the file-resolver path remains a first-class flow. Without verification, a regression in PM or breakdown's two-hop logic could silently break the file-resolver path. Without the prose update, users would be confused about which entry point to choose.
