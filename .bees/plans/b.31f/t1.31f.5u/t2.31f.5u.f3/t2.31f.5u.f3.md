## Context

Epic `t1.31f.5u` (parent Bee `b.31f`) introduces a new `/bees-write-sdd` skill that authors or revises an SDD ticket as a `t1=Doc` child of a Spec Bee. Mirror of sibling Epic `t1.31f.by` (`/bees-write-prd`), but for Software Design Document content rather than Product Requirements. Composable: invokable solo (`/bees-write-sdd <spec-bee-id>`) for spec revisions, and invokable inline from the redesigned `/bees-plan` (sibling Epic `t1.31f.y2`) via the Skill tool.

**SRD → SDD rename.** Apiary's source skill is `skills/write-srd/skill.md`. We rename SRD → SDD throughout the new skill for consistency with this repo's existing Documentation Locations contract key (`Internal architecture docs (SDD)` in CLAUDE.md). Engineer must apply this rename at every reference point — frontmatter description, Overview prose, body section names, helper-script names if any, and the resulting child-ticket title (which must be exactly `SDD`, not `SRD`).

The Specs hive contract values fixed by sibling Epic `t1.31f.ho`'s Subtasks are consumed by this Task verbatim:

- Hive normalized name: `specs` (lowercase) when targeting `bees create-ticket --hive specs`.
- Tier name for the SDD ticket: `t1`.
- Title for the SDD ticket: exactly `SDD` (case-sensitive — sibling Epic `t1.31f.4u`'s PM two-hop lookup uses exact-match title to differentiate PRD from SDD).
- Status values: `drafted` (during authoring), `ready` (when authoring completes); no other statuses.

Sibling Epic `t1.31f.by` (`/bees-write-prd`) is the PRD-side analog. Both skills should share structural conventions but each owns its own SKILL.md. Sibling Epic `t1.31f.y2` (`/bees-plan` redesign) is the inline caller. README skill-catalog update for `/bees-write-sdd` is sibling Task 2 of this Epic (separately tracked) — not in this Task's scope.

## What Needs to Change

`skills/bees-write-sdd/SKILL.md` is the single new file. The skill must implement:

1. **Skill scaffolding.** YAML frontmatter (`name: bees-write-sdd`, specific `description`, `argument-hint: "<spec-bee-id>"`). The description must mention the Specs hive, the SDD-ticket-as-`t1=Doc`-child shape, and that the skill is composable (solo for revisions; inline from `/bees-plan` via the Skill tool).

2. **Required sections in the SDD ticket body.** Adapt apiary's `skills/write-srd/skill.md` required scaffolding (renamed SRD → SDD throughout). The required sections in this order:

   - Codebase exploration findings (architecture, affected modules, existing patterns, data models, test fixtures, configuration — apiary's required scaffolding is right; keep it)
   - Numbered SR-style requirement groups, organized by domain/concern
   - Test Fixtures section (apiary requires this — we keep it; prevents hardcoded values from sneaking past test-helper conventions)
   - Existing Behavior (what must NOT change)
   - Documentation (what docs must be written after implementation)
   - **Background and rationale** (NEW — same reasoning as sibling Epic `t1.31f.by`'s NEW sections)
   - **Decisions and rejected alternatives** (NEW — captures architectural decisions and trade-offs considered)

   The two NEW sections must always be present in the SDD body — filled with substance or explicitly marked empty, but never silently omitted (same disambiguation-rationale logic as sibling `.by`).

3. **Codebase research via Explore agent.** Apiary's `/write-srd` uses the Explore subagent for codebase research; this skill must do the same. The Explore agent (`subagent_type=Explore`) is available in this Claude Code session per the existing skill ecosystem (used by `/bees-plan` etc.). Use it for the "codebase exploration findings" section's content gathering. Apiary's "RESEARCH NEEDED" flag pattern for ambiguous spots is a useful precedent — carry it over so SDD authors can mark spots that need further investigation rather than fabricate guesses.

4. **Mid-conversation context awareness.** Same as sibling Epic `t1.31f.by` — detect rich preceding context and distill into the required sections rather than restart discovery. Detection heuristic and "err toward distill" principle apply identically. Cross-reference `t1.31f.by`'s SKILL.md prose to ensure the two skills' detection language aligns rather than diverges.

5. **Solo invocation.** `/bees-write-sdd <spec-bee-id>` invoked directly creates the SDD child if absent (`bees create-ticket --hive specs --ticket-type t1 --parent <spec-bee-id> --title SDD --body-file <path> --status drafted`) or updates it if present (`bees update-ticket --ids <sdd-id> --body-file <path>`). Default to full-rewrite for body coherence; `bees append-ticket-body` available but not default. Idempotent — re-running on the same Spec Bee does NOT create a duplicate.

6. **Inline invocation via Skill tool.** Sibling Epic `t1.31f.y2`'s redesigned `/bees-plan` will invoke this skill, passing the Spec Bee ID and conversation-distilled scope. Define a clean handoff contract — input shape, output shape — symmetric to sibling `.by`'s contract. Document it in the SKILL.md.

7. **Quality bar.** Adapt apiary's quality guidance from `write-srd/skill.md`: cite real module names from the actual codebase (no fabricated names); include test-fixture conventions explicitly so engineers can't hardcode magic numbers; keep architecture decisions and trade-offs traceable.

8. **Skill-prose constraints (this repo's design rules).** Project-neutral, paired POSIX bash + Windows PowerShell snippets, no bash-only fallback, no hardcoded this-repo paths/IDs. Scratch-file convention honored: `<tempdir>/.bees-workflow/`, `mkdir -p` / `New-Item -ItemType Directory -Force` create-if-absent, never `rm` after.

9. **Bash etiquette.** Single literal command per Bash call; no compound chains, pipes, redirects, shell-variable expansion. Sequential commands as separate Bash blocks. Multi-step variable-bearing logic via Python helper script if needed.

10. **SRD → SDD rename note.** Add a one-line note in the skill's prose (or a comment near the SRD-derived section names) for future readers of apiary docs explaining that this skill renamed SRD → SDD for our Documentation Locations contract-key parity.

## Why

Same reasoning as sibling Epic `t1.31f.by`'s "Why": today's `/bees-plan` Step 4 mutates `docs/sdd.md` at plan time. The redesign moves SDD content into a ticketed child of a Spec Bee, decoupling spec authoring from doc mutation. The `/bees-write-sdd` skill is the SDD-side authoring surface that sibling Epic `t1.31f.y2` (`/bees-plan` redesign) delegates to — without this skill, `.y2` cannot produce the SDD child of the Spec Bee.

The factored-out-skill approach (rather than inlining into `/bees-plan`) is the same decision documented in the parent Bee body's "Decisions and rejected alternatives": composable, re-authorable without re-planning, single-place quality bar, mental-model parity with apiary.
