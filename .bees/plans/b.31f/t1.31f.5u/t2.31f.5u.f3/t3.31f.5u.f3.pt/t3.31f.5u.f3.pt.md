## Context

The sibling Subtask lands the scaffold of `skills/bees-write-sdd/SKILL.md` and the full solo-invocation flow against an existing Spec Bee, including Explore-agent codebase research, the seven required SDD sections (`Codebase exploration findings`, `Requirements`, `Test Fixtures`, `Existing Behavior`, `Documentation`, `Background and rationale`, `Decisions and rejected alternatives`), the `RESEARCH NEEDED` ambiguity-flag pattern, and idempotent create-or-update branching. This Subtask layers on the two remaining behaviors required by the parent Task (`t2.31f.5u.f3`):

1. **Mid-conversation context awareness** (parent Task body item 4). When the skill is invoked from a substantive preceding conversation — either as a delegated sub-skill from sibling Epic `t1.31f.y2`'s redesigned `/bees-plan`, or directly mid-discussion when the user has been planning a feature with the assistant — the skill must distill that conversation into the seven required SDD sections rather than restart with apiary-style discovery. The detection heuristic and the "distill, propose draft for review" branch belong to this Subtask.

2. **Inline-invocation contract via the Skill tool** (parent Task body item 6). Sibling Epic `t1.31f.y2`'s redesigned `/bees-plan` will invoke this skill via the Skill tool, passing the Spec Bee ID it just created plus the conversation-distilled scope information. This Subtask defines and documents the clean handoff contract — the input shape `.y2` passes, the output shape this skill produces — symmetric to sibling `.by`'s contract (defined in `t3.31f.by.7h.xa`). Per parent Task body cross-Task contracts section, the contract should mirror sibling `.by`'s for consistency so `.y2` can dispatch both writers with the same shape.

This Subtask does not modify the solo-invocation flow the sibling Subtask lands; it adds branching logic before discovery (to detect rich prior context) and adds a dedicated section in the SKILL.md documenting the Skill-tool handoff contract. After this Subtask, the parent Task's full acceptance set is met (modulo the Specs hive existing in the target repo, which sibling Epic `t1.31f.ho`'s Subtask provides, and the inline path being exercised end-to-end, which sibling Epic `t1.31f.y2` provides).

## What Needs to Change

Edit `skills/bees-write-sdd/SKILL.md` (created by the sibling Subtask) to add two new behaviors. Existing prose stays; new prose layers on top.

1. **Mid-conversation detection branch.** Insert a step at the start of the workflow (before the discovery step the sibling Subtask lands) that branches on whether the prompt window contains substantive prior context about the feature.

   - **Detection heuristic (skill author's call on exact wording).** The heuristic must tilt toward distilling rather than restarting when the choice is ambiguous (per the parent Task body item 4 explicit guidance: "err toward distill"). Indicative signals to enumerate in the prose: rich back-and-forth in the same session about the feature's design, architectural constraints, or alternatives; a `Description` argument passed to the skill that itself contains substantive scope information beyond a one-line title; an explicit hint from the Skill-tool caller (sibling `.y2`) that distilled scope is being passed in.
   - **Cross-reference sibling `/bees-write-prd`.** Read the mid-conversation detection prose in `skills/bees-write-prd/SKILL.md` (landed by sibling Epic `t1.31f.by`'s Subtask `t3.31f.by.7h.xa`) and align this skill's detection language with it — same indicative signals, same err-toward-distill principle, same finite-choice approval gate. Divergence between the two skills' detection language is itself a defect: when `/bees-plan` invokes both writers in sequence on the same Spec Bee, both should make the same distill-vs-restart decision.
   - **Distill branch.** When the heuristic fires, skip the apiary-style discovery questions the sibling Subtask lands. Instead: read the prior context, distill it into the seven required SDD sections (especially `## Background and rationale`, `## Decisions and rejected alternatives`, the `## Requirements` requirement groups, and the high-level scope of `## Existing Behavior` — these are precisely where prior-conversation richness should land). The Explore agent codebase-research dispatch still runs because the conversation rarely supplies real module names or test-fixture conventions; the distill branch replaces only the user-interview discovery step, not the codebase research. Present the draft to the user for review before the create-or-update branch fires. Use `AskUserQuestion` for the approval gate (finite choices: approve / revise / cancel) per CLAUDE.md `## AskUserQuestion usage`.
   - **Restart branch.** When the heuristic does NOT fire (cold solo invocation), proceed with the sibling Subtask's discovery flow unchanged.
   - **Document the err-toward-distill principle inline** so future maintainers don't tighten the heuristic into a stricter "only fire when X is unambiguously true" gate, which would defeat the design intent.

2. **Inline-invocation contract section.** Replace the placeholder the sibling Subtask left in the Overview's "two invocation paths" listing with a dedicated `## Inline invocation via the Skill tool` section (or equivalently-titled subsection within Workflow) that documents the contract end-to-end:

   - **Input shape (caller → this skill).** Document what sibling Epic `t1.31f.y2`'s `/bees-plan` passes when it invokes this skill via the Skill tool. At minimum: the Spec Bee ID (the parent for the SDD `t1=Doc` child) and the distilled scope content from the planning conversation. The exact shape (free-text args string vs structured JSON-in-prose) is the skill author's call BUT must mirror sibling `/bees-write-prd`'s contract shape (per parent Task body cross-Task contracts section: "Should mirror sibling `.by`'s contract for consistency"). Read `skills/bees-write-prd/SKILL.md`'s contract section first, then mirror its shape here for parity. Whatever shape is chosen, document it as a stable contract — future maintainers of `.y2` rely on it.
   - **Output shape (this skill → caller).** Document what this skill produces and reports back when invoked inline. At minimum: the SDD ticket ID it created or updated, plus the final status (`drafted` vs `ready`). This is what `.y2` needs to confirm both PRD and SDD children are written before transitioning the Spec Bee from `drafted` to `ready`. Mirror sibling `.by`'s output shape for parity.
   - **Behavioral guarantees.** Same idempotency, same seven required sections, same `drafted → ready` lifecycle as solo invocation — the inline path is functionally identical to solo from the Spec Bee's perspective; only the discovery-vs-distillation branch differs (inline always takes the distill branch because `.y2` always passes substantive distilled scope). The Explore agent codebase-research dispatch still runs in the inline path.
   - **Cross-reference to the mid-conversation detection.** Note that inline invocation is the canonical case where the distill branch fires.
   - **Cross-reference sibling `/bees-write-prd`.** Note that the contract shape mirrors `/bees-write-prd`'s deliberately so `/bees-plan` can dispatch both writers with the same shape.

3. **No regression to the sibling Subtask's solo flow.** The solo-invocation workflow the sibling Subtask lands continues to work identically when invoked without substantive prior context. The branching added here is a guard, not a replacement. The Explore agent dispatch, the seven required sections, the `RESEARCH NEEDED` ambiguity-flag pattern, and the SRD → SDD rename note all remain in place.
