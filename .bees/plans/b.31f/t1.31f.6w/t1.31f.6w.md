## Outcome

`agents/doc-writer.md` updated to take responsibility for appending or
updating `### Feature: <title>` subsections in the project's cumulative
PRD/SDD docs (paths from CLAUDE.md `## Documentation Locations`)
post-implementation, reflecting what was actually built. Resolves the
redundancy where today's `/bees-plan` writes feature design speculatively
and doc-writer also writes it post-impl based on the diff.

## Scope

1. **doc-writer responsibility expansion.** Today's doc-writer reviews
   the Engineer's diff and updates customer-facing and internal
   architecture docs based on what was actually built. Expand its
   responsibilities to include:
   - Append a new `### Feature: <title>` subsection to the project's
     cumulative PRD (path from CLAUDE.md `Project requirements doc
     (PRD)`) under the existing `## Per-feature scope` header (or
     create the header if absent).
   - Append a new `### Feature: <title>` subsection to the project's
     cumulative SDD (path from CLAUDE.md `Internal architecture docs
     (SDD)`) under the existing `## Per-feature design` header.
   - Update the project's customer-facing docs (README) for any
     getting-started, configuration, or deployment changes, as today.
   - `### Feature: <title>` content reflects what was actually built —
     drawn from the diff and the implementation, NOT from the Plan
     Bee body or PRD/SDD ticket-children's forward-looking content.

2. **Source of truth for `<title>`.** The Plan Bee's title. The
   doc-writer must read the Plan Bee (via the Grandparent Bee chain
   for execute mode, or via the Plan Bee linked from the Issue's
   `up_dependencies` for fix mode) to obtain the title and use it
   verbatim as the `### Feature: <title>` heading text.

3. **Idempotency.** If a `### Feature: <title>` subsection already
   exists in the cumulative doc (e.g., this is an update to an already-
   shipped feature, or an Epic-by-Epic build where earlier Epics
   already added a stub), the doc-writer must update the existing
   subsection rather than appending a duplicate.

4. **Skip when not applicable.** Some features (pure refactors,
   internal-only infrastructure) don't warrant cumulative-doc entries.
   Apply the same categorization heuristic from today's `/bees-plan`
   Step 4 table (the table itself moves to doc-writer's prose):
   - Internal refactor / pure-tech: no PRD or SDD entry.
   - Architecture-only changes: SDD entry only.
   - User-facing features: both PRD and SDD entries.
   The categorization happens during the doc-writer pass, informed by
   the actual diff and Plan Bee body.

5. **Execute-mode dispatch prompt update.**
   `skills/bees-execute/SKILL.md` Doc Writer dispatch prompt currently
   focuses on "review the Engineer's diff for doc gaps." Extend to
   include the cumulative-PRD/SDD-append responsibility, with a
   reference to the categorization heuristic.

6. **Fix-mode dispatch prompt update.** `skills/bees-fix-issue/SKILL.md`
   Doc Writer dispatch is similar — fixes that change behavior may
   need cumulative-doc updates. Apply the same expansion.

## Acceptance criteria

- A feature executed end-to-end via `/bees-execute` (against a Plan
  Bee created by the new `/bees-plan` flow from Epic 4) results in
  appropriate `### Feature: <title>` subsections appearing in
  `docs/prd.md` and `docs/sdd.md`, written by the doc-writer agent.
- The cumulative-doc subsections describe what was actually built
  (cite real module names, accurate behavior, current API contracts).
  Verifiable by inspection against the diff.
- Re-running an Epic from the same Plan Bee (e.g., a fix-up) updates
  the existing subsection rather than creating a duplicate.
- Pure refactors with no user/system-visible behavior change result in
  NO new cumulative-doc subsection (skip-when-not-applicable branch).
- `/bees-fix-issue` runs that change behavior also produce
  cumulative-doc updates as appropriate.

## Implementation notes

- The categorization table from today's `/bees-plan` Step 4 (Product
  feature / Architecture-reliability / Deployment-infra / CI-testing /

  for cumulative-doc updates. No new tool requirements.
- doc-writer must read CLAUDE.md `## Documentation Locations` to
  resolve the actual PRD and SDD paths (existing behavior; just
  expand the operations performed against those paths).
