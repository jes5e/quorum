## Context

Parent Task `t2.31f.6w.21` expands `agents/doc-writer.md` to take ownership of cumulative project PRD/SDD updates post-implementation. Sibling Epic `t1.31f.y2` is stripping `/bees-plan` Step 4's plan-time doc-mutation entirely; after that lands, cumulative project docs need a new actor to keep them current. The doc-writer is the right actor because it sees the actual diff and runs after implementation.

This Subtask delivers the core responsibility expansion: the cumulative-PRD and cumulative-SDD update behavior with `<title>` resolution, idempotency, and CLAUDE.md-keyed path resolution. The companion Subtask lifts the categorization heuristic that decides which docs (PRD-and-SDD vs SDD-only vs neither) to update.

## What Needs to Change

Modify `agents/doc-writer.md` to add a new section (e.g., `## Cumulative project doc updates` or folded into `## Responsibilities` / `## Instructions` — pick the cleanest fit consistent with the existing structure) documenting the following behaviors. Preserve all existing prose unchanged.

1. **Cumulative PRD update.** After the Engineer's diff lands and the doc-writer is dispatched, the doc-writer appends or updates a `### Feature: <title>` subsection under `## Per-feature scope` in the project's PRD. If `## Per-feature scope` is absent, create it. The PRD path is resolved via CLAUDE.md `## Documentation Locations` `Project requirements doc (PRD)` key — never hardcoded. Subsection content reflects what was ACTUALLY built (drawn from the diff and implementation), NOT forward-looking content from the Plan Bee body or PRD/SDD ticket-children.

2. **Cumulative SDD update.** Same pattern, for the SDD path resolved via `Internal architecture docs (SDD)` key. Append or update `### Feature: <title>` under `## Per-feature design`; create the header if absent. Content reflects actual implementation, not forward-looking spec.

3. **Source of truth for `<title>`.** The Plan Bee's title field, used verbatim as the `### Feature: <title>` heading text. No paraphrasing. Document the traversal recipe for both modes:
   - **Execute mode**: Subtask → Task → Epic → Plan Bee chain, via the parent linkage already exposed by `bees show-ticket`. Read the Plan Bee's title from there.
   - **Fix mode**: traverse the Issue's `up_dependencies` to find a linked Plan Bee, read its title.

   The standalone-Issue case (Issue with no Plan Bee in `up_dependencies`) is a real but edge-case shape this Task does NOT specify a fallback for — the parent Task body and Epic body describe only the linked-Plan-Bee path. If a standalone Issue is encountered at runtime, the doc-writer should surface that as an out-of-spec condition rather than improvise a `<title>`. Handling that case is deferred to a follow-up Issue if it surfaces in practice.

4. **Idempotency.** If a `### Feature: <title>` subsection already exists in the cumulative doc (e.g., this is a refinement of an already-shipped feature, or an Epic-by-Epic build where an earlier Epic added a stub), the doc-writer updates the existing subsection rather than appending a duplicate. Detection rule: search for the literal heading text `### Feature: <title>` (exact match, case-sensitive on `<title>`) and replace the existing subsection's content (everything from the heading up to the next `### ` heading or end-of-file) rather than appending a new section with the same heading.

5. **Path resolution via CLAUDE.md keys.** Document explicitly that PRD path comes from CLAUDE.md `## Documentation Locations` `Project requirements doc (PRD)` and SDD path from `Internal architecture docs (SDD)`. Do NOT hardcode `docs/prd.md` or `docs/sdd.md` — that violates this repo's design rule 1 (project-neutral skill prose). The customer-facing-docs path resolution from `Customer-facing docs` already in the agent's existing prose stays unchanged.

6. **Skill-prose constraints.** Project-neutral; no this-repo paths or Bee IDs hardcoded; no language-specific commands. Any shell snippets ship as paired POSIX bash + Windows PowerShell variants. Note however that `agents/doc-writer.md`'s frontmatter does NOT include `Bash` in its tool allowlist by design — so any traversal logic that would require shell goes through the orchestrating execution skill rather than being executed directly. Document this constraint: the doc-writer reads the Plan Bee title from context already provided by the dispatching skill (or via `Read` against ticket files exposed to the agent) rather than calling `bees show-ticket` itself.

7. **Preserve existing behavior.** Do not modify the agent's existing responsibilities (mode divergence prose, model default and runtime override, customer-facing docs review, status-transition handoff prose, no-Bash design rationale). Only ADD the cumulative-PRD/SDD-update responsibility.

Defer the categorization heuristic (which features get PRD entries, which get SDD entries, which get neither) to the companion Subtask. For this Subtask, document the cumulative-update mechanism assuming the categorization decision has been made; the categorization Subtask will land the heuristic that drives the decision.
