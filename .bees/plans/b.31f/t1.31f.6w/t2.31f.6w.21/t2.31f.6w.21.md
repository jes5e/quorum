## Context

Epic `t1.31f.6w` (parent Bee `b.31f`) expands the `doc-writer` agent's responsibilities to take ownership of cumulative project PRD/SDD updates post-implementation. With sibling Epic `t1.31f.y2`'s `/bees-plan` redesign no longer pre-writing `### Feature: <title>` subsections at plan time, the cumulative project docs need to be updated *after* implementation lands by an actor that can see what was actually built. The `doc-writer` agent is that actor.

This Task delivers the canonical contract change: expand `agents/doc-writer.md` to include the cumulative-doc-update responsibility. Sibling Task 2 of this Epic separately updates the Doc Writer dispatch prompts in `skills/bees-execute/SKILL.md` and `skills/bees-fix-issue/SKILL.md` to acknowledge the new responsibility.

## What Needs to Change

`agents/doc-writer.md` is the single file modified by this Task. The existing agent reviews the Engineer's diff and updates customer-facing and internal architecture docs based on what was actually built. Expand its responsibilities to include:

1. **Cumulative PRD update.** After the Engineer's diff lands and the doc-writer is dispatched (in execute mode or fix mode), the doc-writer must:
   - Read the Plan Bee title (the source of truth for the `<title>` in `### Feature: <title>`).
   - Append a new `### Feature: <title>` subsection to the project's cumulative PRD (path resolved from CLAUDE.md `## Documentation Locations` `Project requirements doc (PRD)` key) under the existing `## Per-feature scope` header. If the header is absent, create it.
   - The subsection content reflects what was ACTUALLY built — drawn from the diff and the implementation, NOT from the Plan Bee body or PRD/SDD ticket-children's forward-looking content.

2. **Cumulative SDD update.** Same pattern, for the Internal architecture docs (SDD) path:
   - Append a new `### Feature: <title>` subsection under the existing `## Per-feature design` header (create if absent).
   - Content reflects actual implementation, not forward-looking spec.

3. **Customer-facing docs (README) update.** As today's doc-writer already does — update README for any getting-started, configuration, or deployment changes the diff introduces.

4. **Source of truth for `<title>`.** The Plan Bee's title field. The doc-writer must read the Plan Bee:
   - **Execute mode**: traverse the Grandparent Bee chain (Subtask → Task → Epic → Plan Bee) to find the Plan Bee, read its title.
   - **Fix mode**: traverse the Issue's `up_dependencies` to find a linked Plan Bee, read its title.
   - Use the title verbatim as the `### Feature: <title>` heading text. No paraphrasing.

5. **Idempotency.** If a `### Feature: <title>` subsection already exists in the cumulative doc (e.g., this is an update to an already-shipped feature, or an Epic-by-Epic build where earlier Epics already added a stub), the doc-writer must update the existing subsection rather than appending a duplicate. Detection: search for the literal heading text `### Feature: <title>` (exact match, case-sensitive on the title) and replace the existing subsection's content rather than appending a new section with the same heading.

6. **Skip when not applicable.** Some features don't warrant cumulative-doc entries:
   - Pure internal refactors (no user-visible or system-visible behavior change): no PRD or SDD entry.
   - Architecture-only changes (no user-facing surface): SDD entry only.
   - User-facing features: both PRD and SDD entries.

   The categorization heuristic comes from the table that today's `/bees-plan` Step 4 carries (sibling Epic `t1.31f.y2`'s strip Subtask `t3.31f.y2.9q.42` removes that table from `/bees-plan`'s prose). Lift the table into `agents/doc-writer.md` — same categories, different actor and timing. Specifically, document these categories:
   - Internal refactor / pure-tech: skip both PRD and SDD.
   - Architecture-only changes: SDD entry only.
   - Deployment / infrastructure / CI / testing changes: SDD entry only (unless customer-visible behavior changes too).
   - User-facing features: both PRD and SDD entries.

   The categorization happens during the doc-writer pass, informed by the actual diff and the Plan Bee body's "What" / "Why" / "Acceptance criteria" sections.

7. **Skill-prose constraints (custom-subagent contract — same rules as SKILL.md).** Project-neutral; paired POSIX bash + Windows PowerShell snippets where any new shell commands are shown; single-literal-command Bash etiquette; no hardcoded this-repo paths or ticket IDs. The doc-writer agent has Read, Edit, Write, Grep tools per its existing frontmatter — sufficient for cumulative-doc updates. No new tool requirements.

8. **Path resolution.** The doc-writer must resolve PRD and SDD paths via CLAUDE.md `## Documentation Locations` lookup keys (`Project requirements doc (PRD)`, `Internal architecture docs (SDD)`, `Customer-facing docs`). Do NOT hardcode `docs/prd.md` or `docs/sdd.md` — that is a design rule 1 violation in skill prose.

9. **Preserve existing behavior.** The doc-writer's other responsibilities (review the Engineer's diff for doc gaps, update customer-facing docs for getting-started/config/deployment changes, etc.) are not modified by this Task — only the cumulative-PRD/SDD-update responsibility is added.

## Why

Today's `/bees-plan` Step 4 mutates `docs/prd.md` and `docs/sdd.md` at plan time, which sibling Epic `t1.31f.y2` strips out. After `.y2` lands, cumulative project docs would never get updated unless someone takes over the responsibility. The post-implementation doc-writer pass is the right actor for that work because: (a) it sees the actual diff (no speculation), (b) it can categorize accurately based on observable change (not on plan-time intent), and (c) it idempotently handles re-runs (e.g., Epic-by-Epic builds where earlier Epics added a stub the later Epic refines).

The categorization heuristic moves to `agents/doc-writer.md` because the heuristic is fundamentally about classifying real change — that classification is a doc-writer concern, not a plan-time concern.
