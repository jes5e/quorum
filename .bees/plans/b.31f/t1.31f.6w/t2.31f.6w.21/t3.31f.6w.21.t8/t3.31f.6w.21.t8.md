## What Needs to Change

Modify `agents/doc-writer.md` to add categorization-heuristic prose alongside the cumulative-doc-update behavior added by the companion Subtask. Recommended placement: a subsection within or adjacent to the cumulative-PRD/SDD-update section, so the categorization decision feeds directly into the update mechanism.

Document the following four categories (collapsing /quo-plan's 5 rows into 4 — Architecture/reliability and Deployment/infra share the SDD-only outcome and may merge into a single row for clarity, or stay separate; pick whichever reads cleanest):

1. **Internal refactor / pure-tech** (no user-visible or system-visible behavior change): skip both PRD and SDD entries. Examples — code dedup, helper extraction, internal renames.

2. **Architecture-only changes** (perf, caching, retry logic, internal structure changes, dependency upgrades that affect architecture but not user-facing surface): SDD entry only. No PRD entry.

3. **Deployment / infrastructure / CI / testing changes** (Helm, CI workflows, monitoring, smoke tests, test helpers): SDD entry only — UNLESS the change also affects customer-visible behavior (e.g., a new Helm chart that changes setup instructions a user follows), in which case both PRD and SDD entries. The customer-facing-docs (README) update path the doc-writer already handles is independent of this categorization — README updates are driven by getting-started/configuration/deployment changes the diff introduces, regardless of whether PRD or SDD entries are warranted.

4. **User-facing features** (new endpoints, behavior changes, new user-visible capabilities): both PRD and SDD entries.

Present the categorization as a markdown table (mirroring the source table in `/quo-plan` Step 4 for visual consistency) plus a paragraph or two of guidance on how to apply it. The guidance should call out:

- **Inputs the doc-writer uses for categorization.** The actual diff (primary signal — what surface area the change touches), the Plan Bee body's `## What Needs to Change` / `## Why` / `## Acceptance Criteria` sections (secondary signal — confirms intent matches observed change), and the file paths touched (heuristic — changes confined to internal modules without entry-point or contract surface usually mean refactor; changes to public APIs, CLI surfaces, configuration schemas usually mean user-facing).

- **Edge cases.**
   - A change that BOTH refactors internal structure AND adds user-facing behavior is categorized by the user-facing dimension (both PRD and SDD entries).
   - A change that adds a new architectural capability USED by the same change to deliver user-facing behavior is also categorized by the user-facing dimension.
   - When in doubt between two adjacent rows (e.g., architecture-only vs deployment), pick the broader-scope row — over-documenting in cumulative docs is recoverable; under-documenting requires re-running the doc-writer pass.

- **Relationship to the customer-facing docs (README) responsibility.** README updates remain driven by the existing diff-review pass for getting-started / configuration / deployment changes, regardless of which categorization row the change lands in. The categorization heuristic governs cumulative PRD/SDD entries only.
