## Outcome

`agents/pm.md` and `skills/quo-breakdown-epic/SKILL.md` updated to
perform two-hop lookup: read `reference_materials`, follow the `bees`
resolver to a Spec Bee, walk the Spec Bee's children for PRD and SDD
content. Existing `file-path` resolver path and body-as-spec fallback
remain functional.

## Scope

1. **PM agent (`agents/pm.md`) two-hop lookup.** Today's PM reads
   spec sources from `reference_materials` directly (via the
   `file-path` resolver) and falls back to the Plan Bee body. Update
   to add the `bees`-resolver branch:
   - Read `reference_materials`. For each entry:
     - If `resolver` is `file-path` (or omitted, i.e. default), read
       the file at the given path. Existing behavior.
     - If `resolver` is `bees`, the value is a ticket ID. Show the
       Spec Bee, enumerate its `t1=Doc` children, and read each
       child ticket's body. Identify PRD vs SDD by ticket title
       (`PRD` and `SDD` respectively, per Epics 2 and 3).
   - Fall back to Plan Bee body if `reference_materials` is empty
     (existing behavior; this Plan Bee itself uses this fallback).

2. **`/quo-breakdown-epic` two-hop lookup.** Same change to
   `skills/quo-breakdown-epic/SKILL.md`. The skill's PM dispatch
   prompt and its own Spec Traceability Review step both read
   `reference_materials`; both need the `bees`-resolver branch added.

3. **Scoped-marker integration with the bees resolver.** The existing
   Scoped-marker machinery (`Scoped to '### Feature: <title>' ...`
   line in a Plan Bee body, parsed by
   `skills/quo-breakdown-epic/scripts/scoped_marker_resolver.py`)
   was designed for cumulative file-based PRD/SDD. With Specs Bees,
   each feature has its own Spec Bee — no cumulative file, no need
   for marker-based scoping on the `bees` resolver path.
   - When the resolver is `bees`, skip Scoped-marker resolution
     entirely (the Spec Bee IS already feature-scoped).
   - When the resolver is `file-path`, run Scoped-marker resolution
     as today (preserves `/quo-plan-from-specs --feature` flow).

4. **Execution-skill PM dispatch prompts.** `skills/quo-execute/SKILL.md`
   and `skills/quo-fix-issue/SKILL.md` both dispatch the PM agent
   with prompts that mention "PRD/SDD linked from the Bee's
   `reference_materials`." Update those prompts to acknowledge the
   `bees` resolver path so the PM agent doesn't get confused. The
   actual lookup logic lives in `agents/pm.md` (the agent does the
   work); the dispatch prompts just need to not contradict.

5. **Documentation updates.** `docs/doc-writing-guide.md` `Querying
   tickets` section and `Reference materials` definition need to
   document the `bees` resolver usage and the two-hop lookup pattern.

## Acceptance criteria

- A PM agent dispatched against a Plan Bee whose `reference_materials`
  is `[{"value":"<spec-bee-id>","resolver":"bees"}]` correctly
  retrieves both PRD and SDD child-ticket bodies and uses them as the
  spec source for spec-traceability and drift checks. Verifiable by
  running `/quo-execute` against a Plan Bee created via the new
  `/quo-plan` (Epic 4) flow and checking the PM's report cites
  spec content.
- The same PM agent dispatched against a Plan Bee whose
  `reference_materials` uses the `file-path` resolver (legacy or
  `/quo-plan-from-specs`-created) still works as today, including
  Scoped-marker-narrowed scoping.
- A PM agent dispatched against a Plan Bee with empty
  `reference_materials` falls back to the Plan Bee body (existing
  body-as-spec behavior).
- `/quo-breakdown-epic` Spec Traceability Review correctly cites
  PRD §X and SDD §Y when the resolver is `bees`, with the §
  references resolving against the child ticket bodies.

## Implementation notes

- The `bees show-ticket --ids <spec-bee-id>` output includes a
  `children` array with child ticket IDs. Use it to enumerate, then
  `bees show-ticket --ids <child-id>` (or batch) to fetch bodies.
- Title-based PRD/SDD differentiation: the writer skills (Epics 2/3)
  always use exactly `PRD` and `SDD` as titles, so case-sensitive
  exact-match is sufficient.
- The `bees execute-freeform-query` recipes in `docs/doc-writing-guide.md`
  may be useful for one-call enumerate-and-fetch patterns.
- Don't break the asymmetric Path A vs Path B handling in PM's
  Scoped-marker integration — it's load-bearing for the file-resolver
  path. Add a third pre-branch: "if resolver is `bees`, skip marker
  resolution entirely."
