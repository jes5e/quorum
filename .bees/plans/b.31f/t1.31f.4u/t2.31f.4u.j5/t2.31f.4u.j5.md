## Context

Epic `t1.31f.4u` (parent Bee `b.31f`) updates the PM agent and breakdown skill to perform two-hop `reference_materials` lookup: read `reference_materials`, follow the `bees` resolver to a Spec Bee, walk the Spec Bee's children for PRD and SDD content. This Task delivers the PM-agent half of the change — `agents/pm.md` is updated to add the `bees`-resolver branch in its spec-source resolution logic. The existing `file-path` resolver path and the body-as-spec fallback both remain functional.

Sibling Tasks under this Epic:
- **Task 2** of this Epic (`t2.31f.4u.<id>`) — `skills/bees-breakdown-epic/SKILL.md` two-hop lookup (the same change applied to the breakdown skill's PM dispatch prompt and Spec Traceability Review).
- **Task 3** of this Epic — execution-skill PM dispatch prompt updates (in `skills/bees-execute/SKILL.md` and `skills/bees-fix-issue/SKILL.md`) plus `docs/doc-writing-guide.md` updates documenting the `bees` resolver and the two-hop lookup pattern.

This Task focuses solely on `agents/pm.md`'s two-hop lookup logic.

## What Needs to Change

`agents/pm.md` is the file modified by this Task. Today's PM agent reads spec sources from `reference_materials` directly (via the `file-path` resolver) and falls back to the Plan Bee body when `reference_materials` is empty. Update to add a `bees`-resolver branch in the `reference_materials` resolution logic.

1. **Add `bees`-resolver branch.** When parsing `reference_materials`, for each entry:
   - If `resolver` is `file-path` (or omitted, i.e. default), read the file at the given path. **Existing behavior unchanged.**
   - **NEW**: If `resolver` is `bees`, the value is a ticket ID. Show the Spec Bee via `bees show-ticket --ids <spec-bee-id>`, enumerate its `t1=Doc` children (read from the `children` array in the response), and read each child ticket's body via `bees show-ticket --ids <child-id>`. Identify PRD vs SDD by ticket title (exactly `PRD` and exactly `SDD` respectively, case-sensitive — established by sibling Epic `t1.31f.by`'s `t3.31f.by.7h.ba` and sibling Epic `t1.31f.5u`'s `t3.31f.5u.f3.9i`).
   - Fall back to Plan Bee body if `reference_materials` is empty (existing body-as-spec behavior). The parent Bee `b.31f` itself uses this fallback (it's in bootstrap mode); preserve compatibility.

2. **Title-based PRD/SDD differentiation.** Use exact-match (case-sensitive) on the title field: `title == "PRD"` for PRD content, `title == "SDD"` for SDD content. Per the cross-Task contracts established in sibling Epics `t1.31f.by` and `t1.31f.5u`, the writer skills always use exactly `PRD` and `SDD` as titles, so case-sensitive exact-match is sufficient and unambiguous.

3. **Scoped-marker integration on `bees`-resolver path.** The existing Scoped-marker machinery (`Scoped to '### Feature: <title>' from <prd-path> and <sdd-path>.` line in a Plan Bee body, parsed by `skills/bees-breakdown-epic/scripts/scoped_marker_resolver.py`) was designed for cumulative file-based PRD/SDD. With Spec Bees, each feature has its own Spec Bee — no cumulative file, no need for marker-based scoping on the `bees` resolver path.
   - **When the resolver is `bees`, skip Scoped-marker resolution entirely.** The Spec Bee IS already feature-scoped (one Spec Bee per feature), so there is no marker-based subsection narrowing needed.
   - **When the resolver is `file-path`, run Scoped-marker resolution as today.** Preserves the `/bees-plan-from-specs --feature` cumulative-doc flow that sibling Epic `t1.31f.67` regression-tests.
   - Today's `agents/pm.md` has asymmetric Path A vs Path B handling for the Scoped-marker. Add a third pre-branch: "if resolver is `bees`, skip marker resolution entirely." Do NOT break the existing Path A vs Path B handling for the file-resolver path — it is load-bearing for the file-resolver flow.

4. **Cross-reference: enumeration helper.** The bees CLI's `bees execute-freeform-query --query-yaml '<yaml>'` is the canonical way to enumerate Spec Bee children programmatically. The simpler `bees show-ticket --ids <spec-bee-id>` returns a `children` array directly (an array of child ticket IDs), which is sufficient for the two-hop lookup. The PM agent prose can use either approach; pick the simpler `show-ticket --ids` + read-each-child approach unless there's a reason to use the freeform query (e.g., title-filtering at query time). Document the chosen approach inline.

5. **Preserve existing behavior.** The PM agent's other responsibilities (per-Task scope review, Epic-wide spec traceability, drift checks) are not modified by this Task. Only the `reference_materials` resolution / spec-source-acquisition logic is extended.

6. **Skill-prose constraints.** This file (`agents/pm.md`) is a custom-subagent contract (markdown with YAML frontmatter), not a SKILL.md, but the same design rules apply: project-neutral prose, paired POSIX bash + Windows PowerShell snippets where any new shell commands are shown, single-literal-command Bash etiquette, no hardcoded this-repo paths or ticket IDs.

## Why

Sibling Epic `t1.31f.y2`'s `/bees-plan` redesign emits Plan Bees whose `reference_materials` is `[{"value":"<spec-bee-id>","resolver":"bees"}]`. The PM agent dispatched against such a Plan Bee must resolve that pointer to read the actual PRD and SDD content (now stored as ticket bodies, not file content). Without this Task's update, the PM agent would either fail to find the spec content or fall back to body-as-spec mode, missing the rich PRD/SDD content that the Spec Bee's children carry.

The asymmetric branching (skip marker resolution on `bees`, run it on `file-path`) reflects a fundamental difference: Spec Bees are per-feature (no cumulative co-mingling, so no marker needed), while cumulative file-based PRD/SDD on disk needs the marker for feature-subsection scoping. Both modes coexist after this Epic lands; the PM agent must handle both correctly.
