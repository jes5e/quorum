## Context

Parent Task `t2.31f.4u.j5` (Epic `t1.31f.4u`, parent Bee `b.31f`) extends the PM agent's `reference_materials` resolution logic to support a `bees` resolver that points at a Spec Bee in the `specs` hive. The Spec Bee carries `t1=Doc` children titled exactly `PRD` and `SDD` (case-sensitive); the PM agent must walk that two-hop path to read PRD/SDD content for spec-traceability and drift checks.

Today's `agents/pm.md` resolves spec content one of two ways:
1. From the Grandparent Bee's `reference_materials` array (treating each entry's `value` as a file path on disk — the implicit `file-path` resolver).
2. From the Grandparent Bee body itself when `reference_materials` is null/empty (body-as-spec fallback). The parent Bee `b.31f` itself uses this mode (bootstrap), and that compatibility must be preserved.

The Scoped-marker machinery in the existing file is asymmetric across two execution-skill dispatch paths (Path A under `/bees-execute` hard-fails on helper exit 2; Path B under `/bees-fix-issue` falls back best-effort). That asymmetry is explicitly load-bearing per the existing `## Asymmetric error-handling` section and must remain unchanged for the file-resolver path. Sibling Tasks under this Epic make the matching change in `skills/bees-breakdown-epic/SKILL.md` (Task 2) and in the execution-skill PM dispatch prompts plus `docs/doc-writing-guide.md` (Task 3); this Subtask is scoped strictly to `agents/pm.md`.

## What Needs to Change

Modify `agents/pm.md` to add the `bees`-resolver two-hop branch and the Scoped-marker skip-on-bees rule, while preserving every existing behavior on the file-resolver path and on the body-as-spec fallback.

1. **Add `bees`-resolver branch in the spec-source resolution prose.** When iterating entries in the Grandparent Bee's `reference_materials`, dispatch on the entry's `resolver` field:
   - `resolver` is `file-path` (or omitted, default) — read the file at `value`. Existing behavior, unchanged.
   - `resolver` is `bees` — treat `value` as a Spec Bee ID. Run `bees show-ticket --ids <spec-bee-id>` and read the `children` array from the response (the Spec Bee's `t1=Doc` children). For each child ID, run `bees show-ticket --ids <child-id>` and read its `title` and `body`. Match `title == "PRD"` (exact, case-sensitive) for PRD content and `title == "SDD"` (exact, case-sensitive) for SDD content. Use those bodies as the spec source in place of file content.
   - `reference_materials` is null/empty — body-as-spec fallback (existing behavior, unchanged). The parent Bee `b.31f` itself relies on this and must remain compatible.

2. **Title-match contract.** Document inline that PRD vs SDD differentiation is by exact-match (case-sensitive) on the child ticket's `title` field — `PRD` and `SDD` literally. Reference that this is a cross-Epic contract established by sibling Epics (PRD title and SDD title), without baking specific ticket IDs into the prose.

3. **Scoped-marker skip-on-bees rule.** Add a pre-branch at the top of the `## Spec-source scoping (Scoped-marker integration)` section: when the entry that produced the spec source had `resolver == "bees"`, skip Scoped-marker resolution entirely (do not write a temp file, do not invoke the helper, do not parse exit codes). Spec Bees are already feature-scoped (one Spec Bee per feature), so marker-based subsection narrowing is irrelevant on that path. The existing Path A vs Path B asymmetric handling (and the load-bearing `## Asymmetric error-handling` section that documents the intent) MUST remain unchanged for the file-resolver path — only the new pre-branch short-circuits the marker work for the `bees` path.

4. **Enumeration approach.** Use the simpler `bees show-ticket --ids <spec-bee-id>` plus per-child `bees show-ticket --ids <child-id>` recipe, since `show-ticket` returns a `children` array directly. The freeform-query route (`bees execute-freeform-query --query-yaml '<yaml>'`) is also acceptable but adds complexity without clear benefit for this two-hop walk; pick one and document it inline. Cross-reference `docs/doc-writing-guide.md` `## Querying tickets` for callers who want title-filtered enumeration.

5. **Preserve all other PM-agent responsibilities verbatim.** The `## Responsibilities`, `## Model default and runtime override`, in-flight `/bees-code-review` and `/bees-doc-review` orchestration prose, time-budget short-circuit, cross-Task / cross-Epic interaction-check section, final report contract, and shell-command etiquette section are NOT modified by this Subtask.

6. **Skill-prose constraints (custom-subagent contract — same rules as SKILL.md).** Project-neutral prose; no hardcoded this-repo Bee IDs (e.g., do not bake `b.31f` or any `t1.31f.*` ID into `agents/pm.md`). Any new shell snippets ship as paired POSIX bash + Windows PowerShell variants. Every Bash tool call shown is a single literal command — no compound chains, pipes between commands, redirects, diagnostic tails, `$VAR` / `$(...)` / backticks, or multi-line `-c` heredocs. Where multi-step logic or variable-bearing flow is needed, instruct the caller to use the `Write` tool to author a Python helper file and run it with one Bash call.
