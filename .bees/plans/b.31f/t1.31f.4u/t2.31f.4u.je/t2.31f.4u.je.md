## What Needs to Change

`skills/quo-breakdown-epic/SKILL.md` is the file modified by this Task. There are two distinct touch-points within the skill:

1. **PM dispatch prompts.** The breakdown skill dispatches a PM agent at two boundaries: per-Task PM review (Section 4's `#### Per-Task PM dispatch`) and Epic-wide Spec Traceability Review (Section 5). Both dispatch prompts must reference the `bees`-resolver branch added in sibling Task 1 (`agents/pm.md`). The PM agent's actual lookup logic lives in `agents/pm.md`; this skill's dispatch prompts only need to NOT contradict the new logic. Specifically:
   - If the dispatch prompt mentions "PRD/SDD linked from the Bee's `reference_materials`," update to acknowledge the `bees` resolver path: spec content may come from file-resolver paths OR from Spec Bee children via the `bees` resolver, depending on what the Plan Bee's `reference_materials` carries.
   - The dispatch prompt should not duplicate the lookup logic — refer to `agents/pm.md` as the authoritative spec for resolution.

2. **Skill's own Spec Traceability Review (Section 5).** The breakdown skill performs its own spec traceability review (the `Spec Traceability Review` prose embedded in the PM dispatch prompt for the Epic-wide review). The Source column in the traceability table must accommodate both file-resolver citations (`PRD §X / SDD §Y`) and `bees`-resolver citations. For the `bees`-resolver path, the table should cite Spec Bee child ticket IDs and section/bullet references within those tickets, e.g., `<prd-ticket-id> §<section>` or `Spec Bee t1=Doc child PRD §<section>`. Document the citation format inline so PM agents executing the review use a consistent shape.

3. **Scoped-marker integration with `bees` resolver (mirror of sibling Task 1's logic).** The breakdown skill's existing Scoped-marker handling (Section 2 — "Check for the Scoped-marker on the parent Bee") runs on the parent Bee body via `scoped_marker_resolver.py`. With the `bees` resolver:
   - **When resolver is `bees`, skip Scoped-marker resolution entirely.** The Spec Bee is already feature-scoped (one Spec Bee per feature); there is no cumulative file to scope into a `### Feature: <title>` subsection.
   - **When resolver is `file-path` (or default), run Scoped-marker resolution as today.** Preserves `/quo-plan-from-specs --feature` mode that sibling Epic `t1.31f.67` regression-tests.
   - Add a pre-branch in the Section 2 prose: detect `reference_materials`'s resolver kind first; if `bees`, skip the marker; if `file-path` or empty, run the existing marker-detection helper as today.

4. **Read `reference_materials` correctly.** Section 2's existing prose reads `reference_materials` as a list of file paths. Update to handle the polymorphic shape: each entry has a `value` and an optional `resolver` field. Iterate the list, branch on `resolver`, and accumulate spec content per the appropriate path (file content for `file-path`, child-ticket bodies for `bees`).

5. **Cross-skill consistency with `agents/pm.md`.** Sibling Task 1 establishes the canonical lookup shape in `agents/pm.md`. This Task's prose in `skills/quo-breakdown-epic/SKILL.md` MUST mirror that shape — same title-match (exact-match on `PRD` / `SDD`), same enumerate-children-via-`show-ticket` recipe, same Scoped-marker skip-on-bees rule. Divergence is a defect.

6. **Preserve existing behavior.** The breakdown skill's other responsibilities (Task decomposition, Subtask body authoring, ticket creation, status transitions, commit, next-steps menu) are not modified by this Task. Only the `reference_materials` resolution / spec-source-acquisition logic is extended.

7. **Skill-prose constraints.** Project-neutral; paired POSIX + PowerShell snippets where applicable; single-literal-command Bash etiquette; no hardcoded this-repo paths or ticket IDs.
