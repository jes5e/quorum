## Context

Epic `t1.31f.4u` updates the PM agent and breakdown skill to perform two-hop `reference_materials` lookup. Sibling Tasks 1 and 2 of this Epic update `agents/pm.md` and `skills/bees-breakdown-epic/SKILL.md` respectively. This Task closes the loop: update the PM dispatch prompts in the two execution skills (`skills/bees-execute/SKILL.md` and `skills/bees-fix-issue/SKILL.md`) so they don't contradict the new lookup logic, AND update `docs/doc-writing-guide.md` to document the `bees` resolver and the two-hop lookup pattern.

Sibling Tasks:
- **Task 1** of this Epic — `agents/pm.md` two-hop lookup (the canonical PM agent logic).
- **Task 2** of this Epic — `skills/bees-breakdown-epic/SKILL.md` two-hop lookup (the breakdown skill's own usage).

This Task focuses on the execution skills' PM dispatch prompts AND the doc-writing-guide updates that document the new reference-materials shape.

## What Needs to Change

Three files modified:

1. **`skills/bees-execute/SKILL.md`** — locate the PM agent dispatch prompt (where the skill spawns a PM agent for per-Task or Bee-wide review). Today's prompt mentions "PRD/SDD linked from the Bee's `reference_materials`." Update the prompt's prose to acknowledge the `bees`-resolver path:
   - The PM agent reads `reference_materials` and resolves it via the canonical logic in `agents/pm.md` (which now branches on resolver kind).
   - Spec content may come from file-resolver paths (`/bees-plan-from-specs` flow) OR Spec Bee `t1=Doc` children via the `bees` resolver (`/bees-plan` flow).
   - Defer to `agents/pm.md` as the authoritative spec for the lookup logic; do NOT duplicate the resolution prose in the dispatch prompt.

   The actual lookup logic does not live here — it lives in `agents/pm.md` (sibling Task 1). This Task only ensures the dispatch prompt does not contradict the new logic.

2. **`skills/bees-fix-issue/SKILL.md`** — same change. Locate the PM dispatch prompt (where the skill spawns a PM agent). Apply the same prose update: acknowledge `bees` resolver path, defer to `agents/pm.md`. Note: `/bees-fix-issue` deals with Issues, not Plan Bees, so its PM dispatch typically doesn't read `reference_materials` from the Issue itself (Issues don't have it). However, when an Issue's resolution requires consulting the originating Plan Bee (e.g., if the Issue surfaces a divergence from the Plan), the PM dispatch prompt may transitively read `reference_materials` from the linked Plan Bee — verify the prose handles this case correctly.

3. **`docs/doc-writing-guide.md`** — two prose updates:
   - **`Reference materials` section (or equivalent)** — document the `bees` resolver: format `[{"value":"<bee-id>","resolver":"bees"}]`, when to use it (Spec Bees with t1=Doc children), how downstream skills resolve it (two-hop lookup via `bees show-ticket --ids`), and when to use the alternative `file-path` resolver (cumulative on-disk PRD/SDD via `/bees-plan-from-specs`).
   - **`The Scoped-marker contract` section** — clarify that the marker now applies ONLY to `/bees-plan-from-specs --feature` mode (sibling Epic `t1.31f.y2`'s `/bees-plan` redesign no longer emits markers, since Spec Bees are already feature-scoped). The marker remains valid for the file-resolver path; nothing else about the marker's contract changes.

## Why

Without these prose updates, the execution skills' PM dispatch prompts and the doc-writing-guide's reference-materials documentation would still describe a file-paths-only world, contradicting the new two-hop lookup logic in `agents/pm.md` (sibling Task 1) and `skills/bees-breakdown-epic/SKILL.md` (sibling Task 2). PM agents reading the dispatch prompt would see contradictory instructions — the prompt says "read PRD/SDD files," but the actual `agents/pm.md` logic says "branch on resolver."

The doc-writing-guide updates ensure future skill authors and reviewers know the `bees` resolver exists, when to use it, and how it relates to the file-resolver and Scoped-marker flows. These are skill-author-facing docs, not customer-facing — but they are load-bearing for future skill changes that touch `reference_materials`.
