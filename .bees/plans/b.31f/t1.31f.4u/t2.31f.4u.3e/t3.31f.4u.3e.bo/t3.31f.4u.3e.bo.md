## Context

The `Reference materials` glossary entry in `docs/doc-writing-guide.md` (currently line 258) describes only the bees CLI's built-in `file-path` resolver. Sibling Epic `t1.31f.y2`'s `/bees-plan` redesign introduces a second resolver mode used in `reference_materials` entries: the `bees` resolver, which carries a ticket ID rather than an on-disk path. Entries take the shape `[{"value":"<bee-id>","resolver":"bees"}]` and let one Plan Bee point at another Bee (e.g., a per-feature Plan Bee pointing at a Spec Bee whose body and `t1=Doc` children hold the spec content). Downstream skills (`bees-breakdown-epic`, `bees-execute`'s PM dispatch, `bees-fix-issue`'s PM dispatch, the PM agent itself) need to follow these references with a two-hop lookup: resolve `reference_materials` to a target Bee ID, run `bees show-ticket --ids <bee-id>`, and treat the resolved Bee body (or its children) as the spec source.

The doc-writing-guide is the canonical place for this contract — sibling Tasks `t2.31f.4u.j5` (agents/pm.md) and `t2.31f.4u.je` (skills/bees-breakdown-epic/SKILL.md) plus the engineer-side execution-skill SKILL.md changes all reference this guide for the resolver semantics, so the guide entry must land before / alongside those edits.

## What Needs to Change

Expand the `Reference materials` bullet in the `## Project terminology` section of `docs/doc-writing-guide.md` (currently a single line at line 258) to:

1. Keep the existing description of the `file-path` resolver (default; entries are paths to on-disk source documents like PRD/SDD).
2. Add a description of the `bees` resolver: entries take the shape `[{"value":"<bee-id>","resolver":"bees"}]`; the value is a ticket ID, not a path; entries are resolved by `bees show-ticket --ids <bee-id>` and the resolved Bee's body (or its `t1=Doc` children) is the spec source.
3. Document when each resolver applies: `file-path` for canonical on-disk PRD/SDD specs (the `/bees-plan-from-specs` path); `bees` for inter-Bee references where one Plan Bee anchors against a Spec Bee's children as authoritative scope (the `/bees-plan` redesign path under sibling Epic `t1.31f.y2`).
4. Document the two-hop lookup pattern downstream skills must follow when they encounter a `bees`-resolver entry: hop 1 is reading the entry's `value` from the parent Bee's `reference_materials`; hop 2 is `bees show-ticket --ids <value>` to fetch the referenced Bee. For Spec Bees, the consumer further enumerates the `t1=Doc` children (PRD + SDD) by exact title-match. Reference `agents/pm.md` and `skills/bees-breakdown-epic/SKILL.md` for the canonical recipe.
5. Preserve the existing null/empty-`reference_materials` clause: when the entire field is null/empty on a Plan Bee, the Plan Bee body itself is the authoritative spec. The `bees` resolver does not displace this fallback — it is a third source mode (path on disk / referenced Bee + children / own Bee body), not a replacement for the body-as-spec case.

The entry may grow from a single bullet to a short multi-line bullet or a small sub-list under the `Reference materials` term — keep it scoped to terminology, not a full how-to. The how-to belongs in the consuming skills' prose (sibling Tasks).
