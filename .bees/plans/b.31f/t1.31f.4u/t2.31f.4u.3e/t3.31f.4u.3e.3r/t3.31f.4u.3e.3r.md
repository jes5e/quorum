## Context

The `## The Scoped-marker contract` section of `docs/doc-writing-guide.md` (currently lines 165-203) currently describes the marker as carried on `/bees-plan-from-specs --feature`-authored Plan Bees, but ambient guide prose and historical context could lead a reader to assume `/bees-plan` Plan Bees may also carry the marker. Sibling Epic `t1.31f.y2` redesigns `/bees-plan` so it no longer emits Scoped-markers — the marker is now exclusively a `/bees-plan-from-specs --feature` artifact. The guide must say so unambiguously, and downstream consumer skills (`bees-breakdown-epic`, the PM agent, `bees-execute`'s PM dispatch, `bees-fix-issue`'s PM dispatch) must continue to apply the marker only when present, never inferring it from a `/bees-plan`-authored Bee.

## What Needs to Change

Update `## The Scoped-marker contract` in `docs/doc-writing-guide.md` (lines 165-203) to:

1. Make the opening sentence narrower and explicit: the marker is emitted only by `/bees-plan-from-specs --feature "<title>"`. Today's opening already names that mode; tighten the language so a reader cannot infer that `/bees-plan` ever emits a marker. Add an explicit one-line clarification that `/bees-plan` Plan Bees do NOT carry a Scoped-marker — they use `reference_materials` (with either resolver) or fall back to body-as-spec, but never the marker.
2. Preserve the marker grammar, path resolution, subsection extraction rule, hard-fail rules, bundled-parser description, and `bees-fix-issue` consumer note exactly as they read today — none of those mechanics change for this Task. Only the scope-of-applicability framing changes.
3. Where the section currently says "A Plan Bee authored via `/bees-plan-from-specs --feature \"<title>\"`" (line 167), keep that phrasing and reinforce it with the new clarifying sentence so the contract reads as `/bees-plan-from-specs --feature`-only, full stop.
4. The downstream-consumer paragraphs ("When the marker is present, ...", "`bees-fix-issue` consumer note") remain valid — consumers still apply the marker when present and fall back to the full doc / body-as-spec when absent. Leave their semantics intact; only update wording if it implies `/bees-plan` is a possible marker source.

The edit is small and surgical — the section is already mostly correct; the goal is to remove any residual ambiguity introduced by the parallel `/bees-plan` redesign in sibling Epic `t1.31f.y2`.
