## Context

Epic `t1.31f.ho` adds a third hive (`Specs`) to quorum alongside the existing `Plans` and `Issues` hives. Sibling Task 1 colonizes the hive in `/bees-setup`; sibling Task 2 adds precondition gates to the three execution skills. This Task updates `README.md` so that (a) the customer-facing hive catalog accurately documents the new hive, and (b) users upgrading from an older quorum installation know to re-run `/bees-setup` to pick up the new hive without disturbing existing ones.

This Task is documentation-only — no skill prose, no helper script, no agent contract changes. The README is the authoritative customer-facing doc per CLAUDE.md `## Documentation Locations`.

## What Needs to Change

Two surfaces in `README.md`:

1. **Hive table.** README has a hive catalog (the table or list that documents `Plans`, `Issues`, what each hive holds, and how it is shaped). Add a row/entry for `Specs` describing:
   - Display name and normalized name (`Specs` / `specs`).
   - Child tiers: `t1=Doc/Docs`. PRD and SDD are both `t1=Doc` children differentiated by ticket title (`PRD` vs `SDD`).
   - Statuses: `drafted`, `ready` (two states only, in contrast to Plans's four-state ladder and Issues's two-state `open → done`).
   - Allowed resolvers: `bees` — so a Plan Bee's `reference_materials` can point at a Spec Bee.
   - One-line summary of what the hive holds (Spec Bees: per-feature spec docs as `t1=Doc` children for the redesigned `/bees-plan` flow).

2. **Upgrading-from-older-versions section.** README has an existing upgrading-from-older-versions section (recently extended in commit `efd1093`). Add a one-liner noting that re-running `/bees-setup` against a repo that already has `Plans` and `Issues` will detect the missing `Specs` hive and prompt for its path, without disturbing the existing hives. Keep the tone and length consistent with the existing entries — this is one bullet or one short paragraph, not a new section.

The doc-writer should keep the addition strictly factual and avoid speculating about future flows that are owned by sibling Epics — this Task documents the Specs hive itself, not how `/bees-plan` (sibling Epic 4) will use it or how PM (sibling Epic 5) will resolve the `bees` resolver. Those are separate Epics; their docs land later when those Epics execute.

## Why

The README is what a user reads to understand what hives exist and how to upgrade an existing installation. Without these two updates, a user re-running `/bees-setup` would see an unexpected new prompt for a hive they have never heard of, and a user reading the catalog would see a hive in `bees list-hives` that the README does not describe. Both are user-experience cliffs.

The Specs hive's role in the broader workflow (storing per-feature spec content as ticketed `t1=Doc` children to replace today's plan-time doc mutation) is part of the Plan Bee body's narrative; that broader narrative does not need to land in the README until the relevant downstream Epics ship. This Task is scoped narrowly to the hive itself.
