## Outcome

Add a new `Specs` hive to the bees-workflow and wire it into the existing
setup and precondition machinery so the rest of this Plan Bee's Epics
have somewhere to land.

## Scope

1. **Colonize the Specs hive (in `/bees-setup`).**
   - Hive name: `Specs`. Path: chosen by the user during `/bees-setup`,
     same UX pattern as the existing Plans/Issues hive prompts.
   - Child tiers: `t1` → `Doc / Docs`. PRD and SDD will both be `t1`
     children differentiated by ticket title.
   - Status values: `drafted`, `ready`. Two states only — Plan Bee
     dependency on Spec Bee transitions on `ready`; no in-progress or
     done states for spec docs (see Plan Bee body's "Decisions and
     rejected alternatives" for rationale).
   - Allowed resolvers: `bees` (so the Plan Bee's `reference_materials`
     can point at the Spec Bee). Other resolvers can be added later.

2. **Detect-and-add path for existing repos.** When `/bees-setup` runs
   on a repo that already has Plans + Issues but lacks Specs, it must
   prompt for a path and colonize-hive without disturbing the existing
   hives. The recent `efd1093` "upgrading from older versions"
   pattern is the precedent to follow.

3. **Hard-fail preconditions.** Add the Specs-hive-presence check to:
   - `/bees-execute` — already has a "Run /bees-setup first." pattern;
     extend the missing-hive list to include Specs.
   - `/bees-fix-issue` — same pattern; extend identically.
   - `/bees-breakdown-epic` — same pattern.

   Hard-fail message must direct the user to re-run `/bees-setup` to
   add the missing hive, matching the README's upgrade-path prose.

4. **README + upgrading-from-older-versions update.** Document the
   Specs hive in the README's hive table. Add a one-liner to the
   upgrading-from-older-versions section noting that re-running
   `/bees-setup` adds Specs without disturbing existing hives.

## Acceptance criteria

- Fresh `/bees-setup` run on a brand-new repo produces all three hives
  (Plans, Issues, Specs) with the correct tier and status configurations.
- Re-running `/bees-setup` on an existing repo (one that has Plans +
  Issues but not Specs) prompts for the Specs path, colonizes it, and
  leaves the other two hives unchanged. Verifiable by running
  `bees list-hives` before and after.
- `/bees-execute`, `/bees-fix-issue`, and `/bees-breakdown-epic` each
  emit `Run /bees-setup first. — Specs hive is not colonized for this
  repo.` (or equivalent) when invoked against a repo missing Specs.
  Verifiable by temporarily removing the Specs hive and running each.
- `bees list-hives` returns three hives whose normalized names are
  `plans`, `issues`, `specs` (after fresh setup or upgrade).
- README's hive table includes Specs with its tier/status info.
- Upgrading-from-older-versions section mentions the Specs addition.

## Implementation notes

- Hive colonization on POSIX and Windows: follow `apiary-setup`'s
  paired-shell-snippet pattern, which `/bees-setup` already does for
  Plans/Issues.
- `colonize-hive` accepts `--child-tiers JSON` and `--status-values JSON`;
  pass them explicitly rather than relying on global config inheritance,
  per the same pattern in apiary's setup.
- Don't hardcode the path; ask via `AskUserQuestion` the way the
  existing Plans/Issues prompts do (sibling-to-repo vs in-repo
  options at minimum, plus user-supplied path).
