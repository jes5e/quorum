## Context

The fast-path `#### Re-register each on-disk hive` subsection in `skills/bees-setup/SKILL.md` (currently lines 141-208) handles the new-machine case: hive directories are committed to the repo and present on disk, but the per-machine `~/.bees/config.json` has no scope entry for this repo. The existing prose branches on hive name: `issues` and `plans` are treated as canonical (the skill applies known defaults verbatim, no user prompt), while any other hive name falls into the `Unknown hive names` branch which walks the user through inline `AskUserQuestion` prompts for child tiers and status values.

Once the Specs hive is part of the canonical set (the prior Subtasks of this Task), a fresh clone on a new machine will have `.bees/specs/` on disk just like `.bees/issues/` and `.bees/plans/`. Without an update here, the fast-path will treat `specs` as an unknown hive and force the user through avoidable prompts every time they re-register on a new machine — defeating the fast-path's whole purpose. This Subtask elevates `specs` to canonical status in the fast-path branching so it gets the same defaults-applied-verbatim treatment as `issues` and `plans`.

This Subtask depends on the Subtask that establishes the Specs hive contract values in the `## Valid configuration` reference section. It does not depend on the slow-path-extension Subtask — the slow-path and fast-path Subtasks modify different regions of the same file and can land in either order after the contract Subtask.

## What Needs to Change

In `skills/bees-setup/SKILL.md`, inside the `#### Re-register each on-disk hive` subsection (currently lines 141-208):

1. **Update the canonical-vs-unknown split.** The current paragraph reads `**Canonical hives (`issues` and `plans`)** — apply the canonical defaults verbatim.` and `**Unknown hive names** (anything other than `issues` or `plans`) — ...`. Change both phrases to include `specs`: `**Canonical hives (`issues`, `plans`, and `specs`)**` and `**Unknown hive names** (anything other than `issues`, `plans`, or `specs`)`.

2. **Add a Specs-specific re-registration block** parallel to the existing per-hive blocks for `issues` (lines 148-150 / 159-161) and `plans` (lines 152-156 / 163-167). The Specs block must:
   - Run `bees colonize-hive --name specs --path "<discovered-path>" --scope "<repo-root>/**" --allowed-resolvers '["bees"]'` (note the inline `--allowed-resolvers` flag — Specs requires it; Issues and Plans do not).
   - Run `bees set-types --scope hive --hive specs --child-tiers '{"t1":["Doc","Docs"]}'`.
   - Run `bees set-status-values --scope hive --hive specs --status-values '["drafted","ready"]'`.
   - Ship as paired POSIX bash + Windows PowerShell snippets, matching the surrounding code blocks' shape exactly (per design rule 2 in this repo's CLAUDE.md).

3. **No changes to the `#### Detect` subsection** (lines 134-140) or to the JSON shape of the `detect_fast_path.py` script's output — that script is hive-name-agnostic and already correctly enumerates whatever `.bees/<hive>/` directories exist on disk; the fast-path eligibility check ("at least one on-disk hive marker exists") doesn't need to know the names.

4. **No changes to the `#### Diagnose`, `#### Confirm and exit`, or other adjacent subsections** — they're hive-count-driven, not hive-name-driven, and already work for any N.

5. **No changes to `skills/bees-setup/scripts/detect_fast_path.py`.** The detector reports on-disk hives by reading directory names; it makes no assumption about which names are canonical, so adding `specs` as a canonical hive in the prose alone is sufficient.

## Key Files

- `skills/bees-setup/SKILL.md` — extend the `#### Re-register each on-disk hive` subsection to make `specs` a third canonical hive name in the fast-path branching, with its own per-hive registration block.

## Acceptance Criteria

- The fast-path `#### Re-register each on-disk hive` subsection enumerates exactly three canonical hive names: `issues`, `plans`, and `specs`. The unknown-hive-names branch is correspondingly described as triggering for any name other than those three.
- A new-machine re-registration of a repo whose on-disk markers include `.bees/specs/` does NOT prompt the user for child tiers or status values for the Specs hive — the canonical defaults are applied without prompts (matching the existing UX for `issues` and `plans`).
- The Specs canonical block in this subsection runs `bees colonize-hive --name specs --path "<discovered-path>" --scope "<repo-root>/**" --allowed-resolvers '["bees"]'`, then `bees set-types --scope hive --hive specs --child-tiers '{"t1":["Doc","Docs"]}'`, then `bees set-status-values --scope hive --hive specs --status-values '["drafted","ready"]'`, in that order.
- Every shell snippet added in this Subtask ships as paired POSIX bash + Windows PowerShell variants. No bash-only fallback.
- `skills/bees-setup/scripts/detect_fast_path.py` is not modified by this Subtask.
- A `python -m pyflakes skills/*/scripts/*.py` lint run from the repo root continues to pass.
