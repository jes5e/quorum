## Context

Task `t2.31f.ho.8h` introduces a new `Specs` hive into `/quo-setup`. The skill's `## Valid configuration` reference section (lines 18-41 of `skills/quo-setup/SKILL.md`) currently only documents the `Issues` and `Plans` hives — their child tiers and allowed status values. This section is the in-skill contract that the rest of the SKILL.md prose references when prompting for, colonizing, and validating hives. Before the fresh-install and upgrade-add code paths can be extended to handle Specs (Subtasks 2 and 3 of this Task), the contract values for the Specs hive must be stated in this canonical place.

This Subtask is documentation-only — it adds a new subsection to the reference block. No `AskUserQuestion` prompts, no `bees` CLI invocations, no shell snippets are added here.

## What Needs to Change

In `skills/quo-setup/SKILL.md`, inside the `## Valid configuration` section (currently lines 18-41), add a new `### Specs Hive` subsection alongside the existing `### Issues Hive` and `### Plans Hive` subsections. The new subsection must document four contract values fixed by the parent Task body:

- **Child tiers**: `t1 — Doc / Docs` (one tier only — both PRD and SDD will be `t1=Doc` children differentiated by exact-match title, not by tier).
- **Status values**: `drafted` and `ready` only — no `in_progress` or `done`. The Spec Bee's transition to `ready` is what gates the downstream Plan Bee dependency; a status of `done` is intentionally excluded because spec content is amended over the project lifetime, not closed.
- **Allowed resolvers**: must include `bees` (so a Plan Bee's `reference_materials` can carry `[{"value":"<spec-bee-id>","resolver":"bees"}]`).
- **Hive shape**: top-level hive (not nested under any other hive), same as `Plans`.

Use the same prose shape and bullet formatting as the existing `### Issues Hive` and `### Plans Hive` subsections — the Plans-hive subsection is the closest precedent because it also has child tiers, and the Issues-hive subsection is the closest precedent for documenting allowed status values explicitly. Keep wording project-neutral per design rule 3 in this repo's CLAUDE.md.

No other section of the file is modified by this Subtask. Subtasks 2 and 3 will reference the contract values stated here.
