## Context

Epic `t1.31f.ho` (parent Bee `b.31f`) introduces a new `Specs` hive into the quorum so a redesigned `/quo-plan` (sibling Epic `t1.31f.y2`) can store PRD/SDD spec content as `t1=Doc` child tickets of a Spec Bee instead of mutating cumulative project docs at plan time. This Task colonizes the `Specs` hive in `/quo-setup` â€” both for fresh repos and as an upgrade-add for repos that already have `Plans` and `Issues` hives but no `Specs` hive.

Spec content authoring (the `t1=Doc` children themselves), the `bees`-resolver lookup logic in PM/breakdown, doc-writer post-impl responsibility, and the `/quo-plan` redesign itself all live in sibling Epics â€” they are NOT in this Task's scope. This Task only delivers the hive itself, ready to be filled by future Epics.

## What Needs to Change

`skills/quo-setup/SKILL.md` is the single skill modified by this Task. Two pathways must be added or extended:

1. **Fresh-repo path.** When `/quo-setup` runs against a brand-new repo, the user is prompted (via `AskUserQuestion`, paired POSIX + PowerShell snippet pattern, same UX as the existing Plans/Issues prompts) for the `Specs` hive path â€” sibling-to-repo, in-repo, and user-supplied path options at minimum. After the path is chosen, `bees colonize-hive` is invoked with the Specs-specific tier and status JSON arguments.

2. **Detect-and-add (upgrade) path.** When `/quo-setup` re-runs on a repo that already has `Plans` and `Issues` colonized but does not yet have `Specs`, the skill detects the gap via `bees list-hives` (looking for a hive whose `normalized_name` is `specs`), then prompts for the path and invokes `bees colonize-hive` only for the missing hive â€” without re-prompting for or disturbing the existing two. This mirrors the precedent established by commit `efd1093` (the upgrading-from-older-versions pattern); follow the existing prose shape rather than inventing a new one.

The Specs-hive colonize-hive call must pass these contract values explicitly (do NOT rely on global config defaults â€” pass `--child-tiers` and `--status-values` JSON the same way the existing Plans/Issues calls do):

- Hive display name: `Specs`. Normalized name (auto-derived by `bees`): `specs`.
- Child tiers JSON: `[{"singular": "Doc", "plural": "Docs"}]` â€” one tier (`t1`) only. Both PRD and SDD will be `t1=Doc` children differentiated by the title field (`PRD` vs `SDD`), not by tier.
- Status values JSON: `["drafted", "ready"]` â€” exactly two states; no `in_progress` or `done`. The Spec Bee's transition to `ready` is what gates the Plan Bee's downstream dependency.
- Allowed resolvers: `bees` (so a Plan Bee's `reference_materials` can carry `[{"value":"<spec-bee-id>","resolver":"bees"}]`).

Both code paths must use the OS-paired snippet shape (POSIX bash + Windows PowerShell at minimum) the rest of `/quo-setup` already uses for Plans/Issues colonization. Do NOT introduce a bash-only fallback. Do NOT hardcode a single path; let the user pick.

## Why

Today's `/quo-plan` Step 4 mutates `docs/prd.md`, `docs/sdd.md`, and the README at plan time. The redesign moves spec content into ticketed-form (a Spec Bee with PRD and SDD `t1=Doc` children) so plan-without-execute no longer pollutes project docs and so parallel planning of multiple unrelated features stays coherent. The Specs hive is the storage substrate that enables the redesign â€” without it, none of the downstream Epics in this Plan Bee can land.

The detect-and-add path is required because most repos using quorum already have Plans + Issues from earlier setup runs; an upgrade path that adds only the missing hive (without re-asking about the existing two) is the smooth user experience.

## Cross-Task contracts established by this Task

This Task fixes the following contract values that the rest of this Epic and downstream Epics depend on. Any sibling Task or downstream Epic that consumes the Specs hive must use these exact values:

- **Hive display name**: `Specs` (capitalized). **Normalized name**: `specs` (lowercase). Sibling Task 2 of this Epic must use `specs` (normalized) when checking presence via `bees list-hives`.
- **Tier shape**: `t1=Doc/Docs`. Sibling Epics 2 (`t1.31f.by`, `/quo-write-prd` skill) and 3 (`t1.31f.5u`, `/quo-write-sdd` skill) must create their child tickets at the `t1` tier. Downstream Epic 5 (`t1.31f.4u`, PM + breakdown two-hop lookup) must enumerate Spec Bee children at the `t1` tier and differentiate PRD vs SDD by exact-match title (`PRD` and `SDD`).
- **Status values**: `drafted` and `ready` only. No `in_progress` or `done` for spec docs. Sibling Epic 4 (`t1.31f.y2`, `/quo-plan` redesign) must transition the Spec Bee to `ready` after PRD and SDD child tickets are written; downstream consumers gate on `ready`.
- **Allowed resolver**: `bees`. Sibling Epic 4 (`t1.31f.y2`) must set `reference_materials` to `[{"value":"<spec-bee-id>","resolver":"bees"}]` when creating the Plan Bee. Downstream Epic 5 (`t1.31f.4u`) must accept this shape in PM and breakdown two-hop lookup logic.
- **Failure-message vocabulary**: when the `Specs` hive is missing on an existing repo, the upgrade-add prompt language and any subsequent precondition hard-fail (Task 2 of this Epic) must consistently refer to the hive as the **Specs hive** (capitalized), matching the README hive table that Task 3 of this Epic adds.
