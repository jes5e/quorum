## Context

The slow-path `### Hive Configuration` section of `skills/bees-setup/SKILL.md` (currently lines 324-363) is the prose that handles both fresh-install hive colonization and the detect-and-add upgrade path. Its existing logic — `bees list-hives` to detect what's present, `AskUserQuestion` to pick a location strategy for any missing hives, then `bees colonize-hive` followed by `bees set-types` and `bees set-status-values` for each missing hive — is structurally already correct for the detect-and-add upgrade pattern (it only colonizes hives that are missing). What's currently missing is: (a) the Specs hive is not in the canonical-hives set the prose enumerates, (b) the location-strategy prompt's bullet examples only mention `issues` and `plans` paths, and (c) there is no `--allowed-resolvers '["bees"]'` flag on any existing colonize-hive snippet — but Specs requires it.

This Subtask depends on the Subtask that establishes the Specs hive contract values in the `## Valid configuration` reference section (the canonical contract that this section's logic must align with).

Verified bees CLI flag names (from `bees colonize-hive --help`): `--name`, `--path`, `--scope`, `--child-tiers` (JSON dict mapping tier keys to `[singular, plural]` arrays — NOT array-of-objects), `--status-values` (JSON array), `--allowed-resolvers` (JSON list). There is no separate `set-allowed-resolvers` subcommand; `--allowed-resolvers` must be passed inline on `colonize-hive`.

## What Needs to Change

In `skills/bees-setup/SKILL.md`, modify the `### Hive Configuration` subsections so the canonical hive set is `Issues`, `Plans`, and `Specs`:

1. **`#### Create or validate` (currently lines 334-363).** Update the prose so:
   - The `bees list-hives` validation step covers all three canonical hives (Issues, Plans, Specs) — not just two.
   - The `AskUserQuestion` location-strategy prompt's bullet examples enumerate paths for all three hives. Specifically, the **In-repo** option's example becomes `<repo>/.bees/issues`, `<repo>/.bees/plans`, and `<repo>/.bees/specs`; the **Sibling-to-repo** option's example becomes `<project-parent>/<repo>-issues`, `<project-parent>/<repo>-plans`, and `<project-parent>/<repo>-specs`.
   - The conditional language `If both hives are missing, ask once and apply the chosen strategy to both.` is updated to handle 1-3 missing hives (e.g., "If multiple hives are missing, ask once and apply the chosen strategy to all of them. If only one is missing, scope the question to just that hive.").
   - Add a Specs-specific colonize block parallel to the implicit Issues and Plans blocks. The Specs block must (per the parent Task body's contract): pass `--allowed-resolvers '["bees"]'` inline on `bees colonize-hive`, then call `bees set-types --scope hive --hive specs --child-tiers '{"t1":["Doc","Docs"]}'`, then call `bees set-status-values --scope hive --hive specs --status-values '["drafted","ready"]'`. Use the exact JSON-dict shape `{"t1":["Doc","Docs"]}` for `--child-tiers` (NOT `[{"singular":"Doc","plural":"Docs"}]` — the parent Task body's worked example used the wrong shape; the actual CLI takes the dict-of-arrays shape, confirmed against `bees colonize-hive --help`). Ship paired POSIX bash + Windows PowerShell snippets per design rule 2 in this repo's CLAUDE.md — no bash-only fallback.
   - Wherever the existing prose generically describes "the above hives" or refers to creating Issues and Plans, ensure Specs is treated identically as a first-class canonical hive (no second-class "optional add-on" framing — the parent Task body says Specs is required for the redesigned `/bees-plan` to function).

2. **No changes to the `#### Scope requirement` subsection** (lines 328-332) — its guidance is hive-name-agnostic and already correct for Specs.

3. **No changes to the existing in-flight prompt logic.** The detect-and-add upgrade case (a re-run on a repo that already has Plans + Issues) is naturally handled by the existing "only missing hives are prompted for" flow — once Specs is in the canonical set, the existing logic correctly: (a) detects Specs as missing on `bees list-hives`, (b) prompts only for the Specs hive's path strategy, (c) leaves the other two hives untouched.

4. **Failure-message vocabulary.** Where prompts mention the hive by name in user-facing text, use `Specs hive` (capitalized) per the parent Task body's cross-Task contract on naming.
