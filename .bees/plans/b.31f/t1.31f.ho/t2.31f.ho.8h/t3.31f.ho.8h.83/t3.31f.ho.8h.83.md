## Context

`docs/doc-writing-guide.md` `## Project terminology` (line 256) defines what a "Hive" is for skill authors and reviewers:

> **Hive** — a bees collection. The workflow uses two: **Plans** (top-level, with t1/t2/t3 = Epic/Task/Subtask) and **Issues** (no children).

This bullet is the canonical definition that downstream contributors consult when authoring skill prose. Once Task `t2.31f.ho.8h` lands and `/bees-setup` colonizes a Specs hive, the "two hives" claim becomes factually wrong. Leaving it stale risks future skill authors writing prose that ignores the third hive (e.g., "all hives" enumerations, hive-scoped recipes that hard-code a `hives.* in (plans, issues)` shape).

The bullet immediately following at line 257 defines a **Bee** as "a ticket inside a hive" with examples "Plan Bee" and "Epic" — that wording is already generic enough to accommodate Spec Bees, but the doc reads more naturally if a Spec Bee is also called out as an example.

No other parts of the doc-writing-guide need updates:
- The `## Querying tickets` filter terms (line 96) document `hive=<name>` and `hive~<regex>` filters that already work for any hive name including `specs` — no change needed.
- The worked query examples (lines 114-141) all hard-code a specific hive (`hive=issues`, `hive=plans`, or no hive filter) and remain valid for their stated purpose. The status-snapshot example at lines 137-141 enumerates `[type=bee, hive=plans]` plus children — its prose framing is "Status snapshot — all Plan Bees plus their children", which is still accurate; it is not claiming to cover all hives. No edit required.
- The Scoped-marker contract (lines 165-203) is about Plan Bees specifically — Specs are unrelated and need no mention.

Scope-bounded: this is a non-README doc edit (sibling Task `t2.31f.ho.b2` owns README). It does NOT add any guidance about Spec-content authoring (that's sibling Epic `t1.31f.y2`); it only updates the Hive terminology bullet so the doc-writing-guide accurately enumerates the hives a contributor will encounter.

1. **`## Project terminology` → Hive bullet (line 256)** — replace "The workflow uses two: Plans … and Issues …" with a three-hive enumeration. Suggested wording (engineer should match the cross-Task contracts from Task `t2.31f.ho.8h`):
