## Context

Epic `t1.31f.ho` requires that the three execution skills hard-fail with `Run /quo-setup first.` (plus a one-line note about what is missing) when invoked against a repo that does not have the new `Specs` hive colonized. This Task adds that precondition gate to all three skills, mirroring the existing precondition logic that already checks for `Plans` and the required CLAUDE.md sections.

Sibling Task 1 of this Epic colonizes the `Specs` hive itself (in `/quo-setup`); sibling Task 3 of this Epic updates the README hive table and upgrading-from-older-versions section. This Task only adds the precondition gate ŌĆö it does NOT re-document the hive nor re-author the colonize-hive logic.

## What Needs to Change

Three SKILL.md files must be updated, one per execution skill:

- `skills/quo-execute/SKILL.md` ŌĆö extend the `## Preconditions` section's hive-presence check (today checks for `Plans`) to also verify a hive whose `normalized_name` is `specs` is present. If absent, hard-fail with `Run /quo-setup first. ŌĆö Specs hive is not colonized for this repo.`
- `skills/quo-fix-issue/SKILL.md` ŌĆö same change to that skill's `## Preconditions` section.
- `skills/quo-breakdown-epic/SKILL.md` ŌĆö same change to that skill's `## Preconditions` section.

The check must be implemented exactly the same way the existing `Plans` check is implemented (so future maintainers see one consistent pattern, not three subtly different ones). It must use the canonical `bees list-hives` output and look for `normalized_name == "specs"` (lowercase, matching what `/quo-setup` colonizes per Task 1 of this Epic). Do NOT match on display name (`Specs`) or any other field.

The hard-fail message must direct the user to re-run `/quo-setup` to add the missing hive ŌĆö matching the README upgrading-from-older-versions prose (which sibling Task 3 of this Epic authors). Suggested wording:

```
Run /quo-setup first. ŌĆö Specs hive is not colonized for this repo. Re-run /quo-setup to add the Specs hive without disturbing existing hives (Plans, Issues).
```

The engineer working this Task should keep the wording aligned across all three skills. Single-source the message string mentally; if it varies per skill in any way other than skill-specific framing already present in their preamble prose, that is a sign of accidental divergence.

## Why

Without this gate, `/quo-execute`, `/quo-fix-issue`, and `/quo-breakdown-epic` would silently fail (or fail with confusing downstream error messages) when invoked against a Plan Bee whose `reference_materials` carries a `bees`-resolver pointer to a Spec Bee in a hive that doesn't exist on disk. The gate makes the failure explicit and fixable in one step (re-run `/quo-setup`). It mirrors the precedent established by the existing Plans-hive check, the CLAUDE.md `## Documentation Locations` check, and the CLAUDE.md `## Build Commands` check.
