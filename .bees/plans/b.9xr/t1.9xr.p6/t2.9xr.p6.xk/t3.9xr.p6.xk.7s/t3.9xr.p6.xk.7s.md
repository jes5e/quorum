## Context

The new `## Backend dispatcher` subsection (Subtask 4.D1) introduces a fourth design rule, but `## Review criteria for skill changes` currently lists only the original three. Without a matching review-criterion bullet, `bees-code-review` and `bees-doc-review` have no explicit hook to flag direct `bees ...` calls when reviewing skill-prose or helper-script changes in this repo. This subtask makes the new rule reviewable.

This subtask is the in-scope companion to Subtask 4.D1 — the new design rule and its review-criterion bullet are two halves of one design-rules update; the split exists so each lands in its own CLAUDE.md section without entangling the diffs.

## What Needs to Change

- Add a fourth bullet to the existing list under `## Review criteria for skill changes` covering direct `bees <subcommand>` invocations in `SKILL.md` prose or in `skills/<name>/scripts/` helper scripts, including any restating of dispatcher verb argv or JSON output shapes inline (which would defeat the cross-Task contract). Mirror the wording style of the existing three bullets (one-line statement plus parenthesized rule reference, e.g., "(rule: backend dispatcher)").
- Preserve the additive-criteria framing already in that section: the new bullet is an additional check layered on top of each review skill's standard checks, applies only to changes inside this repo, and does not travel with the review skills when they run against downstream projects.
- Do not change the existing three bullets, the additive-criteria paragraph, or the project-scoped caveat — they are load-bearing prose and a separate concern from this addition.
