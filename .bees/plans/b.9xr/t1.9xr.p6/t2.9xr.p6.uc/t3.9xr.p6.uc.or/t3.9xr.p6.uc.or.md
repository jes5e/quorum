## Context

Six skills share a common surface — `execute-freeform-query`, `show-ticket`, `create-ticket`, `update-ticket`, and `list-hives` — and three of them (`bees-plan`, `bees-execute`, `bees-file-issue`) include a particularly awkward shape: a piped `bees list-hives | python3 -c '<JSON parse>'` POSIX snippet paired with a `(bees list-hives | ConvertFrom-Json) | Where-Object ...` PowerShell snippet to extract a hive path. Both shapes violate CLAUDE.md "Bash etiquette" (compound pipes, multi-line `-c` scripts, `$(...)` capture). The dispatcher migration is the right moment to also retire that pattern.

## What Needs to Change

- Replace every direct `bees <subcommand>` call across these six skills with the dispatcher equivalent, using the verb names, flag names, and JSON output shape documented in `ticket_backend.py`'s module docstring as the single source of truth: `bees-plan/SKILL.md`, `bees-plan-from-specs/SKILL.md`, `bees-breakdown-epic/SKILL.md`, `bees-execute/SKILL.md`, `bees-fix-issue/SKILL.md`, `bees-file-issue/SKILL.md`.
- For body-bearing `create-ticket` calls (Plan Bees, Epics, Tasks, Subtasks, Issues), preserve the "author body to a temp file via the `Write` tool, then pass `--body-file <path>`" pattern. The dispatcher's `create` verb must accept the same body-file flag (per its docstring) — if the docstring's `create` contract uses a different flag name, surface that in the migrated prose verbatim.
- For status-only updates and single-line bodies, the dispatcher's `update` verb should accept the same status/body flags as the bees CLI surface they replace.
- For the three "find hive path" call sites in `bees-plan`, `bees-execute`, and `bees-file-issue`, retire the `bees list-hives | python3 -c '...'` POSIX + `(bees list-hives | ConvertFrom-Json) | Where-Object ...` PowerShell snippet pair. **Recommended pattern:** prose calls `python3 ".../ticket_backend.py" list-spaces > <temp-file>`, then uses the `Read` tool to load the JSON and select the matching hive's path in agent reasoning rather than in shell. This keeps every Bash call a single literal command, eliminates both the pipe and the inline `python3 -c` script, and makes the OS-paired snippets collapse to a single Python invocation. If `ticket_backend.py`'s docstring specifies a different mechanism (e.g., a dedicated helper verb), use that instead and call out the alternative in the migrated prose.
- Preserve all existing semantics: the `Run /bees-setup first.` precondition checks; the bees-execute `up_dependencies` resolution; the bees-fix-issue up-front validation that issue IDs are valid + open; the Plan Bee `egg=null` author-from-disk behavior; the bees-breakdown-epic Epic-children walk.
- Preserve all OS-conditional logic that is NOT a bees CLI call (temp file paths under `/tmp` vs `$env:TEMP`, line-continuation `\` vs backtick). Only the dispatcher invocations themselves collapse to one Python call.
- Do NOT touch any tier-label scheme, beads-specific branching, or the backend-pick prompt — those are Epic B's territory. Flag any subtask drift in that direction.
- Skill prose does NOT restate verb argv shapes or JSON output schemas inline — those live in the dispatcher's module docstring.
