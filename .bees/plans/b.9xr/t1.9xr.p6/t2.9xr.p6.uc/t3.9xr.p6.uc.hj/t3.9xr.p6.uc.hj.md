## Context

The dispatcher in `skills/_shared/scripts/ticket_backend.py` (Task 1) is the new single entry point for all ticket-backend interactions in skill prose. `bees-status/SKILL.md` is the smallest, fully read-only skill — it issues one `list-hives` and three `execute-freeform-query` calls inside a single fenced bash block. Migrating it first establishes the dispatcher-invocation pattern in skill prose with minimal risk before tackling the more complex skills.

## What Needs to Change

- Replace every direct `bees <subcommand>` call in `bees-status/SKILL.md` prose with a `python3 "<base-dir>/../_shared/scripts/ticket_backend.py" <verb> ...` call, using the verb names, flag names, and JSON output shape documented in `ticket_backend.py`'s module docstring as the single source of truth.
- Convert the existing single fenced block into the dispatcher invocation; since `python3` is cross-platform, collapse to one snippet rather than maintaining paired POSIX/PowerShell variants. State the cross-platform nature inline so future editors don't re-add OS-paired blocks.
- Preserve the meaning of each call: `list-hives` → dispatcher's `list-spaces` verb; the three `execute-freeform-query` calls → dispatcher's `query` verb with the same YAML payload semantics.
- Confirm the resulting prose follows CLAUDE.md "Bash etiquette in this repo" — one literal command per Bash call; do not introduce pipes, compounds, or `$(...)` expansions.
- Do not touch the workflow-stages table, decision-logic tree, or output-format sections; only the four CLI calls change.
- Skill prose does NOT restate verb argv shapes or JSON output schemas inline — those live in the dispatcher's module docstring; the skill calls and consumes.
