# Task 2: Migrate all 11 SKILL.md files to call the dispatcher (and update doc-writing-guide)

## Context

With the dispatcher landed in Task 1, every direct `bees <subcommand>` call in skill prose becomes a `python3 "<base-dir>/.../_shared/scripts/ticket_backend.py" <verb> ...` call. This Task migrates the prose across all 11 skills and updates the relevant sections of `docs/doc-writing-guide.md` so future skill authors follow the new pattern.

Three skills (`bees-plan`, `bees-execute`, `bees-file-issue`) currently have an awkward `bees list-hives | python3 -c '...'` POSIX + `(bees list-hives | ConvertFrom-Json) | Where-Object ...` PowerShell snippet pair to extract a hive path. Both shapes violate CLAUDE.md "Bash etiquette in this repo" (compound pipes, multi-line `-c` scripts). Migration is the right moment to retire that pattern using the dispatcher's `list-spaces` verb plus the Read tool.

## Cross-Task contract dependency

The verb names, flag names, and JSON output shapes used in the migrated prose match the contract documented in `ticket_backend.py`'s module docstring. The docstring is the single source of truth — skill prose does NOT restate envelope shapes inline.

This Task does NOT add a `## Ticket Backend` writer to `bees-setup` (sibling Epic B's job). It does NOT introduce beads-specific branching, README updates, `tier:` labeling, or hard-fail extension to require `## Ticket Backend`. Any subtask drift toward those items must be flagged.

## What Needs to Change

- Replace every direct `bees <subcommand>` invocation in skill prose with the dispatcher equivalent across all 11 skills.
- Collapse paired POSIX/PowerShell snippet pairs to a single Python invocation where the only OS-conditional content was the bees CLI call.
- Retire the `bees list-hives | python3 -c '...'` piped pattern in `bees-plan`, `bees-execute`, and `bees-file-issue` using the dispatcher's `list-spaces` + Read-tool flow.
- Update `docs/doc-writing-guide.md` `## Querying tickets` recipes to use the dispatcher's `query` verb (YAML payloads unchanged).
- Update `docs/doc-writing-guide.md` `## OS-conditional shell blocks` to note dispatcher calls collapse to one labeled block.

## Why

Backend neutrality at the skill-prose layer. No skill should know whether the underlying backend is bees or beads. The dispatcher absorbs that decision; Epic B then introduces the beads branch inside the dispatcher and skill prose is unaffected.

## Success Criteria

- `grep -nE "bees (execute-freeform-query|show-ticket|create-ticket|update-ticket|list-hives|colonize-hive|set-types|set-status-values)" skills/*/SKILL.md` returns zero matches (Epic AC #2).
- Every SKILL.md prose snippet using the dispatcher is a single literal command per CLAUDE.md "Bash etiquette in this repo" — no pipes, no `&&`, no `$(...)`, no multi-line `python3 -c`.
- `/bees-status` produces output byte-for-byte identical (modulo timestamps) before/after the migration (Epic AC #1).
- A smoke `/bees-plan "Add a hello-world function"` creates a Plan Bee + one Epic in the same shape as before (Epic AC #3).
- `python -m pyflakes skills/_shared/scripts/*.py skills/*/scripts/*.py` exits 0 (Epic AC #4).
