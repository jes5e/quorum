# Task 3: Verify helper scripts don't subprocess `bees` directly

## Context

Epic A's Scope explicitly calls out: *"Verify and revise `bees-setup`'s `detect_fast_path.py` if it shells out to `bees` directly."* Pre-collected inventory shows none of the four bundled helpers (`file_list_resolver.py`, `detect_fast_path.py`, `force_clean_team.py`, `check_agent_teams.py`) currently subprocess to bees, so this Task is verification-and-document. If any helper does shell to bees, it must be routed through the dispatcher.

This Task can run any time after Task 1 reaches `done` (the dispatcher must exist as the refactor target). It runs in parallel with Task 2.

## What Needs to Change

- Grep `skills/*/scripts/*.py` (excluding `skills/_shared/scripts/ticket_backend.py` itself) for any pattern that subprocesses to the bees CLI.
- Explicitly include `skills/bees-setup/scripts/detect_fast_path.py` in the verification scope (Epic Scope names it).
- If the grep is empty (expected outcome), record the verification commands and outputs in the commit message; make no file changes.
- If any helper does shell to bees, refactor it to call `ticket_backend.py` instead, preserving the helper's external contract (CLI argv shape, exit codes, JSON output).

## Why

Belt-and-suspenders for Epic A's "bees stays the only backend, but only one place knows that" invariant. Helper scripts that route around the dispatcher would leak backend coupling and break Epic B's beads branch.

## Success Criteria

- Verification grep across `skills/*/scripts/*.py` (excluding the dispatcher) returns zero direct bees invocations, OR all hits have been refactored to route through the dispatcher.
- `detect_fast_path.py` is explicitly named in the verification scope (per Epic Scope bullet).
- The commit message records the exact grep commands and outputs for reviewer confirmation.
- No helper's external contract (argv shape, exit codes, JSON output) changes.
- `python -m pyflakes` on any refactored helper exits 0.
