## Context

The remaining four verbs are heavier. `create` and `update` translate dispatcher flags into a single `bees create-ticket` / `bees update-ticket` call. `setup-spaces` is a composite — one dispatcher call fans out to a bees `colonize-hive` plus optional `set-types` plus `set-status-values`, replacing the three-call sequence in current `bees-setup` prose. `resolve-spec` on bees is a thin convenience: it reads a Bee's `egg` field via `bees show-ticket` and emits it on stdout — skill-side egg resolution is Epic B's concern (when backend=beads).

All four verbs go in one subtask because they share the subprocess plumbing established in Subtask 1.3 and one expert pass through them produces a consistent shape.

**Forward-compatibility constraint.** The `setup-spaces` verb's argv shape and dispatch path must gate the bees branch on the explicitly-resolved backend value (from Subtask 1.2's reader), not on an implicit assumption that bees is the only backend. Epic B will add the beads branch inside the same verb handler; the seam must already be there in Epic A.

## What Needs to Change

Implement the bees branches of:

- `create` — flags mirror `bees create-ticket`: `--ticket-type`, `--title`, `--hive`, `--body | --body-file`, `--parent`, `--children`, `--up-deps`, `--down-deps`, `--tags`, `--status`, `--egg`. Forward each to bees. Stdout is bees' JSON pass-through (the contract field skill prose reads is `tickets[0].ticket_id`). The `--body | --body-file` mutual exclusion is enforced by argparse, mirroring bees' own behavior.
- `update` — flags mirror `bees update-ticket`: `--ids` (variadic), `--status`, plus any other update fields bees supports that current skill prose uses (status-only updates are the only shape in current prose; check `bees update-ticket --help` and forward whatever's there to keep the migration faithful). Stdout pass-through.
- `setup-spaces` — one composite call per hive. Flags: `--hive <name>`, `--path <path>`, `--scope <scope>`, `--egg-resolver <abs-path>` (optional — omit on hives that don't need a resolver), `--child-tiers '<json>'` (optional — when omitted, no `set-types` call), `--status-values '<json>'` (required). Internally the dispatcher runs `bees colonize-hive`, then optionally `bees set-types`, then `bees set-status-values`, in that order. Any non-zero exit aborts the sequence and propagates the failing call's stderr. Stdout on success is a small JSON envelope `{"status": "success", "hive": "<name>", "steps_run": ["colonize-hive", "set-types", "set-status-values"]}` — this is a new shape (no equivalent in raw bees) so document it in the module docstring. Skill prose for `bees-setup` collapses three OS-paired snippet pairs into a single Python call per hive.
- `resolve-spec` — flag: `--id <bee-id>`. On bees, runs `bees show-ticket --ids <id>` internally and emits a JSON envelope `{"status": "success", "ticket_id": "<id>", "egg": <egg-value-as-bees-returned-it>}` on stdout. The `egg` field is whatever bees returned (string, JSON array, or null). Skill-side resolver invocation (calling `file_list_resolver.py` from the dispatcher) is explicitly out of scope — Epic B adds it on the beads branch.

For each verb, refresh the module docstring's per-verb JSON-shape entry to match what's actually emitted. For `setup-spaces`, the verb dispatch must read the resolved backend value from Subtask 1.2's reader and route to the bees branch only when backend=`bees`; the seam for Epic B's beads branch must be visible in the code structure.

## Acceptance Criteria

- `python3 ticket_backend.py create --ticket-type bee --hive plans --title "x" --body "y" --status drafted` creates a ticket equivalent to the same `bees create-ticket` call (verify by running both against a throwaway test hive and comparing the resulting `bees show-ticket` outputs modulo `ticket_id` / `created_at` / `guid`).
- `python3 ticket_backend.py update --ids <id> --status done` updates the ticket to the same end state as `bees update-ticket --ids <id> --status done`.
- `python3 ticket_backend.py setup-spaces --hive plans --path <p> --scope <s> --egg-resolver <r> --child-tiers '<json>' --status-values '<json>'` produces the same end-state in `~/.bees/config.json` as the current three-call sequence in `bees-setup` SKILL.md, and the success envelope lists the steps that ran.
- `python3 ticket_backend.py setup-spaces --hive issues --path <p> --scope <s> --egg-resolver <r> --status-values '<json>'` (no `--child-tiers`) skips the `set-types` call and lists only `colonize-hive` and `set-status-values` in `steps_run`.
- `python3 ticket_backend.py resolve-spec --id <bee-id-with-non-null-egg>` emits a JSON envelope whose `egg` field equals the egg value bees returned for that Bee.
- `python3 ticket_backend.py resolve-spec --id <bee-id-with-null-egg>` emits a JSON envelope with `egg: null`.
- Any non-zero exit from a wrapped bees call propagates its exit code and stderr through the dispatcher unchanged.
- `setup-spaces`'s dispatch path explicitly reads the resolved backend value from Subtask 1.2's reader and routes to the bees branch only when backend=`bees` — the seam for Epic B's beads branch is visible in the code structure (a reviewer can identify where to drop in the beads case without reverse-engineering the verb).
- The module docstring's per-verb JSON-shape entries match the verbs' actual output for `setup-spaces` and `resolve-spec` (the two verbs with new envelope shapes that aren't direct bees pass-throughs).
- `python -m pyflakes skills/_shared/scripts/ticket_backend.py` exits 0.
