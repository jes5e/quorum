# Epic B — Add beads adapter, backend-pick step, and tier labels

**Status at completion.** Users can pick `beads` at `/bees-setup`; full chain works on either backend.

**Dependencies.** Blocked by Epic A — Epic A must reach `done` before this Epic can start.

## Description

Build out the beads side of the dispatcher introduced in Epic A. Inside `ticket_backend.py`, each verb gains a beads branch that composes `bd --db <repo>/.beads-{plans,issues}/dolt ...` calls with the right flag mappings (verified equivalents from the freeform-query inventory). The `setup-spaces` verb gains beads-specific logic to `bd init` two databases under `.beads-plans/` and `.beads-issues/` with `issue_prefix` and `status.custom` configured per database. The `resolve-spec` verb runs `file_list_resolver.py` skill-side when backend=beads (since beads has no CLI-registered resolver hook).

`bees-setup` gains a backend-pick step at the top — auto-detect from on-disk markers (`.bees/<hive>/.hive/identity.json` or `.beads-plans/dolt/`); prompt via `AskUserQuestion` when neither is present. The chosen backend is written to a new CLAUDE.md `## Ticket Backend` section (one of the values `bees` or `beads`). The Prerequisites and Hive Configuration sections of `bees-setup` SKILL.md become backend-conditional; the Egg Resolver subsection runs only when backend=bees.

All skill prose that currently fails with `Run /bees-setup first.` when the existing two contract sections are missing extends that check to also require `## Ticket Backend`. The dispatcher reads the section once at startup and dispatches accordingly.

The `tier:` labeling cleanup applies on **both** backends at create-time: every Plan Bee gets `tier:bee`, every Epic gets `tier:t1`, every Task gets `tier:t2`, every Subtask gets `tier:t3`. On bees this is informational; on beads it lets queries like "all Plan Bees" use `bd list --label tier:bee` instead of a jq post-filter.

README gets the changes specified in the PRD's acceptance criteria: intro, Requirements, Why-this-exists bullets, Status-vocabulary column header, `/bees-setup` description, plus a new "Ticket backend" section between "After install" and "The skills."

## Scope

- Implement beads branch for all seven dispatcher verbs in `ticket_backend.py`.
- Implement `setup-spaces --backend beads` in the dispatcher: `bd init` for both databases, `bd config set status.custom` for each, label-vocabulary verification.
- Add backend-pick step to `bees-setup` SKILL.md (auto-detect or `AskUserQuestion` prompt).
- Add `## Ticket Backend` writer to `bees-setup` SKILL.md.
- Make `bees-setup` SKILL.md's Prerequisites + Hive Configuration sections backend-conditional.
- Make every skill that currently checks the two contract sections also check `## Ticket Backend`.
- Add `tier:<level>` labels at create-time in the dispatcher's `create` verb (both backends).
- Update README per scope: intro, Requirements, Why-this-exists, Status-vocabulary column header, `/bees-setup` description, Coming-soon, Credits, plus the new "Ticket backend" section.
- Update CLAUDE.md `## Documentation Locations` if any new doc files get added (none planned).

## Acceptance criteria

1. **Demo: fresh-repo bees path (regression).** Agent creates a fresh git repo, runs `/bees-setup` and picks `bees`, then runs `/bees-plan` with a canned scope, then `/bees-breakdown-epic`, then `/bees-execute`. The chain reaches `done` on the Plan Bee. Equivalent to the pre-Epic-A behavior.
2. **Demo: fresh-repo beads path.** Agent creates a fresh git repo, runs `/bees-setup`, picks `beads` when prompted, and confirms `.beads-plans/dolt/` and `.beads-issues/dolt/` exist with `bd --db .beads-plans/dolt config get status.custom` returning `drafted:active,ready:active,done:done`. Then runs `/bees-plan` with the same canned scope and confirms a Plan Bee + one Epic exist via `bd --db .beads-plans/dolt list --label tier:bee --json`. Then runs `/bees-execute` and reaches `done`.
3. **Demo: backend-pick auto-detect.** Agent runs `/bees-setup` on a repo that already has `.bees/plans/.hive/identity.json` markers and confirms the skill auto-detects `bees` without prompting. Same for `.beads-plans/dolt/` → auto-detects `beads`.
4. **Demo: tier labels present.** On the beads run, `bd --db .beads-plans/dolt show <bee-id> --json | jq '.labels'` includes `tier:bee`. On the bees run, `bees show-ticket --ids <bee-id>` returns labels including `tier:bee`.
5. **Demo: missing-section hard-fail.** Agent removes the `## Ticket Backend` section from CLAUDE.md and runs `/bees-execute`; the skill exits with `Run /bees-setup first.` plus the missing-section note.
6. **README check.** Agent shows the rendered new "Ticket backend" section appears between "After install" and "The skills" in the README, matching the PRD's wording outline.

- A test suite for the skills repo (separate plan, ships after this).
- Beads' "stealth mode" / sync-branch / `bd dolt push` lifecycle hooks beyond what `/bees-setup` configures.
