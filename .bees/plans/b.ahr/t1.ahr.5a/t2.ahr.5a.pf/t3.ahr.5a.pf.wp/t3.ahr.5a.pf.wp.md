## Context

`/quo-file-issue`'s External-reference branch (Sub-steps A/B/C around `skills/quo-file-issue/SKILL.md:219-291`) currently has no dedupe check — re-running `/quo-file-issue --reference <url>` (or, after sibling Task 1 lands, `/quo-file-issue <url>` via the bare-URL Mode-fork extension) against the same URL silently mints a duplicate Issue. This breaks `/quo-fix-issue <url>` retry flows: when a fix run fails partway through and the user retries the file-then-fix loop, they hit a fresh duplicate Issue every time.

The in-conversation-capture branch already has a duplicate-title check at Step 2 (`skills/quo-file-issue/SKILL.md:105-117`) that surfaces existing open Issues by title and lets the user pick between using-existing and filing-new. That check stays as-is — this Subtask adds the equivalent dedupe behavior to the External-reference branch, keyed on `reference_materials.value` (URL match) rather than title.

This Subtask is the sole implementation Subtask for Task t2.ahr.5a.pf. The Task is one cohesive prose edit: add a dedupe sub-step in the External-reference branch, encode the bees query path, encode the three-way `AskUserQuestion`, and verify the cross-Task contracts.    The prompt body MUST include the matched ticket's ID, title, and status (the `report: [title, ticket_status]` projection in step 1 of the query recipe supplies all three). Encode the prompt-body shape inline as a literal example so the agent does not under-supply context.
