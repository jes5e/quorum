## Context

`/quo-file-issue`'s Step 2 today has a duplicate-title check for the in-conversation-capture branch — it scans existing open Issues by title and, on a hit, surfaces the existing ticket via `AskUserQuestion` so the user can choose between using the existing ticket and filing a new one. The External-reference branch has no equivalent check today: re-running `/quo-file-issue --reference <url>` (or, after Task 1, `/quo-file-issue <url>`) against the same URL silently mints a duplicate Issue.

This is a real problem when a `/quo-fix-issue <url>` run fails partway through and the user retries — they hit a fresh, duplicate Issue every time. This Task adds a dedupe path that queries open Issues for one whose `reference_materials.value` matches the URL exactly, and surfaces a three-way `AskUserQuestion` on hit.

This is the third of three Tasks in Epic t1.ahr.5a. The dedupe path lives inside the External-reference branch, so it applies to *both* entry surfaces (bare URL from Task 1 and the existing `--reference` / `--from-github` flag forms).

## What Needs to Change

- Extend the External-reference branch in `skills/quo-file-issue/SKILL.md` with a dedupe sub-step that runs BEFORE filing. The sub-step queries open Issues (status=`open` only — closed-and-archived issues MUST NOT block re-filing) for one whose `reference_materials.value` matches the URL exactly.
- Implementer chooses between two query paths at implementation time:
  - **One-step path:** a single `bees execute-freeform-query` with `reference_materials` in a stage-filter clause — only viable if the bees CLI supports `reference_materials` as a stage-filter field. Verify empirically before relying on this.
  - **Two-step path (canonical fallback):** freeform-query for `[type=bee, hive=issues, status=open]` returning `[ticket_id]`, then batch `bees show-ticket --ids <id1> <id2> ...` and filter client-side on each ticket's `reference_materials[*].value`. The bees CLI rejects `reference_materials` in the `report` clause (confirmed empirically), so a single freeform-query that *reports* `reference_materials` is NOT viable — both paths must read it via `bees show-ticket`.
- On a single match: surface the existing ticket via `AskUserQuestion` per CLAUDE.md `## AskUserQuestion usage` (multi-choice) with three finite choices: `Use existing` (return the matched ticket ID, set `action=reused-existing` in the inline-invocation return shape, skip filing); `File new` (file a fresh ticket anyway, set `action=created`); `Cancel` (abort the skill cleanly with no partial state). The matched ticket's ID, title, and status MUST appear in the prompt body so the user can decide intelligently.
- On multiple matches (rare but possible from prior partial runs): surface all candidates via `AskUserQuestion` with one option per candidate (labeled `Use existing (b.<id>)` or similar), plus `File new` and `Cancel`.
- Cancelling at the dedupe prompt MUST exit cleanly with no half-filed ticket and no partial mutation visible to programmatic callers (this is part of the SR-4.3 / behavioral-guarantees contract published in sibling Task 2).
- The duplicate-title check for in-conversation-capture mode (today's Step 2 behavior) MUST remain intact — the new dedupe path is for the External-reference branch only.

## Why

Without dedupe, re-running `/quo-fix-issue <url>` after any failed run mints a fresh duplicate Issue, polluting the bees workspace and confusing the user. Scoping the dedupe to status=`open` only is intentional: closed-and-archived Issues are already-fixed bugs, and surfacing them on a re-file would confuse the user into choosing "use existing" against a closed ticket. The three-choice `AskUserQuestion` (`Use existing` / `File new` / `Cancel`) gives the user control over the case where the existing ticket is stale or the user genuinely wants a fresh ticket.
