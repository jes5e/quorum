## Context

`/quo-file-issue` today routes positional arguments through a "Mode fork" that distinguishes interactive (no-args), in-conversation-capture (free-text description), and external-reference (`--reference <url>` or `--from-github <url>`) modes. A bare URL passed as a positional argument is currently treated as a description string — almost certainly not the user's intent.

This Task is the first of three in Epic t1.ahr.5a. It changes the user-facing entry surface so a positional argument matching `^https?://` auto-routes to the existing External-reference branch — no flag required. The body of the External-reference branch (thin-body authoring, URL-pattern resolver-name selection, `bees create-ticket --reference-materials` invocation) is unchanged. Existing flag forms remain accepted as silent no-op aliases.

## What Needs to Change

- Update the Mode fork in `skills/quo-file-issue/SKILL.md` to recognize a `^https?://` positional argument as a third routing input alongside flag-form (`--reference`, `--from-github`) and free-text description. URL-shaped positionals route to the existing External-reference branch; the branch body is unchanged.
- Update the `argument-hint` frontmatter to surface the bare-URL invocation form alongside the flag forms and the free-text description form.
- Update the `### Usage` section to list the bare-URL invocation alongside the flag-form invocations, with one example line each.
- Detection rule: a token is a URL iff it starts with `http://` or `https://` after argument tokenization. URLs embedded *inside* a quoted description string (e.g., `/quo-file-issue "Fix the bug at https://example.com/foo"`) MUST continue to be treated as part of the description, not auto-detected.
- The `--reference` and `--from-github` flag forms MUST continue to work without modification — they become silent no-op aliases on top of detection. Their documentation in skill prose stays for backward compat and discoverability.

## Why

The flag requirement is documentation noise: a URL in argument position is unambiguous, and no other plausible interpretation exists. Dropping the flag requirement removes a friction point that compounds when `/quo-fix-issue` (Epic 2) starts dispatching `/quo-file-issue` programmatically — the cleaner UX dominates the small surface-area cost.

## Cross-Task contracts

- **Routes-to-External-reference contract.** This Task establishes that a bare-URL positional routes through the same External-reference branch as `--reference <url>`. Sibling Task 3 (dedupe by `reference_materials.value`) lives inside that branch and depends on this routing — its dedupe path applies to *both* the bare-URL entry and the flag-form entry, since they converge on the same branch body.

## Success Criteria

- `/quo-file-issue https://github.com/owner/repo/issues/123` (no flag) produces an Issue identical in shape to today's `/quo-file-issue --reference https://github.com/owner/repo/issues/123`: thin body with the `external reference:` convention line, `reference_materials: [{"value":"<url>","resolver":"github-issue"}]`, status=`open`.
- `/quo-file-issue --reference <url>` and `/quo-file-issue --from-github <url>` continue to produce identical Issue shapes to the bare-URL path.
- `/quo-file-issue` (interactive, no args) and `/quo-file-issue <description>` (free-text capture) continue to dispatch identically to today.
- `/quo-file-issue "Fix the bug at https://example.com/foo"` is treated as a free-text description (URL inside a quoted string is not auto-detected).
- The `argument-hint` frontmatter and `### Usage` section both list the bare-URL form alongside the flag forms.
- Skill-prose changes pass project design rules: no language-specific commands (rule 1), POSIX + PowerShell pairing for any new shell snippets (rule 2), project-neutral phrasing (rule 3), scratch-file convention preserved.
