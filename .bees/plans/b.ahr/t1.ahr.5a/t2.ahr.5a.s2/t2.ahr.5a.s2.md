## Context

`/quo-fix-issue` (Epic 2) needs to dispatch `/quo-file-issue` programmatically through the Skill tool, capturing the new Issue ticket ID so it can substitute the URL in its working list. Today, `/quo-file-issue`'s Step 5 reports back to the user in prose only — there is no published contract a programmatic caller can parse without scraping prose by regex (fragile to wording changes).

The canonical precedent is `skills/quo-write-prd/SKILL.md`'s `## Inline invocation via the Skill tool` section, which `/quo-plan` consumes when dispatching the writer skills inline. This Task adds a parallel contract section to `/quo-file-issue` so Epic 2 can wire the file-then-fix transition against a stable contract surface.

This is the second of three Tasks in Epic t1.ahr.5a. The contract is *published* here but not yet *consumed* — Epic 2's `/quo-fix-issue` is what consumes it.

## What Needs to Change

- Append a new `## Inline invocation via the Skill tool` section near the bottom of `skills/quo-file-issue/SKILL.md`, mirroring the section structure used by `skills/quo-write-prd/SKILL.md`.
- The section MUST document three contract surfaces:
  - **Input shape (caller → this skill).** A free-text `args` string carrying the URL as a positional value (e.g., `url: <url>`). Document any optional context fields the caller may supply.
  - **Output shape (this skill → caller).** A structured final assistant message carrying at minimum: `issue_ticket_id` (the new Issue's ticket ID, or the matched ticket ID when the dedupe path returns); `issue_status` (`open` on success — Issue tickets only support `open` and `done`); `action` (`created` for a freshly-filed Issue, `reused-existing` when the dedupe `Use existing` branch was taken).
  - **Behavioral guarantees.** The user-facing `AskUserQuestion` gates inside `/quo-file-issue` (in-conversation capture's distill-vs-restart, External-reference's body-confirmation, dedupe disambiguation from sibling Task 3) MUST still fire when invoked inline. The Skill-tool caller does NOT short-circuit those gates.
- The contract section's idempotency prose MUST state: re-running the skill against the same URL produces an idempotent result — a duplicate file does not create a duplicate ticket; the dedupe path returns the existing ticket ID with `action=reused-existing`.

## Why

A published contract surface is the single-source-of-truth handoff between `/quo-file-issue` and any programmatic caller. Without it, `/quo-fix-issue` (Epic 2) would either re-implement URL handling locally — duplicating the URL-pattern resolver-name table — or scrape Step 5's prose with a regex. Both alternatives lock in fragility. The inline-invocation contract section is small (one section, ~30-50 lines) and follows an existing precedent.

## Cross-Task contracts

- **`action` vocabulary contract.** This Task publishes the `action` value vocabulary: `created` and `reused-existing`. Sibling Task 3 (dedupe) implements the path that emits `action=reused-existing`. The two Tasks MUST agree on the exact spelling — `reused-existing` (lowercase, hyphenated). Reviewers cross-check the contract section authored here against the dedupe-path behavior authored in Task 3.
- **Gates-still-fire contract.** This Task's `Behavioral guarantees` block names the gates that fire on the inline path, including the dedupe disambiguation gate from Task 3. If Task 3's dedupe gate set changes, this Task's prose MUST be revised to match.
