## Context

This subtask implements Task t2.ahr.5a.s2 in full. `/quo-fix-issue` (Epic 2 in this Bee, t1.ahr.5a's sibling Epic) needs to dispatch `/quo-file-issue` programmatically through the Skill tool and parse a structured response carrying the new Issue's ticket ID. Today, `/quo-file-issue`'s Step 5 only reports back to a human reader in prose — there is no published machine-readable contract. The canonical precedent for an inline-invocation contract section is `skills/quo-write-prd/SKILL.md`'s `## Inline invocation via the Skill tool` section (lines 315-366 of that file), consumed by `/quo-plan` when it dispatches the writer skills inline.

Preconditions:
- `skills/quo-file-issue/SKILL.md` exists with the current sectioning (Overview, House style, Usage, Preconditions, Steps with the Mode fork + Steps 0-5 + External-reference branch).
- `skills/quo-write-prd/SKILL.md` is read first as the structural template for the new section.
- The work is documentation-only: it does NOT alter any of the existing invocation paths (`/quo-file-issue` interactive, `/quo-file-issue <description>`, `--reference <url>`, `--from-github <url>`, or the bare-URL form added by sibling Task 1). It does NOT alter the dedupe behavior added by sibling Task 3.
- This is a single coherent edit. The Task is small (one new section, ~30-50 lines) and the cross-Task contracts (`action` vocabulary, gates-still-fire) are content-internal to the same section, so further decomposition would add overhead without value.

## What Needs to Change

Append a new `## Inline invocation via the Skill tool` section near the bottom of `skills/quo-file-issue/SKILL.md`, after Step 5 (`### 5. Report back`, currently ending at line 340). The section MUST mirror the structure used by `skills/quo-write-prd/SKILL.md`'s `## Inline invocation via the Skill tool` section: a short framing paragraph, then four sub-headings — `### Input shape (caller → this skill)`, `### Output shape (this skill → caller)`, `### Behavioral guarantees`, and `### Cross-reference` (or a brief idempotency / lifecycle treatment folded into Behavioral guarantees as the precedent does).

Content requirements for the new section:

1. **Framing paragraph.** State that the section is the stable contract for callers that invoke `/quo-file-issue` through the Skill tool rather than as a user-typed slash command. Name `/quo-fix-issue` as the canonical caller (it dispatches `/quo-file-issue` when it encounters a URL form requiring a freshly-filed Issue). State that other future callers MAY also invoke this skill via the Skill tool and must match the contract documented here.

2. **Input shape (caller → this skill).** Document the free-text `args` string the Skill-tool caller passes. The string carries the URL as a positional `url:` field. Recommended shape:
   ```
   url: <url>
   ```
   Document that any optional context fields the caller may supply (e.g., a `summary:` block carrying a pre-distilled 2-3 sentence summary so the External-reference branch's `WebFetch` step can be skipped) are OPTIONAL. The skill parses the `args` string, captures the URL, and routes execution through the Mode fork's external-reference branch (the skill treats an inline-invocation `url:` payload as equivalent to `/quo-file-issue --reference <url>`).

3. **Output shape (this skill → caller).** When the workflow completes, the skill returns a structured final assistant message naming AT LEAST these three fields:
   - **`issue_ticket_id`** — the new Issue's ticket ID. On the dedupe-reuse path (when sibling Task 3's dedupe `Use existing` branch is taken), this is the matched existing ticket ID rather than a freshly-minted one.
   - **`issue_status`** — `open` on success. The Issues hive only supports `open` and `done`, so this is always `open` when the skill returns successfully (the close-out flip to `done` is owned by `/quo-fix-issue`, not by this skill).
   - **`action`** — exactly one of two values: `created` for a freshly-filed Issue, `reused-existing` when the dedupe path returned an existing ticket. The vocabulary is exactly `created` and `reused-existing` (lowercase, hyphenated). This vocabulary is published here and consumed by the dedupe path implemented in sibling Task 3 — the two MUST agree on the exact spelling.

4. **Behavioral guarantees.** Include an explicit `User-facing AskUserQuestion gates still fire on the inline path` bullet naming, at minimum:
   - The in-conversation capture path's distill-vs-restart approval gate (Step 1a's distilled-draft `Approve`/`Revise`/`Cancel` gate). Note: although a Skill-tool `url:` payload routes through the External-reference branch, the in-conversation capture path's gate is named here for completeness — if a future inline path drops the URL flag, the capture-gate is still expected to fire.
   - The External-reference branch's body-confirmation step (the path where the user is asked in prose for a one- or two-sentence summary when neither prior conversation nor `WebFetch` produces a useful summary, per the existing External-reference branch sub-step A.3).
   - The dedupe disambiguation gate added by sibling Task 3 (when the dedupe lookup matches an existing open Issue, the user is asked via `AskUserQuestion` whether to use the existing ticket, file anyway, or cancel).

   The bullet MUST state that the Skill-tool caller does NOT short-circuit any of these gates — the user owns the approval, not the caller.

5. **Idempotency.** Include explicit prose stating that re-running the skill against the same URL produces an idempotent result — a duplicate file does NOT create a duplicate ticket. The dedupe path returns the existing ticket ID with `action=reused-existing`. This is the observable behavior callers and reviewers can verify by invoking the skill twice in a row against the same URL. (This idempotency claim depends on sibling Task 3's dedupe implementation; the contract published here and the dedupe behavior in Task 3 must match.)

6. **Lifecycle.** State that the Issue ticket is created at `status=open` (the only status the Issues hive supports for new tickets) and that the inline path does NOT flip the Issue to `done`. The close-out flip is owned by `/quo-fix-issue` per its Section 6 close-out, not by this skill or its callers.

7. **Cross-reference / Future-maintenance note** (optional, mirroring the precedent's `### Cross-reference` block). State that the gate set named in Behavioral guarantees is keyed off the gates that fire in the existing skill flow — if a future Task adds, removes, or renames a user-facing gate, this contract section MUST be revised to match.

Design-rule preservation:
- No language-specific commands, file extensions, or manifest filenames in the new section (rule 1).
- If any shell snippets are added (none are required for a documentation-only section), they MUST ship as paired POSIX bash + Windows PowerShell blocks (rule 2). The precedent in `skills/quo-write-prd/SKILL.md`'s contract section adds no shell snippets — emulate that.
- Phrasing remains project-neutral (rule 3): the section MUST NOT reference quorum-internal ticket IDs, this repo's paths, or workflow specifics that wouldn't make sense in a downstream installation.
- No scratch-file convention concerns arise (the section authors no `--body-file` payloads).
