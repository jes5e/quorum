## What Needs to Change

In `skills/quo-fix-issue/SKILL.md`:

1. **Frontmatter `argument-hint` update (line 4).** The current frontmatter is:
   ```
   argument-hint: "[<issue-id> | <id1> <id2> ... | all]"
   ```
   Update it to surface the URL form and the mixed-list form alongside the existing forms — for example:
   ```
   argument-hint: "[<issue-id> | <id-or-url> ... | <url> | all]"
   ```
   The exact wording is the engineer's call as long as the URL and mixed-list forms are visible in the hint. The hint is displayed by Claude Code in slash-command discovery surfaces; new-user discoverability is the goal.

2. **Overview / Section 1 prose updates (lines 9-13 and lines 50-55).** Update the `## Overview` bullet list at lines 9-13 to enumerate the new fifth invocation form (URL) and the mixed-list form (IDs and URLs interleaved), with one short example per form. Update Section 1's mode-classification bullets (around lines 52-55) to enumerate the URL token shape and the mixed-list shape alongside the four existing shapes, with one example each. Suggested examples (the engineer may revise wording for project-neutral phrasing):
   - URL-only: `/quo-fix-issue https://github.com/example/repo/issues/123`
   - Mixed list: `/quo-fix-issue b.cnb https://github.com/example/repo/issues/123 b.xet`

   The "do NOT sort — the user's order is intentional; earlier issues may be prerequisites for later ones" invariant at line 55 MUST remain intact verbatim. Updates to Section 1's bullet list MAY add new bullets but MUST NOT modify the line-55 prose.

   Project-neutral example URLs: prefer `https://example.com/issues/123` or `https://github.com/example/repo/issues/123` over real-world URLs. Per design rule 3, no quorum-internal references in the new prose.

3. **File-then-fix transition announcement (new, in Section 1).** When `/quo-fix-issue` is invoked with one or more URL tokens, the skill MUST announce the file-then-fix transition to the user before the URL-resolution sub-step runs. Add prose to Section 1 stating this requirement, with a recommended announcement string (e.g., `"Filing URL(s) as Issue(s) first, then fixing."`). The announcement is informational console output — it is NOT an `AskUserQuestion` gate; it does not block the run. State that the announcement fires only when at least one URL is present in the post-tokenization working list (no announcement on the all-IDs path).

4. **Post-resolution working-list display (new, in Section 1).** After the URL-resolution sub-step from Subtask 1 completes (i.e., all URL tokens have been resolved or soft-failed), and BEFORE the upfront `bees show-ticket --ids` validation pass at line 59, the skill MUST display the post-resolution working list to the user. Specify the display as informational output — a labeled markdown list showing each input position alongside the resolved ID, with URL positions called out as filed-from-URL (and the `action` value from Subtask 1's captured return — `created` vs `reused-existing` — surfaced so the user can see when dedupe matched). Recommended display shape:
   ```
   Post-resolution working list:
   1. b.cnb (input: b.cnb)
   2. b.<new-id> (input: <url>, action: created)
   3. b.xet (input: b.xet)
   ```
   This is a display-only step. NO `AskUserQuestion`. NO ability to re-order. The display lets the user confirm prerequisite ordering survived the substitution; if the user is unhappy with the result they can `Ctrl-C` and re-run with corrected positional order.

   State explicitly that the display fires only when at least one URL token was present in the input (skipped on the all-IDs path).

5. **Existing invocation forms continue to work.** Confirm in the prose updates that all four existing forms — interactive, `all`, single ID, list of IDs — continue to dispatch identically and their `argument-hint` slots and Overview bullets remain. The new prose is purely additive (new bullets, not edits to existing bullets); the line-55 invariant is verbatim-preserved.

Design-rule preservation:
- No language-specific commands, file extensions, or manifest filenames in the new prose (rule 1). Example URLs in the prose use neutral hosts (`https://example.com/...` or `https://github.com/example/repo/...`) — no real-world repos beyond GitHub's generic shape.
- If any shell snippets are added (none are expected for this Subtask — the work is prose, frontmatter, and display formatting), they MUST ship as paired POSIX bash + Windows PowerShell blocks (rule 2).
- Phrasing remains project-neutral (rule 3): no quorum-internal ticket IDs from this Bee in the new prose; the example IDs `b.cnb`, `b.xet` already used in the existing list-mode examples (lines 12, 58) are project-neutral placeholder ticket IDs and may be reused for parallel structure.
- No scratch-file convention concerns arise (no `--body-file` payloads).
