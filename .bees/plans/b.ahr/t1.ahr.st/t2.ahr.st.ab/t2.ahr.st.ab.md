## Context

`/quo-fix-issue` today accepts only bees ticket IDs (`all`, single ID, or a list of IDs) as positional arguments. URLs in argument position are mishandled — Section 1's token classifier rejects them. This Task adds full URL handling to `/quo-fix-issue` by extending its Section 1 token classifier to recognize URLs, dispatching `/quo-file-issue` (Epic 1's published `## Inline invocation via the Skill tool` contract) to file each URL as an Issue, substituting the resolved ticket ID *in place* in the working list, and threading TaskList progress entries for the file-then-fix transition.

This is the second of two Epics in Bee `b.ahr` and the user-facing payoff: after this Task lands, `/quo-fix-issue <url>` and `/quo-fix-issue <id1> <url> <id2>` Just Work end-to-end — file-then-fix in a single command, with the user's prerequisite ordering preserved by in-place substitution.

This Task's scope is bounded to `skills/quo-fix-issue/SKILL.md`. The cumulative-doc subsections (`### Feature:` blocks in `docs/prd.md` and `docs/sdd.md`) and the README skill-table row updates ship in sibling Task 2 of this Epic.

## What Needs to Change

In `skills/quo-fix-issue/SKILL.md`:

1. **Token classifier extension (Section 1, lines 48-67).** Recognize tokens matching `^https?://` as a fifth token-shape category alongside the four existing categories (zero / `all` / single-ID / list-of-IDs). Detection rule: a token is a URL iff it starts with `http://` or `https://` after argument tokenization. The existing list-mode tokenization (split on whitespace and/or commas) is unchanged — URLs space-and-comma-delimit identically.

2. **URL-resolution sub-step (new sub-step in Section 1).** Add a new sub-step that runs:
   - **AFTER** the existing `#### Choose agent model preference` sub-step (lines 69-78) and the `#### Validate isolation strategy` sub-step (lines 80-102) — the user's model-preference and isolation-strategy choices apply to the entire run including the file-then-fix transition; isolating before resolution lets the user opt into a fresh branch that scopes the file-from-URL commits.
   - **BEFORE** the existing `bees show-ticket --ids <id1> <id2> ...` upfront-validation pass (line 59) — so validation operates on the post-resolution ticket-ID list (no URL tokens fed to `bees show-ticket`).

   For each URL token in the working list, the sub-step MUST:
   - Dispatch `/quo-file-issue` via the Skill tool with the URL as the bare-positional argument, consuming Epic 1 Task 2's published inline-invocation contract (`## Inline invocation via the Skill tool` section in `skills/quo-file-issue/SKILL.md`).
   - Capture the returned `issue_ticket_id` from the structured return shape.
   - Substitute the URL token *in place* in the working list with the resolved ID. The list-mode prerequisite-ordering invariant from Section 1 (line 55: "do NOT sort — the user's order is intentional; earlier issues may be prerequisites for later ones") MUST be preserved: a URL appearing at position N in the input list becomes the ticket ID at position N in the resolved list, never appended at the tail.
   - Treat Skill-tool dispatch failures (errors, user-cancel at one of `/quo-file-issue`'s gates, `Cancel` at the dedupe prompt from Epic 1 Task 3) per the existing soft-fail-on-bad-ID pattern at line 59 — drop the failed URL from the working list, report the failure to the user, and continue with remaining tokens. The run errors out only when no valid tokens remain (mirrors the existing "no valid IDs" behavior).

3. **TaskList naming convention extension (lines 263-271).** Add a `file-from-url-<n>` category to the documented naming convention, where `<n>` is the 1-based index of the URL token in the user-supplied positional-argument list. Each entry MUST be marked `pending` when created, `in_progress` when the Skill-tool dispatch lands (`Skill(...)` returns), and `completed` when the new ticket ID is captured (or, on the dedupe `Use existing` path from Epic 1 Task 3, when the matched ticket ID is captured). The existing per-issue TaskList categories (sections 3 / 4 / 6 — `<role>-<issue-id>` shape) MUST be unchanged; the new `file-from-url-<n>` category is additive, prefacing the per-issue entries.

4. **Mixed-list ordering surface display.** After URL resolution completes (before per-issue dispatch begins), the skill MUST display the post-resolution working list to the user as informational output (e.g., a labeled markdown list showing each input position and the resolved ID, with URL positions called out as filed-from-URL). This is a display-only step — NO `AskUserQuestion`, NO ability to re-order. The display lets the user confirm prerequisite ordering survived the substitution; the existing isolation-strategy prompt (which already gates the run) remains the cancel point if the resolution looks wrong.

5. **Frontmatter and Section 1 prose.** Update the `argument-hint` frontmatter to surface the URL and mixed-list invocation forms alongside the existing forms (`all`, single ID, list of IDs). Update Section 1's prose (the section that documents the four existing token-shape categories) to enumerate the fifth (URL) and the mixed-list form, with one example each. The "do NOT sort" invariant prose at line 55 MUST remain intact; extend the surrounding prose to clarify that URL tokens are substituted in place and do not violate the invariant.

6. **Invocation-feedback prose.** When `/quo-fix-issue` is invoked with one or more URL tokens, the skill MUST announce the file-then-fix transition before doing it (e.g., "Filing URL(s) as Issue(s) first, then fixing.") so the user understands why a `/quo-file-issue` step appears in the trace. Place this announcement at the top of the new URL-resolution sub-step, before the per-URL Skill-tool dispatch loop.

7. **Existing invocation forms unchanged.** All five existing argument shapes — interactive (no args), `all`, single ID, list of IDs, and the existing AskUserQuestion gates in Sections 2-6 — MUST continue to dispatch identically to today. The new fifth shape and the new sub-step are additive.

## Why

Without URL handling in `/quo-fix-issue`, the user has to manually run `/quo-file-issue` first to mint a ticket ID, capture the ID, then invoke `/quo-fix-issue <id>` — a clumsy two-step that translates directly into adoption friction when starting from a GitHub Issue / Linear ticket / other web link. Wiring the file-then-fix transition through Epic 1's published Skill-tool contract gives the user a single-command entry point. The in-place substitution preserves the existing prerequisite-ordering invariant in list mode; the soft-fail-on-bad-tokens preserves the existing list-mode validation behavior.
