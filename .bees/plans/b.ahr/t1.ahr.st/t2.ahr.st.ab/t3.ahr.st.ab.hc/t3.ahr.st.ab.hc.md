## Context

`/quo-fix-issue`'s Section 1 token classifier (lines 48-67 of `skills/quo-fix-issue/SKILL.md`) today recognises four token shapes: zero / `all` / single-ID / list-of-IDs. URLs in argument position are mishandled — the upfront `bees show-ticket --ids` validation pass at line 59 rejects them. This Subtask wires URL handling into Section 1: it adds URL as a fifth token shape, adds a new URL-resolution sub-step that dispatches `/quo-file-issue` inline via the Skill tool to convert each URL to an Issue ticket ID, substitutes the resolved ID in place in the working list (preserving user-supplied prerequisite ordering), threads soft-fail handling for Skill-tool dispatch errors / user-cancels, and adds the `file-from-url-<n>` TaskList progress category. Sibling Subtask 2 of this Task owns the user-visible surface (frontmatter, post-resolution display, Section 1 prose, file-then-fix transition announcement); this Subtask owns the wiring.

Dependency: Epic 1 of this Bee (`t1.ahr.5a`) publishes the `## Inline invocation via the Skill tool` section in `skills/quo-file-issue/SKILL.md`. Input: free-text `args` string carrying `url: <url>`. Output: structured return naming `issue_ticket_id`, `issue_status` (always `open`), `action` ∈ {`created`, `reused-existing`}. Behavioral guarantees: user-facing `AskUserQuestion` gates fire on the inline path (distill-vs-restart approval, External-reference body-confirmation, dedupe disambiguation). Re-invoking with the same URL is idempotent — the dedupe path returns the existing ticket ID with `action=reused-existing`. Epic 1 is implemented before this Task in the execution order; consume the published contract verbatim.

The canonical inline-dispatch precedent is `skills/quo-plan/SKILL.md`'s Step 4b (around lines 196-226), which calls the Skill tool with `skill="quo-write-prd"` (and `"quo-write-sdd"`), passes a free-text `args` payload, and captures structured return fields (`prd_ticket_id`, `prd_status`, `action`). Mirror this dispatch shape.

The "do NOT sort — the user's order is intentional; earlier issues may be prerequisites for later ones" invariant inside Section 1 line 55 is load-bearing for prerequisite ordering. URL substitution MUST be in-place (a URL at input position N becomes the resolved ticket ID at position N), never appended at the tail.

## What Needs to Change

In `skills/quo-fix-issue/SKILL.md`:

1. **Token classifier — add URL as a fifth token shape (Section 1, lines 48-67).** Recognise tokens matching `^https?://` as URLs alongside the four existing categories. Detection rule: a token is a URL iff it starts with `http://` or `https://` after the existing whitespace-and-comma argument tokenization (which is unchanged — URLs space-and-comma-delimit identically to ticket IDs). Update Section 1's bullet list of mode classifications to enumerate the URL and mixed-list shapes alongside the existing four. The "do NOT sort" invariant prose at line 55 MUST remain intact verbatim.

2. **New URL-resolution sub-step in Section 1.** Add a new third sub-step under `### 1. Determine which issues to fix`, placed:
   - **AFTER** `#### Choose agent model preference` (lines 69-78) and `#### Validate isolation strategy` (lines 80-102) — the user's model-preference and isolation-strategy choices apply to the entire run including the file-then-fix transition; isolating before resolution lets the user opt into a fresh branch that scopes the file-from-URL commits.
   - **BEFORE** the existing `bees show-ticket --ids <id1> <id2> ...` upfront-validation pass at line 59 — so validation operates on the post-resolution ticket-ID list (no URL tokens fed to `bees show-ticket`).

   Sub-step heading: `#### Resolve URL tokens to Issue tickets via /quo-file-issue` (or near-equivalent). Sub-step body MUST:
   - For each URL token in the working list, dispatch `/quo-file-issue` via the Skill tool with `args` set to a free-text payload of the shape `url: <url>`. Mirror the dispatch shape from `skills/quo-plan/SKILL.md` Step 4b. Cross-reference Epic 1's published `## Inline invocation via the Skill tool` section in `skills/quo-file-issue/SKILL.md` as the consumed contract.
   - Capture from the structured return: `issue_ticket_id`, `issue_status`, `action` (∈ {`created`, `reused-existing`}). Document that all three fields are read; the `action` value is informational (used in Subtask 2's post-resolution display) and not load-bearing for the substitution itself.
   - Substitute the URL token *in place* in the working list with the resolved `issue_ticket_id`. Spell out the in-place semantics: a URL appearing at position N in the user-supplied input becomes the ticket ID at position N in the resolved list, NEVER appended at the tail. State that this preserves the line-55 "do NOT sort" prerequisite-ordering invariant.
   - Treat Skill-tool dispatch failures (errors raised by the Skill tool, user-cancel at one of `/quo-file-issue`'s `AskUserQuestion` gates including the dedupe `Cancel` choice from Epic 1's Task 3, or any non-success structured return) per the existing soft-fail-on-bad-ID pattern at line 59: drop the failed URL from the working list, report the failure to the user, and continue with the remaining tokens. The run errors out only when no valid tokens remain after URL resolution AND the subsequent `bees show-ticket --ids` validation pass at line 59 (i.e., the failure check is cumulative across both gates, mirroring line 59's existing "if *no* IDs are valid, exit with an error" clause).

3. **TaskList naming convention — add `file-from-url-<n>` category (lines 263-271).** Append a new bullet to the issue-scoped naming-convention list:
   - **URL-resolution Skill-tool dispatches** — Name: `file-from-url-<n>`, where `<n>` is the 1-based index of the URL token in the user-supplied positional-argument list (e.g., `file-from-url-1`, `file-from-url-2` for two URLs). Each entry is created `pending` when the orchestrator decides to dispatch `/quo-file-issue`, set `in_progress` when the Skill-tool dispatch lands, and set `completed` when the new ticket ID is captured (or, on the dedupe `Use existing` path from Epic 1's Task 3, when the matched ticket ID is captured). State that the dispatch-failure / user-cancel soft-fail path also closes out the entry — mark it `completed` (with the failure reason recorded via `metadata.activity` per the existing prose around line 261 if useful) and clear it from the active set.

   The existing per-issue TaskList categories (Sections 3 / 4 / 6 — `<role>-<issue-id>` shape) MUST remain unchanged; the new `file-from-url-<n>` category is purely additive. The existing "deterministic so two concurrent invocations cannot collide and unambiguous so any reader can map a TaskList entry back to its Issue" framing (lines 265-266) MUST extend to the new category — the 1-based positional index `<n>` is what makes it deterministic and unambiguous (not the URL itself, which may be long, repeat after dedupe, or contain reserved characters).

4. **Existing invocation forms unchanged.** All four existing argument shapes — interactive (zero tokens), `all`, single ID, list of IDs — and the four existing token-classifier branches MUST continue to dispatch identically. The new fifth shape and the new sub-step are purely additive.
