## What this Epic delivers

Rewrite `skills/bees-fix-issue/SKILL.md` using the same reconciliation-loop + ephemeral-Agent pattern from Epic B, applied at issue scope.

## Acceptance criteria

1. `bees-fix-issue/SKILL.md` no longer contains any of: `TeamCreate`, `TeamDelete`, `Agent Teams`, `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`, `force_clean_team`, `check_agent_teams`, `teammateMode`, `metadata.blocked_on`, "graduated escalation".
2. Skill prose references the new subagent types by name.
3. Same reconciliation-loop, warm/cold, and TaskList-progress patterns as Epic B, scoped to one issue per cycle (vs. one Epic per cycle).
4. Single-issue, list, and `all` modes all work as before — no UX regression in mode selection.
5. Optional-PM behavior preserved: simple fixes skip PM, complex fixes include it. The decision logic moves from `bees-fix-issue` skill prose to a small orchestrator-direct check (no Agent needed for the decision itself).
6. **Demo:** `/bees-fix-issue <id>` runs end-to-end on a real (or stand-in) issue without manual prods. Run also for `all` mode against 2-3 dummy issues to validate batch-mode behavior.

## Codebase-green check

`bees-fix-issue` runs end-to-end. `bees-execute` (post-Epic B) and `bees-breakdown-epic` (still on Agent Teams pre-Epic D) unaffected. Other skills unaffected.

## Implementation notes

- **Issue-scoped agent names.** `engineer-<issue-id>` (e.g., `engineer-veq` for issue b.veq). Mirror Epic B's scoping pattern.
- **No `.bees/` state mutation in commits.** Current behavior preserved — only the issue's actual code/test/doc changes get committed; no ticket-status diff is staged. (This differs from `bees-execute` which DOES stage `.bees/` because Tasks/Subtasks have status changes.)
- **Branch-isolation prompt** (current lines ~83-100): preserved as orchestrator-direct AskUserQuestion — no Agent involved.
- **Pre-validation of issue IDs** (batch mode) preserved as orchestrator-direct — no Agent involved.
