## Context

Section 7 (lines 393-417 pre-rewrite) of `skills/quo-fix-issue/SKILL.md` handles per-issue close-out: shutdown agents, mark issue done, format + git commit, `TeamDelete` + `force_clean_team` fallback, output summary. This Task removes the team-lifecycle residue and preserves the commit procedure with **explicit no-`.bees/`-staging discipline**, which differs from `quo-execute`'s commit procedure. Pattern reference: the post-rewrite `skills/quo-execute/SKILL.md` Section 4.1, with intentional divergence at the staging-rule layer.

## What Needs to Change

### DELETE team-lifecycle residue

- DELETE: "Send shutdown requests to all agents on the team" (line 397) — cold-dispatched Agents already exited.
- DELETE: `TeamDelete` + `force_clean_team` fallback prose (line 404) — no team to delete; no helper to invoke.

### PRESERVE issue-status mark-done step

Mark the issue's bees ticket `status=done` (line 398) before committing, so any out-of-band ticket-state propagation is consistent. (The status flip itself is a metadata change handled by the bees CLI; it does not produce a working-tree change that needs staging in this skill — see "no-`.bees/`-staging" below.)

### PRESERVE commit procedure with explicit no-`.bees/`-staging discipline

Adapt the staging procedure (lines 399-403):

1. Run the **Format** command from CLAUDE.md `## Build Commands` to normalize formatting.
2. Run `git status` to see the full set of modified and untracked files.
3. Stage files related to this issue's actual code, test, and doc changes — agent-reported files plus formatting changes to files that were touched by this issue's Agents.
4. **Explicit no-`.bees/`-staging note.** Unlike `quo-execute` (which DOES stage `.bees/` because Tasks/Subtasks accumulate status changes that must land in the per-Task commit), `quo-fix-issue` does NOT stage `.bees/`. Issues live in the `issues` hive and have no in-commit ticket-status accumulation; the issue's status flip from `open` → `done` is captured by the bees CLI metadata transition and does not need a working-tree commit. Preserve this current behavior — do NOT model after `quo-execute`'s staging rule.
5. **Do NOT blindly `git add -A`.** Other agents or background processes may have in-flight changes in the working tree. Review each modified file and only stage it if it's plausibly related to this issue.
6. Commit with a descriptive message per system/project git guidance. NEVER push to remote — committing only.

### Preserve the per-issue summary block

Lines 407-415: the Markdown summary block format is preserved unchanged in shape. The `**Ignored Review Feedback**` field is consumed from Task 3's ignored-feedback discipline.

### Preserve the batch-vs-single mode flow control

Line 417: in batch mode (`all` or list mode), proceed to the next issue (loop back to step 2 / Validate Issue). In single mode, continue to step 8 (Post-Completion Review).

### Mark per-issue TaskList tasks `completed`

After the commit lands, mark the per-issue TaskList tasks (`engineer-<issue-id>`, `test-writer-<issue-id>`, `doc-writer-<issue-id>`, `pm-<issue-id>` if dispatched, plus the reviewer tasks from Task 3) as `completed` and clear them from the active set. There is no Agent shutdown to perform — the per-issue cold dispatches already complete-and-exit when each Agent returns.
