## Context

Section 3 (lines 125-321 pre-rewrite) of `skills/bees-fix-issue/SKILL.md` is the heart of the Agent Teams substrate: complexity assessment + team formation + lifecycle prose + message-flow choreography + pre-dispatch state check + verbatim-ticket-body rule + `blocked_on` signal protocol + inline role prose for Engineer / Test Writer / Doc Writer / PM. This Task replaces the team-driven flow with a reconciliation loop that dispatches fresh ephemeral background `Agent` invocations per role, with all role prose externalized to `agents/<role>.md`. Pattern reference: the post-rewrite `skills/bees-execute/SKILL.md` Section 3 (already shipped under Epic 8s).

## What Needs to Change

### Preserve orchestrator-direct complexity assessment

Keep the Simple-vs-Complex classification and rule-set verbatim (single-file change / mechanical refactor / test-only / doc-only → simple; 3+ source files / public API or proto or storage schema / auth or security or concurrency / PRD-or-SDD-described behavior / non-obvious side effects → complex). The decision logic lives in skill prose — no Agent is dispatched to make this call. PM is dispatched only for complex fixes; for simple fixes the PM dispatch is skipped.

### Replace team formation with per-role ephemeral background dispatch

For each role determined by the complexity decision, dispatch a fresh ephemeral background `Agent` invocation:

- Engineer (when source code changes): `Agent(subagent_type="engineer", run_in_background=true, prompt=<dispatch prompt>)`
- Test Writer (when tests change): `Agent(subagent_type="test-writer", run_in_background=true, prompt=...)`
- Doc Writer (always — Doc Writer decides if docs need updating): `Agent(subagent_type="doc-writer", run_in_background=true, prompt=...)`
- Product Manager (complex fixes only, gated by the orchestrator-direct check): `Agent(subagent_type="pm", run_in_background=true, prompt=...)`

### Issue-scoped agent naming for TaskList progress UI

Each dispatched Agent gets a corresponding TaskList task using the convention `<role>-<issue-id>` — e.g., `engineer-veq` for issue `b.veq`, `pm-veq` for the PM on the same issue. State transitions follow the same pattern as the rewritten `bees-execute`: `pending` (queued) → `in_progress` (dispatched) → `completed` (Agent reported done; bees ticket transitioned to `status=done` where applicable). Issue-scope is the equivalent of Subtask-scope in `bees-execute` — there are no sub-issue work units in `bees-fix-issue`.

### Reconciliation loop tick

Same three-phase tick as the rewritten `bees-execute` Section 3:

1. **Read state.** bees ticket state (`bees show-ticket`, `bees execute-freeform-query`) + TaskList for in-flight Agents + git state.
2. **Reconcile.** Dispatch ready Agents (any role whose precondition is met and that has no in-flight Agent for this issue); persist completions (mark TaskList `completed`, advance to next role); when all implementer Agents return, advance to Section 4 (Review Loop).
3. **Yield.** Event-driven only — no `/loop`, `ScheduleWakeup`, `CronCreate`, polling. The harness fires the next tick on Agent completion notifications or user input.

Replicate the "Anti-pattern: no clock primitives" subsection from the rewritten `bees-execute/SKILL.md` Section 3.

### Cold dispatch only — vs SDD's warm-Agent intent

Mirror the rewritten `bees-execute/SKILL.md` Section 3's "Per-Subtask cold dispatch (vs SDD's warm-Agent intent)" subsection at issue scope. Reference the same divergence-tracking Issue **`b.x9w`** ("Re-probe SendMessage-without-Agent-Teams as warm-Agent token-cost optimization"). Cold dispatch is the consequence of removing Agent Teams: `SendMessage` is unavailable without `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`, so warm-Agent dispatch is not possible in this substrate.

### Hub-and-spoke via substrate, no recursive delegation

Replicate the "Hub-and-spoke via substrate" and "Recursive delegation: not supported" subsections from the rewritten `bees-execute/SKILL.md` Section 3. Per the [Claude Code sub-agents docs](https://docs.claude.com/en/docs/claude-code/sub-agents), "Subagents cannot spawn other subagents" — only the top-level orchestrator may dispatch Agents. Workers do not message each other; the diff is the handoff between roles.

### Dispatch prompt: quote the issue body verbatim

Preserve the verbatim-quoting rule from current lines 199-207 — the rule itself stays load-bearing, but it must be relocated and reframed: it now governs the orchestrator's Agent dispatch prompt construction, not a team-lead message-bus dispatch. Read the issue via `bees show-ticket --ids <issue-id>` and embed the returned body in the dispatch prompt as a quoted block. Do not summarize, paraphrase, or "clean up" identifier spellings. Framing prose around the quoted block (e.g., "your gating precondition is met — start now") is fine; the body itself stays untouched.

### Scoped-marker PM dispatch wiring (Path B)

When dispatching the PM (complex-fix path only), the dispatch prompt must include a `<scoped-marker-resolver-path>` substitution resolved at runtime via `<this skill's base directory>/../bees-breakdown-epic/scripts/scoped_marker_resolver.py`. The placeholder lets `agents/pm.md` run **Path B** (`up_dependencies`-based opportunistic discovery, best-effort fallback) — Path A is for `bees-execute` (Grandparent Bee, hard-fail). The dispatch prompt's context must signal Path B selection: it names an Issue ID and the issue's `up_dependencies` array, with no Grandparent Bee. The orchestrator does NOT inline marker grammar, parsing, or the helper invocation itself — `agents/pm.md` already owns those (lines ~88-142 of the post-Epic-A pm.md).

Provide POSIX bash and Windows PowerShell snippets that show the path the orchestrator embeds in the PM dispatch prompt, mirroring the rewritten `bees-execute/SKILL.md` Section 3's "Scoped-marker PM dispatch wiring" subsection.

- Team lifecycle subsection (lines 153-163) — `TeamCreate`, `TeamDelete`, `force_clean_team` references.
