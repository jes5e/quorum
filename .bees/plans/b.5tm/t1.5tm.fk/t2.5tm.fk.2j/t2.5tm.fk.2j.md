## Context

Section 4 (lines 323-374 pre-rewrite) of `skills/quo-fix-issue/SKILL.md` forms a review Team after the implementation team finishes, with a graduated escalation ladder (~10 min / ~20 min / ~30 min nudges + ps-check verification) for idle teammates. This Task replaces Team formation with three concurrent ephemeral reviewer `Agent` dispatches, gated by which implementer ran during the issue's implementation pass. Pattern reference: the post-rewrite `skills/quo-execute/SKILL.md` Section 5.

## What Needs to Change

### Replace review Team formation with three concurrent ephemeral reviewer Agents

Same dispatch shape as the rewritten `quo-execute/SKILL.md` Section 5. Issue-scoped TaskList naming consumed from Task 2:

- `Agent(subagent_type="code-reviewer", run_in_background=true, prompt=...)` → TaskList task `code-reviewer-<issue-id>`
- `Agent(subagent_type="test-reviewer", run_in_background=true, prompt=...)` → TaskList task `test-reviewer-<issue-id>`
- `Agent(subagent_type="doc-reviewer", run_in_background=true, prompt=...)` → TaskList task `doc-reviewer-<issue-id>`

Reviewer role contracts (responsibilities, model assignment, gating, Skill-tool dispatch instructions) live in the role files; the orchestrator's job is to dispatch, not to carry the role's prose. Drop the inline reviewer Instructions blocks (lines 347-368).

### Conditional spawn — only dispatch a reviewer whose corresponding implementer ran

If the Engineer Agent ran during this issue's implementation pass → dispatch the code-reviewer Agent. If the Test Writer ran → dispatch the test-reviewer. If the Doc Writer ran → dispatch the doc-reviewer. Mirror the rewritten `quo-execute/SKILL.md` Section 5's conditional-spawn prose. The corollary: a doc-only fix dispatches only the doc-reviewer (no code-reviewer or test-reviewer); a code+test fix without a Doc Writer pass dispatches only code-reviewer + test-reviewer.

### Replace graduated escalation with reconciliation-loop tick

DELETE the "Don't wait silently on idle teammates — graduated escalation" subsection (lines 330-345), including the four-rung ladder (~10 min / ~20 min / ~30 min / proceed-and-log) and the ps-check verification block (`ps -ef | grep`, `Get-Process | Where-Object`, `tasklist | findstr`). The reconciliation-loop tick (defined in Task 2) replaces this — the orchestrator yields after dispatching the reviewer Agents and the harness fires the next tick on completion notifications. There is no idle teammate to escalate to; cold-dispatched Agents either return or are still running.

### Iteration loop: replace "reform the first team" with re-dispatching fresh ephemeral implementer Agents

Lines 369-374 currently say "reform the first team" when reviewer feedback warrants follow-up work. Replace with re-dispatching fresh ephemeral implementer Agents per the dispatch shape established in Task 2 (Engineer / Test Writer / Doc Writer / PM as needed). The PM re-dispatch follows the same complex-vs-simple gate from Task 2 (skip PM if the original fix was simple).

### Preserve the ignored-feedback-must-go-in-summary discipline

Lines 373-374 establish that ignored reviewer feedback MUST be included in the per-issue summary block (Task 4 / Section 7). Preserve verbatim. The Summary block in Section 7 has a `**Ignored Review Feedback**` field that consumes this list.

## Why

Without this rewrite, the review loop's reviewer dispatches and idle-handling prose remain incompatible with the cold-dispatch substrate established in Task 2.

## Cross-Task contracts

- **Issue-scoped TaskList naming** consumed from Task 2 — reviewer Agents use the `<role>-<issue-id>` convention.
- **Conditional-spawn rule** mirrors Task 2's role-gate logic: if Task 2's implementer dispatch did not run a particular implementer (e.g., a doc-only fix did not dispatch the Engineer), Task 3 must not dispatch the corresponding reviewer (no code-reviewer for a doc-only fix). The conditional rule must be expressed in skill prose, mirroring the post-rewrite `quo-execute` Section 5 phrasing.
- **PM re-dispatch on iteration** follows Task 2's complex-vs-simple gate. Re-dispatching the PM unconditionally would drift from the Epic's "optional-PM behavior preserved" acceptance criterion.
- **Three subagent type names** (`code-reviewer`, `test-reviewer`, `doc-reviewer`) match the canonical list from Task 1's Preconditions section.
