## Context

This Subtask is the source-side rewrite for Task 1 of Epic t1.5tm.fk. Epic 8s shipped the canonical pattern in `skills/bees-execute/SKILL.md` Preconditions section (Subtask t3.5tm.8s.6z.4w); this Subtask mirrors it at issue scope for `skills/bees-fix-issue/SKILL.md`. The substrate switch from Agent Teams to ephemeral background `Agent` invocations against the seven custom subagent types Epic A authored under `agents/` makes the current `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` precondition obsolete and replaces it with a check that the seven required subagent types are registered in the running Claude Code session.

Per Epic A's `## Probe results — verified update` finding, custom subagents are loaded by Claude Code at session start only — mid-session installs require a restart or `/agents` to hot-reload. A missing subagent_type at skill invocation therefore almost always means the user installed the agents but did not restart Claude Code, or did not install them at all.

The verification mechanism is the same hybrid the canonical pattern ships:

- **Procedural gate (load-bearing primary).** If a dispatch later in the run hits an `Agent type '<name>' not found`-style error from the Agent tool for any of the seven required subagent types, the orchestrator STOPS, emits the hard-fail message, and exits — no fallback to `general-purpose`, no skipping the dispatch, no improvising substitutes.
- **Upfront fast-fail (opportunistic, belt-and-braces).** Before any dispatch is attempted, run `claude agents` to enumerate registered subagents and verify each of the seven literal names is present. The pattern is already shipped and verified in `skills/bees-execute/SKILL.md`; mirror it.

This Subtask only edits `skills/bees-fix-issue/SKILL.md`. The on-disk `check_agent_teams.py` helper stays in place — its deletion is downstream-Epic scope.
