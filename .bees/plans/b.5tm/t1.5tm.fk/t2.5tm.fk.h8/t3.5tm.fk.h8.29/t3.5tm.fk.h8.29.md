## Context

This Subtask is the first of four in Task 6 of Epic t1.5tm.fk. PRD AC #6 demands an end-to-end demo of the rewritten `bees-fix-issue` covering both single-issue mode and `all` mode (or list mode against 2-3 dummy issues to validate batch-mode behavior). PRD AC #4 requires the optional-PM behavior to be preserved (simple skips PM, complex includes PM).

This Subtask identifies or authors the fixture Issue(s) the next two demo Subtasks will run against. Three open Issues already exist in the repo (`b.dkw`, `b.ehy`, `b.x9w`); the executing Engineer may use those as fixtures, or may author small stand-in Issues specifically for the demo. Either is acceptable as long as the fixtures are:

- **Small enough to complete cleanly** (single-file change, mechanical refactor, doc-only, or test-only fix).
- **Large enough to exercise the substrate**: at least one Agent dispatch (Engineer or Doc Writer), the Section 4 reviewer loop, and the Section 7 commit.
- **A mix of simple and complex** if practical, so the optional-PM gate is exercised across the batch (e.g., one doc-only, one source-code-only, one mixed). Complexity classification follows the rule-set in Section 3 (single file → simple; 3+ files / public API / proto / storage schema / auth / security / concurrency / PRD-or-SDD-described / non-obvious side effects → complex).

This Subtask is research/observation; no source files are modified beyond optional fixture-Issue creation via the bees CLI.

## What Needs to Change

- Decide whether to use existing open Issues (`b.dkw`, `b.ehy`, `b.x9w`) or author small stand-in fixtures specifically for the demo. Document the choice.
- If authoring fixtures, create them via the bees CLI in the `issues` hive (e.g., a small README typo fix, a one-line config change, a docstring correction). Each fixture's body should follow the same shape as a normally-filed issue (Context, What Needs to Change, Acceptance Criteria) so the orchestrator and dispatched Agents have enough to act on without manual prods.
- Capture the fixture Issue IDs (with statuses at start) into a notes artifact for Subtask 6.4 to consume.
- Aim for a small mix: ideally 2-3 fixture Issues spanning at least one simple (no PM expected) and at least one complex (PM expected) fix. If only existing Issues are used, document each one's complexity classification so Subtasks 6.2-6.3 can predict the optional-PM behavior.

Do NOT modify any SKILL.md, helper script, README, SDD, or `agents/<role>.md` during this Subtask. Do NOT run `/bees-fix-issue` itself yet — that is Subtasks 6.2 and 6.3.

## Key Files

- No source files modified. Optional new fixture Issues created in the `issues` hive via the bees CLI.

## Acceptance Criteria

- A list of fixture Issue IDs (existing or newly authored) is captured for Subtasks 6.2 and 6.3 to consume.
- For each fixture, the complexity classification (simple or complex) is recorded so the optional-PM behavior can be predicted in Subtasks 6.2-6.3.
- The fixture set spans at least 2 issues for `all`/list-mode validation, ideally 3.
- If new fixture Issues were authored, their initial statuses (`open`) are recorded so Subtask 6.4 can confirm they were marked `done` after the demo.
- No SKILL.md, helper script, README, SDD, or `agents/<role>.md` file was modified.
