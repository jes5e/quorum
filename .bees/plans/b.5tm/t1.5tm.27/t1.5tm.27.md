## What this Epic delivers

Author the seven role-specific subagent definitions that the rewritten execution skills will reference, and extend the install instructions so they ship to the right place.

- Top-level `subagents/` directory at the repo root
- One `<role>.md` file per role:
  - `engineer.md` (model: opus; tools: Bash, Edit, Read, Write, Grep)
  - `test-writer.md` (model: opus; tools: Bash, Edit, Read, Write, Grep)
  - `doc-writer.md` (model: opus by default — downstream user picks at /bees-execute time; tools: Read, Edit, Write, Grep)
  - `pm.md` (model: configurable; tools: Bash, Read, Skill, Grep)
  - `code-reviewer.md` (model: opus; tools: Bash, Read, Grep, Skill)
  - `doc-reviewer.md` (model: configurable; tools: Bash, Read, Grep, Skill)
  - `test-reviewer.md` (model: opus; tools: Bash, Read, Grep, Skill)
- Each file: YAML frontmatter (`name`, `description`, `model`, `tools`) + body with role-specific instructions lifted from the current inline blocks in `skills/bees-execute/SKILL.md` (lines ~290-389 — Engineer, Test Writer, Doc Writer, PM Instructions) and `skills/bees-fix-issue/SKILL.md` (analogous blocks). Strip Agent Teams-specific prose: `metadata.blocked_on` convention, "self-trigger because team-lead might miss your ping," any references to `TaskUpdate metadata.blocked_on` or the graduated escalation ladder.
- README's `## Install` section (Option A POSIX, Option A PowerShell, Option B POSIX, Option B PowerShell — four sites) extended with `cp` / `Copy-Item` lines for `subagents/` alongside the existing `skills/` lines. Two-tier paths: `~/.claude/subagents/` (global) and `<repo>/.claude/subagents/` (per-project).
- Probe verifies that an `Agent` invocation against a custom subagent type loads correctly when installed in `~/.claude/subagents/`. Document the result (success or failure mode) in this Epic's done-status notes — that result feeds Epic B's recursive-delegation probe and informs whether reviewers/PM use `Skill` or invoke review skills directly.

## Post-Step-8-review correction (b.51d polish, commit 9c97dd1, 2026-05-03)

The Step 8 post-completion reviewer of b.51d flagged that the `bees-fix-issue` Scoped-marker check should not stay a no-op — it should opportunistically wire through the existing `up_dependencies` field. That fix landed in commit `9c97dd1` ("Polish b.51d Step 8 review items"). **It supersedes the bees-fix-issue-portion of the cross-cutting note above.**

**Correction to the `pm.md` lift target.** Instead of carrying a "documented no-op" note from `bees-fix-issue/SKILL.md`, `pm.md` must now carry the **active `up_dependencies`-based opportunistic marker discovery procedure** from the post-9c97dd1 `bees-fix-issue/SKILL.md` (around lines 275-305). The procedure:

1. After validating `up_dependencies` blocker statuses, iterate the array and for each ID query whether it resolves to a Bee in the `plans` hive (using `bees execute-freeform-query` with `[id=<dep-id>, type=bee, hive=plans]`).
2. For each Plans-hive Bee found, extract `tickets[0].body` from the query result (do NOT dump the whole JSON envelope — escaped `
` in the body string would silently break the marker line-scan) and write it to a temp file via the `Write` tool.
3. Run `<base>/../bees-breakdown-epic/scripts/scoped_marker_resolver.py` against that temp file.
4. If the helper reports `scoped: true`, scope spec docs to the matching `### Feature: <title>` subsection. If `scoped: false`, malformed, missing-doc, or missing-heading: **best-effort fallback** to full-doc spec content. **First entry with a valid scoped marker wins** on tie-break.
5. Clean up the temp file with paired POSIX (`rm`) + PowerShell (`Remove-Item`) snippets after each helper invocation.

**Dual-use of `up_dependencies`.** The procedure deliberately reuses `up_dependencies` for both its primary blocker role AND optional scope-context discovery. `pm.md` must carry this dual-use note explicitly so future maintainers do not conflate the two roles silently.

**SDD impact.** SDD's `bees-fix-issue` Key components tag was updated from "Marker consumer (no-op today)" to "Marker consumer (via up_dependencies-link)" in the same commit. Epic F's planned SDD updates must preserve the new tag, not regress it.

The originally-planned `pm.md` tool allowlist (Bash, Read, Skill, Grep, plus Write per the cross-cutting note above) is unchanged by this correction — the up_dependencies discovery uses Bash + Write, both already on the list.
