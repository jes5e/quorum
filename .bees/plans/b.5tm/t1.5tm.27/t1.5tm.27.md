## What this Epic delivers

Author the seven role-specific subagent definitions that the rewritten execution skills will reference, and extend the install instructions so they ship to the right place.

- Top-level `subagents/` directory at the repo root
- One `<role>.md` file per role:
  - `engineer.md` (model: opus; tools: Bash, Edit, Read, Write, Grep)
  - `test-writer.md` (model: opus; tools: Bash, Edit, Read, Write, Grep)
  - `doc-writer.md` (model: opus by default — downstream user picks at /bees-execute time; tools: Read, Edit, Write, Grep)
  - `pm.md` (model: configurable; tools: Bash, Read, Skill, Grep)
  - `code-reviewer.md` (model: opus; tools: Bash, Read, Grep, Skill)
  - `doc-reviewer.md` (model: configurable; tools: Bash, Read, Grep, Skill)
  - `test-reviewer.md` (model: opus; tools: Bash, Read, Grep, Skill)
- Each file: YAML frontmatter (`name`, `description`, `model`, `tools`) + body with role-specific instructions lifted from the current inline blocks in `skills/bees-execute/SKILL.md` (lines ~290-389 — Engineer, Test Writer, Doc Writer, PM Instructions) and `skills/bees-fix-issue/SKILL.md` (analogous blocks). Strip Agent Teams-specific prose: `metadata.blocked_on` convention, "self-trigger because team-lead might miss your ping," any references to `TaskUpdate metadata.blocked_on` or the graduated escalation ladder.
- README's `## Install` section (Option A POSIX, Option A PowerShell, Option B POSIX, Option B PowerShell — four sites) extended with `cp` / `Copy-Item` lines for `subagents/` alongside the existing `skills/` lines. Two-tier paths: `~/.claude/subagents/` (global) and `<repo>/.claude/subagents/` (per-project).
- Probe verifies that an `Agent` invocation against a custom subagent type loads correctly when installed in `~/.claude/subagents/`. Document the result (success or failure mode) in this Epic's done-status notes — that result feeds Epic B's recursive-delegation probe and informs whether reviewers/PM use `Skill` or invoke review skills directly.

## Acceptance criteria

1. The user can `cat subagents/engineer.md` (and the other six) and see a YAML frontmatter block followed by role-specific instructions.
2. After installing this repo's `subagents/*` to `~/.claude/subagents/`, an `Agent(subagent_type: "engineer", prompt: "Read README.md and tell me the project's tagline", run_in_background: false)` invocation from a fresh Claude Code session in any repo returns a response that reflects engineer-role behavior.
3. README's `## Install` Option A and Option B both show extended copy commands for `subagents/` in both POSIX and PowerShell variants.
4. Running `/bees-execute`, `/bees-fix-issue`, or `/bees-breakdown-epic` continues to work as before — still using Agent Teams. This Epic is purely additive infrastructure; it does not flip any skill yet.

All 11 existing skills continue to work as today. The change is additive — new directory, new files, README additions only. No existing skill prose is touched in this Epic.

## Implementation notes

- **Tool allowlists.** PM and the three reviewers need `Skill` to invoke `/bees-code-review`, `/bees-doc-review`, `/bees-test-review`. Engineer / Test Writer / Doc Writer don't need `Skill`.
- **Verbatim role-prose lifting.** The current PM Instructions block in `bees-execute/SKILL.md` (~lines 342-389) is large and includes spec-traceability checks, cross-Task interaction checks, and the time-budget short-circuit. Lift faithfully into `pm.md` — do not paraphrase load-bearing prose. The b.hf8 lesson on verbatim ticket-body quoting applies analogously here.
- **Recursive-delegation feasibility is NOT determined in this Epic.** That belongs to Epic B's probe. Epic A only verifies single-level subagent invocation.
- **Frontmatter shape.** Per Claude Code's subagent definition format: `name`, `description`, `model`, `tools` are the four fields the harness reads. Other fields are free-text and ignored by the harness but visible to the subagent itself when it loads its own definition.
