- One `<role>.md` file per role:

  - `test-writer.md` (model: opus; tools: Bash, Edit, Read, Write, Grep)

### Root-cause analysis

- The `Agent` tool's harness lists only built-in subagent types (`claude-code-guide`, `Explore`, `general-purpose`, `Plan`, `statusline-setup`). None of the seven newly-installed custom subagent types are registered.
- The install location (`~/.claude/subagents/`) matches Claude Code's documented convention.
- The frontmatter shape on the installed files is well-formed (validated on `engineer.md`).
- Most parsimonious explanation: Claude Code sessions cache the subagent registry at startup; new files dropped into `~/.claude/subagents/` mid-session are not hot-reloaded. A fresh post-install session is required to validate Acceptance Criterion #2.
- Secondary structural finding (surfaced earlier in this Task before the probes): the `Agent` tool itself is **not exposed** inside Agent Teams teammate processes — only the team-lead's top-level session has `Agent`. This means substrate-2 (`Agent`-based ephemeral subagents) and substrate-1 (Agent Teams) are **mutually exclusive in a single execution stack**. Epic B's recursive-delegation probe author MUST read this before assuming substrate co-existence is possible.

### Implications for downstream Epics

- **Epic A — README install instructions need a "restart Claude Code after copying" step.** Without it, a user running the install commands inside a running session won't see the new subagent types until restart. Recommend filing this as a follow-up Issue or amending Task `na`'s README edit before closing this Bee. This is the single highest-priority follow-up surfaced by the probes.
- **Epic B (`/bees-execute` rewrite)** and **Epic C (`/bees-fix-issue` rewrite)**: when their orchestrator dispatches `Agent(subagent_type="engineer"|"pm"|...)`, the dispatch will only succeed in sessions started after install. - **T2 and T3 contracts**: both currently held as ASSERTED-BUT-UNVERIFIED — Probes 2 and 3 failed at the same upstream blocker as Probe 1, so each individual contract's actual behavior is not distinguished. They all hold pending a post-restart re-probe. Epic B's planning should treat T2 and T3 contracts as ASSERTED-BUT-UNVERIFIED rather than confirmed.
- **Substrate exclusivity finding** (from the secondary structural finding above): Agent Teams (substrate-1) and ephemeral `Agent`-tool subagents (substrate-2) cannot co-exist in a single execution stack. Epic B's rewrite must complete the substrate switch wholesale at session-entry time; partial migration (e.g., dispatching `Agent` calls from inside a still-running team) is structurally impossible.
