## Context

The Doc Writer subagent updates documentation either (in execute mode) by executing pre-planned doc Subtasks plus a diff-review pass over the Engineer's output, or (in fix mode) by reviewing the Engineer's diff for doc gaps and updating ad-hoc. Its current Instructions blocks live in `skills/bees-execute/SKILL.md` Step 3 (Doc Writer entry — note the explicit divergence note vs. `bees-fix-issue`'s Doc Writer) and `skills/bees-fix-issue/SKILL.md`. **Model nuance:** unlike Engineer and Test Writer, the Doc Writer's runtime model is user-chosen (Opus or Sonnet) at the start of a `/bees-execute` or `/bees-fix-issue` run per CLAUDE.md `## Model assignment in execution skills`. The file ships with `model: opus` as the shipped default; downstream Epics B and C document the user-override mechanism when they wire the orchestrator's runtime model selection into the Agent invocation.

## What Needs to Change

- Ensure the top-level `subagents/` directory exists at the repo root (`mkdir -p subagents/` is idempotent).
- Create `subagents/doc-writer.md` with YAML frontmatter and a markdown body.
- Frontmatter:
  - `name: doc-writer`
  - `description`: lead with the verb; state scope as authoring or updating customer-facing and internal architecture docs based on the Engineer's diff or pre-planned doc subtasks, against the project's doc writing guide.
  - `model: opus` (shipped default; runtime override to `sonnet` is handled by the orchestrating execution skill — record the deferral in a body note so the apparent inconsistency with the role's user's-choice model assignment is explained).
  - `tools: [Read, Edit, Write, Grep]`. The Doc Writer does NOT get `Bash` — its work is read/edit/write of doc files only.
- Body lifts the Doc Writer Instructions from `skills/bees-execute/SKILL.md` Step 3 (preferring the execute variant since it covers both pre-planned doc subtasks and the diff-review pass; the fix-issue variant covers only the diff-review pass and is a strict subset). Retain:
  - Use the doc writing guide referenced in CLAUDE.md `## Documentation Locations`.
  - Execute customer-facing docs subtasks; execute internal architecture docs subtasks.
  - Review the Engineer's work for doc gaps the pre-planned subtasks may have missed; update accordingly.
  - Mark each Subtask `status=in_progress` and `status=done` (artifact-based handoff signal — even though Bash is not in the allowlist, status transitions are routed through the orchestrator).
  - The fix-mode-vs-execute-mode divergence note from the existing block (preserved as a short note explaining the role behaves slightly differently when invoked from `bees-fix-issue` (no pre-planned subtasks; pure diff-review and ad-hoc updates) versus `bees-execute` (pre-planned subtasks first, then diff-review)).

This pattern is shared with T2.S3 (doc-reviewer.md) and T3.S1 (pm.md).
