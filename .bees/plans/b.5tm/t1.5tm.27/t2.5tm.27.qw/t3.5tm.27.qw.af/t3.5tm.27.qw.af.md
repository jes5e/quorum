## Context

The Engineer subagent is the implementation worker invoked by the future reconciliation-loop orchestrator (Epic B in `bees-execute`, Epic C in `bees-fix-issue`) to execute implementation Subtasks. Its current inline Instructions block lives in `skills/bees-execute/SKILL.md` Step 3 Engineer entry; an analogous block lives in `skills/bees-fix-issue/SKILL.md`. Lifting this prose into a definition file is the core deliverable. The parent Task's "Agent-Teams-prose-stripping convention" governs what is dropped during the lift.

## What Needs to Change

- Ensure the top-level `subagents/` directory exists at the repo root (`mkdir -p subagents/` is idempotent and fine to run from this subtask even if T1.S2/T1.S3 also create it).
- Create `subagents/engineer.md` with YAML frontmatter and a markdown body.
- Frontmatter:
  - `name: engineer`
  - `description`: lead with the verb. State scope precisely as implementing code changes for an assigned Subtask or set of Subtasks against the project's specs and best-practices guides. Vague descriptions cause mis-invocation; this is the `description` precision rule from `docs/doc-writing-guide.md` `## The frontmatter contract`.
  - `model: opus` (Engineer is always Opus per CLAUDE.md `## Model assignment in execution skills` — not user-configurable).
  - `tools: [Bash, Edit, Read, Write, Grep]`.
- Body lifts the Engineer Instructions block from `skills/bees-execute/SKILL.md` Step 3 (preferring its variant since it covers per-Subtask iteration; the `bees-fix-issue` variant differs in not having pre-planned subtasks). Specifically retain:
  - Read the assigned ticket (Subtask in execute mode, Issue body in fix mode) using the bees CLI.
  - Review internal architecture docs referenced in CLAUDE.md `## Documentation Locations`.
  - Review existing code; review the engineering best practices guide referenced in CLAUDE.md `## Documentation Locations`.
  - Execute implementation following the ticket's Context / What Needs to Change / Key Files / Acceptance Criteria.
  - Mark each Subtask `status=in_progress` when starting, `status=done` when done (artifact-based handoff signal in the new substrate; preserve it).
  - Compile-check discipline: look up `Compile/type-check` from CLAUDE.md `## Build Commands` and run after each subtask; skip rung when entry is empty (interpreted languages); also run `Lint` at narrow scope.
  - Test-scope discipline: Narrow test + Lint while iterating, deferring Full test to the Task's authoritative `.T` subtask. Reference lookup keys by exact contract names (`Compile/type-check`, `Format`, `Lint`, `Narrow test`, `Full test`).
  - Shell-command etiquette block (one literal command per invocation, no diagnostic tails, no `$VAR`/`$(...)`, no compounds, prefer first-class tools, prefer Python helper file over inline `-c`/heredoc).
- Strip during the lift (parent Task's Agent-Teams-prose-stripping convention):
  - The "Self-trigger" bullet (the new substrate is one-shot per Agent invocation; there is no "top of every turn" semantic).
  - The "Surface blocked state via `blocked_on` metadata" bullet and any `TaskUpdate metadata.blocked_on` references.
  - Any named-team-scoped agent-naming prose ("e.g., `engineer-xb`").
  - Any "do not wait for further pings from the team-lead" framing.
- Keep the body project-neutral: no references to this repo's paths, ticket IDs, or workflow specifics. The contract-key lookups to `## Build Commands` / `## Documentation Locations` stay because those keys live in the *target* repo's CLAUDE.md at runtime.
