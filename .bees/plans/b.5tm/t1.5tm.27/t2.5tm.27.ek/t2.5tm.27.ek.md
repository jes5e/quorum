## Context

Epic A acceptance criterion #2 requires verification that `Agent(subagent_type="engineer", prompt="Read README.md and tell me the project's tagline", run_in_background=false)` from a fresh Claude Code session in any repo returns a response that reflects engineer-role behavior after the seven new subagent files are installed. This Task runs that probe and two narrower probes that validate inter-Epic contracts locked by Tasks T2 and T3, then persists the results to this Epic's body so downstream Epics (B's recursive-delegation probe; C's `bees-fix-issue` rewrite) can read them.

The probes are SINGLE-LEVEL ONLY — recursive delegation feasibility is explicitly out of scope per the Epic body and belongs to Epic B's probe.

This Task is split into two subtasks: T5.S1 runs the probes and synthesizes a structured finding; T5.S2 persists the finding to the Epic body via `bees update-ticket`. T5.S2 depends on T5.S1.

## What Needs to Change

Run three probes from a fresh Claude Code session against subagent files installed at `~/.claude/subagents/` per T4's README instructions:

- **Probe 1 — engineer-role behavior** (canonical Acceptance Criterion #2 probe): `Agent(subagent_type="engineer", prompt="Read README.md and tell me the project's tagline", run_in_background=false)`. Verify the response reflects engineer-role behavior (reads README, returns tagline, does not exhibit off-role behavior).
- **Probe 2 — pm subagent with orchestrator-passed helper path** (validates T3.S2 contract): invoke pm with a dispatch prompt that includes a placeholder helper path pointing at a non-standard location (e.g., a copy of `scoped_marker_resolver.py` placed at `/tmp/probe-helper/scoped_marker_resolver.py`). Verify pm.md uses the prompt-passed path verbatim — not a hardcoded path, not a re-derivation. This is the **single most important** sub-probe — without it, T3's helper-resolution-path contract is asserted but not validated.
- **Probe 3 — code-reviewer with `Skill`-tool dispatch** (validates T2 contract): invoke code-reviewer against a small fixture diff. Verify code-reviewer.md invokes `/bees-code-review` via the `Skill` tool from inside a subagent context and returns structured findings.

Synthesize a probe-result finding with these structured fields:
- Per-probe outcome: PASS / FAIL.
- Per-probe failure mode (if FAIL): specific identification using the failure-mode vocabulary.
- Per-probe verbatim or representative response excerpt.
- **Implications for downstream Epics** — note covering helper-resolution-mechanism viability (orchestrator-pass-through vs. fallback) and `Skill`-tool-from-subagent viability (whether reviewers/PM invoke skills via Skill or some alternative).
- **Probe environment provenance**: host OS, Claude Code version, target repo, install command used.
- Explicit note that **warm-Agent / `name:` parameters were NOT tested** (Epic B scope) and **nested Agent invocations were NOT tested** (Epic B scope) — recorded so Epic B's probe author doesn't assume any prior validation.

Persist the finding to the Epic body via a single `bees update-ticket --ids <epic-id> --body <markdown>` invocation. The persistence appends a "Probe results" section; pre-existing Epic body content is byte-for-byte preserved. Do NOT change the Epic's `ticket_status` in this Task — that is the orchestrator's responsibility.

## Why

Acceptance criterion #2 is a hard go/no-go for whether downstream Epics B/C/D can rely on the new substrate. Recording the outcome in the Epic body (rather than only in conversational message history) makes it durable input — Epic B reads the Epic body in a fresh session and consumes the finding without re-running probes or re-deriving outcomes.

## Cross-Task contracts (surfaced for sibling Tasks and downstream Epics)

- **Single-level scope-fence**: T5 probes do NOT spawn nested Agent invocations. The probe artifacts (fixture prompts, fixture content) contain zero `Agent(subagent_type=...)` calls inside any subagent's prompt or body. Recursive-delegation feasibility is sibling Epic B's probe scope.
- **Probe outcome recording schema**: per-probe outcome / failure mode / response excerpt / implications / environment provenance. This shape is what sibling Epic B reads to decide flat-vs-nested orchestration.
- **Failure-mode-feeds-Epic-B contract**: per Epic body, T5 outcomes inform whether reviewers/PM use `Skill` or invoke review skills directly. If Probe 3 (Skill-tool from subagent) FAILS, sibling Epic B/C must adapt their reviewer-wiring strategy. Surface the contingency explicitly in the persisted finding.
- **Three-probe minimum**: engineer (canonical Acceptance Criterion #2), pm (T3.S2 contract validation), code-reviewer (T2 `Skill`-tool contract validation). Other four roles (test-writer, doc-writer, doc-reviewer, test-reviewer) inherit the same shape and are assumed-good if these three pass — exhaustive seven-role probing is low-ROI scope creep.
- **Cross-OS coverage**: POSIX is mandatory. Windows PowerShell coverage is best-effort; if Windows access is unavailable at probe time, document the gap explicitly so sibling Epics B/C/D verify there.
- **Install-state precondition**: probe must run from a fresh CC session in any repo against subagents installed via T4's README instructions. T5 has up_dependencies on T1, T2, T3, T4.
- **Probe artifact location**: fixtures used by probes (fixture Plan Bee body, fixture diff, copy of helper at non-standard path) MUST NOT be committed to `subagents/` or `skills/` — those are install-deliverable directories. Use `/tmp/` or document the location in done-status notes.
- **Helper-script side-effects**: Probe 2 places a copy of `scoped_marker_resolver.py` at a non-standard path for the path-passthrough test. This is a probe artifact, not a contract change — the helper at `skills/bees-breakdown-epic/scripts/scoped_marker_resolver.py` is unchanged.
