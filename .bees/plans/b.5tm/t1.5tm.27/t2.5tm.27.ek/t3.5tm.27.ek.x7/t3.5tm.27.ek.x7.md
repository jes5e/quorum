## Context

Epic A acceptance criterion #2 specifies that the probe result is recorded in this Epic's done-status notes. The bees workflow's persistence point for an Epic-level finding is the Epic ticket body, mutated via `bees update-ticket --ids <epic-id> --body <markdown>`. T5.S1's deliverable is the synthesized finding; this subtask persists it. Persisting the finding (rather than relying on the team-lead's per-Task summary or conversational message history) makes it durable input for Epic B's recursive-delegation probe and informs Epic B/C decisions about whether reviewers and PM dispatch `Skill`-tool invocations versus invoking review skills directly.

This subtask depends on T5.S1.

## What Needs to Change

- Append a "Probe results" section to the Epic ticket body (Epic ID is `t1.5tm.27`) via `bees update-ticket --ids t1.5tm.27 --body-file <path>`. Author the new full body to a temp file via the `Write` tool (the existing body content + the new "Probe results" section appended) and pass `--body-file` to avoid command-injection issues with multi-section markdown.
- The persisted section structure should carry, at minimum:
  - **Probe 1 — engineer-role behavior**: outcome (PASS / FAIL), failure mode if FAIL, representative response excerpt.
  - **Probe 2 — pm orchestrator-passed-path**: outcome (PASS / FAIL), failure mode if FAIL, primary-vs-fallback distinction, representative response excerpt.
  - **Probe 3 — code-reviewer Skill-tool dispatch**: outcome (PASS / FAIL), failure mode if FAIL, representative response excerpt.
  - **Implications for Epics B / C / D**: the implications note synthesized in T5.S1, covering helper-resolution-path-mechanism viability and any downstream-Epic decisions the result informs.
  - **Probe environment**: host OS, Claude Code version, target repo, install command used.
  - **Explicit disclaimers**: warm-Agent / `name:` parameters NOT tested (Epic B scope); nested Agent invocations NOT tested (Epic B scope).
- Preserve the existing Epic body content — append the probe-results section rather than overwriting. The Epic body already contains its scope statement, acceptance criteria, and Task list; the probe-results section sits at the end as a "## Probe results" subheading.
- The finding text should be concrete and self-contained: an Epic-B engineer reading this body in a fresh session must be able to understand the finding without re-running the probes. Avoid prose like "the probe passed" without context — instead, "Probe 1 (engineer-role behavior): PASS. The engineer subagent returned the README's tagline verbatim from a fresh session in `<repo-name>`. Response excerpt: `<...>`."
- The persistence is a single `bees update-ticket --ids t1.5tm.27 --body-file <path>` invocation. Do NOT run multiple update-ticket invocations to layer the section in piece by piece — body update is wholesale, and partial updates risk losing existing body content.
- Remove the temp file after the bees command exits (paired POSIX `rm` and Windows `Remove-Item` cleanup).
- **Do NOT mark the Epic itself as `status=done` in this subtask.** The Epic-status transition is the orchestrator's responsibility at the end of the Epic per `bees-execute`'s Step 4.2 inter-Epic-interaction-checkpoint flow, not this subtask's scope. T5.S2 only updates the body, not the status.
