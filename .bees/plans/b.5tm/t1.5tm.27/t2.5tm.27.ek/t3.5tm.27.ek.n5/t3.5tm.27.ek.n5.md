## Context

Epic A acceptance criterion #2 requires verification that `Agent(subagent_type="engineer", prompt=..., run_in_background=false)` from a fresh Claude Code session in any repo returns a response that reflects engineer-role behavior. Two narrower probes are added to validate inter-Epic contracts locked in T2 and T3: that orchestrator-passed paths reach pm.md intact (T3.S2 contract), and that `Skill`-tool dispatch from inside a subagent body works (T2 contract).

All probes are SINGLE-LEVEL ONLY — recursive delegation is explicitly out of scope per the Epic body and belongs to Epic B's probe.

The probe execution is a future-executor activity (run during `/quo-execute` when this Epic runs); this subtask description tells the future executor what to do.

**Preconditions:**
- T1/T2/T3 have landed all seven subagent files.
- T4 has landed the README install instructions; the executor follows those instructions to install the subagent files.

## What Needs to Change

The future executor should:

- Install the seven `subagents/*.md` files to `~/.claude/subagents/` per T4's Option A README instructions. Use the same shell etiquette this repo enforces (paired POSIX + PowerShell snippets per `docs/doc-writing-guide.md` `## OS-conditional shell blocks`).
- Open a **fresh Claude Code session** in any repo (the quorum repo is a reasonable default since the README is present for the engineer probe). Fresh-session is load-bearing — the only way to verify the subagent files are loaded by Claude Code's startup discovery rather than carried-over context.

- **Probe 1 — engineer-role behavior** (canonical Acceptance Criterion #2 probe):
  - Probe form: `Agent(subagent_type="engineer", prompt="Read README.md and tell me the project's tagline", run_in_background=false)`.
  - Record:
    - Whether the Agent invocation returned at all (vs. immediately erroring with "subagent type not found").
    - The verbatim returned text (or representative excerpt if long).
    - Whether the response reflects engineer-role behavior (reads README, returns tagline, does NOT exhibit off-role behavior such as proposing tests, refusing on a "not in scope" technicality, returning empty/error response).

- **Probe 2 — pm subagent with orchestrator-passed helper path** (validates T3.S2 contract):
  - Place a copy of `scoped_marker_resolver.py` at a non-standard path (e.g., `/tmp/probe-helper/scoped_marker_resolver.py`).
  - Invoke pm with a dispatch prompt that includes the placeholder helper path pointing at that non-standard location AND a small fixture Plan Bee body (with or without a marker — either is valid for this probe).
  - Probe form: `Agent(subagent_type="pm", prompt="<dispatch prompt with scoped_marker_resolver_path: /tmp/probe-helper/scoped_marker_resolver.py and a fixture Bee body>", run_in_background=false)`.
  - Record:
    - Whether pm.md USED the prompt-passed path (not a hardcoded path, not a re-derivation). Verify by inspecting the response/transcript for the path it actually invoked.
    - Whether the helper invocation succeeded against the fixture and pm.md returned scoped or unscoped behavior consistent with the fixture's marker presence/absence.
    - Whether pm.md fell back to the canonical-install-location strategy (defensive belt-and-braces) — if so, that's a partial pass: the primary mechanism failed, but the fallback worked. Distinguish between primary-path success and fallback success in the recording.

- **Probe 3 — code-reviewer with `Skill`-tool dispatch** (validates T2 contract):
  - Construct a small fixture diff (e.g., a one-file Python or Markdown change in `/tmp/probe-fixture/`).
  - Probe form: `Agent(subagent_type="code-reviewer", prompt="<dispatch prompt scoped to the fixture diff>", run_in_background=false)`.
  - Record:
    - Whether code-reviewer.md invoked `/bees-code-review` via the `Skill` tool from inside the subagent context.
    - Whether the underlying review skill loaded and returned findings.
    - Whether the response was structured findings (severity tags + file:line + suggested fixes) or some other shape.
    - If `Skill` from inside subagent FAILS, record exact error message and harness state — this is critical signal for Epic B/C reviewer-wiring strategy.

- **Synthesize a probe-result finding** with structured fields:
  - Per-probe outcome: PASS / FAIL.
  - Per-probe failure mode (if FAIL): specific identification — `subagent type not found` / `prompt not delivered` / `response empty` / `response content unrelated to role` / `response indicates wrong tools accessible` / `path string mangled` / `path string dropped from prompt context` / `subagent did not recognize placeholder semantic` / `Skill tool unavailable from subagent context` / `underlying review skill failed to load from subagent context` / other (describe).
  - Per-probe verbatim or representative response excerpt.
  - **Implications for Epics B / C / D** — note covering helper-resolution-mechanism viability (orchestrator-pass-through vs. fallback) and `Skill`-tool-from-subagent viability.
  - **Probe environment provenance**: host OS (POSIX or Windows PowerShell), Claude Code version, target repo, install command used.
  - Explicit disclaimers: **warm-Agent / `name:` parameters were NOT tested** (Epic B scope) and **nested Agent invocations were NOT tested** (Epic B scope) — recorded so Epic B's probe author doesn't assume any prior validation.
- The synthesized finding is the deliverable handed to T5.S2 for persistence.

**Scope fences:**
- Probes are SINGLE-LEVEL ONLY. Do NOT invoke any prompt that asks a subagent to spawn another Agent. Recursive-delegation is Epic B's scope.
- Do NOT extend the probe to all seven subagent types. Three is enough — engineer (canonical AC#2), pm (T3.S2 contract), code-reviewer (T2 contract). The other four roles inherit the same shape and are assumed-good if these three pass.
- Do NOT modify any subagent file or SKILL.md based on probe results. If a subagent file is broken, file the finding in T5.S2's Epic-body record; remediation is a follow-up Epic concern.
- Probe fixtures (`/tmp/probe-helper/`, `/tmp/probe-fixture/`) MUST NOT be committed to `subagents/` or `skills/` — those are install-deliverable directories.
