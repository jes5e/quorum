## Context

The Code Reviewer subagent wraps the existing `/bees-code-review` skill in a thin subagent definition so the future reconciliation-loop orchestrator can dispatch code review as a fresh-eyes background `Agent` invocation. Per the PRD's acceptance criteria and the SDD's hub-and-spoke specialist model, all three reviewer subagents are **always cold** by design — they must contain no warm-Agent / SendMessage / per-Task-reuse prose. The current inline reviewer Instructions blocks in `skills/bees-execute/SKILL.md` (Step 5 final-Bee review) and `skills/bees-fix-issue/SKILL.md` (Step 4 review loop) are already thin (a single "Invoke the /bees-code-review skill" bullet), so the lift is mostly framing — the heavy lifting stays in the wrapped review skill.

## What Needs to Change

- Ensure the top-level `subagents/` directory exists at the repo root (idempotent).
- Create `subagents/code-reviewer.md` with YAML frontmatter and a markdown body.
- Frontmatter:
  - `name: code-reviewer`
  - `description`: lead with the verb; state scope precisely as performing a fresh-eyes code review of the work just produced by the Engineer, via the project's `/bees-code-review` skill, returning structured findings to the orchestrator.
  - `model: opus` (always — not user-configurable per CLAUDE.md `## Model assignment in execution skills`).
  - `tools: [Bash, Read, Grep, Skill]`. `Skill` is mandatory so the body can invoke `/bees-code-review`.
- Body: keep it thin and concise. The body should:
  - State the role's purpose in one or two sentences (review the output of the Engineer; provide feedback where the work was not up to standards).
  - Direct the subagent to invoke the `/bees-code-review` skill via the `Skill` tool against the diff or scope indicated in the orchestrator's prompt (the orchestrator passes the relevant scope — diff range, ticket ID, or both — in the Agent invocation prompt).
  - Direct the subagent to return findings to the orchestrator in a structured form (numbered list with severity tags such as `blocker` / `suggestion` / `nit`) consistent with the wrapped skill's existing output contract.
  - State explicitly that the reviewer always runs cold (fresh eyes) and must not assume any context from prior invocations.
- Strip during the lift (parent Task's Agent-Teams-prose-stripping convention):
  - Any "self-trigger" / "top of every turn" prose.
  - Any `metadata.blocked_on` / TaskUpdate references.
  - Any team-lead-ping / escalation-ladder prose.
  - Any named-team-scoped agent-naming prose.
  - Any warm-Agent / SendMessage / per-Task-reuse prose.
- Keep the body project-neutral: the only project-specific reference allowed is the wrapped skill name `/bees-code-review`.
- Any shell snippet must follow paired POSIX + PowerShell labelled blocks; thin wrapper is unlikely to need shell.
