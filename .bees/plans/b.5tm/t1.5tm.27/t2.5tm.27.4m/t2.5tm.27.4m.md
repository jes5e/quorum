## Context

The three reviewer subagent definitions (`code-reviewer`, `test-reviewer`, `doc-reviewer`) wrap the existing review skills (`/bees-code-review`, `/bees-test-review`, `/bees-doc-review`) so the reconciliation-loop orchestrator (Epic B in `bees-execute`, Epic C in `bees-fix-issue`) can dispatch reviews as fresh-eyes background `Agent` invocations. The current inline reviewer Instructions blocks in `skills/bees-execute/SKILL.md` Step 5 and `skills/bees-fix-issue/SKILL.md` Step 4 are already thin (a single "Invoke the /bees-<x>-review skill" bullet under a Responsibilities + Model heading), so the lift is mostly framing — the heavy lifting stays in the wrapped review skill where it already lives.

This Task is purely additive — no existing skill SKILL.md is touched.

## What Needs to Change

Create three reviewer subagent definition files in `subagents/`:

- `subagents/code-reviewer.md` — YAML frontmatter + thin wrapper body invoking `/bees-code-review` via the `Skill` tool, returning structured findings.
- `subagents/test-reviewer.md` — same shape; wraps `/bees-test-review`.
- `subagents/doc-reviewer.md` — same shape; wraps `/bees-doc-review`. Includes a short body note explaining the model-default + runtime-override interaction (mirrors T1.S3 doc-writer.md pattern).

Tool allowlist for all three: `Bash, Read, Grep, Skill`. The `Skill` tool inclusion is **load-bearing** — it is the only mechanism for a subagent to invoke an installed skill. Without `Skill` in the allowlist, reviewer wrappers cannot run their underlying skill.

Models per SDD `### Feature: Ephemeral-Agent Orchestration`:
- code-reviewer.md, test-reviewer.md: `model: opus` (always — not user-configurable per CLAUDE.md `## Model assignment in execution skills`)
- doc-reviewer.md: `model: opus` (shipped default; runtime override to `sonnet` handled by Epic B/C — same pattern as doc-writer.md and pm.md)

Body shape (thin wrapper):
- One or two sentences stating the role's purpose (review the output of Engineer / Test Writer / Doc Writer; provide feedback where the work was not up to standards).
- Direct the subagent to invoke the corresponding `/bees-<x>-review` skill via the `Skill` tool against the scope indicated in the orchestrator's prompt.
- Define the input contract — the orchestrator passes scope (diff range, ticket ID, or both) in the dispatch prompt; the wrapper does not compute scope itself.
- Define the output contract — return findings as a structured list (severity tags such as `blocker` / `suggestion` / `nit`, file:line refs, suggested fixes, verdict). Reference the wrapped skill's existing output shape rather than redefining it.
- State explicitly that the reviewer **always runs cold** (fresh-eyes by design per PRD acceptance criterion). The body MUST NOT carry warm-Agent / SendMessage-resume / per-Task-reuse prose.

Strip Agent-Teams-specific prose per the T1 contract (self-trigger, blocked_on, escalation ladder, named-team-scoped agent names, Team-Lead choreography).

## Why

Centralizing review-skill invocation in subagent definitions keeps the orchestrator prose in Epic B/C simple — Epic B's reconciliation loop dispatches by `subagent_type` rather than embedding `Skill`-tool invocation logic in skill prose.

## Cross-Task contracts (surfaced for sibling Tasks and downstream Epics)

- **Frontmatter schema reuse**: `name`, `description`, `model`, `tools` in that order — same as T1.
- **Strip list reuse**: T1's Agent-Teams-substrate strip list applies identically to T2.
- **`Skill`-tool dispatch contract**: each wrapper invokes its corresponding review skill via the `Skill` tool with `skill: "bees-code-review"` (or doc-review / test-review). Underlying skill names are stable; preserve them verbatim. Sibling subtask T3.S1 (pm.md) carries the same `Skill`-tool dispatch convention for in-flight `/bees-code-review` and `/bees-doc-review` invocations during PM's per-Task review.
- **Cold-start invariant**: all three reviewer subagents always run cold per PRD acceptance criterion. T2 wrappers MUST NOT carry warm-Agent or SendMessage-resume language. The cold-start contract applies to sibling Epics B/C/D — they MUST dispatch reviewer subagents fresh per invocation, never via SendMessage to a named warm Agent.
- **Output contract uniformity**: all three reviewers return findings in the same structured shape (severity tags + file:line refs + suggested fixes + verdict) so Epic B/C orchestrators can treat reviewer outputs uniformly.
- **PRD scope-fence**: PRD `### Feature: Ephemeral-Agent Orchestration` "Out of scope" explicitly says "Modifications to the three review skills (bees-code-review, bees-doc-review, bees-test-review) beyond ensuring they invoke cleanly from a subagent context. Their prose stays as-is." T2 wrappers MUST NOT modify `skills/bees-code-review/SKILL.md`, `skills/bees-doc-review/SKILL.md`, or `skills/bees-test-review/SKILL.md`.
- **Doc Reviewer model-default note**: same pattern as T1.S3 doc-writer.md and T3.S1 pm.md.
