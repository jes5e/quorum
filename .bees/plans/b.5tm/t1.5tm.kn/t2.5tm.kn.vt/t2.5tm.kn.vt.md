## Context

Epic `t1.5tm.kn`'s acceptance criteria 3 and 4 require live-demo verification:

- AC3: `/bees-setup` runs end-to-end on a fresh repo with no Agent Teams or display-backend prompts and no prompts about `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`.
- AC4: The fast-path detection (`skills/bees-setup/scripts/detect_fast_path.py`) still works on multi-machine clones — it's not Agent Teams-related, but the cleanup should not regress it.

This Task gates the Epic's "ready for shipped" status. It runs after Tasks 1 (`bees-setup/SKILL.md` cleanup), 2 (contributor-facing docs cleanup), and 3 (helper-script deletion) land.

The verification follows this repo's demo-runbook pattern: dispatched research Agents do **static verification** (read the post-cleanup `bees-setup/SKILL.md` end-to-end, simulate the walk-through against the Build Commands and Documentation Locations contract keys, confirm no remaining settings-write step, confirm `detect_fast_path.py` invocation prose is intact) **plus author a manual runbook** the user runs from a fresh Claude Code session against a temp-directory fresh repo. Live execution is the user's responsibility from a fresh session — the orchestrator does not run it from inside this skill invocation.

## What Needs to Change

1. **Static verification (research-mode dispatch deliverable).** Read `skills/bees-setup/SKILL.md` end-to-end after Task 1 lands and walk through it as if a fresh user were invoking `/bees-setup` for the first time on a fresh repo. Confirm:
   - No step prompts the user about Agent Teams.
   - No step prompts the user about `teammateMode`.
   - No step writes to `~/.claude/settings.json` for `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` or `teammateMode`.
   - Hive-colonization, `## Documentation Locations` write, `## Build Commands` write, and the optional PRD/SDD bootstrap step are all still present and functional.
   - `detect_fast_path.py` invocation prose (helper script + bash/PowerShell snippets) is unchanged.
   - The skill prose still follows the three non-negotiable design rules (language-agnostic, OS-paired POSIX + PowerShell snippets, project-neutral) and the scratch-file convention from `CLAUDE.md`.
   - The skill prose's Bash-tool calls follow the `CLAUDE.md` `## Bash etiquette in this repo` etiquette (single literal command per call, no compound chains, no diagnostic tails, etc.).
2. **Author a manual end-to-end runbook** under the namespaced workflow scratch dir per `CLAUDE.md` `## Scratch-file convention` — `/tmp/.bees-workflow/<task-id>-runbook.md` on POSIX, `$env:TEMP\.bees-workflow\<task-id>-runbook.md` on Windows — that the user can execute from a fresh Claude Code session. **The runbook is a transient artifact and is NOT committed to the repo** (matches the demo-runbook precedent on Epic `t1.5tm.o1` Task `t2.5tm.o1.7t`). The runbook must include:
   - Steps to set up a temp-directory fresh repo (`git init` in a fresh `<tempdir>/.bees-workflow/`-shape directory) so `/bees-setup` runs against a non-colonized state.
   - The `/bees-setup` invocation and the expected interactive flow (which prompts to expect, which to NOT see).
   - A checklist of observable outcomes mapped to AC3 and AC4.
   - A multi-machine clone simulation step that exercises `detect_fast_path.py` (e.g., set up a parallel clone with hives already colonized in the original tree, run `/bees-setup` in the second clone, observe the fast-path branch fires).
   - Per the demo-runbook precedent, **observations from the live run are appended to this Task body's `## Probe results` section** (NOT to the runbook file). The orchestrator records the user-reported observations back into the bees-stored Task body as the closing artifact for the Epic.
