## Context

Per the bees-workflow demo-runbook precedent (Epic `t1.5tm.o1` Task `t2.5tm.o1.7t`, Subtask `t3.5tm.o1.7t.sz`), the demo for this Epic is structured as: dispatched Agents author a markdown runbook, the user executes the runbook manually from a fresh Claude Code session, and observations get appended to the parent Task body's `## Probe results` section. This Subtask owns the runbook authoring.

The runbook lives under the namespaced workflow scratch directory per `CLAUDE.md` `## Scratch-file convention`: `/tmp/.bees-workflow/<task-id>-runbook.md` on POSIX and `$env:TEMP\.bees-workflow\<task-id>-runbook.md` on Windows, where `<task-id>` is the parent Task's bees ID (`t2.5tm.kn.vt`). Create the `.bees-workflow` subdirectory if absent (`mkdir -p` POSIX; `New-Item -ItemType Directory -Force` Windows). Never delete the runbook file — files under `<tempdir>/.bees-workflow/` accumulate intentionally so the user can find runbooks for past demos in one well-known place. The runbook is NOT committed to the repo.

This Subtask depends on the static-verification Subtask (sibling). Authoring a runbook against broken state would produce a runbook that describes the wrong demo flow; the static-verification Subtask must PASS before runbook authoring starts. If the sibling surfaces a regression, the regression is routed back to Task 1/2/3 per the parent Task's Cross-Task contracts and this Subtask waits for the rework to land.

Acceptance criteria for the demo itself (not for this Subtask, but for the runbook to encode) are derived from Epic `t1.5tm.kn` AC3 (no Agent Teams / `teammateMode` / `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` prompts) and AC4 (fast-path detection works on multi-machine clones).
