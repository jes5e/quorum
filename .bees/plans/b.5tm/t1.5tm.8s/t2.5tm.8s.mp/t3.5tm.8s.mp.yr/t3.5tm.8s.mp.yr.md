## Context

The doc writing guide's `## When you're updating an existing skill` section currently contains a rule (rule 3) stating: "Agent Teams is required for `quo-execute` and `quo-fix-issue`. Both skills spawn a team unconditionally — there is no single-agent fallback path. Don't introduce conditional 'if-Teams-off' branches when editing those skills; rely on the precondition check to hard-fail when Agent Teams isn't enabled." Once Section 3 of `skills/quo-execute/SKILL.md` is rewritten in Task 2 (Subtasks 2.1 + 2.2), that rule becomes wrong about `quo-execute`.

The rule must be **split** so it remains correct about `quo-fix-issue` (still Agent-Teams-based; out of scope for Epic 8s) while accurately describing `quo-execute`'s new substrate (per-Subtask ephemeral `Agent` dispatch with a subagent-registry hard-fail precondition rather than an Agent-Teams-env-var hard-fail precondition).

This Subtask ships in the same commit as Subtasks 2.1 + 2.2 (Section 3 SKILL.md rewrite) for the same atomicity reason as Subtask 2.D1.

## What Needs to Change

Revise rule 3 under `## When you're updating an existing skill` in `docs/doc-writing-guide.md` to:

- Retain the "Agent Teams is required, no fallback" guidance for `quo-fix-issue`.
- Add equivalent guidance for `quo-execute` describing its new substrate: per-Subtask ephemeral Agent dispatch is the unconditional substrate; the precondition check that hard-fails is now a subagent-registry check (the seven required subagent types must be registered), not an `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` env-var check; do not introduce conditional fallback branches when editing the skill.

Keep the rule's structural intent intact: a precondition gate is the single point of substrate enforcement for each skill, and skill prose downstream of that gate must not branch on substrate availability.
