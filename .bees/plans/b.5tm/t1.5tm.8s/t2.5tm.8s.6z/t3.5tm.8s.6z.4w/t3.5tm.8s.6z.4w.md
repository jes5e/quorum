## Context

This Subtask is the source-side rewrite for Task 1 of Epic t1.5tm.8s. Epic 8s switches `bees-execute` from the experimental Agent Teams substrate to ephemeral background `Agent` tool invocations against the seven custom subagent types Epic A authored under `agents/` at the repo root. The current Preconditions section in `skills/bees-execute/SKILL.md` hard-fails when `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` is not `"1"` and points at the bundled `check_agent_teams.py` helper for verification — both are obsolete under the new substrate.

Epic A's `## Probe results — verified update` finding established that Claude Code loads custom subagents at session start; mid-session installs require a restart or `/agents` hot-reload. A missing subagent_type at skill invocation therefore almost always means the user installed the agents but did not restart Claude Code, or did not install them at all. The replacement precondition must surface that condition clearly at the start of the run, before any dispatch is attempted.

**Verification mechanism — chosen approach.** The new precondition uses a hybrid of two complementary mechanisms:

- **Primary: procedural gate.** Document at the top of the Preconditions section that if any of the seven subagent types fails to dispatch at run-time (the orchestrator will see an `Agent type '<name>' not found` style error from the Agent tool), the orchestrator STOPS, prints the hard-fail message directing the user to verify the install per `README.md` `## Install` AND restart Claude Code or run `/agents` to hot-reload, then exits. This is the load-bearing check — it is honest about Claude Code's session-load semantics, and it cannot be bypassed by token-budget pressure or model creativity, because it fires at the natural failure point.
- **Opportunistic: `claude agents` shell-out fast-fail.** Per Claude Code's official sub-agents docs, `claude agents` lists all configured subagents from the command line without starting an interactive session. If the executing Engineer's research confirms that (a) `claude agents` is safe to invoke from inside a running Claude Code session (no recursion, no session-state corruption) and (b) its output is parseable in a way that allows the orchestrator to verify each of the seven names is present, the new "Verifying the subagents precondition" subsection ships paired POSIX bash + Windows PowerShell snippets that invoke `claude agents` and check for the seven names, surfacing the hard-fail message before any dispatch is attempted. This is a belt-and-braces structural check — it catches the missing-restart case earlier and saves the user one wasted dispatch turn.

**Engineer research gate before authoring.** Before authoring the verification subsection, the Engineer must verify both safety properties of `claude agents` against authoritative sources (Claude Code docs first per CONTRIBUTING.md `## Verifying external-system contracts`; an actual probe only if the docs are insufficient). If either property is not confirmed, ship the procedural gate alone — that is still a valid hard-fail check, just at-first-dispatch rather than upfront — and record the disconfirmation in the new subsection's prose so a future contributor knows why the shell-out is absent. Do NOT ship a `claude agents` snippet without the safety confirmation.

This Subtask only edits `bees-execute/SKILL.md`. The on-disk `check_agent_teams.py` helper stays in place — its deletion is Epic E's scope. The cross-Task contract for Epic t1.5tm.8s is that the precondition gate is the single point of enforcement for subagent registration; Tasks 2-7 of this Epic will not add fallback logic for missing subagent types and will assume all seven are registered when they dispatch.
