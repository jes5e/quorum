## Context

Per the team-lead's saved demo-runbook pattern, the demo for this Epic is structured as: dispatched Agents author a markdown runbook, the user executes the runbook manually from a fresh Claude Code session, and observations get appended to this Task body's `## Probe results` section. This Subtask owns the runbook authoring.

The runbook lives under the namespaced workflow scratch directory per CLAUDE.md `## Scratch-file convention`: `/tmp/.bees-workflow/<task-id>-runbook.md` on POSIX and `$env:TEMP\.bees-workflow\<task-id>-runbook.md` on Windows. Create the `.bees-workflow` subdirectory if absent (`mkdir -p` POSIX; `New-Item -ItemType Directory -Force` Windows). Never delete the runbook file — files under `<tempdir>/.bees-workflow/` accumulate intentionally so the user can find runbooks for past demos in one well-known place.

The runbook codifies the demo steps the user will execute manually after the rewrite lands. Acceptance criteria for the demo itself (not for this Subtask, but for the runbook to encode) are derived from Epic AC#5 ("`/bees-breakdown-epic <epic-id>` produces same-quality Tasks/Subtasks as the current Agent Teams version, verified by spot-checking the output structure against the existing template") plus the new ordering / dispatch-shape verifications introduced by this Epic.

## What Needs to Change

Author a markdown runbook at `/tmp/.bees-workflow/<task-id>-runbook.md` (POSIX path; PowerShell-equivalent path on Windows). The runbook content covers:

**Preflight:**

- Confirm subagent install per `README.md` `## Install` and Claude Code restart (or `/agents` hot-reload). Without this, the Preconditions section's hard-fail message is the expected outcome — that's a valid demo observation, not a failure.
- Pick or create a small synthetic stand-in Bee + Epic suitable for breakdown. Sizing: a single Epic whose scope reasonably decomposes into 2-3 Tasks. Provide concrete suggestions for synthetic Bee/Epic shape (e.g., a "rename CLI flag X to Y across N skill files" Bee with a single Epic, or a "add new logging hook to module Z" Bee with a single Epic; the runbook author picks one and writes it as the example).

**Demo execution (from a fresh Claude Code session):**

- Invoke `/bees-breakdown-epic <epic-id>`.
- Watch for, and record, the observations below in order. Each becomes an Acceptance Criterion of the demo (not of this runbook-authoring Subtask):
  1. Subagent-precondition check fires upfront — `claude agents` is invoked, all seven required subagent names are verified, the run proceeds (or hard-fails with the documented message if the user skipped the restart).
  2. Research-mode dispatch shape is observed — `Agent(subagent_type=<role>, run_in_background=true, prompt=<research-mode preamble + ...>)` invocations appear in the harness with the literal subagent-type values `engineer`, `doc-writer`, `test-writer`, `pm` (whichever ones the orchestrator dispatches for the synthetic Epic).
  3. TaskList naming convention follows the documented pattern — `engineer-research-<task-scope-id>`, `pm-research-<epic-id>`, etc. Names are deterministic and scope-suffixed.
  4. Spec Traceability Review via PM Agent runs BEFORE `bees create-ticket` is invoked by the orchestrator (Task 2 reorder verification). The PM Agent's output (a traceability table with all rows OK) is observable before the orchestrator's first `bees create-ticket` invocation.
  5. The Mandatory Subtask Description Template is applied to every Subtask the orchestrator creates — each Subtask body contains all four sections (`## Context`, `## What Needs to Change`, `## Key Files`, `## Acceptance Criteria`).

**Spot-check:**

- After the run completes, fetch the created Tasks and Subtasks via `bees show-ticket --ids <ids>` and confirm: (a) all Subtask bodies contain the four template sections; (b) Tasks and Subtasks are at status `ready`; (c) the Epic is at status `ready`.

**Recording observations:**

- The runbook concludes with a section instructing the user to append their observations to this Task body's `## Probe results` section via `bees update-ticket`. Provide the canonical update shape (write observations to a temp file under `/tmp/.bees-workflow/`, then `bees update-ticket --ids <task-id> --body-file <path>`).

The runbook is portable — paired POSIX and Windows PowerShell paths and commands per CLAUDE.md design rule 2.
