## What Needs to Change

Replace the existing fourth bullet in `## Status / type renames history` (currently beginning `**Skill renames code-review / doc-review / test-review → bees-code-review / ...`) with a single rewritten bullet that captures three load-bearing claims:

1. **Misinvocation surface from `/<x>-review` substring matches.** Claude Code's skill loader matched `/code-review`, `/doc-review`, and `/test-review` against the user's natural-language requests and over-invoked them in unrelated contexts. The `bees-` prefix on its own did not eliminate this because the matcher still saw the `-review` suffix as a viable invocation trigger.
2. **Standalone-invocation value did not justify a generic skill name.** The previous entry's precedent — that standalone invocation was supported and worth optimizing the name for — proved low-value in practice; the dominant invocation path is the orchestrated review cycle inside `/quo-execute` and `/quo-fix-issue`. Standalone use exists but is not load-bearing on the naming choice.
3. **Role-based names mirror `agents/<role>.md`.** The new names (`quo-engineer-review`, `quo-doc-writer-review`, `quo-test-writer-review`) align with the role contracts in `agents/`, making the orchestration-side mental model consistent across the skill catalog and the dispatched-subagent contracts.

The rewrite must **replace, not append** — the existing bullet's text (including the `standalone invocation remains supported via the existing dual-mode prose` precedent line) is gone after this edit. Do not preserve the old bullet alongside a new one. The history section's other bullets (hive-rename, status-rename, skill-prefix, and any others) are unchanged.

The rewrite must stay in the existing history-entry editorial format (date-prefixed bullet, brief decision-and-reasoning prose) so the section's voice is consistent with the other entries.
