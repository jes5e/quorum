# Task 2: Run final repo-wide verification (Lint + grep)

Covers SDD `## Requirements` SR-6.1 and SR-6.2.

## Context

Second Task of Epic `t1.mak.ka`. After Epic 1 (skill internals), Epic 2 (callers), Epic 3 (top-level docs), and Task 1 of this Epic (open-ticket sweep) all complete, this Task runs the final repo-wide verification:

1. The repo's `Lint` build command (`python -m pyflakes skills/*/scripts/*.py`, per `CLAUDE.md` `## Build Commands`) must return no errors. The helper scripts (`detect_fast_path.py`, `scoped_marker_resolver.py`) do not reference the renamed skill directories, so the rename should be a no-op for lint — but the verification step still runs to catch any regression.
2. A repo-wide `grep -rE "bees-code-review|bees-test-review|bees-doc-review" .` must return matches **only** inside the three documented exception classes:
   - (a) the rewritten `CONTRIBUTING.md` "Status / type renames history" entry (the prior `code-review` → `bees-code-review` rename is being narrated; preservation of old names there is correct);
   - (b) closed/done bees ticket files under `.bees/` (immutable historical snapshots);
   - (c) the eight self-referential rename-narrative tickets under `.bees/plans/b.mak/` (`b.mak`, `t1.mak.eo`, `t1.mak.su`, `t1.mak.84`, `t1.mak.ka`) and `.bees/specs/b.mam/` (`b.mam`, `t1.mam.qr`, `t1.mam.a2`).

This Task is the final acceptance signal for the entire Bee `b.mak`. Any unexpected old-name match outside the documented exceptions is a defect in one of the prior Epics and must be surfaced for triage.

Per SDD `## Codebase exploration findings`: this Epic's net work surface is essentially zero — both verifications are expected to pass cleanly. The Task ships as a deliberate verification step (cleaning per-Epic commit-message audit trail of "the verification ran").

## What Needs to Change

1. **Run Lint.** Execute `python -m pyflakes skills/*/scripts/*.py` from the repo root. Expected: no errors.

2. **Run repo-wide grep.** Execute `grep -rE "bees-code-review|bees-test-review|bees-doc-review" .` (or the project's preferred recursive grep — the `Grep` first-class tool is acceptable). Inspect each match:
   - If the match is inside the rewritten `CONTRIBUTING.md` "Status / type renames history" entry (the bullet that narrates the prior `code-review` → `bees-code-review` rename), it is an expected exception per PRD AC#14.
   - If the match is inside a closed/done bees ticket file under `.bees/` (verify by reading the ticket file's `ticket_status` field — `done` or any closed status), it is an expected exception per PRD AC#14.
   - If the match is inside one of the eight self-referential rename-narrative tickets (`b.mak`, `t1.mak.eo`, `t1.mak.su`, `t1.mak.84`, `t1.mak.ka` under `.bees/plans/b.mak/` and `b.mam`, `t1.mam.qr`, `t1.mam.a2` under `.bees/specs/b.mam/`), it is an expected exception per PRD AC#14.
   - Any other match is an **unexpected** old-name reference and indicates a defect in Epic 1, 2, 3, or this Epic's Task 1. Surface the file path and line number; do NOT silently fix it (the gap belongs to whichever Epic owns that file's surface).

3. **Report the verification outcome.** If both Lint and grep pass cleanly per the above, the Task is `done`. If either surfaces an unexpected finding, halt and surface the finding for triage rather than auto-fixing.

## Cross-Task contracts

- **Up-blocked at the Epic level by `t1.mak.eo`, `t1.mak.su`, and `t1.mak.84`** (predecessor Epics) plus by **Task 1 of this Epic** (open-ticket sweep). If the open-ticket sweep surfaces an unexpected stray ticket and updates it, this Task's grep should run *after* that update lands.
- **No source-code, doc, or ticket-body edits.** This Task is verification-only. If grep surfaces an unexpected match, halt and report; do not silently fix the offending file.
- **Per `CLAUDE.md` `## Bash etiquette`.** Each command runs as its own Bash call — no compound chains, no diagnostic tails. If grep is invoked via Bash rather than the first-class `Grep` tool, it must be a single literal command.

## Why

Per SDD SR-6.1 / SR-6.2 and PRD `## Acceptance Criteria` #13 / #14. The Lint verification catches any Python-helper regression (the bundled scripts are not expected to be affected by the rename, but verification runs to be sure). The repo-wide grep is the authoritative final check that the rename is observable as a single coherent transition: every old-name reference outside the three documented exception classes should be gone.

## Success Criteria

- `python -m pyflakes skills/*/scripts/*.py` returns no errors (PRD AC#13).
- Repo-wide `grep -rE "bees-code-review|bees-test-review|bees-doc-review" .` returns matches **only** inside the three documented exception classes: (a) the rewritten `CONTRIBUTING.md` history entry, (b) closed/done bees tickets, (c) the eight self-referential rename-narrative tickets under `.bees/plans/b.mak/` and `.bees/specs/b.mam/`. (PRD AC#14.)
- Any match outside the three exception classes is surfaced for triage and the Task is held in `in_progress` until the offending Epic is patched.
