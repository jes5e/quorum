# Task 1: Run the open-ticket sweep across Plans and Issues hives

Covers SDD `## Requirements` SR-5.1 and SR-5.2.

## Context

First Task of Epic `t1.mak.ka`. After Epics 1, 2, and 3 land, this Task runs a `bees execute-freeform-query` across the Plans and Issues hives filtered to `ticket_status` in (`drafted`, `ready`, `in_progress`, `open`) with bodies containing any of the three old skill names. The expected result post-Epic-1-2-3 is exactly the **eight self-referential tickets** that describe the rename narrative itself:

- Spec Bee `b.mam`
- Its two `t1=Doc` children: `t1.mam.qr` (PRD), `t1.mam.a2` (SDD)
- Plan Bee `b.mak`
- `b.mak`'s four Epics: `t1.mak.eo`, `t1.mak.su`, `t1.mak.84`, `t1.mak.ka`

These eight tickets correctly describe the rename narrative and **must not be modified**. Their bodies legitimately reference the old skill names because they ARE the rename's specification and breakdown.

For any open/in-progress/drafted/ready ticket the query surfaces **beyond** those eight self-references, this Task updates the body in place via `bees update-ticket --body-file <path>` with the new names substituted. Done/closed tickets are not touched per the project's "closed/done tickets are immutable historical snapshots" policy.

Per SDD `## Codebase exploration findings`: "Net Epic 4 work surface: zero. The open-ticket sweep is a verification-only Epic — the `bees execute-freeform-query` for open/in-progress/drafted/ready tickets containing old-name references should return exactly the eight self-referential tickets enumerated above, all of which are correctly preserved." However, this is only the *expected* result; the Task still runs the query as the authoritative check.

**Note on filter syntax.** The Epic body and SDD show an illustrative YAML query shape (`hive=plans, status_in=[drafted,ready,in_progress]; hive=issues, status=open; filter: body_contains_any: [...]`). The actual filter vocabulary supported by the bees CLI at execution time may differ — use `docs/doc-writing-guide.md` `## Querying tickets` recipes plus `bees execute-freeform-query --help` to construct the exact valid query. The result-set shape is what matters: enumerate every open/in-progress/drafted/ready ticket whose body matches any of `bees-code-review|bees-test-review|bees-doc-review`.

## What Needs to Change

1. **Run the sweep query.** Use `bees execute-freeform-query` to enumerate every ticket with `ticket_status` in (`drafted`, `ready`, `in_progress`, `open`) across the Plans and Issues hives whose body contains `bees-code-review`, `bees-test-review`, or `bees-doc-review`. Capture the resulting ticket IDs and titles.

2. **Compare against the expected self-reference set.** The eight expected ticket IDs are: `b.mam`, `t1.mam.qr`, `t1.mam.a2`, `b.mak`, `t1.mak.eo`, `t1.mak.su`, `t1.mak.84`, `t1.mak.ka`. If the result set is exactly those eight, no update work is needed and Task 1 is verification-only.

3. **For any ticket beyond the eight self-references, update its body in place.** Read the current body via `bees show-ticket --ids <id>`, write the body to a temp file under `<tempdir>/.bees-workflow/` (per `CLAUDE.md` `## Scratch-file convention`), apply the three textual substitutions (`bees-code-review` → `bees-engineer-review`, `bees-test-review` → `bees-test-writer-review`, `bees-doc-review` → `bees-doc-writer-review`), and run `bees update-ticket --ids <id> --body-file <temp-path>`. Do **not** touch the eight self-references; do **not** touch any done/closed ticket regardless of body content.

## Cross-Task contracts

- **Up-blocked at the Epic level by `t1.mak.eo`, `t1.mak.su`, and `t1.mak.84`.** All source-file rename work must complete before the open-ticket sweep is meaningful (otherwise the sweep would surface in-flight source-edit tickets that are mid-rename).
- **Eight-ticket exception list.** The eight self-references are documented in PRD `## Acceptance Criteria` #12 and SDD SR-5.1; preserving them is correct, not a defect.
- **Done/closed tickets are out of scope.** Per PRD AC#12 and SDD `## Decisions and rejected alternatives` ("Decision: Sweep open/in-progress bees tickets, leave done/closed tickets untouched"), historical tickets are immutable.
- **No source-code edits.** Sibling Epics' scope (already done).
- **No top-level doc edits.** Sibling Epic `t1.mak.84`'s scope (already done).
- **Task 2 of this Epic depends on this Task** — Task 2's repo-wide verification grep would surface unexpected old-name matches in ticket bodies if this Task missed something.

## Why

Per SDD SR-5.1 / SR-5.2 and PRD `## Functional Requirements` #9 / `## Acceptance Criteria` #12. In-flight tickets that an Engineer or PM agent reads mid-execution would surface old skill names and trigger wrong-tool invocations during the work; the sweep keeps the workflow coherent. The expected near-zero work surface (just the 8 self-references) is documented; if it turns out non-zero, the sweep does the cleanup.

## Success Criteria

- The `bees execute-freeform-query` open-ticket sweep returns exactly the eight self-reference tickets (`b.mam`, `t1.mam.qr`, `t1.mam.a2`, `b.mak`, `t1.mak.eo`, `t1.mak.su`, `t1.mak.84`, `t1.mak.ka`) — no fewer, no more.
- If the sweep surfaces any tickets beyond the eight self-references, those tickets' bodies have been updated in place via `bees update-ticket --body-file` with the three substitutions applied, and a re-run of the sweep returns exactly the eight self-references.
- Done/closed tickets are unchanged (verification by inspection — no `bees update-ticket` call should target a ticket whose `ticket_status` is `done` or `closed`).
- The eight self-reference tickets are unchanged (their bodies legitimately preserve the old names as historical / specification context per PRD AC#12).
