## Codebase exploration findings

### Architecture

The change spans four distinct kinds of skill-prose surface in `/Users/jesseg/code/quorum`:

1. **In-flow review skills** — four skills under `skills/`, each invoked by an orchestrator during a review loop.
2. **Embedded reviewer dispatch prompts** — three inline-prompted general-purpose Agent dispatches embedded directly in skill prose (one each in `quo-fix-issue`, `quo-execute`, `quo-plan`).
3. **Reviewer subagent wrappers** — three thin pass-through agent role files under `agents/`.
4. **Orchestrator R2/R3 dispatch logic** — two orchestrator skills (`quo-fix-issue` and `quo-execute`) that consume review findings and dispatch follow-up implementers.

These four surfaces share no central registry — each one's contract is documented in its own SKILL.md. The change has to land coherently across all four; the backwards-compat shim (FR-7 in the PRD) is what lets the rollout proceed incrementally rather than as one atomic landing.

### Data models

#### Compromise tracker entry shape

The compromise tracker is a single markdown file accumulating one section per accepted compromise. Each section uses this canonical shape:

```markdown
## Compromise <n>

- **Finding (verbatim):** <severity tag> <fix-path enumeration with depth tags> <description>
- **Fix paths surfaced by reviewer:**
  - (a) [depth:<depth>] <description>
  - (b) [depth:<depth>] <description>
  - ...
- **Decision:** <one of: User picked path (a) | User picked path (b) | ... | User picked Defer to follow-up Issue | User picked Accept the limitation | Auto-routed (a) per single-path refactor-locally rule | User overrode auto-route after post-completion challenge>
- **Rationale:** <user's stated reason if surfaced via AskUserQuestion, or the auto-route rule that fired>
- **Follow-up Issue:** <ticket ID if filed via /quo-file-issue, else `none`>
```

The shape is markdown so the file is human-readable and `Read`-tool-consumable by a dispatched Agent. The `<n>` discriminator is a 1-based counter scoped to the current run's tracker file, NOT a globally unique identifier — the tracker is session-scoped per FR-4.

#### Finding shape with two-dimensional tagging

The extended finding shape that in-flow review skills emit:

```
<n>. `<severity>` <one or more fix-path lines, each with its own depth tag> — <description>
```

Where each fix-path line is rendered:

```
(<letter>) [depth:<trivial-tweak|refactor-locally|re-architect>] <description of that fix path>
```

Single-fix-path findings continue to be valid: the finding still carries a `(a)` fix-path block, but only one. The shape is uniform whether the reviewer enumerated 1 path or 4, which simplifies the orchestrator's parser.

The `[preferred]` token is **optional** and, when present, sits immediately after the `[depth:<...>]` token (before the path description), rendered `(<letter>) [depth:<...>] [preferred] <description of that fix path>`. A reviewer appends it to **at most one** fix-path line per finding to signal the path it recommends among multiple enumerated alternatives; it is meaningful only for multi-path findings (a single-path finding has no preference to express). The token is strictly downstream of the parser-relevant `(<letter>) [depth:<...>]` head, so it does not affect the orchestrator's `(num-paths, max-depth)` tuple derivation — it is consumed solely by the routing-decision gate (SR-5.1), which maps the `[preferred]`-tagged path to the `(Recommended)` marker on its `AskUserQuestion` choice. When no fix-path line carries `[preferred]`, the gate marks no path `(Recommended)` (graceful degradation).

#### `reference_materials` shape

The bees CLI's `reference_materials` JSON shape (`[{"value":"<id>","resolver":"<resolver>"}]`) is unchanged. The compromise tracker is NOT a `reference_materials` resolver — it's a session-scoped markdown file, NOT a ticket reference.

## Requirements

### SR-1 — Two-dimensional finding tagging

- **SR-1.1** — The output-shape sections of `skills/quo-engineer-review/SKILL.md`, `skills/quo-doc-writer-review/SKILL.md`, `skills/quo-test-writer-review/SKILL.md`, and `skills/quo-spec-review/SKILL.md` document the extended finding emission shape: severity tag + one or more fix-path lines with depth tags + description. For the three implementer-side skills, this is the *first* introduction of severity tagging — the affected sections (`### Step 4 / ### Step 6: Generate Work Item List` in each) document `blocker` / `suggestion` / `nit` plus the depth dimension.
- **SR-1.2** — The output-shape sections enumerate fix paths per finding. Single-path findings remain valid; multi-path findings are required when multiple plausible alternatives exist. The reviewer skills' prose teaches the enumeration discipline with **at least one worked example per depth bucket** (`trivial-tweak`, `refactor-locally`, `re-architect`) per skill — so reviewers have concrete anchors for depth classification. See `## Background and rationale` → "Assumptions and calibration mechanisms" for the calibration rationale.
- **SR-1.3** — The embedded reviewer dispatch prompts in `skills/quo-fix-issue/SKILL.md` Section 8 (lines 517–530), `skills/quo-execute/SKILL.md` Section 6 (lines 449–483), and `skills/quo-plan/SKILL.md` Step 5e (lines 465–540) extend their output-contract sections to require severity + per-fix-path depth tags plus enumerated fix paths. The dispatched general-purpose Agents see the extended contract in their dispatch prompts.
- **SR-1.4** — The reviewer-agent frontmatter contracts in `agents/code-reviewer.md`, `agents/doc-reviewer.md`, `agents/test-reviewer.md` are updated to mention the depth dimension explicitly alongside the existing severity-tag mention. The agents continue to defer to their wrapped skills' output contracts; this is a pass-through documentation update, not a behavioral change.
- **SR-1.5** — The output-shape sections of the four in-flow review skills and the embedded reviewer dispatch prompts (per SR-1.1 / SR-1.3), plus `skills/quo-plan/SKILL.md` Step 5e, define an **optional** `[preferred]` emission marker on the fix-path line, positioned immediately after the `[depth:<...>]` token and before the path description. A reviewer emits `[preferred]` on at most one fix-path line per finding to signal its recommended path among multiple enumerated alternatives; the marker is meaningful only for multi-path findings and is strictly optional (its absence is fully valid and is the graceful-degradation default). This producer-side requirement closes the asymmetry whereby SR-5.1's consumer prose marked a reviewer-preferred path `(Recommended)` without any emission requirement defining how a reviewer signals that preference. The marker sits strictly downstream of the parser-relevant `(<letter>) [depth:<...>]` head and does not affect the `(num-paths, max-depth)` tuple derivation (SR-2.4).

### SR-2 — Orchestrator routing discipline

- **SR-2.1** — `skills/quo-fix-issue/SKILL.md` gains a new section titled exactly "Orchestrator discipline: routing review findings", positioned immediately before Section 5's review-loop content (current line 401). The new section contains the deterministic routing table.
- **SR-2.2** — `skills/quo-execute/SKILL.md` gains the same titled section, positioned immediately before Section 5's review-loop content (current line 414). The two sections carry near-identical prose; per the project's reuse convention, divergence between them is a defect, not an intentional difference.
- **SR-2.3** — The routing table follows the Step 4c Loop-back UX precedent (`skills/quo-plan/SKILL.md` lines 254–258): two columns, input-tuple → routing-action. Mechanically followable: given any `(num-paths, max-depth-across-paths)` input, exactly one routing decision applies.
- **SR-2.4** — Routing decisions: `(1, trivial-tweak)` → auto-dispatch; `(1, refactor-locally)` → auto-dispatch + log to compromise tracker per SR-3.2; `(1, re-architect)` → user gate before dispatching; `(N>1, any)` → user gate before dispatching.
- **SR-2.5** — The new section includes an anti-pattern callout block titled "ANTI-PATTERN — do not write this:" forbidding scope-bounding directives inside R2/R3 dispatch prompts. The callout names the specific forbidden phrasings (e.g., `"out of scope for this issue"`, `"out of scope for <id>"`, `"prefer option (a)"`, `"Do NOT add X — out of scope"`) and directs the orchestrator to surface scope-bounding decisions via `AskUserQuestion` instead.
- **SR-2.6** — Backwards-compatibility shim: findings emitted without a depth tag (legacy review-skill emission during rollout, or hand-authored findings from a future caller) are treated by the routing table as if `re-architect` depth — i.e., they route to the user gate.

### SR-3 — Session-scoped compromise tracker

- **SR-3.1** — The compromise tracker is a single markdown file written under the project's standard scratch-file convention (CLAUDE.md `## Scratch-file convention`). Canonical filename: `compromises-YYYYMMDD-HHMM-<short-suffix>.md`, where `YYYYMMDD-HHMM` is a UTC timestamp (e.g., `20260520-1714`) generated once at the start of the run and `<short-suffix>` is a short collision-resistant random string. The timestamp prefix makes tracker files debuggably identifiable across multiple runs accumulated in `<tempdir>/.quorum/` over time — without it, the user has no easy way to map an old tracker file back to a specific session.
- **SR-3.2** — One tracker entry is appended for each of: user picks `Defer to follow-up Issue` at a scope-bounding gate; user picks `Accept the limitation` at a scope-bounding gate; orchestrator auto-routes a `(1, refactor-locally)` finding; user explicitly overrides an auto-route decision after the post-completion review challenges it (per SR-6.7). Each entry uses the shape in `## Codebase exploration findings` → "Data models" → "Compromise tracker entry shape".
- **SR-3.3** — The tracker persists across orchestrator-yield events within a single run (it is the file system, not in-memory state, so yields are inherently safe). It MUST NOT persist across separate `/quo-fix-issue` or `/quo-execute` runs — the orchestrator generates a new `<session-suffix>` at the start of each run so previous-run files are visible but not accidentally appended-to.
- **SR-3.4** — `/quo-fix-issue` Section 7's per-Issue summary and the analogous summary surface in `/quo-execute` Section 7 (or wherever the run-end summary lives) gain an "Accepted compromises" section listing the tracker contents. When the tracker is empty (no compromises accepted), the section is omitted entirely.

### SR-4 — Post-completion review reframe

- **SR-4.1** — The dispatch prompt at `skills/quo-fix-issue/SKILL.md` Section 8 lines 517–530 is rewritten to lead with a "challenge accepted compromises" phase. The new structure: (a) consume the compromise tracker as input, (b) judge solution quality holistically beyond logged compromises, (c) then sweep for discrete defects. Findings emit one of `[compromise-challenge]` / `[design]` / `[defect]` tags.
- **SR-4.2** — The dispatch prompt at `skills/quo-execute/SKILL.md` Section 6 lines 449–483 receives the same rewrite. The two prompts carry near-identical structure; per SR-2.2 the divergence between them is a defect.
- **SR-4.3** — The compromise tracker is passed into both dispatch prompts via **file path** (not inlined content); the dispatched general-purpose Agent reads the tracker via its `Read` tool. This mechanism is documented in `## Decisions and rejected alternatives` below.
- **SR-4.4** — Each emitted finding additionally carries the severity tag and per-fix-path depth tag established by SR-1. The depth tag is informative — the post-completion sweep is the final pre-merge gate, so any finding it emits is candidate to gate the run regardless of depth.
- **SR-4.5** — The post-completion review's `[compromise-challenge]` phase explicitly checks **depth-tag plausibility** for auto-routed `(1, refactor-locally)` findings. When the reviewer judges that the routed fix's depth was misjudged and the true depth is `re-architect`, the reviewer emits a `[compromise-challenge]` finding with the misclassification as the substance; the orchestrator then routes the user per SR-6.7. This is the depth-calibration feedback loop — see `## Background and rationale` → "Assumptions and calibration mechanisms".
- **SR-4.6** — The post-completion review's `[compromise-challenge]` phase additionally checks **fix-path enumeration plausibility** for every finding the in-flow reviewer surfaced. When the post-completion reviewer judges that the in-flow reviewer should have enumerated an additional plausible fix path the user might have wanted to choose from (i.e., the in-flow reviewer surfaced one path and routed via auto-dispatch under SR-2.4's single-path branch, but the post-completion reviewer can see a second plausible path that warrants user-gate routing), the post-completion reviewer emits a `[compromise-challenge]` finding naming the under-enumeration. SR-4.5 and SR-4.6 are complementary backstops covering two distinct slip-throughs in the routing-table's user-gate prescription: SR-4.5 catches when *depth* was misjudged for a surfaced path; SR-4.6 catches when a plausible path *wasn't surfaced at all*. Without SR-4.6, an in-flow reviewer who under-enumerates fix paths can silently route auto-dispatch decisions past the user even when the depth tag is correct. The orchestrator's response to an SR-4.6 finding follows the same shape as SR-6.7: user gate with three choices (`File follow-up Issue to surface the missing path` / `Accept the under-enumeration and proceed` / `Pause to discuss`).

### SR-5 — User-facing gate prose

- **SR-5.1** — When the orchestrator routing table fires a user gate for a multi-path or single-path-re-architect finding, the `AskUserQuestion` invocation:
  - Includes the finding verbatim in the question text.
  - Includes the reviewer-surfaced fix paths with their depth tags as the answer choices.
  - Adds a `Defer to follow-up Issue` option AND a `Cancel` option.
  - Marks the path the reviewer flagged as preferred (if any) as `(Recommended)`, mirroring the Step 5e Shape A/B/C pattern in `skills/quo-plan/SKILL.md` lines 551–577.
  - **Cancel semantics:** In `/quo-fix-issue`, `Cancel` at this gate aborts the **current Issue's fix run** — the orchestrator stops processing that Issue and proceeds to the next Issue in the batch (in list / `all` mode), or ends the run cleanly if no more Issues remain. In `/quo-execute`, `Cancel` aborts the **current Task's execution** — the orchestrator stops processing that Task and proceeds to the next Task, or ends the run if no more Tasks remain. The Cancel choice does NOT abort the entire `/quo-fix-issue` or `/quo-execute` run; that level of abort remains the user's `Ctrl-C` option. This per-Issue / per-Task scoping is the per-batch-unit abort the user opted into when invoking the skill.

- **SR-5.2** — When the orchestrator considers scope-bounding a finding (rather than picking between fix paths), the `AskUserQuestion` invocation:
  - Includes the finding verbatim in the question text.
  - Offers exactly the three finite choices: `Fix properly now`, `Defer to follow-up Issue`, `Accept the limitation`.
  - Does NOT inline the scope-bounding decision as a directive in a subsequent dispatch prompt — see SR-2.5 anti-pattern callout.
  - **No `Cancel` option at this gate.** Scope-bounding is a per-finding decision; a `Cancel` here would have ambiguous semantics (cancel the finding's fix? the Issue? the run?). The three meaningful choices are the only choices. The user retains `Ctrl-C` for run-level abort if needed.

- **SR-5.3** — The end-of-run summary's "Accepted compromises" section (per SR-3.4) renders one bullet per tracker entry, surfacing the finding, the chosen path, the rationale, and the follow-up Issue ID (or `none`).
- **SR-5.4** — When the post-completion reviewer emits one or more `[compromise-challenge]` findings, the orchestrator's preamble (rendered as prose before presenting the findings) names this explicitly so the user reads the section with the right framing. Mirror the Step 5e verdict-keyed preamble pattern in `skills/quo-plan/SKILL.md` (the existing `approve` / `revise-recommended` / `escalate-to-user` preamble shapes).

### SR-6 — Edge-case and error-handling behavior

- **SR-6.1** — Malformed finding tags (severity not in `{blocker, suggestion, nit}` or depth not in `{trivial-tweak, refactor-locally, re-architect}`): orchestrator treats the finding as if `re-architect` depth (per SR-2.6 shim) AND surfaces the parse failure to the user so the reviewer's emission can be corrected.
- **SR-6.2** — User picks `Defer to follow-up Issue` but `/quo-file-issue` fails or the user cancels at one of its gates: orchestrator does NOT silently ship the soft fix; instead, re-prompts with the same three choices.
- **SR-6.3** — Compromise tracker file is missing or unreadable when the post-completion review runs: the post-completion review proceeds but flags the missing tracker explicitly in its output so the missing hand-off is visible.
- **SR-6.4** — Routing table somehow returns multiple decisions for a single finding (impossible by construction, but defensive): orchestrator defaults to user gate and surfaces the routing ambiguity.
- **SR-6.5** — Tracker accumulates >10 entries in a single run: the end-of-run summary surfaces them all (no truncation), with a short prologue noting the volume.
- **SR-6.6** — User picks a deep fix path at a gate but the implementer's diff turns out to implement the cheap path: post-completion review's `[compromise-challenge]` phase MUST catch the mismatch — the tracker records what the user asked for, the diff is what shipped, the reviewer compares.
- **SR-6.7** — **Auto-route depth-misclassification recovery.** When the post-completion review's `[compromise-challenge]` phase (per SR-4.5) flags an auto-routed `(1, refactor-locally)` decision as misjudged (the depth should have been `re-architect`), the orchestrator MUST surface this to the user with `AskUserQuestion` offering three finite choices:
  - `File follow-up Issue to revisit the depth decision` — orchestrator dispatches `/quo-file-issue` via the Skill tool capturing the depth-mismatch finding with the original compromise-tracker entry as context. The original tracker entry's `Follow-up Issue` field is updated to reference the new Issue ID.
  - `Accept the misjudgment and proceed` — orchestrator appends an "explicit override" entry to the same compromise tracker file capturing the user's decision to accept the misclassification despite the post-completion challenge. The new entry uses the same canonical shape (per the Data models section) with `Decision: User overrode auto-route after post-completion challenge` and the post-completion reviewer's challenge text as the Rationale.
  - `Pause to discuss` — orchestrator stops the post-completion flow at this point and surfaces a prose discussion with the user. The user can re-issue any of the three choices after the discussion completes.
  - This closes the auto-route audit loop: when reviewer mis-classification slips a depth past the routing-table gate, the post-completion challenge catches it and gives the user a remediation path rather than silently accepting the cheap fix.

## Background and rationale

### Proximate trigger

The b.bjp incident in this repo. `b.bjp` was an Issue requesting the addition of `/quo-plan`'s fresh-eyes plan-review gate (Step 5e). During the `/quo-fix-issue` run for b.bjp, two distinct review-loop findings were surfaced by in-flow reviewers, downgraded by the orchestrator (this very Claude instance, in the prior conversation that produced this SDD) via inline scope-bounding directives in R2 dispatch prompts, and shipped as known limitations:

1. **`findings:` payload contract drift.** The Code Reviewer and the Section 8 post-completion reviewer both flagged that `/quo-plan`'s Step 5e Revise-routing prose claimed plan-review findings had the same shape as spec-review findings, when in fact the shape differed (`target:` tag vs PRD/SDD-section anchor). The shipped fix rephrased the claim to honestly describe the shape difference, rather than broadening the writer-side contract to actually accept both shapes. The shipped prose at `skills/quo-plan/SKILL.md` (the b.bjp commit) acknowledged that "the writer's Revise pass is expected to absorb the differently-shaped items pragmatically" — a soft contract that would silently degrade if the writer skills ever tightened their `findings:` field expectations.
2. **Cancel-orphan foot-gun.** The Code Reviewer flagged that the new Step 5e Cancel branch promised idempotent resume, but Step 5a had no Plan-Bee-reuse detection (only Step 4a had Spec-Bee detection). The shipped fix rewrote the Cancel-branch prose to honestly tell the user to manually delete orphan `drafted` Plan Bee + Epic tickets before re-invoking `/quo-plan`. The R2 dispatch prompt to the Engineer explicitly said: `"Do NOT add a new Plan-Bee-reuse detection branch to Step 5a in this pass — that is out of scope for b.bjp."` — verbatim scope-bounding inside the dispatch prompt.

Both compromises were caught by the user only by manually invoking a separate fresh Claude Code session with holistic-judgment framing (`"could you please do a thorough review of all the recent work we did on issue b.bjp and look specifically for if the issue was well understood and we implemented a truly useful and well executed solution"`). That external session found both issues immediately and recommended both real fixes (broaden the writer-side contract; add Plan-Bee-reuse detection mirroring 4a). The user then asked the in-conversation Claude to fix both — which it did, demonstrating that the failure was workflow-level, not Claude-instance-level. Both fixes shipped successfully in a follow-up commit.

### Root cause

The current quorum workflow's orchestrator (the team-lead Claude running `/quo-fix-issue` or `/quo-execute`) has unilateral authority over three classes of decisions that, taken together, allow soft fixes to ship without user input:

1. **Pre-narrowing fix paths inside R2/R3 dispatch prompts.** When a reviewer surfaces a finding, the orchestrator decides which fix path to ask the Engineer to implement. The Engineer then implements the chosen path. The user never sees the menu.
2. **Scope-bounding findings inside R2/R3 dispatch prompts.** When the orchestrator decides a finding's full fix is "out of scope" for the current Issue, it inlines that as a directive in the dispatch prompt. The user never sees the scope-bounding decision.
3. **Downgrading fix depth.** When a reviewer surfaces a finding tagged `[suggestion]` whose underlying fix actually requires re-architecting, the orchestrator can treat it as discretionary because severity alone doesn't capture fix-depth.

Severity tags (`blocker` / `suggestion` / `nit`) describe *how important fixing-at-all is*, not *how much the fix-path choice matters*. The two dimensions are orthogonal: a `blocker`-severity finding can have a trivial-tweak fix path that the orchestrator can safely pick alone; a `suggestion`-severity finding can have a re-architect fix path that should belong to the user.

### Two design moves

This work depends on two recognitions that came up during the post-b.bjp design conversation:

1. **Findings have two orthogonal dimensions.** Severity says "how important is fixing-at-all"; depth says "what does fixing this cost". The orchestrator's routing decision should be based on the depth dimension (or the multi-path-or-not flag), not on severity alone.
2. **The orchestrator's authority to pre-narrow and scope-bound is the structural mechanism that lets soft fixes ship.** The depth dimension gives the orchestrator a deterministic basis for deciding when to gate the user; the anti-pattern callout removes the inline scope-bounding habit. The post-completion review reframe is necessary as a backstop (it catches compromises that legitimately accumulate even after the routing rules apply) but it is NOT sufficient on its own — by the time the post-completion review runs, the compromises have already shipped as part of the R2 commit.

### Why earlier post-completion-only fix was rejected

An earlier, narrower scoping of this work was to *only* reframe the Section 8 / Section 6 post-completion review dispatch prompts ("lead with holistic judgment, then defects"). The user pushed back: "is this really enough?". Honest analysis showed it wasn't — the post-completion review fires too late, after the orchestrator's compromise decisions have already shipped. Earlier gates at the routing-decision moment are the only structural fix; the post-completion reframe is necessary as a backstop but not load-bearing alone.

### Why this work is appropriately one cohesive Plan Bee

The five mechanical pieces (two-dimensional tagging, routing table, anti-pattern callout, compromise tracker, post-completion reframe) reinforce each other:

- Two-dimensional tagging without orchestrator routing rules is unused signal.
- Routing rules without two-dimensional tagging are unimplementable.
- The compromise tracker only matters if the orchestrator-discipline rules generate compromise entries.
- The post-completion review reframe only matters if it has a compromise tracker to challenge against.
- The anti-pattern callout only matters if the routing-decision gate exists as the surface that replaces the forbidden directives.

Shipping any one piece alone is a partial fix. One cohesive Plan Bee with several Epics representing the mechanical pieces is the right shape.

### Two-phase landing pattern

While this is one cohesive Plan Bee (the five mechanical pieces reinforce each other and shipping any one alone is a partial fix per the prior subsection), the 10 Epics decompose naturally into two phases with respect to the `b.wii` dependency. This is a landing-order recommendation, **not** a re-decomposition of the Plan Bee — the Epic dependency wiring already enforces Phase 2's ordering, and `/quo-breakdown-epic` can run on all 10 Epics regardless of which phase the user is in.

- **Phase 1 — Emission enrichment (Epics 1–6).** Per-skill emission extensions (severity + per-fix-path depth + enumerated fix paths across `/quo-engineer-review`, `/quo-doc-writer-review`, `/quo-test-writer-review`, `/quo-spec-review`, and `/quo-plan` Step 5e) plus reviewer-agent wrapper frontmatter alignment. All six Epics are independent (`up_dependencies` empty for each). **Phase 1 has zero `b.wii` exposure** — no new user gates fire under Phase 1; the depth tags become emitted but are not yet consumed by any orchestrator routing rule. Standalone value: severity tags become emitted for the first time across three implementer-side review skills (a real improvement in itself); depth + fix-path enumeration land everywhere; the reviewer-agent wrapper contracts get aligned with reality (those wrappers' frontmatter mentions severity tags today but the wrapped skills don't emit them — Phase 1 closes that pre-existing contract drift).

- **Phase 2 — Routing + tracker + post-completion reframe (Epics 7–10).** The orchestrator-discipline routing rules + anti-pattern callout + scope-bounding gate (Epic 7), the session-scoped compromise tracker with four append triggers (Epic 8), the "Accepted compromises" end-of-run summary surface (Epic 9), the post-completion review reframe + SR-4.5 / SR-4.6 plausibility checks + SR-6.7 three-choice recovery gate (Epic 10). **Phase 2 inherits the `b.wii` fragility per `## Risk / Known Dependencies` above** — the new user gates the routing table and recovery gate fire use the same prose-strengthening adherence mechanism that `b.wii` documents as intermittently failing.

**Recommended landing order:** Phase 1 first, Phase 2 second. Phase 2 should land either (a) after `b.wii` has a resolved structural fix, or (b) with explicit user acknowledgement at the start of the `/quo-execute` run for Phase 2 that the new gates may fail-as-prose intermittently. `/quo-execute b.ut9` will naturally land Phase 1 first (Epics 1–6 have no dependencies and surface before Epic 7 in any topological ordering), so the phasing is enforced by the existing dependency wiring without requiring `/quo-execute` to learn a "phase" concept. The user controls the Phase 1 → Phase 2 boundary by either continuing the run after Phase 1 completes or pausing to wait on `b.wii`.

### Why the orchestrator-adherence bug is a separate Issue (b.wii) rather than absorbed into this work

A second narrative thread, surfaced during this SDD's review pass: this design's user gates depend on orchestrator adherence (the orchestrator actually firing `AskUserQuestion` when the routing table or scope-bounding logic says it should). That adherence mechanism is prose strengthening (per `b.fpm`) and has demonstrably failed in the very session that authored this SDD. The structural fix for adherence does not have a settled design — candidate directions include harness-level enforcement and `tool_choice` injection at the API level, but no design conversation has resolved which to pursue.

Bundling the unsettled structural fix into this PRD/SDD's scope would be the exact soft-fix pattern this work is meant to prevent. Filing it as a separate Issue (`b.wii`) and documenting the dependency as a Risk callout (`## Risk / Known Dependencies` above) is the disciplined move: it gives the structural-fix conversation a discrete home, lets this SDD ship its mechanism with the current (fragile) adherence layer underneath, and ensures that when `b.wii` resolves, this SDD's gates inherit the fix automatically.

### Assumptions and calibration mechanisms

The design's user-gate firing decisions depend on reviewers classifying fix depth (`trivial-tweak` / `refactor-locally` / `re-architect`) plausibly. This is a load-bearing assumption (mirrored in the PRD's `## Assumptions`) but not a fire-and-forget one — without an explicit calibration mechanism, two failure modes are possible:

- **Reviewers systematically err high** (everything tagged `re-architect`): every finding fires a user gate; the friction-targeting goal regresses; users see the design as "too many gates" and resort to reflexive Approve clicks that re-hide the substance.
- **Reviewers systematically err low** (everything tagged `trivial-tweak`): the depth gate stops firing for real `re-architect`-grade fixes; the design silently fails because the orchestrator auto-routes work that should have been a user gate.

The SDD addresses this with three layered calibration mechanisms:

1. **Per-skill worked examples anchoring depth classification (SR-1.2).** Each in-flow review skill's prose includes at least one worked example per depth bucket so reviewers anchor their classifications against concrete instances of `trivial-tweak`, `refactor-locally`, and `re-architect`. This is a per-skill SKILL.md edit, included in Epics 1–4's acceptance criteria. The examples ground the depth definitions so reviewers don't drift on personal interpretation.

2. **Post-completion plausibility checks (SR-4.5).** The post-completion review's `[compromise-challenge]` phase explicitly evaluates whether the depth tags on auto-routed findings were plausible. When the reviewer judges a `(1, refactor-locally)` decision was actually `re-architect` work, the reviewer emits a `[compromise-challenge]` finding with the misclassification as the substance. This is the per-run feedback channel.

3. **Drift-revisit expectation (documented, not code-enforced).** When the post-completion review surfaces sustained patterns of depth misclassification across multiple runs (observable via the `[compromise-challenge]` rate on auto-routed findings in the end-of-run summaries), the team revisits the per-skill worked examples (mechanism 1) to recalibrate. This is documented as an expectation in this subsection, not a code-level acceptance criterion. The trigger is qualitative ("sustained pattern of misclassification observed in summary reports"); the revisit shape is a SKILL.md prose edit refining the worked examples to better disambiguate the buckets the reviewers were getting wrong.

These three mechanisms are layered, not redundant: (1) anchors classification at emission time, (2) catches misclassification at post-completion time, (3) drives prose-level recalibration when (2) surfaces patterns. Future readers (Engineer / PM / Doc Writer dispatched by `/quo-execute`) and future maintainers (anyone editing the in-flow review skills) MUST preserve this layered structure rather than relying on one mechanism alone. Removing (3) without a replacement turns the design into a fire-and-forget assumption that the user pushed back against in the plan-review gate; removing (2) breaks the auto-route audit loop (SR-6.7 depends on it); removing (1) leaves reviewers without anchors.

## Decisions and rejected alternatives

### Decisions

- **Compromise tracker as a markdown file under `<tempdir>/.quorum/`, not TaskList entries.** TaskList signals active in-flight work; compromise entries are post-decision session metadata that the post-completion review consumes as input data. Mixing them muddles TaskList semantics. Mirrors the existing scratch-file convention precedents (Plan Bee body, Epic body, etc.).
- **Routing table follows the `skills/quo-plan/SKILL.md` Step 4c Loop-back UX table shape** (lines 254–258). Two columns: input condition, routing action. Same prose discipline (`AskUserQuestion` choices listed, counter-anchor clause present). Keeps future readers' cognitive load low by reusing an established prose pattern.
- **Three depth buckets: `trivial-tweak`, `refactor-locally`, `re-architect`.** Three is enough to capture the practical decision space cleanly: trivial and refactor-locally auto-route; only re-architect (or multi-path) gates. Two buckets (`trivial` / `significant`) was considered and rejected because the `refactor-locally` middle ground is genuinely distinguishable in reviewer judgement and useful for the auto-route-but-log-to-tracker case.
- **Reviewers carry the burden of enumerating fix paths.** The orchestrator may augment a reviewer's enumeration but MUST NOT silently narrow it. The orchestrator's role is routing and synthesis, not design authority. Captured as the SR-1.2 requirement and the SR-2.5 anti-pattern callout.
- **Anti-pattern callout lives alongside the orchestrator-discipline section** (SR-2.5) rather than as a separate file or a CONTRIBUTING.md addition. Keeps the rule near the routing table that depends on it; one SKILL.md edit per orchestrator skill.
- **Backwards-compatibility shim treats missing-depth findings as `re-architect`-depth** (SR-2.6). Errs toward user gate when uncertain. Lets the rollout land incrementally — the orchestrator routing rule can ship before all the in-flow review skills update their emission, and the shim ensures the system is safe during the gap.
- **Compromise tracker passed to post-completion review via file path (`Read`-tool mechanism), not inlined into the dispatch prompt.** Smaller dispatch prompts amortize prompt cache better across multi-Epic runs. Establishes a new repository-wide pattern (`Read` against `.quorum/` artifacts) since no existing precedent exists.
- **Single-path `refactor-locally`-depth findings auto-route + log to the compromise tracker** (per SR-2.4 and SR-3.2). They do NOT fire a user gate. The depth is non-trivial but the path is unambiguous; the tracker entry suffices for retrospective audit (and the post-completion review's `[compromise-challenge]` phase will challenge the choice if it was wrong). Earlier drafts framed this as an open architectural question; now committed. The auto-route audit loop is closed by SR-6.7's depth-misclassification recovery gate, which fires when the post-completion review flags an auto-routed depth as misjudged.
- **Cancel semantics at the routing gate are per-Issue (in `/quo-fix-issue`) and per-Task (in `/quo-execute`), not whole-run.** The user opted into the multi-finding batch when invoking the skill; a single bad routing decision should let them skip the offending unit cleanly without invasive `Ctrl-C`. The scope-bounding gate does NOT include a `Cancel` option because the semantics would be ambiguous (cancel which level?); the user retains `Ctrl-C` for run-level abort if needed.
- **Depth calibration mechanism is three-layered: per-skill worked examples + post-completion plausibility checks + drift-revisit expectation.** Documented in `## Background and rationale` → "Assumptions and calibration mechanisms". The user surfaced this as a plan-review-gate finding (the design previously assumed reviewer calibration without naming a mechanism). The three layers are intentional and additive: examples anchor classification at emission time, post-completion checks catch misclassification per-run, drift-revisit drives prose-level recalibration when patterns emerge across runs.
- **Structural fix for narrate-instead-of-do (orchestrator-adherence) is filed as a separate Issue (`b.wii`), not bundled into this work.** The SR-level requirements above depend on the orchestrator firing `AskUserQuestion` when prescribed; that mechanism (prose strengthening per `b.fpm`) is currently fragile. Bundling the unsettled structural fix would be the soft-fix pattern this design is meant to prevent. Tracked separately as `b.wii` with a Risk callout above; this SDD lands on the current mechanism with explicit acknowledgement.

### Rejected alternatives

- **Adding `severity:critical` above `blocker`.** Rejected. The problem isn't severity granularity; it's the orthogonal depth dimension. A `blocker`-severity finding with a `trivial-tweak` fix path still auto-routes correctly under the new rules. The failure mode is `suggestion`-severity findings with `re-architect` fix paths getting treated as discretionary — that mismatch is what the depth dimension addresses, not severity granularity.
- **TaskList as the compromise tracker mechanism.** Rejected. TaskList signals active in-flight work; compromise entries are post-decision session metadata that the post-completion review consumes as input data. Reading TaskList entries as input data is awkward — they aren't a list of tasks to do, they're a list of decisions already made.
- **A new `compromise-reviewer` subagent under `agents/`.** Rejected. The post-completion review reframe (SR-4) already empowers the existing `general-purpose` reviewer to challenge compromises. A new subagent bumps the precondition-check count in `/quo-execute` and `/quo-breakdown-epic` (seven becomes eight) and `/quo-fix-issue` (eight becomes nine) without commensurate value.
- **Blanket user gate per finding regardless of depth.** Rejected. Most findings are `trivial-tweak` depth (typos, prose nits); blanket gates would create unbearable friction. The depth-aware routing targets friction at exactly the decisions that matter.
- **Persistent compromise tracker across runs (a ticketed bees artifact).** Rejected. Compromises are scoped to a single run's review-loop discipline. Cross-run persistence would conflate accepted-compromise records with active-work tracking, and the post-completion review only needs current-run context to do its job.
- **Extending `/quo-spec-review`'s contract to also cover plan-substance critique.** Rejected for the same reasons it was rejected during b.bjp's design: two skills with distinct concerns (checklist content review vs holistic plan critique) are cleaner than one with two modes.
- **Only reframing the post-completion review** (and not adding routing rules or the compromise tracker). Rejected. The post-completion review fires too late — by the time it runs, the orchestrator has already shipped the compromise. The framing fix is necessary as a backstop but not sufficient on its own.
- **Pre-narrowing fix paths inside R2/R3 dispatch prompts but adding a "challenge the narrowing later" gate.** Rejected. This would be a complex two-phase mechanism that achieves what a simple user-gate-before-narrowing already does. The simpler design wins.
- **Inlining the compromise tracker contents into the post-completion review dispatch prompt.** Considered and set aside in favor of the `Read`-against-file-path mechanism. Inline is viable but increases dispatch-prompt size, which works against prompt-cache amortization on multi-Epic runs. Captured here so a future contributor knows the alternative was considered.
- **A two-bucket depth ladder (`trivial` / `significant`).** Rejected. Loses the `refactor-locally` middle ground that the orchestrator can safely auto-route (with a tracker entry as the audit-time backstop). Three buckets capture the practical decision space cleanly without unnecessary granularity.
- **Gating the user on single-path `refactor-locally`-depth findings.** Considered and rejected. The path is unambiguous; gating would add friction for a decision the reviewer has already made. Tracker logging + post-completion `[compromise-challenge]` (per SR-6.7's recovery loop) is sufficient backstop.
- **Bundling the structural fix for orchestrator adherence into this SDD's scope.** Rejected. No settled design exists for the structural fix (candidates include harness-level enforcement, `tool_choice` injection, pre-commitment self-check). Adding scope without a design path is over-promising — exactly the failure mode this design is meant to prevent. Tracked as `b.wii` with a Risk callout above.
- **Treating orchestrator-adherence as a separate `## Existing Behavior` carve-out rather than a Risk.** Considered and rejected. The adherence-mechanism preservation IS noted in `## Existing Behavior` (the trailer-pattern bullet), but the load-bearing dependency on the broken adherence is the substantive point — naming it as a Risk surfaces the dependency direction and gives future readers a place to look when the gates misbehave.
- **A `Cancel` option at the scope-bounding gate.** Rejected. The three meaningful choices for a scope-bounding decision are `Fix properly now` / `Defer to follow-up Issue` / `Accept the limitation`; a `Cancel` here would be ambiguous (cancel the finding's fix? the Issue? the run?). The user retains `Ctrl-C` for run-level abort if needed. Per-Issue / per-Task scoping for the routing gate's `Cancel` choice (which IS present, per SR-5.1) is the cleaner contract.
- **Whole-run `Cancel` at the routing gate.** Considered and rejected. The user opted into the multi-finding batch when invoking `/quo-fix-issue all` or `/quo-execute <bee-id>`; a single bad routing decision shouldn't kill the whole batch. Per-Issue / per-Task abort lets the user skip the offending unit cleanly without losing the rest of the work; `Ctrl-C` remains the whole-run escape hatch.
- **Reviewer-depth calibration as fire-and-forget assumption.** Considered and rejected. The plan-review gate flagged this as a load-bearing assumption with no mechanism. The three-layered calibration mechanism (per-skill examples + post-completion plausibility checks + drift-revisit expectation) replaces the fire-and-forget assumption with an explicit, layered feedback loop. Documented in `## Background and rationale` → "Assumptions and calibration mechanisms".
- **Auto-route depth-misclassification as silent acceptance.** Rejected. Earlier drafts treated the `(1, refactor-locally)` auto-route as effectively unchallengeable post-shipping — the tracker entry was an audit record but had no remediation path. SR-6.7 closes the loop: when the post-completion review's `[compromise-challenge]` phase flags an auto-routed depth as misjudged, the user gets a three-choice gate (file follow-up Issue / accept the misjudgment / pause to discuss) rather than a silent ship.
