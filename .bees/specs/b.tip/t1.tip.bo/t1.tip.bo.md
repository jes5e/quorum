## Problem Statement

The quorum workflow has a recurring failure mode where reviewers find substantive design weaknesses but the orchestrator (the team-lead Claude running `/quo-fix-issue` or `/quo-execute`) ships soft fixes anyway. The orchestrator currently has unilateral authority to (a) pick between fix paths surfaced by a reviewer, (b) scope-bound findings as "out of scope" inside an R2/R3 dispatch prompt, and (c) downgrade fix depth — all without surfacing those decisions to the user. The post-completion review then evaluates the shipped state as the new baseline, with defect-enumeration framing that does not invite pushback on accepted compromises.

The user observable: real foot-guns, contract drifts, and design weaknesses ship as known limitations and accumulate over time. The b.bjp incident is one data point: two real issues (`findings:` payload contract drift between `/quo-plan`'s plan-review gate and the writer skills, and a Cancel-orphan foot-gun in `/quo-plan` Step 5e) were caught by in-flow reviewers, downgraded to soft fixes by the orchestrator via verbatim "Do NOT add detection — out of scope" directives in the R2 dispatch prompt, and shipped. The user only caught them by manually running a fresh-eyes review with holistic-judgment framing in a separate Claude Code session — proving the failure is workflow-level, not Claude-instance-level.

This problem affects every user of `/quo-fix-issue` and `/quo-execute`. It worsens as those skills are invoked more frequently, because soft-fix accumulation is monotonic. It matters now because the project's compounding self-improvement loop (the skills review each other's output and inform each other's design) means every uncorrected soft-fix becomes an upstream cause for future failures.

## Goals

1. **Compromises become explicit, user-owned decisions, not silent orchestrator picks.** When a reviewer surfaces a finding with multiple plausible fix paths, OR a single fix path of substantial depth, the user picks the path via `AskUserQuestion` before any implementer agent is dispatched.
2. **The "out of scope" directive disappears from R2/R3 dispatch prompts.** Scope-bounding becomes a user gate (`Fix properly now` / `Defer to follow-up Issue` / `Accept the limitation`), never an inline directive the orchestrator writes to itself or to a dispatched agent.
3. **Accepted compromises remain visible after the run ends.** A session-scoped compromise tracker accumulates one entry per accepted compromise; the end-of-run summary surfaces the tracker; the post-completion review consumes the tracker as input and is empowered to challenge specific compromise decisions.
4. **The b.bjp failure mode would have been caught by the new gates.** Replaying b.bjp under these rules: both surviving issues (`findings:` contract drift and Cancel-orphan foot-gun) fire user gates rather than shipping as soft fixes. This is the validation criterion the design must satisfy.
5. **Friction is targeted, not blanket.** The most common finding shape — trivial-tweak severity-`nit` typo fixes — auto-routes silently. User gates fire only on decisions that materially affect the shipped solution's design quality.

## Non-Goals / Out of Scope

- **Changing the severity vocabulary.** `blocker` / `suggestion` / `nit` stays. The problem isn't severity granularity — it's the missing orthogonal depth dimension. Adding more severity buckets without depth would not fix the failure mode.
- **Introducing new reviewer subagent roles under `agents/`.** Depth tagging and compromise tracking are content-level extensions to existing review skills and orchestrator prose. The seven-subagent precondition counts in `/quo-execute` and `/quo-breakdown-epic` and the eight-subagent count in `/quo-fix-issue` are unchanged.
- **Changing `/quo-spec-review`'s checklist contents or output shape beyond emission tagging.** The twelve PRD checks and seven SDD checks stay intact. Only the per-finding emission extends with the depth dimension.
- **Changing `/quo-plan` Step 5e's plan-review gate substance.** Step 5e is already holistic-judgment framed and already produces the kind of findings this PRD's routing rules need. The only change to Step 5e is that its embedded reviewer's emission learns the depth dimension so its findings flow into the new orchestrator routing cleanly.
- **Changing `/quo-breakdown-epic`'s PM Spec Traceability Review.** That review is a coverage matrix (every spec requirement → at least one Task / Subtask), not the failure mode this PRD addresses.
- **Persisting the compromise tracker across runs as a ticketed bees artifact.** Compromises are session-scoped workflow state; persisting them across runs would conflate accepted-compromise records with active-work tracking. Cross-run persistence is a deliberate future-work omission, not a defect.
- **Fixing the narrate-instead-of-do orchestrator-adherence bug.** This PRD's new user gates depend on the orchestrator actually firing them when prescribed; that adherence mechanism is currently prose-strengthening (per `b.fpm`) and has demonstrably failed (per `b.wii`). The structural fix for adherence is tracked separately as `b.wii` — see `## Risk / Known Dependencies` below. This PRD does NOT attempt the structural fix; it lands on the current mechanism with explicit acknowledgement of the fragility.

## Functional Requirements

The functional requirements are grouped by the surface where each requirement applies.

### FR-1 — In-flow review skills emit two-dimensional findings

The four in-flow review skills (`/quo-engineer-review`, `/quo-doc-writer-review`, `/quo-test-writer-review`, `/quo-spec-review`) and the two embedded reviewer prompts (the post-completion sweep dispatch in `/quo-fix-issue` Section 8 and the analogous sweep in `/quo-execute` Section 6) and the embedded plan-review dispatch in `/quo-plan` Step 5e MUST emit findings that carry, for each finding:

- A severity tag, exactly one of `blocker`, `suggestion`, or `nit`.
- One or more fix paths. Each path carries its own depth tag, exactly one of `trivial-tweak`, `refactor-locally`, or `re-architect`.

Single-fix-path findings remain valid when there is genuinely only one plausible fix. The burden is on the reviewer to think about plausible alternatives and surface them when they exist.

### FR-2 — Orchestrator routes findings per a deterministic table

`/quo-fix-issue` and `/quo-execute` MUST contain an "Orchestrator discipline: routing review findings" section that defines a routing decision for every finding shape:

| Number of fix paths | Maximum depth across paths | Orchestrator routing                         |
|---------------------|---------------------------|---------------------------------------------|
| 1                   | trivial-tweak             | Auto-dispatch the implementer with the fix  |
| 1                   | refactor-locally          | Auto-dispatch the implementer with the fix  |
| 1                   | re-architect              | User gate before dispatching                |
| > 1                 | (any)                     | User gate before dispatching                |

Given any `(num-paths, max-depth)` tuple, exactly one routing decision applies. The orchestrator does not invent its own classification; it reads the reviewer's emitted tags and consults the table.

### FR-3 — Scope-bounding is a user gate, never a dispatch-prompt directive

When the orchestrator decides a finding's full fix should be deferred or accepted-as-known-limitation, the decision MUST surface via `AskUserQuestion` with finite choices:

- `Fix properly now` — proceed with the deep fix path the reviewer surfaced.
- `Defer to follow-up Issue` — file a follow-up Issue via `/quo-file-issue` capturing the deferred work; ship the soft fix (or no fix) in the current run.
- `Accept the limitation` — no fix in the current run, no follow-up Issue filed; document the limitation in the compromise tracker and surface it in the end-of-run summary.

The orchestrator MUST NOT inline phrases like "Do NOT do X — out of scope for this Issue" inside an R2/R3 dispatch prompt to an implementer agent. The anti-pattern callout in the orchestrator-discipline section forbids these directives and gives the user the choice instead.

### FR-4 — A session-scoped compromise tracker captures every accepted compromise

`/quo-fix-issue` and `/quo-execute` MUST maintain a session-scoped compromise tracker that captures one entry per accepted compromise. An accepted compromise is any of:

- The user picked `Defer to follow-up Issue` at a scope-bounding gate.
- The user picked `Accept the limitation` at a scope-bounding gate.
- A finding routed auto-dispatch under the `refactor-locally`-depth single-path branch (the orchestrator did not gate the user, but the depth was non-trivial — captured for retrospective audit).
- The user picked `Accept the misjudgment and proceed` at the post-completion review's auto-route depth-misclassification recovery gate (per the SDD's SR-6.7 recovery gate). The orchestrator records an explicit-override entry in the tracker capturing the user's decision to accept the misjudgment despite the post-completion challenge.

Each tracker entry MUST carry:

- The finding verbatim (severity tag, fix paths with depth tags, descriptive body).
- The decision the user made (or the auto-route decision for `refactor-locally`).
- A one-line rationale (user's stated reason, or the auto-route rule that fired).
- The follow-up Issue ticket ID if one was filed; `none` otherwise.

The tracker MUST persist across orchestrator-yield events within a single run (it is not lost when the orchestrator yields waiting for a background Agent), and MUST NOT persist across separate runs (a fresh `/quo-fix-issue` invocation starts with a fresh tracker).

### FR-5 — The compromise tracker surfaces in the end-of-run summary

At end-of-run (Section 7's per-Issue summary in `/quo-fix-issue`, the equivalent summary surface in `/quo-execute`), the compromise tracker contents MUST appear as a labeled section. When the tracker is empty (no compromises were accepted during the run), the section MUST be omitted entirely so the summary stays clean.

### FR-6 — The post-completion review challenges accepted compromises

The post-completion review dispatch prompt (Section 8 in `/quo-fix-issue`, Section 6 in `/quo-execute`) MUST lead with a "challenge accepted compromises" phase that consumes the compromise tracker as input. The reviewer MUST evaluate each tracked compromise on its own merits and push back when the chosen path is not defensible. Findings MUST be tagged as one of:

- `[compromise-challenge]` — pushback on a specific tracker entry.
- `[design]` — holistic finding about solution quality not tied to a tracked compromise.
- `[defect]` — discrete bug / prose / contract issue.

Each finding additionally carries the severity tag and per-fix-path depth tag established by FR-1.

**Tracker hand-off mechanism.** The orchestrator passes the compromise tracker into the dispatched post-completion reviewer as a **file path**, not as inlined content; the dispatched general-purpose Agent reads the tracker via its `Read` tool. This keeps the dispatch prompt small (better prompt-cache amortization on multi-Epic runs) and leaves the tracker's markdown structure intact for the reviewer to parse.

### FR-7 — Backwards-compatibility shim during rollout

The orchestrator routing table (FR-2) MUST tolerate findings that lack depth tags (older reviewer emission or hand-authored findings from a future call site). A finding without a depth tag MUST be treated as if `re-architect` depth, which routes the orchestrator to a user gate. The shim errs toward user input when uncertain, consistent with the workflow's broader "err toward reuse / err toward surface" principles.

## Edge Cases and Error Handling

- **Reviewer emits malformed finding tags.** When a finding's severity tag is not exactly one of `blocker` / `suggestion` / `nit`, or a fix path's depth tag is not exactly one of `trivial-tweak` / `refactor-locally` / `re-architect`, the orchestrator MUST treat it as if depth-tagged `re-architect` (per FR-7) AND surface the malformed finding to the user with the parse failure noted, so the reviewer's emission can be corrected for future runs.
- **User picks `Defer to follow-up Issue` but `/quo-file-issue` fails or is cancelled.** The orchestrator MUST NOT silently ship the soft fix; instead, surface the failure to the user and re-prompt with the original gate's three choices. The user can re-attempt the defer, accept the limitation explicitly, or pick the deep fix path.
- **The compromise tracker file is missing or unreadable when the post-completion review runs.** The post-completion review MUST proceed but flag the missing tracker explicitly so the user knows the "challenge accepted compromises" phase ran with empty input. This guards against silent regression where a future skill change accidentally drops the tracker hand-off.
- **A reviewer surfaces a finding the orchestrator cannot route deterministically** (e.g., the routing table somehow returns multiple decisions, which should be impossible by construction). The orchestrator MUST default to user gate and surface the routing ambiguity to the user, so the issue is visible rather than papered over.
- **A single run accumulates so many compromise tracker entries that surfacing them in the end-of-run summary becomes unwieldy** (more than ~10). The summary MUST surface them anyway in full, with a short prologue noting the volume — the alternative (truncation) would hide exactly the signal the tracker exists to preserve.
- **The user picks the deep fix path at a gate, but the implementer agent's diff turns out to implement the cheap path anyway** (because the depth was misjudged or the dispatch prompt was unclear). The post-completion review's `[compromise-challenge]` phase MUST catch this — the tracker entry records what the user asked for; the diff is what shipped; the reviewer flags the mismatch.
- **The orchestrator emits the user-gate prescription as prose and yields instead of firing the gate** (the narrate-instead-of-do failure mode tracked separately as `b.wii`). This is a known dependency, not an edge case this PRD itself fixes — see `## Risk / Known Dependencies` below.

## Non-Functional Requirements

- **Cross-platform compliance.** Any new shell snippets for the compromise tracker (file path resolution, write-append patterns) MUST come in paired POSIX bash + Windows PowerShell variants per the project's CLAUDE.md design rule #2. No bash-only fallback.
- **Project-neutral / stack-agnostic.** The orchestrator routing rules MUST NOT assume language-specific tooling, build commands, or test frameworks. The rules read finding tags and route accordingly; the underlying language of the project being worked on is irrelevant.
- **Scratch-file convention.** The compromise tracker file MUST live under `<tempdir>/.quorum/` (per the project's scratch-file convention in CLAUDE.md), MUST be created via the existing `mkdir -p` / `New-Item -ItemType Directory -Force` patterns, and MUST NOT be deleted by the skill (the OS reclaims `<tempdir>` on its own cadence).
- **No new tools, agents, or bees state.** The compromise tracker is a markdown file; routing is orchestrator prose; finding tags are emission extensions. The change adds zero new precondition checks to any of the existing entry-point skills (`/quo-fix-issue`, `/quo-execute`, `/quo-breakdown-epic`).
- **Backwards compatibility during rollout.** Per FR-7, the system MUST tolerate mixed-emission runs (some reviewers depth-tagging, others not) so the rollout can land incrementally without requiring an atomic all-or-nothing change.

## UI/UX Requirements

The user-visible surface changes are:

- **New `AskUserQuestion` gate shape when the orchestrator decides to scope-bound a finding** (per FR-3 and SR-5.2). Three finite choices: `Fix properly now` / `Defer to follow-up Issue` / `Accept the limitation`. The question text includes the finding verbatim and one-line context for why the gate is firing. **Cancel semantics:** there is NO `Cancel` option at this gate. Scope-bounding is a per-finding decision; a `Cancel` here would have ambiguous semantics (cancel the finding's fix? the Issue? the run?). The user retains `Ctrl-C` for run-level abort if needed.

- **New `AskUserQuestion` gate shape when the orchestrator routes a multi-path or single-path-re-architect finding** (per FR-2 and SR-5.1). Choices are one per reviewer-surfaced fix path PLUS a `Defer to follow-up Issue` option AND a `Cancel` option. Each fix-path choice's description includes the depth tag so the user sees the cost of the chosen path. The Recommended marker is applied to the path the reviewer flagged as preferred when the reviewer surfaced a preference; otherwise no path is marked Recommended.

  **Cancel semantics at this gate:** in `/quo-fix-issue`, `Cancel` aborts the **current Issue's fix run** — the orchestrator stops processing that Issue and proceeds to the next Issue in the batch (in list / `all` mode), or ends the run cleanly if no more Issues remain. In `/quo-execute`, `Cancel` aborts the **current Task's execution** — the orchestrator stops processing that Task and proceeds to the next Task, or ends the run if no more Tasks remain. The Cancel choice does NOT abort the entire `/quo-fix-issue` or `/quo-execute` run; that level of abort remains the user's `Ctrl-C` option. This scoping is per-Issue or per-Task because the user opted into the multi-finding batch when they invoked the skill, and a single bad routing decision in one Issue / Task should not invalidate the whole batch — but the per-Issue / per-Task abort lets the user skip the offending unit cleanly without invasive `Ctrl-C`.

- **End-of-run summary gains a "Accepted compromises" section when the tracker is non-empty.** The section lists one bullet per tracker entry showing the finding, the chosen path, the rationale, and the follow-up Issue ID (or `none`). When the tracker is empty, the section is omitted.
- **Post-completion review prose preamble surfaces "compromises challenged" outcomes prominently.** When the post-completion reviewer emits one or more `[compromise-challenge]` findings, the orchestrator's preamble before presenting the findings says so explicitly so the user reads the section with the right framing.

All gates use `AskUserQuestion` per CLAUDE.md `## AskUserQuestion usage` (multi-choice only). No fake "Use my own answer" options pointing at the auto-appended `Chat about this` slot.

The output format is documented in each skill's SKILL.md and the dispatch-prompt templates produce findings in the documented shape on at least one end-to-end run of each skill.

The table is mechanically followable: given any `(num-paths, max-depth-across-paths)` tuple, exactly one routing decision applies (auto-dispatch vs user-gate-then-dispatch).

The anti-pattern callout in the SKILL.md prose names these forbidden phrasings explicitly.
4. A compromise tracker file at `<tempdir>/.quorum/compromises-<session>.md` exists after any `/quo-fix-issue` or `/quo-execute` run that surfaced at least one user gate or auto-routed at least one `refactor-locally`-depth finding. Each tracker entry carries the finding, fix paths surfaced, decision, rationale, and follow-up Issue ID if filed.
5. The end-of-run summary in `/quo-fix-issue` and `/quo-execute` shows the compromise tracker contents (or omits the section entirely if the tracker is empty). A user reading the summary can immediately see which compromises were accepted.
6. The post-completion review dispatch prompt in `/quo-fix-issue` Section 8 and `/quo-execute` Section 6 includes a `[compromise-challenge]` instruction phase that consumes the compromise tracker as input. Findings emitted by the post-completion reviewer carry one of `[compromise-challenge]` / `[design]` / `[defect]` tags.
7. **b.bjp walk-through validation.** Replaying the b.bjp run mentally under the new rules: the `findings:` contract drift finding (Code Reviewer + Section 8) surfaces as multi-path (option (a) trivial-tweak vs option (b) re-architect) → user gate fires. The Cancel-orphan foot-gun finding (Code Reviewer) surfaces as multi-path (option (a) trivial-tweak prose rewrite vs option (b) re-architect Plan-Bee-reuse detection) → user gate fires. The orchestrator does NOT inline "Do NOT add detection — out of scope" directives. Both issues land with the user-chosen fix depth.
8. A backwards-compatibility regression test: invoking the orchestrator's routing logic on a finding with no depth tag results in a user gate firing (the shim treats missing-depth as `re-architect`). This guarantees the rollout can land incrementally.

## Assumptions

- **Reviewers can reasonably classify fix depth.** The reviewer-side burden of judging `trivial-tweak` vs `refactor-locally` vs `re-architect` is small relative to the cost of the failure mode this PRD addresses. Misclassifications err toward `re-architect` (user gate), which is the safer default.
- **Users prefer occasional friction over silent foot-guns.** The friction added by user gates on multi-path and single-path-re-architect findings is acceptable to users in exchange for visibility into compromise decisions. The b.bjp incident's user response (manually running a fresh-eyes review in a separate session) is evidence that the user wants this visibility today and is paying friction cost out-of-band to get it.
- **The post-completion review can be trusted with `[compromise-challenge]` authority.** A fresh-eyes reviewer reading the compromise tracker can meaningfully evaluate whether a chosen compromise was defensible. The reviewer is not constrained by the orchestrator's prior framing because the reviewer is dispatched cold and reads the tracker as input data, not as authoritative state.
- **In-flow reviewers can be updated in lockstep with the orchestrator routing change** (via Change A's depth-tagging emission), so the backwards-compatibility shim (FR-7) is a rollout safety net rather than a long-term contract surface.
- **Orchestrator adherence to user-gate prescriptions is best-effort, not guaranteed.** This PRD's gates use the same prose-strengthening adherence mechanism as `b.fpm` and `/quo-spec-review`'s trailers; that mechanism has demonstrably failed (per `b.wii`). The PRD lands on the current mechanism with explicit risk acknowledgement — see `## Risk / Known Dependencies` below.

## Risk / Known Dependencies

This PRD has one load-bearing dependency on a separate, unresolved Issue, and that dependency is honestly documented here rather than buried.

**Dependency on `b.wii` (Narrate-instead-of-do failure mode persists after b.fpm's prose-strengthening fix).**

- **What `b.wii` is.** A known bug in the workflow's orchestrator-adherence mechanism: when skill prose prescribes a tool call via the second-person imperative + counter-anchor pattern (e.g., `**Your next tool call MUST be AskUserQuestion** ... Do not produce a text response describing this gate — call the tool directly.`), the orchestrator sometimes emits the prescription as prose and yields the turn instead of firing the prescribed tool call. `b.fpm` was a prior fix attempt that strengthened the trailer prose; the bug persists. `b.wii` was filed during this PRD's authoring after the failure was reproduced live in the very session — `/quo-spec-review`'s Shape 2 trailer was emitted as text and the gate did not fire until the user surfaced the failure.
- **Why this PRD depends on `b.wii`.** This PRD's new user gates (per FR-3, FR-5, FR-6 surfaces and the orchestrator-discipline routing rules they add to `/quo-fix-issue` and `/quo-execute`) inherit the same prose-strengthening adherence mechanism. The gates prescribe `**Your next tool call MUST be AskUserQuestion**`-shaped trailers; the orchestrator must actually fire those gates rather than narrate them. If `b.wii` is not structurally fixed, the gates this PRD adds are exposed to the same failure mode at new surfaces.
- **Dependency direction and shape.** This PRD lands on the **current** adherence mechanism (the only one available); it does NOT attempt the structural fix for `b.wii`. The structural fix needs a design conversation (candidate directions: harness-level enforcement, `tool_choice` injection, pre-commitment self-check, etc. — see `b.wii`'s body for the full enumeration). If `b.wii` resolves before this PRD's Epics land, the gates become reliable from the start. If `b.wii` is still open when this PRD ships, the gates work in the common case but inherit the same fragility — the orchestrator may emit a trailer as prose and yield, requiring the user to surface the missed gate (the same manual remediation that already exists for `/quo-spec-review`'s and `/quo-plan` Step 5e's existing gates).
- **Why this PRD is being landed anyway.** Bundling the structural fix into this PRD without a settled design would be exactly the soft-fix pattern this PRD is meant to prevent. The right move is to land this PRD's mechanisms (routing table, compromise tracker, anti-pattern callout, post-completion reframe) using the current adherence mechanism, document the dependency honestly here and in the SDD, and resolve `b.wii` separately with its own design conversation. When `b.wii` resolves, this PRD's gates inherit the structural fix at no additional cost — there is nothing to revise here because the gate-firing mechanism is referenced through the trailer convention, not duplicated.
- **What changes if `b.wii` is rejected or de-prioritized.** This PRD's value is not zero in that case — it still removes the orchestrator's pre-narrowing and scope-bounding authority, surfaces compromise decisions to the user, and gives the post-completion review the compromise tracker to challenge. The user-gate firing remains best-effort under prose strengthening alone. The fallback for missed gates remains the same as today: the user notices and prompts the orchestrator to fire it.

This is the only known dependency. Other risks (reviewer mis-classification of depth, runaway tracker volume, etc.) are addressed in `## Edge Cases and Error Handling` and `## Assumptions`.

## Background and rationale

The b.bjp incident is the proximate trigger but not the root cause. The root cause is structural: the quorum workflow's orchestrator has been granted unilateral authority over fix-path selection and scope-bounding because in most cases the orchestrator's pick is the right one and surfacing every decision would be unbearable friction. But the workflow currently has no mechanism to distinguish "this decision is one the orchestrator can make safely" from "this decision shapes the solution and should be the user's." Severity tags (`blocker` / `suggestion` / `nit`) don't capture that distinction because severity is about how important fixing-at-all is, not about how much the fix-path choice matters.

The b.bjp review-of-review chain made the gap visible. Quorum's reviewers found both surviving issues (`findings:` contract drift and Cancel-orphan foot-gun) in-flow. The orchestrator wrote dispatch prompts to the R2 Engineer that pre-narrowed the fix paths in the prompts ("Option (a) is cheaper / lower-risk fix. Prefer (a) unless..." for one; "Do NOT add a new Plan-Bee-reuse detection branch to Step 5a in this pass — that is out of scope for b.bjp." for the other). The R2 Engineer dutifully implemented the cheap option and shipped. The post-completion review evaluated the shipped state as the new baseline.

The user discovered the failures by manually invoking a fresh Claude Code session with holistic-judgment framing ("could you please do a thorough review of all the recent work we did on issue b.bjp and look specifically for if the issue was well understood and we implemented a truly useful and well executed solution"). That session found both issues and challenged the chosen compromises. The workflow needs to bring that holistic challenge inside the workflow, not leave it to the user's discretion to invoke separately.

Two design moves came together to produce this PRD. The first was the realization that findings have two orthogonal dimensions, not one: severity and depth. The second was the realization that the orchestrator's authority to pre-narrow fix paths and scope-bound findings is the structural mechanism that lets soft fixes ship. The depth dimension gives the orchestrator a deterministic basis for deciding when to gate the user; the anti-pattern callout removes the inline scope-bounding habit. The compromise tracker and the post-completion review reframe are the backstop: when a compromise *is* accepted (legitimately), it remains visible and challengeable rather than baked into the new baseline.

The earlier framing of "just reframe the post-completion review" was rejected because the post-completion review fires too late — by the time it runs, the orchestrator has already shipped the compromise as part of the R2 commit. Earlier gates at the routing-decision moment are the only structural fix; the post-completion reframe is necessary as a backstop but not sufficient.

A second narrative thread, surfaced during this PRD's review pass: the design depends on orchestrator adherence to user-gate prescriptions, and that adherence mechanism (prose strengthening) has demonstrably failed. Rather than absorbing the structural fix into this PRD's scope without a settled design, the structural fix was filed as a separate Issue (`b.wii`) and the dependency was documented as a Risk callout above. This separation is itself an example of the discipline this PRD is trying to install — naming a real failure mode, documenting the dependency, and shipping the work that has a settled design rather than rolling unsettled scope into a working plan.

## Decisions and rejected alternatives

### Decisions

- **Compromise tracker is a markdown file under `<tempdir>/.quorum/`, not a TaskList artifact.** TaskList signals active in-flight work; the tracker is post-decision session metadata that the post-completion review consumes. Mixing them muddles TaskList semantics and makes the tracker awkward to read as input data.
- **Depth has three buckets: `trivial-tweak`, `refactor-locally`, `re-architect`.** Three buckets capture the practical decision space cleanly: the orchestrator auto-routes the first two; only the third (or multi-path findings of any depth) fires a user gate. Two buckets (`trivial` / `significant`) was considered and rejected — `refactor-locally` is genuinely distinguishable from `re-architect` in reviewer judgement and useful for the auto-route-but-log-to-tracker case.
- **Reviewers carry the burden of enumerating fix paths.** Orchestrators may augment a reviewer's enumeration but MUST NOT silently narrow it. The orchestrator's role is routing and synthesis, not design authority. This is the structural shift the PRD is built around.
- **Anti-pattern callout lives alongside the orchestrator-discipline section** in both `/quo-fix-issue` and `/quo-execute` SKILL.md, not in a separate file. Same prose, same SKILL.md edit. Keeps the rule near the routing table that depends on it.
- **Backwards-compatibility shim treats missing-depth-tag findings as `re-architect`-depth.** Errs toward user gate when uncertain. Lets the rollout land incrementally without an atomic all-or-nothing dependency between the reviewer skills and the orchestrator routing rules.
- **Single-path `refactor-locally`-depth findings auto-route + log to the compromise tracker.** They do NOT fire a user gate. The depth is non-trivial but the path is unambiguous, so the tracker entry suffices for retrospective audit (and the post-completion review will challenge it if the choice was wrong). Previously framed as an open question; now committed.
- **Compromise tracker is passed to the post-completion review as a file path, not inlined content.** The dispatched general-purpose Agent uses its `Read` tool to consume the tracker. Keeps the dispatch prompt smaller, better for prompt-cache amortization. Previously framed as an open question; now committed.
- **Cancel semantics at the routing gate are per-Issue (in `/quo-fix-issue`) and per-Task (in `/quo-execute`), not whole-run.** The user opted into the multi-finding batch when they invoked the skill; a single bad routing decision should let them skip the offending unit cleanly without invasive `Ctrl-C`. The scope-bounding gate does NOT include a `Cancel` option because the semantics would be ambiguous (cancel which level?); the user retains `Ctrl-C` for run-level abort.

### Rejected alternatives

- **Adding `severity:critical` above `blocker`.** Rejected. The problem is not severity granularity; it is the missing orthogonal depth dimension. A `blocker`-severity finding with a `trivial-tweak` fix path can still auto-route correctly under the new rules. The failure mode is `suggestion`-severity findings with `re-architect` fix paths getting treated as discretionary — that mismatch is what the depth dimension addresses, not severity granularity.
- **TaskList as the compromise tracker mechanism.** Rejected. TaskList signals active in-flight work; compromise entries are post-decision session metadata that the post-completion review consumes as input data. Reading TaskList entries as input data is awkward — they aren't a list of tasks to do, they're a list of decisions already made.
- **Adding a new `compromise-reviewer` subagent under `agents/`.** Rejected. The post-completion review reframe already empowers the existing `general-purpose` reviewer to challenge compromises. A new subagent bumps the precondition-check count in `/quo-execute` and `/quo-breakdown-epic` (seven becomes eight) and `/quo-fix-issue` (eight becomes nine) without commensurate value.
- **Blanket user gate per finding regardless of depth.** Rejected. Most findings are `trivial-tweak` depth (typos, prose nits); blanket gates would create unbearable friction. The depth-aware routing targets friction at exactly the decisions that matter.
- **Persistent compromise tracker across runs (a ticketed bees artifact).** Rejected. Compromises are scoped to a single run's review-loop discipline. Cross-run persistence would conflate accepted-compromise records with active-work tracking, and the post-completion review only needs current-run context to do its job.
- **Extending `/quo-spec-review`'s contract to also cover plan-substance critique.** Rejected for the same reasons it was rejected during b.bjp's design: two skills with distinct concerns (checklist content review vs holistic plan critique) are cleaner than one with two modes.
- **Only reframing the post-completion review** (and not adding routing rules or the compromise tracker). Rejected. The post-completion review fires too late — by the time it runs, the orchestrator has already shipped the compromise. The framing fix is necessary as a backstop but not sufficient.
- **Pre-narrowing fix paths inside R2/R3 dispatch prompts but adding a "challenge the narrowing later" gate.** Rejected. This would be a complex two-phase mechanism that achieves what a simple user-gate-before-narrowing already does. The simpler design wins.
- **Gating the user on single-path `refactor-locally`-depth findings.** Considered and rejected. The path is unambiguous; gating would add friction for a decision the reviewer has already made. Tracker logging + post-completion challenge is sufficient backstop.
- **Inlining the compromise tracker contents into the post-completion review dispatch prompt.** Considered and rejected. Increases dispatch-prompt size which works against prompt-cache amortization on multi-Epic runs. File-path-via-`Read` is the recommended mechanism.
- **Bundling the structural fix for orchestrator adherence into this PRD's scope.** Rejected. No settled design exists for the structural fix; bundling without a design path is over-promising. Filed as `b.wii` with explicit Risk callout here.
- **Treating orchestrator-adherence as a `## Non-Goals / Out of Scope` non-goal rather than a Risk.** Considered and rejected. The failure mode is not a deliberately-excluded surface — it is an active dependency this PRD's gates rely on. Documenting it under Non-Goals would understate the risk; documenting it under Risk / Known Dependencies surfaces the dependency direction and gives future readers a place to look when the gates misbehave.
- **A `Cancel` option at the scope-bounding gate.** Rejected. The three meaningful choices for a scope-bounding decision are `Fix properly now` / `Defer to follow-up Issue` / `Accept the limitation`; a `Cancel` here would be ambiguous (cancel the finding's fix? the Issue? the run?). The user retains `Ctrl-C` for run-level abort if needed. Per-Issue / per-Task scoping for the routing gate's `Cancel` choice (which IS present) is the cleaner contract.
- **Whole-run `Cancel` at the routing gate.** Considered and rejected. The user opted into the multi-finding batch when they invoked `/quo-fix-issue all` or `/quo-execute <bee-id>`; a single bad routing decision shouldn't kill the whole batch. Per-Issue / per-Task abort lets the user skip the offending unit cleanly without losing the rest of the work; `Ctrl-C` remains the whole-run escape hatch.
