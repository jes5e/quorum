## Requirements

### SR-1. URL detection in `/quo-file-issue`

- **SR-1.1** — `/quo-file-issue` MUST recognize a positional argument matching `^https?://` as a URL token. Bare positional arguments not matching the URL pattern are treated as the existing in-conversation-capture description (existing behavior, unchanged).
- **SR-1.2** — When a URL token is detected, the skill MUST route to the existing External-reference branch (`skills/quo-file-issue/SKILL.md` lines 219-291). The thin-body authoring, resolver-name selection (sub-step B's URL-pattern table), `--reference-materials` JSON shape, and `bees create-ticket` invocation are unchanged.
- **SR-1.3** — The skill MUST continue to accept `--reference <url>` and `--from-github <url>` as flag-and-value pairs. Detection MUST strip the flag and treat the value as a URL token (entering the same auto-detect path). The flags become silent no-op aliases on top of detection.
- **SR-1.4** — The Mode-fork section's existing routing prose MUST be extended to document URL detection alongside the flag-form routing as an additional routing input, not a replacement. The `argument-hint` frontmatter (line 4) and the `### Usage` section (lines 25-32) MUST surface the bare-URL invocation form.

### SR-2. URL detection in `/quo-fix-issue`

- **SR-2.1** — `/quo-fix-issue` Section 1's token classifier MUST recognize tokens matching `^https?://` as URL tokens — a fifth token-shape category alongside zero / `all` / single-ID / list-of-IDs.
- **SR-2.2** — For each URL token, the skill MUST invoke `/quo-file-issue` via the Skill tool with the URL as the bare-positional argument. The skill MUST consume `/quo-file-issue`'s structured return shape (`issue_ticket_id` per SR-8) to capture the new ticket ID; it MUST NOT re-implement URL handling locally.
- **SR-2.3** — URL tokens MUST be replaced **in place** in the working list by their resolved ticket IDs. Mixed forms (URLs + IDs) MUST be supported. The list-mode prerequisite-ordering invariant from Section 1 line 55 MUST be preserved: a URL appearing at position N in the input list becomes the ticket ID at position N in the resolved list, never appended at the tail.
- **SR-2.4** — URL resolution MUST run BEFORE Section 1's existing `bees show-ticket --ids <id1> <id2> ...` upfront-validation pass at line 59, so the validation pass operates on the post-resolution ticket-ID list (no URL tokens fed to `bees show-ticket`).
- **SR-2.5** — URL resolution MUST run AFTER the **`#### Choose agent model preference`** sub-step (lines 69-78) and the **`#### Validate isolation strategy`** sub-step (lines 80-102). Rationale: the user's model-preference and isolation-strategy choices apply to the entire run, including the file-then-fix transition; isolating before resolution lets the user opt into a fresh branch that scoping the file-from-URL commits.

### SR-3. Duplicate detection by `reference_materials.value`

- **SR-3.1** — Before filing a URL via `/quo-file-issue`'s External-reference branch, the skill MUST query open issues for one whose `reference_materials.value` matches the URL exactly. The query lives inside `/quo-file-issue` (extending its Step 2 duplicate-check), NOT inside `/quo-fix-issue`. Single source of truth.
- **SR-3.2** — The dedupe query MUST be scoped to status=`open` only. Closed-and-archived issues MUST NOT block re-filing.
- **SR-3.3** — On a single match, the skill MUST surface the existing ticket via `AskUserQuestion` per CLAUDE.md `## AskUserQuestion usage` (multi-choice) with finite choices: `Use existing` (use the matched ticket ID and skip filing), `File new` (file a fresh ticket anyway), `Cancel` (abort the skill cleanly). The matched ticket's ID, title, and status MUST be surfaced in the question prompt body so the user can decide intelligently.
- **SR-3.4** — On multiple matches, the skill MUST surface all candidates with one option per candidate (labeled `Use existing (b.<id>)` or similar), plus `File new` and `Cancel`.
- **SR-3.5** — When `Use existing` is chosen, the skill returns from the External-reference branch with the matched ticket ID and `action=reused-existing`. When `File new` is chosen, filing proceeds as today. When `Cancel` is chosen, the skill exits cleanly without partial state.
- **SR-3.6** — Dedupe query implementation: the bees CLI's freeform-query mechanism does NOT permit `reference_materials` in the `report` clause (confirmed empirically). The implementation MUST either (a) use a single-step freeform-query if the bees CLI supports `reference_materials` in stage-filter clauses, or (b) fall back to the two-step path: freeform-query for `[type=bee, hive=issues, status=open]` returning `[ticket_id]`, then batch `bees show-ticket --ids <id1> <id2> ...` and filter client-side on each ticket's `reference_materials[*].value`. The choice between (a) and (b) is a runtime-efficiency decision per the RESEARCH NEEDED flag in `## Codebase exploration findings`.

## Background and rationale

This SDD captures the design that emerged from a single-session conversation in which the user invoked `/quo-fix-issue https://github.com/jes5e/quorum/issues/1` and the skill's argument parser rejected the URL — Section 1's token classifier accepts only `all`, single bees IDs, and lists of bees IDs. The initial framing was "add URL support to `/quo-fix-issue`" as a one-skill change reusing `--from-github` as the marker.

The user pushed back on a deeper design question: "why do we need a `--reference` flag when we pass a GitHub URL?" The flag requirement itself is friction — a URL in argument position is unambiguous, no other interpretation makes sense. That reframe is load-bearing for this design: the cleaner improvement is to drop the flag requirement entirely from BOTH skills and auto-detect URLs in argument position. The two changes (drop the flag in `/quo-file-issue`; add URL handling to `/quo-fix-issue`) are bundled into one feature rather than two, because asymmetry between the two skills (one auto-detects, one still flag-required) would be confusing.

The "no concrete bees resolver yet" gap (no `github-issue` / `linear-issue` / `url` resolver implementation in the bees CLI today) is intentional and pre-existing. `agents/engineer.md` lines 18-19 and `agents/pm.md` lines 45-66 already use `WebFetch` as the canonical external-URL fetch path; the resolver-name string is written for forward compatibility. This feature does NOT introduce concrete bees resolvers — that is separate, owned-elsewhere work.

The dedupe-before-filing requirement came out of recognizing that re-running `/quo-fix-issue <url>` against the same URL (e.g., re-trying after a failed run) would otherwise mint a duplicate Issue. `/quo-file-issue` Step 2 already has a duplicate-title check for in-conversation capture; this feature extends that check to `reference_materials.value` matching for external-reference mode. The two-step query path (freeform-query for IDs → batch `bees show-ticket` to read `reference_materials` → client-side filter) is forced by the bees CLI's constraint that `reference_materials` is not a `report` clause field — verified empirically. Whether it works as a stage-filter clause is the only `RESEARCH NEEDED` item in this SDD; either path satisfies the same Requirements.

The "preserve list-mode prerequisite ordering" requirement reflects the existing invariant in `/quo-fix-issue` Section 1 line 55. URL substitution is in place rather than appending — the invariant survives.

The decision to delegate URL filing in `/quo-fix-issue` to `/quo-file-issue` via the Skill tool (rather than inlining the URL-pattern resolver-name table) preserves the single-source-of-truth invariant. `/quo-plan`'s Skill-tool dispatch of `/quo-write-prd` and `/quo-write-sdd` is the canonical precedent. The structured return shape on `/quo-file-issue` (SR-8) is the new contract surface this feature adds; it mirrors the pattern already published by the writer skills.

## Decisions and rejected alternatives

- **Auto-detect URLs by `^https?://` pattern, not by flag.** Decided over preserving the flag-required model.
  - *Rejected: keep the flag mandatory.* Friction; URL pattern is unambiguous in argument position.
  - *Rejected: detect URLs at any argument position, not just positional.* Risks false positives when a URL appears inside a description string.
- **Keep `--reference` and `--from-github` flags as silent no-op aliases.** Decided over removing them entirely.
  - *Rejected: remove the flags.* Would break existing scripts and muscle memory; silent no-ops cost nothing.
- **Dedupe by `reference_materials.value` matching, scoped to status=`open`.** Decided over no-dedupe and over including closed Issues.
  - *Rejected: no-dedupe.* Re-running `/quo-fix-issue <url>` after a failed run would silently mint duplicates.
  - *Rejected: include closed Issues.* Closed Issues are already-fixed bugs; surfacing them on a re-file would confuse users (the matched ticket is closed, why am I being prompted to "use existing"?).
- **Two-step dedupe query path, with a single-step optimization if the bees CLI supports `reference_materials` as a stage-filter clause.** Decided over forcing one path.
  - *Rejected: hardcode the two-step path.* If the bees CLI supports filter-clause access, the one-step path is strictly faster.
  - *Rejected: hardcode the one-step path.* Unverified at SDD time. The implementation chooses based on the runtime CLI behavior.
- **`/quo-fix-issue` delegates URL filing to `/quo-file-issue` via the Skill tool.** Decided over inlining URL handling in `/quo-fix-issue`.
  - *Rejected: duplicate the URL-pattern resolver-name table in `quo-fix-issue/SKILL.md`.* Two sources of truth would drift over time as new resolvers (`--from-jira`, `--from-asana`, etc.) are added.
  - *Rejected: refactor URL handling into a shared library.* No precedent for shared skill-side libraries in this repo; would add a new abstraction layer for one feature. Skill-tool dispatch is the existing precedent (set by `/quo-plan`'s inline writers).
- **Add `## Inline invocation via the Skill tool` contract section to `/quo-file-issue`.** Decided over relying on prose-shape return parsing.
  - *Rejected: parse `/quo-file-issue`'s Step-5 report-back text by regex.* Fragile to prose changes; the writer skills' precedent is to publish a structured contract section. `/quo-file-issue` should follow the same precedent.
- **Mixed URL+ID forms in `/quo-fix-issue` list mode are supported; URL tokens substitute in place, not appended at the tail.** Decided over not supporting mixed forms or appending.
  - *Rejected: only allow homogeneous lists.* Restricts realistic use cases.
  - *Rejected: append URL-resolved IDs at the tail.* Breaks the existing list-mode prerequisite-ordering invariant.
- **No agent contract changes.** Decided over modifying `engineer.md` / `pm.md`.
  - *Rejected: introduce a new agent path for URL Issues.* `engineer.md` lines 18-19 and `pm.md` lines 45-66 already handle external-URL `reference_materials` via `WebFetch`. No change needed.
- **No bees CLI changes.** Decided over implementing concrete bees resolvers.
  - *Rejected: implement concrete `github-issue` / `linear-issue` / `url` resolvers as part of this feature.* Out of scope; resolver implementation is separately owned, and `WebFetch` already covers the use case.
- **TaskList `file-from-url-<n>` naming.** Decided over reusing the per-issue naming convention.
  - *Rejected: name the entry `engineer-<resolved-id>` etc. with the resolved ID.* The file-then-fix transition happens BEFORE the new ID is known; naming the entry with the resolved ID requires either a placeholder rename (race conditions) or a delayed-create pattern. Indexing by URL position avoids the dependency.
- **URL resolution before upfront-validation pass; URL resolution AFTER `Choose agent model preference` and `Validate isolation strategy` sub-steps.** Decided over running validation first or running resolution before the model/isolation prompts.
  - *Rejected: run validation first.* The validation pass operates on bees ticket IDs; URLs would fail validation. Resolution-then-validation produces a clean post-resolution ID list.
  - *Rejected: run resolution before model/isolation prompts.* The user's model-preference and isolation-strategy choices apply to the entire run including the file-then-fix transition; isolating before resolution lets the user opt into a fresh branch that scopes the file-from-URL commits cleanly.
