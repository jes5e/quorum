## Problem Statement

Quorum users currently cannot run `/quo-fix-issue` against an external URL (GitHub Issue, Linear ticket, or other web link). Today the skill's argument parser accepts only bees ticket IDs, the literal token `all`, or no arguments. Filing a URL as an external-reference Issue is supported by `/quo-file-issue`, but only if the user remembers the `--reference` or `--from-github` flag — passing a bare URL routes to in-conversation capture mode, where the URL becomes the description string (almost certainly not what anyone intends).

This forces a clumsy workflow when starting from a URL: invoke `/quo-file-issue --from-github <url>` (remembering the flag), capture the returned ticket ID, then invoke `/quo-fix-issue <id>`. The flag requirement is documentation noise — a URL in argument position is unambiguous, and no other plausible interpretation exists.

The user is anyone with a GitHub Issue (or Linear ticket, or other external bug-tracker URL) who wants the bees workflow to drive the fix. This becomes the dominant entry point as quorum is adopted into projects that already use external trackers as their primary issue surface — the URL is the natural artifact to hand to the workflow, and any friction at that boundary translates directly into adoption friction.

## Goals

- Make `/quo-file-issue <url>` and `/quo-fix-issue <url>` Just Work without requiring `--reference` or `--from-github`.
- Preserve full backward compatibility: existing invocations using `--reference` and `--from-github` continue to behave identically.
- Eliminate the manual two-step when starting from a URL — `/quo-fix-issue <url>` files-then-fixes end-to-end in a single command.
- Keep the URL-pattern resolver-name table in one place (`quo-file-issue/SKILL.md`) — `/quo-fix-issue` delegates URL filing to `/quo-file-issue` rather than duplicating logic.
- Support mixed URL+ID forms in `/quo-fix-issue` list mode while preserving the existing prerequisite-ordering invariant ("do NOT sort — earlier issues may be prerequisites for later ones").

## Non-Goals / Out of Scope

- Implementing concrete bees-CLI resolvers for `github-issue`, `linear-issue`, or `url`. The agent-side `WebFetch` fallback in `agents/engineer.md` and `agents/pm.md` already covers it and remains the canonical fetch path.
- Changes to agent role contracts (`engineer.md`, `pm.md`, `doc-writer.md`, etc.) — they already handle external-URL `reference_materials` end-to-end.
- Adding URL handling to other skills (`/quo-plan`, `/quo-plan-from-specs`, `/quo-execute`, `/quo-breakdown-epic`). This feature is scoped to the issue-filing-and-fixing pair only.
- Removing the `--reference` and `--from-github` flag literals from skill prose. They remain documented as accepted alias forms.
- Changes to the external-reference Issue body shape (thin 2-3 sentence summary, optional `## Doc divergence noted`, `external reference:` convention line, `reference_materials` JSON shape).
- Authentication-gated URL handling (private GitHub Issues, Linear tickets behind SSO). The `WebFetch` fallback path already covers public URLs; auth-gated URLs continue to fall back to the "Ask the user" branch in `/quo-file-issue`'s external-reference flow.

## Functional Requirements

- **URL detection.** `/quo-file-issue` and `/quo-fix-issue` recognize a positional argument matching `^https?://` as a URL. The detection is purely on the argument shape — no per-host special casing.
- **`/quo-file-issue <url>` behavior.** Routes to the existing external-reference branch. The thin-body shape, the `external reference:` convention line, the resolver-name selection from the URL-pattern table, and the `reference_materials` JSON output remain identical to today's `--reference` behavior.
- **`/quo-fix-issue <url>` behavior.** Files the URL via `/quo-file-issue` (Skill-tool dispatch) and captures the returned ticket ID. The newly-minted ticket then enters the standard fix flow — model preference, isolation prompt, per-issue dispatch loop, end-of-flow commit.
- **Mixed forms in `/quo-fix-issue` list mode.** A list like `b.cnb https://... b.xet` is processed in the user-supplied order. URL tokens are filed first (synchronously, so the new IDs are available for the validation pass), and the resulting ticket IDs replace the URL tokens *in place* in the working list. Bees-ID tokens are unchanged.
- **Backward-compat aliases.** `--reference <url>` and `--from-github <url>` continue to be accepted as flag-and-value pairs in both skills. When detected, the skill strips the flag and treats the value as a URL token (entering the same auto-detect path). No prose-level documentation is removed; both forms remain valid syntax.
- **Duplicate detection before filing.** Before filing a URL, the skill queries open issues for one whose `reference_materials.value` matches the URL. On a hit, the skill surfaces the existing ticket (ID + title) via `AskUserQuestion` with finite choices: `Use existing` (use the matched ticket ID and skip filing), `File new` (file a fresh ticket anyway), `Cancel` (abort the run). The duplicate-detection query is scoped to status=`open` only — closed-and-archived tickets do not block re-filing.
- **Order preservation.** `/quo-fix-issue` list mode's existing rule (do not sort the user-supplied order) extends to URL substitution: a URL appearing at position N in the input list becomes the ticket ID at position N in the resolved list, never appended at the tail.

## Edge Cases and Error Handling

- **Unreachable URL** — when `WebFetch` cannot reach the URL during `/quo-file-issue`'s thin-body authoring (network policy, auth-gated, dead link), the skill falls back to the existing "Ask the user" branch in `/quo-file-issue`'s external-reference flow. Filing still succeeds with a thin body the user provides; downstream `/quo-fix-issue` agents will surface their own `WebFetch` failures via the orchestrator-fail path documented in `agents/engineer.md` and `agents/pm.md`.
- **Mixed list with one bad URL** — if any URL token in a `/quo-fix-issue` list cannot be filed (the user cancels at the dedupe prompt for that URL, or `/quo-file-issue` returns an error), the skill reports the failed URL to the user, drops it from the working list, and continues with the remaining tokens. This mirrors the existing invalid-ID behavior in list mode (do not abort the whole run).
- **All tokens fail** — if no URL tokens can be filed and no valid bees IDs remain, the skill exits with an error before the fix loop begins (mirroring the existing "no valid IDs" behavior).
- **Duplicate detection finds multiple matches** — if more than one open Issue references the same URL (unlikely, but possible from prior partial filings), the skill surfaces all candidates to the user via `AskUserQuestion` with a "Use existing (b.<id>)" option per candidate, plus `File new` and `Cancel`.
- **Bare flag with no value** — `/quo-file-issue --reference` (no URL) continues to error out as today; not a new failure mode.
- **URL detection on a non-positional position** — only positional arguments are checked for URL shape. URLs embedded in the middle of a description string (e.g., `/quo-file-issue "Fix the bug at https://example.com/foo"`) are NOT auto-detected; the description is treated as a free-text string per the existing in-conversation-capture path. Detection rule: a token is a URL iff it starts with `http://` or `https://`.

## Non-Functional Requirements

- **Performance.** No new latency surface beyond the existing external-reference flow's `WebFetch` cost. The dedupe query adds one `bees execute-freeform-query` call before each URL is filed; this is bounded by the open-Issues count and should remain sub-second on realistic project sizes.
- **Cross-platform compatibility.** All new shell snippets in skill prose ship as paired POSIX bash + Windows PowerShell variants per CLAUDE.md design rule 2.
- **Project neutrality.** No language-specific commands, file extensions, or manifest filenames embedded in skill prose per CLAUDE.md design rule 1. The feature works identically on Rust, Node, Python, Go, Java, and unknown stacks.
- **Backward compatibility.** Existing invocations of `/quo-file-issue --reference <url>`, `/quo-file-issue --from-github <url>`, and `/quo-file-issue <description>` (in-conversation capture) continue to work identically. No migration of existing Issue tickets is required — the URL-pattern resolver-name selection is symmetric with what `--reference` already produces.

## UI/UX Requirements

- **Invocation feedback.** When `/quo-fix-issue <url>` is invoked, the skill announces the file-then-fix transition before doing it (e.g., "Filing URL as Issue first, then fixing.") so the user understands why a `/quo-file-issue` step appears in the trace.
- **Dedupe prompt clarity.** When duplicate detection fires, the `AskUserQuestion` includes the existing ticket's ID, title, and status in the prompt body so the user can decide intelligently between `Use existing` and `File new`.
- **Mixed-list ordering surface.** When `/quo-fix-issue` list mode resolves URL tokens to ticket IDs, the post-resolution ID list is shown to the user before the fix loop begins, so the user can confirm prerequisite ordering survived the substitution.
- **TaskList progress entries.** The file-then-fix transition gets its own TaskList entry per URL (e.g., `file-from-url-<n>`) so the progress UI shows the filing step before the per-Issue fix-stage entries from the existing flow.

## Acceptance Criteria

- `/quo-file-issue https://github.com/owner/repo/issues/123` (no flag) produces an Issue identical in shape to today's `/quo-file-issue --reference <url>`: thin body with `external reference:` line, `reference_materials` set to `[{"value":"https://github.com/owner/repo/issues/123","resolver":"github-issue"}]`, status=`open`.
- `/quo-file-issue --reference <url>` and `/quo-file-issue --from-github <url>` continue to work — flags accepted as backward-compat no-ops, producing identical Issue shapes to the bare-URL path.
- `/quo-fix-issue https://github.com/owner/repo/issues/123` (no flag) files the Issue (or reuses an existing match per dedupe), then runs the per-issue fix flow end-to-end, producing one git commit on the chosen branch.
- `/quo-fix-issue b.cnb https://github.com/owner/repo/issues/123 b.xet` processes three items in the user-supplied order. URL token at position 2 becomes a new ticket ID (call it `b.<new>`); the working list resolves to `[b.cnb, b.<new>, b.xet]` and is processed sequentially.
- Before filing a URL, the skill queries open issues for one whose `reference_materials.value` matches the URL. On hit, the skill surfaces the existing ticket via `AskUserQuestion` with `Use existing` / `File new` / `Cancel` choices.
- Resolver name selection uses the existing URL-pattern table in `quo-file-issue/SKILL.md` — no copy of the table appears in `quo-fix-issue/SKILL.md`.
- README skill-table rows for both `/quo-file-issue` and `/quo-fix-issue` mention URL mode in the description column.
- Existing `/quo-file-issue` / `/quo-fix-issue` invocation forms (against bees IDs, against `all` mode, against `--reference` and `--from-github` flags, against in-conversation `<description>` strings) continue to work without modification.

## Assumptions

- The user has network access to the URL or, when network access is unavailable, can supply a 2-3 sentence summary in the existing "Ask the user" fallback branch.
- Open issue volume in any single project is small enough that a `bees execute-freeform-query` over open Issues runs in well under a second; if a project ever has thousands of open Issues, the dedupe query may need a more targeted filter (out of scope for this PRD).
- Users invoking `/quo-fix-issue` with a URL want the file-then-fix behavior — not "look up if this URL is already filed and only fix if so." The `Use existing` option in the dedupe prompt covers the lookup-only case.
- The URL-pattern resolver table in `quo-file-issue/SKILL.md` is correct as-is and does not need changes for this feature.

## Open Questions

- **Skill-tool return-shape contract for `/quo-file-issue`.** The skill's Step 5 reports back to the user with the ticket ID; whether that text-shape is a stable contract for programmatic Skill-tool dispatch from `/quo-fix-issue` needs SDD-level confirmation. If not, this feature may need to introduce a structured return shape on `/quo-file-issue` (a small contract addition, but worth surfacing here).
- **Auto-promote flag-stripping vs no-op semantics.** Should the skill detect `/quo-file-issue --reference https://...` and silently treat it identically to `/quo-file-issue https://...` (the cleanest UX), or should the flag still be honored as a "force external-reference even if the value doesn't match a URL pattern" override? Recommended default: silent no-op (UX cleaner; flag becomes pure documentation noise).
- **Behavior when a URL is passed but the user is in interactive (no-args) `/quo-fix-issue` mode.** Today's `/quo-fix-issue` no-args mode lists open Issues and asks the user to pick one. Should that picker also offer "or paste a URL to file-then-fix"? Recommended default: out of scope for this feature; the user types `/quo-fix-issue <url>` if they have one.
- **Plan Bee linkage for filed-from-URL Issues in `/quo-fix-issue` mixed-list mode.** When a list-mode invocation files multiple URL Issues, should the new Issues' `up_dependencies` be wired to reflect the user's intentional ordering (so later URL-Issues block on earlier URL-Issues completing)? Today's list-mode does not wire dependencies between sibling list-mode tickets; preserving that behavior is the simpler default. Recommended default: do not wire.

## Background and rationale

This feature emerged from a single-session conversation where the user invoked `/quo-fix-issue https://github.com/jes5e/quorum/issues/1` and the skill's argument parser rejected the URL. Initial analysis pointed at "add URL support to `/quo-fix-issue`" as a one-skill change. The user pushed back on a deeper question: "why do we need a `--reference` flag when we pass a GitHub URL?" — the flag requirement itself is friction, since URL pattern is unambiguous in argument position.

That reframe is load-bearing for the design: instead of adding URL support to `/quo-fix-issue` while keeping `/quo-file-issue` flag-required (asymmetric, confusing), drop the flag requirement entirely from both skills. The auto-detect change is small (one branch in each skill's argument parser) and the cleaner UX surface dominates the slightly-larger surface area.

The "no concrete bees resolver yet" gap is intentional — `agents/engineer.md` and `agents/pm.md` already use `WebFetch` as the canonical external-URL fetch path, and writing the canonical resolver name (`github-issue` / `linear-issue` / `url`) future-proofs existing tickets without requiring a CLI implementation today. That decision is preserved here, not re-litigated.

The dedupe-before-filing requirement came out of recognizing that re-running `/quo-fix-issue <url>` against the same URL (e.g., re-trying after a failed run) would otherwise mint a duplicate Issue. Today's `/quo-file-issue` Step 2 has a duplicate-title check for in-conversation capture; this PRD extends that check to `reference_materials.value` for external-reference mode.

The "preserve list-mode ordering" requirement reflects the existing invariant in `/quo-fix-issue` Section 1: "do NOT sort — the user's order is intentional; earlier issues may be prerequisites for later ones." This PRD's design substitutes URL tokens in place rather than appending the new ticket IDs at the tail, so that invariant survives the URL-detection change.

## Decisions and rejected alternatives

- **Auto-detect URLs by `^https?://` pattern, not by flag.** Decided over preserving the flag-required model. The flag is documentation noise; the URL pattern is unambiguous in argument position.
  - *Rejected: keep the flag mandatory.* Friction for users who shouldn't need to remember syntax.
  - *Rejected: detect at any argument position (not just positional).* Risks false positives when a URL appears inside a description string.
- **Keep `--reference` and `--from-github` flags as silent no-op aliases.** Decided over removing the flags entirely.
  - *Rejected: remove the flags entirely.* Would break existing scripts and muscle memory; silent no-op aliases cost nothing.
- **Dedupe by `reference_materials.value` matching, scoped to status=`open`.** Decided over no-dedupe and over including closed Issues.
  - *Rejected: no-dedupe.* Re-running `/quo-fix-issue <url>` after a failed run would silently mint duplicates.
  - *Rejected: include closed Issues in dedupe.* Closed Issues represent already-fixed bugs; surfacing them on a re-file would confuse users.
- **`/quo-fix-issue` delegates URL filing to `/quo-file-issue` via the Skill tool, not by inlining the URL-handling logic.** Decided over duplicating the URL-pattern resolver-name table.
  - *Rejected: duplicate the URL-pattern table in `quo-fix-issue/SKILL.md`.* Two sources of truth would drift over time as new resolvers are added.
- **Mixed URL+ID forms in `/quo-fix-issue` list mode are supported; URL tokens substitute *in place*, not appended at the tail.** Decided over not supporting mixed forms or appending.
  - *Rejected: only allow homogeneous lists.* Restricts realistic use cases.
  - *Rejected: append URL-resolved IDs at the tail.* Breaks the existing list-mode prerequisite-ordering invariant.
- **No agent contract changes.** Decided over modifying `engineer.md` / `pm.md` to handle URLs differently.
  - *Rejected: introduce a new agent path for URL Issues.* `engineer.md` and `pm.md` already handle external-URL `reference_materials` via `WebFetch`; no agent-side change is needed.
- **No bees CLI changes.** Decided over implementing concrete bees resolvers for `github-issue` / `linear-issue` / `url`.
  - *Rejected: implement concrete resolvers as part of this feature.* Out of scope; resolver implementation is owned separately and `WebFetch` already covers the use case.
