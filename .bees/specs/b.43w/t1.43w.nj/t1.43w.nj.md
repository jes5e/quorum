## Background and rationale

The repo recently renamed from `bees-workflow` to `quorum`. The mechanical find-and-replaces (skill prefix `bees-* → quo-*`, repo-name references, workflow-vocabulary references) all landed in commits `001c314`, `1e72130`, and `699128a`. What's left is the **positioning** content in the README, which still reads like a "bees skill set positioned as an alternative to Apiary."

The user's framing for the rewrite is explicit: quorum is **not** a bees skill set, it's a portable SDLC workflow that happens to use bees today. Beads support is on the roadmap (Plan Bee `b.9xr`, "Optional beads backend"), and the design intent is to work with any structured-ticket backend rather than tie quorum to a single one. Reframing the README now — before `b.9xr` ships — sets reader expectations correctly with the project's direction and avoids needing a second README rewrite later.

The user also called out that the existing README anchors heavily on Apiary as a comparison point. While Apiary is a real and useful skill set, comparison framing dates the README and gets in the way of quorum standing on its own merits. The rewrite removes the comparative framing entirely; quorum is described on its own terms.

A scrub-and-audit pass piggy-backed onto the reframing is cheap insurance against drift. Substantial recent changes (`bees-* → quo-*` renames, ephemeral-Agent orchestration rewrite, Spec Bee + `t1=Doc` child redesign, addition of `/quo-spec-review`, rename of the three review skills) mean the README *could* have stale corners. The Explore agent's audit confirmed it does not — the rewrite can therefore be small, focused, and low-risk.

The user explicitly chose to drive the change through `/quo-plan` to dogfood the workflow on a low-risk doc-only change. The motivation is workflow validation: exercising the planning + execution path on a small change exposes any rough edges without putting feature code at risk.

The fact that `b.9xr` already exists as a separate Plan Bee for the actual beads-backend implementation is structurally important for this rewrite. It means the README can credibly name beads as "planned future support" with a concrete tracker behind it, and it means this work can stay strictly README-doc-scoped without risk of mixing in code changes.

## Decisions and rejected alternatives

- **Decision: Reframe and audit, don't rewrite from scratch.** Most of the README is already accurate and well-organized; the diff should be small (~20–30 lines), concentrated in the lead paragraph, the "Why this exists" section, the "Coming soon" Apiary line, and the Credits clause.
  - **Rejected: from-scratch rewrite.** Higher cost, more regression risk on accurate content (install instructions, status vocabulary, workflow diagram). The audit confirmed those sections do not need touching.

- **Decision: Keep the ASCII workflow diagram as-is.** The Explore agent verified all ten skills referenced in the diagram exist; no drift to fix.
  - **Rejected: redraw the diagram into Mermaid.** Out of scope; adds risk; not requested by the user.
  - **Rejected: add `/quo-status` and the three review skills to the diagram.** They're intentionally absent — `/quo-status` is an introspection helper, and the three review skills are dispatched internally by `/quo-execute`/`/quo-fix-issue` rather than being top-level user invocations. The diagram is the *user-facing happy path*, not a complete skill catalog.

- **Decision: Keep the bees CLI install instructions concrete.** The user installs bees today; abstracting the install line ("install the ticket-backend CLI of your choice") makes it non-actionable.
  - **Rejected: abstract install line for forward compatibility with beads.** Trades practical clarity for theoretical portability. When `b.9xr` lands, the install section can grow a backend-pick subsection at that point; it's the right time to do that, not now.

- **Decision: Mention beads explicitly by name and link in the lead paragraph.** Naming the next backend with a link signals concrete direction.
  - **Rejected: generic "and other backends coming" without naming beads.** Vaguer; doesn't communicate the specific direction.

- **Decision: Leave bees-domain vocabulary (Plans / Issues / Specs hives, `t1=Doc` children, "Bee" naming) intact in the rewrite.** This is internal to the workflow, not user-visible install-path concern; abstracting it would inflate scope dramatically and belongs to `b.9xr`'s scope.
  - **Rejected: rename hive vocabulary in lockstep with the README reframing.** Massive scope creep; would touch every skill in the repo plus all `t1=Doc` titles in `docs/sdd.md` `## Per-feature design`.

- **Decision: Scope strictly to `README.md`.** CONTRIBUTING.md, doc-writing-guide.md, and CLAUDE.md may also have stale framing, but bundling those into this Plan Bee inflates scope and review burden. README is the user's first impression and the highest-leverage doc to fix first.
  - **Rejected: bundle the CONTRIBUTING.md / doc-writing-guide.md / CLAUDE.md scrub into this work.** A follow-up `/quo-file-issue` can capture that work cleanly if desired.

- **Decision: Single-Epic plan.** The work is small (one file, ~30-line diff), localized, and has no internal ordering constraints. Splitting into multiple Epics would be over-engineering; one Epic that delivers the full rewrite is the right granularity.
  - **Rejected: split into multiple Epics by section (lead-paragraph Epic, Why-this-exists Epic, Coming-soon Epic, Credits Epic).** Each "section" is a 1–10 line edit; splitting them into Epics would multiply ticket overhead without buying any test/review value.

- **Decision: Drive the change through `/quo-plan` rather than ad-hoc edits.** User explicitly asked for this.
  - **Rejected: ad-hoc edits.** Skips the dogfooding signal the user wanted.

- **Decision: Skip the post-implementation cumulative-doc fold step.** This rewrite has no functional spec impact; there is no `### Feature:` subsection to add to `docs/prd.md` or `docs/sdd.md`.
  - **Rejected: add a `### Feature: README rewrite` subsection to the cumulative project PRD/SDD.** Stretches the cumulative-doc convention to cover doc-only meta-work; that's not what those docs are for.
