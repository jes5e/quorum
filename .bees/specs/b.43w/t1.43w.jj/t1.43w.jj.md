## Problem Statement

The `README.md` for the **quorum** project still reads like a "bees-only" skill set positioned against Apiary as the incumbent. After the recent rename from `bees-workflow` to `quorum` and the design intent to support multiple ticket-system backends (bees today, beads soon, others over time), the README's framing is out of step with the project's actual direction.

Three populations are affected:

- **Prospective users** evaluating quorum — they read "an alternative to Apiary" and conclude the project is narrow in scope or a niche fork, when it's actually a portable SDLC workflow that's deliberately positioned to be backend-agnostic.
- **Current users** who already chose quorum — they get a doc that hasn't caught up with the project's positioning, including stale references to Apiary as a fallback for capabilities quorum doesn't yet ship.
- **Contributors** assessing the project's direction — comparison framing dates the README and obscures the standalone goals quorum is pursuing.

This matters now because Plan Bee `b.9xr` ("Optional beads backend") is already in flight tracking the actual code work to support beads. Reframing the README ahead of `b.9xr` shipping aligns reader expectations with the codebase direction and avoids needing a second README rewrite when beads support lands.

## Goals

- Reframe quorum's `README.md` as a **portable, ticket-system-agnostic Claude Code SDLC workflow**, with bees as the supported backend today and beads explicitly named as planned future support.
- Eliminate every "Apiary" / "apiary" reference from `README.md` (case-insensitive grep returns zero matches), including the comparative "vs Apiary" framing in the "Why this exists" section.
- Audit the workflow diagram and the skill catalog table against the present-day state of `skills/` and `agents/`; correct any drift.
- Keep the rewrite a *positioning + audit* pass, not a from-scratch rewrite — preserve the accurate, useful structure already in place (install instructions, status vocabulary, scratch-file conventions).

## Non-Goals / Out of Scope

- Implementing beads backend support — tracked under Plan Bee `b.9xr`, not this work.
- Editing `CONTRIBUTING.md`, `CLAUDE.md`, `docs/doc-writing-guide.md`, or any in-skill prose for the same reframing — README only. Surface as a follow-up Issue if desired.
- Replacing the concrete bees CLI install commands with abstract "your ticket backend CLI" prose — readers install bees today and need actionable instructions.
- Renaming the bees-domain hive vocabulary (Plans / Issues / Specs) — those names are bees-internal and are a separate, larger concern.
- Restructuring the Status vocabulary table or the Specs-hive lifecycle prose.
- Scrubbing Apiary references in files outside `README.md` — out of scope; surface as a follow-up Issue if desired.

## Functional Requirements

The rewritten `README.md` must:

1. Open with a one-paragraph positioning statement that names quorum as a portable Claude Code SDLC workflow on top of **structured ticket systems**, names bees as the supported backend today, and names beads explicitly as planned future support.
2. Replace the "Why this exists" section's comparative framing with a stand-alone explanation of quorum's design priorities (cumulative project-level docs, language-agnostic, cross-platform, plain-English statuses, PRD/SDD-first or scope-first, idempotent setup).
3. Carry an accurate workflow diagram of the happy path (`/quo-setup` → `/quo-plan` or `/quo-plan-from-specs` → Spec Bee → Plan Bee → `/quo-breakdown-epic` → `/quo-execute` → optional `/quo-file-issue` + `/quo-fix-issue`), audited against the current skill set.
4. Carry a skill catalog table whose row count matches the count of user-facing `quo-*` skills under `skills/` (the `_shared/` directory is internal and is not a row), and whose one-line descriptions are substantively consistent with each skill's `SKILL.md` frontmatter `description` field.
5. Retain the existing install / requirements / `/quo-setup` / "Where docs live" / scratch-files / status-vocabulary / contributing / license sections in their current shape, with only minimal edits required for the reframing or to remove Apiary mentions.
6. Update the "Coming soon: optional skills" section to drop the Apiary fallback line; the section should stand on its own listing planned optional skills.
7. Update the Credits section to drop the Apiary mention; the bees credit (current backend) can remain.
8. Keep cross-platform install instructions (POSIX bash + Windows PowerShell, both shown) intact.

## Edge Cases and Error Handling

- **Diagram drift surfaced during audit.** If the workflow diagram references a skill that has since been renamed, removed, or repositioned in the flow, the diagram must be updated (not just left alone). This is a real defect to fix, not a copy-paste nit.
- **Skill catalog drift surfaced during audit.** If a skill exists in `skills/` but is not in the catalog table, or is in the table but no longer exists in `skills/`, fix the table. If a row's one-line description has diverged substantively from the corresponding `SKILL.md` frontmatter `description`, update the row to reflect the current behavior.
- **Apiary references with surrounding load-bearing prose.** When stripping an Apiary mention, ensure the surrounding paragraph still parses cleanly — do not leave dangling sentence fragments or orphaned headings. Rewrite the immediate context as needed.
- **Bee-credit retention.** The Credits section currently credits both bees (the underlying ticket system) and Apiary (the inspirational skill set). Drop the Apiary credit; the bees credit stays because bees is the current backend and is on the install path.
- **No skill prose, helper script, agent file, or other doc file is touched.** This is a `README.md`-only diff. If during the audit a defect surfaces in a `SKILL.md` frontmatter description, that's filed as a separate Issue, not fixed here.

## Background and rationale

The repo was recently renamed from `bees-workflow` to `quorum`, with the skill prefix renamed from `bees-*` to `quo-*` across all 14 user-facing skills. The mechanical find-and-replaces have all landed; what's left is the positioning content in the README, which still reads like a "bees skill set positioned as an alternative to Apiary."

The user's framing for the rewrite is explicit: quorum is **not** a bees skill set, it's a portable SDLC workflow that happens to use bees today. Beads support is on the roadmap (Plan Bee `b.9xr`), and the design goal is to work with any structured-ticket backend, not just bees. Reframing the README now — before `b.9xr` ships — sets reader expectations correctly with the project's direction and avoids a second rewrite later.

The user also called out that the existing README's positioning anchors heavily on Apiary as a comparison point. While Apiary is a real and useful skill set (and the user does not have a complaint with Apiary itself), the user judged that comparison framing is more confusing than clarifying for new readers, dates the README, and gets in the way of quorum standing on its own merits.

A scrub-and-audit pass piggy-backed onto the reframing is cheap insurance against drift: substantial recent changes (the `bees-* → quo-*` renames, the ephemeral-Agent orchestration rewrite, the Spec Bee + `t1=Doc` child redesign, the addition of `/quo-spec-review`, the rename of the three review skills) mean the README may have stale corners. Catching that drift while we're already touching the file is much cheaper than a separate audit pass later.

The user also explicitly chose to dogfood `/quo-plan` to drive this change, even though the change is small and doc-only. The motivation is workflow validation: exercising `/quo-plan` on a low-risk doc change exposes any rough edges in the planning workflow without putting feature code at risk.

## Decisions and rejected alternatives

- **Decision: Reframe and audit, don't rewrite from scratch.** Most of the README's structural content (skill catalog, install instructions, status vocabulary, scratch-file convention, requirements, "Where docs live", contributing, license) is already accurate and well-organized. The rewrite is a *positioning* change plus a *scrub-and-audit* pass over potentially-stale corners.
  - **Rejected: full rewrite from scratch.** Higher cost, more regression risk on accurate content. The README is mostly correct; a from-scratch rewrite would force re-validating every section that's already fine.

- **Decision: Keep the bees install commands concrete (no `your ticket backend CLI` abstraction).** The user installs bees today; abstracting the install line makes it non-actionable. When beads support lands (`b.9xr`), the install section can grow a backend-pick subsection at that point.
  - **Rejected: replace `bees CLI` install line with generic abstract phrasing.** Breaks the install path for current users; trades practical clarity for theoretical purity.

- **Decision: Mention beads explicitly by name in the lead paragraph.** Naming the next backend with a link signals concrete direction; generic "and other backends coming" alone reads as vague marketing.
  - **Rejected: omit beads name in favor of generic "other backends".** Vaguer; doesn't communicate the specific direction the user wants the README to reflect.

- **Decision: Scope strictly to `README.md`.** `CONTRIBUTING.md`, `docs/doc-writing-guide.md`, and `CLAUDE.md` may also have stale framing, but bundling those into this Plan Bee inflates scope and review burden. The README is the user's first impression and the highest-leverage doc to fix first.
  - **Rejected: bundle the CONTRIBUTING.md / doc-writing-guide.md / CLAUDE.md scrub into this work.** A follow-up `/quo-file-issue` can capture that work cleanly if desired.

- **Decision: Audit the workflow diagram in scope, but only update if genuinely stale.** Stylistic rewrites of a working diagram add risk without clear benefit.
  - **Rejected: redraw the diagram to include optional skills (worktree, multi-repo).** Out of scope — the diagram is the *happy-path* visual, not a complete catalog.

- **Decision: Leave Plan Bee `b.9xr` untouched.** This Plan Bee is README-doc-scoped; the actual beads-backend implementation belongs to `b.9xr`'s scope.
  - **Rejected: combine README rewrite into b.9xr.** Mixing doc-prep with backend implementation makes both reviews harder, and the README rewrite is independently valuable even before beads code lands.

- **Decision: Drive the change through `/quo-plan` rather than ad-hoc edits.** The user explicitly asked to dogfood the workflow on this small, low-risk doc change.
  - **Rejected: ad-hoc edits straight to `README.md`.** Would have skipped the dogfooding signal the user explicitly wanted.
