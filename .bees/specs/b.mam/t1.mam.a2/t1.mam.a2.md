## Background and rationale

This rename is driven by first-hand observation in this repo of agents in fresh Claude Code sessions invoking `/bees-code-review` against generic prompts like "do a thorough code review of the uncommitted changes" — and parallel observations of `/bees-test-review` invoked against generic test-review prompts outside any `bees`-workflow context.

The diagnosis identified two independent matching surfaces driving the misinvocation: the frontmatter `description` advertising standalone use, and the skill names themselves containing the substrings `code-review` / `test-review` / `doc-review`. Both surfaces had to be addressed; renaming alone (keeping the standalone-use prose) would leave the description-match pathway open, and rewriting the description alone (keeping the names) would leave the substring-match pathway open.

The framing question that followed: is the standalone-use story worth defending, or is it the source of the misinvocation cost? Reviewing the unique discipline these skills encode showed most of it is workflow-shaped — the Step 0a stale-snapshot guard, the loop-bounding discipline, the cross-call-site checks. The cross-call-site checks (Step 2 #7 in `bees-code-review`) and the CLAUDE.md best-practices loading are the only pieces that would transfer to standalone use, and Claude Code's built-in `/review` plus a one-line addendum gets you most of that. That made the trade explicit: standalone value is low, misinvocation cost is high.

A confirming-signal cross-check sealed the framing: both `bees-execute` and `bees-fix-issue` already contain anti-pattern callouts (lines 433, 467 / 372, 405) telling the team-lead NOT to invoke any of the three review skills for a final cross-lane sweep — explicitly because each is "scoped to source code / user-facing docs / test files." The new role-based names mirror that lane-scoping exactly. The rename strengthens existing prose in the two execution skills rather than contradicting it.

The four-Epic decomposition (skill internals → callers → top-level docs → ticket sweep) tracks the natural review-and-pause boundaries of the work. Each Epic ends with a `grep`-based verification check that produces a clean success signal, and the sequence supports clean per-Epic commit messages. Epic 4's net work surface turns out to be near-zero (the two open old-name-referencing tickets are this rename's own Spec Bee and PRD child); SR-5 ships it as a verification-only Epic for symmetry with the user's stated decomposition.

## Decisions and rejected alternatives

**Decision: Drop the standalone-use story entirely rather than rename-only or rewrite-description-only.**
Rationale: both root-cause matching surfaces (substring on the name plus the standalone-use prose) need to be closed; addressing only one leaves the other open. Standalone unique value is low (most of what's there is workflow-shaped), and `/review` substitutes adequately for the rest.
*Alternative rejected:* keep the standalone-use story, just rename. Closes the substring-match surface but not the description-match surface — a fresh-session agent would still match generic prompts against the new descriptions if they continue to advertise standalone use.
*Alternative rejected:* rewrite the descriptions but keep the names. Closes the description-match surface but not the substring-match surface — Claude Code's skill-selection mechanism partially keys off the skill name, so generic prompts would still pull these skills into the candidate set.

**Decision: Use role-based names mirroring `agents/<role>.md` filenames.**
Rationale: `bees-engineer-review` (wraps `agents/code-reviewer.md`), `bees-test-writer-review` (wraps `agents/test-reviewer.md`), `bees-doc-writer-review` (wraps `agents/doc-reviewer.md`). These names are legibly orchestrator-only — a fresh-session agent reading the skill catalog sees "review the engineer's work" / "review the test writer's work" / "review the doc writer's work" and can tell at a glance the skill assumes an orchestrating workflow. The new names also reinforce the existing anti-pattern callouts in `bees-execute` / `bees-fix-issue` that name the lane-scoping the rename now mirrors.
*Alternative rejected:* drop the `bees-` prefix. The prefix conveys package origin (per the prior `code-review` → `bees-code-review` rename rationale captured in `CONTRIBUTING.md` line 92) and lets a project with multiple skill providers tell at a glance which come from this repo. The new names retain the prefix.
*Alternative rejected:* "wrapped-review" or "orchestrated-review" suffix. The wrapping is a skill-author concept, not a name a fresh agent would reach for. Role-based names map directly to the workflow vocabulary the rest of the skill set already uses.

**Decision: Do NOT add a new `/bees-review` skill for ad-hoc review.**
Rationale: would re-introduce the substring-match problem this work eliminates. `/review` (Claude Code built-in) is the right ad-hoc tool, and a one-line addendum captures whatever cross-call-site / CLAUDE.md-aware extension a contributor needs at standalone time.
*Alternative rejected:* add `/bees-review` as a thin wrapper over `/review`. Deferred indefinitely; no implementation date.

**Decision: Rewrite (not append) the `CONTRIBUTING.md` "Status / type renames history" entry currently at line 92.**
Rationale: the existing entry says "standalone invocation remains supported via the existing dual-mode prose" — this is the load-bearing precedent future reviewers cite to revert standalone-removal. Replacing it (rather than appending a superseding entry beside it) means the new state is the precedent, not the old one. Two contradictory entries in the same history section would undermine the section's purpose ("what is the current state and why").
*Alternative rejected:* append a new entry beside the existing line 92. Leaves contradictory precedents in the same document; exactly the failure mode the history section is meant to prevent.

**Decision: Sweep open/in-progress bees tickets, leave done/closed tickets untouched.**
Rationale: open tickets reference old names in still-actionable bodies; updating them keeps the workflow coherent. Done/closed tickets are historical snapshots; rewriting them rewrites history without value. Editorial cost (`git log` noise, broken PR-description anchors) outweighs the consistency benefit.
*Alternative rejected:* sweep all tickets including closed ones. Rejected on editorial-cost grounds.
*Alternative rejected:* skip the ticket sweep entirely. Rejected because in-flight tickets that an Engineer or PM agent reads mid-execution would surface old skill names and trigger wrong-tool invocations during the work. (In practice the sweep is near-empty per `## Codebase exploration findings` — the only open old-name-referencing tickets are `b.mam` and `t1.mam.qr`, both of which are this rename's own specification — but symmetric Epic 4 still ships.)

**Decision: Decompose the work into four Epics rather than one monolithic Epic; keep Epic 4 even though its net surface is near-zero. Epics 1 and 2 share a single git commit; Epics 3 and 4 each get their own commit.**
Rationale: the four boundaries (skill internals → callers → top-level docs → ticket sweep) are natural review-and-pause points and let each Epic end with a clean `grep`-based verification check. Keeping Epic 4 even at near-zero net surface preserves the "verify each surface independently" structure: bees-ticket bodies are a different surface from project-doc edits, and a future contributor reading the per-Epic commit history sees the sweep ran as a deliberate verification step.

Epics 1 and 2 must share one git commit (despite being separate Epics in the workflow) to avoid the brief mid-rename inconsistency surfaced in PRD `## Edge Cases and Error Handling` item 1: if Epic 1 (directory rename + body cleanup) commits separately from Epic 2 (caller updates), there is a window where callers reference one set of names and the directories carry the other set, breaking any in-flight `/bees-execute` run that loads `bees-execute/SKILL.md` mid-rename. Combining the two Epics into a single commit closes that window. Epics 3 (top-level docs) and 4 (ticket sweep + verification) operate on artifacts that downstream `/bees-execute` runs do not load mid-execution, so they may land in their own commits without re-introducing the mid-rename inconsistency.
*Alternative rejected:* one monolithic "rename everything" Epic. Too much surface in one review pass; verification checks become harder to scope when everything changes at once. The Epic boundary still serves as a per-surface verification checkpoint even when two adjacent Epics share a commit.
*Alternative rejected:* fold Epic 4 into Epic 3. Considered because the open-ticket sweep's net work surface is near-zero; rejected because bees-ticket bodies are a structurally different surface from project-doc edits, and Epic 3's grep verification doesn't naturally cover ticket files (bees ticket files live under `.bees/` and are organized by status).
*Alternative rejected:* per-Epic individual commits including for Epics 1 and 2. Considered for the cleaner one-Epic-per-commit log readability, but rejected because the brief mid-rename inconsistency window (between Epic 1's commit and Epic 2's commit) is observable to in-flight `/bees-execute` runs and outweighs the log-readability benefit.

**Decision: Treat `skills/bees-spec-review/SKILL.md` and `docs/prd.md` references as in-scope for the AC#14 repo-wide grep.**
Rationale: AC#14 (repo-wide grep returns no matches for old names outside the rewritten history entry and closed tickets) implicitly covers both files. Updating them keeps the grep clean and avoids carving exceptions into the AC. The textual substitutions are mechanical (no behavior change); SR-2.5 covers `skills/bees-spec-review/SKILL.md`, SR-4.5 covers `docs/prd.md`.
*Alternative rejected:* exclude both files and weaken AC#14 to whitelist them. Rejected to keep the AC simple; the textual substitution cost is negligible.
