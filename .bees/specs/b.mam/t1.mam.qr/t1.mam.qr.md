# PRD: Rename and lock down the three review skills

## Problem Statement

Agents in fresh Claude Code sessions, given generic review prompts (e.g., "do a thorough code review of the uncommitted changes"), invoke `/bees-code-review` instead of Claude Code's built-in `/review` or doing a bare review. The same misinvocation has been observed for `/bees-test-review` against generic test-review prompts in non-`bees`-workflow contexts.

The user surface harmed is anyone running Claude Code in this `bees-workflow` repo (or any repo that has the `bees-` skills installed) outside of an active `/bees-execute` or `/bees-fix-issue` cycle. The mis-invoked skill loads orchestrator-shaped guidance — stale-snapshot guards, loop-bounding "be selective" discipline, ticket-resolved scope inference — that is structurally irrelevant to the cold standalone use case. The result is a degraded review experience in the standalone case and a steady churn of "why is this skill behaving like it's in a fix-and-re-review loop?" surprises.

This matters now because the misinvocation rate has been observed first-hand in this repo, and the load-bearing prose that drives it is in two specific places that a tightly-scoped edit can fix: the frontmatter `description` advertising standalone use, and the skill names themselves containing the substrings `code-review` / `test-review` / `doc-review`.

## Goals

- Eliminate the substring-match misinvocation pathway for the three review skills by renaming them to role-based names that mirror `agents/<role>.md`.
- Eliminate the description-match misinvocation pathway by rewriting each frontmatter `description` to lead with workflow scoping and removing the "Standalone use - ad-hoc review" sentence.
- Preserve the orchestrator-aware behavior the three skills already encode (Step 0a stale-snapshot guard, loop-bounding discipline, cross-call-site checks, CLAUDE.md best-practices loading) — this PRD is about reducing misinvocation surface, not refactoring review behavior.
- Update every internal cross-reference and caller in this repo so the rename is observable as a single coherent change, not a partial drift.
- Record the rationale for the rename in `CONTRIBUTING.md`'s "Status / type renames history" so the new state is the precedent future reviewers cite.

## Non-Goals / Out of Scope

- Adding a new `/bees-review` skill for ad-hoc review. Deferred indefinitely — would re-introduce the substring-match problem this PRD eliminates. Claude Code's built-in `/review` is the right ad-hoc tool.
- Editing closed/done bees tickets that reference old skill names. They are historical snapshots and remain immutable.
- Renaming the `agents/<role>.md` files themselves. Their content (descriptions and `Skill`-tool invocations) is updated; their filenames are not.
- Cross-call-site behavior changes inside the three review skills. This PRD covers a rename plus body cleanup, not a behavior refactor.
- Updating per-project install copies at `~/.claude/skills/<old-name>/`. Per `CONTRIBUTING.md`'s "This is the canonical source" principle, install artifacts go stale until the user re-symlinks; the symlink loop in `CONTRIBUTING.md` (updated by this work) picks up the new directory list automatically the next time it runs.
- Changes to any other skill's behavior (`/bees-execute`, `/bees-fix-issue`, etc.) beyond the textual references to the renamed skills.

## Edge Cases and Error Handling

1. **Partial rename observed by an in-flight `/bees-execute` run.** A `/bees-execute` run that loads `bees-execute/SKILL.md` mid-rename could see updated text referring to renamed skills before the renamed skill directories exist on disk. **Mitigation:** complete the rename in a single commit (Epic 1 + Epic 2 land together as one atomic unit), so any external observer either sees the full pre-rename or full post-rename state.
2. **Stale install symlinks at `~/.claude/skills/<old-name>/`.** Users who installed the skills via the symlink loop in `CONTRIBUTING.md` keep symlinks at `~/.claude/skills/bees-code-review/` etc. After the directory rename, those symlinks dangle. **Mitigation:** the updated symlink loop in `CONTRIBUTING.md` lists the new directory names; users re-running it after the rename get fresh symlinks. The PRD does not require removing the old dangling symlinks — Claude Code does not surface a dangling symlink as a registered skill.
3. **Open bees tickets reference old skill names mid-rename.** A ticket in flight (status `in_progress`) that mentions an old skill name in its body becomes inconsistent with the renamed reality. **Mitigation:** Epic 4 sweeps open and in-progress tickets specifically. Done tickets are untouched (historical accuracy outweighs cross-version coherence).
4. **Python lint regression from a directory-name reference inside a script.** Helper scripts (`detect_fast_path.py`, `scoped_marker_resolver.py`) live under `skills/<skill>/scripts/`. They do not currently reference any other skill by name, so the rename should not break them — but a verification step against `python -m pyflakes skills/*/scripts/*.py` after the rename catches any regression.
5. **A caller is missed.** A reference to an old skill name slips through an Epic and lands on disk after Epic completion. **Mitigation:** Epic 2 and Epic 3 each end with a `grep` check for the old skill name strings (`bees-code-review`, `bees-test-review`, `bees-doc-review`) across the files in their scope; the absence of matches is the success signal. Epic 4's open-ticket sweep similarly ends with the same `grep` check across open ticket bodies.
6. **A reviewer reverts the standalone-removal.** Future contributors might propose reintroducing the standalone-use sentence in a description, since the prior `CONTRIBUTING.md` "Status / type renames history" entry endorsed dual-mode. **Mitigation:** the rewrite of that entry (Epic 3) records the new decision and reasoning so the new state is the precedent the next reviewer reads.

## Assumptions

- The misinvocation pathway is driven by the two root causes named in the Problem Statement (substring-match on the skill name plus description-match on the standalone-use sentence). No other discovery mechanism is at play.
- The standalone-use story has low residual value — the unique discipline these skills encode is workflow-shaped (stale-snapshot guard, loop-bounding) and what would transfer to standalone use is mostly already available via Claude Code's built-in `/review` plus a one-line addendum.

- Per-machine install copies at `~/.claude/skills/<old-name>/` will be re-symlinked by users who care about live editing; users who only consumed the install snapshots are not the primary surface for this rename.
- The repo has no test suite, so test-coverage acceptance criteria do not apply. Verification leans on `grep` checks for old-name absence in updated files plus successful Python lint.
- A single coherent rename (skill rename + caller updates landing in adjacent commits) is preferable to a multi-stage rollout. The four-Epic decomposition reflects review-and-pause boundaries, not deployment phases.

## Background and rationale

This rename is driven by first-hand observation in this repo of agents in fresh Claude Code sessions invoking `/bees-code-review` against generic prompts like "do a thorough code review of the uncommitted changes" — and parallel observations of `/bees-test-review` invoked against generic test-review prompts outside any `bees`-workflow context. The diagnosis: two independent matching surfaces both pull these skills into the candidate set:

1. **The frontmatter `description` advertises standalone use.** Each current description includes "Standalone use - ad-hoc review of a diff, worktree, files, or bees ticket" — the exact prose Claude Code's skill-selection mechanism matches against generic review intents.
2. **The skill names themselves contain generic substrings.** `bees-code-review` literally contains `code-review`; same shape for `bees-test-review` and `bees-doc-review`. These substrings latch onto generic review prompts even before description-text-matching enters the picture.

The framing question that followed: is the standalone-use story worth defending, or is it the source of the misinvocation cost? Reviewing the unique discipline these skills encode showed that most of it is workflow-shaped:

- Step 0a's stale-snapshot guard exists because the engineer agent is iterating in parallel — irrelevant standalone in a clean tree.
- The loop-bounded "be selective, don't thrash" discipline is loop-shaped by definition.
- The cross-call-site checks (Step 2 #7 in `bees-code-review`) and the CLAUDE.md best-practices loading are the only pieces that would transfer to standalone use, and `/review` (Claude Code built-in) plus a one-line addendum gets you most of that.

That made the trade explicit: standalone value is low, misinvocation cost is high. Drop the standalone story entirely and lean on `/review` for ad-hoc code review.

A confirming-signal cross-check sealed the framing: both `bees-execute` and `bees-fix-issue` already contain anti-pattern callouts telling the team-lead NOT to invoke any of the three review skills for a final cross-lane sweep — explicitly because each is "scoped to source code / user-facing docs / test files." The new role-based names mirror that lane-scoping exactly. The rename strengthens existing prose in the two execution skills rather than contradicting it; that asymmetry made the rename direction unambiguous.

The decomposition into four Epics tracks the natural scope boundaries of the work (skill internals → callers → top-level docs → ticket sweep) and matches the editorial-pause points where a reviewer would want to inspect the change before continuing.

## Decisions and rejected alternatives

**Decision: Drop the standalone-use story entirely rather than just rename.**
Rationale: descriptions advertising standalone use is one of the two root causes of misinvocation; renaming alone wouldn't solve the description-match pathway. The standalone-use unique value is low (most of what's there is workflow-shaped); `/review` plus a one-line addendum substitutes adequately for the remainder.
*Alternative rejected:* keep the standalone-use story, just rename the skills. Rejected because the description-match pathway would persist — generic review prompts would still match the standalone-use prose even with renamed skills.

**Decision: Use role-based names mirroring `agents/<role>.md`.**
Rationale: `bees-engineer-review` (wraps `agents/code-reviewer.md`), `bees-test-writer-review` (wraps `agents/test-reviewer.md`), `bees-doc-writer-review` (wraps `agents/doc-reviewer.md`). These names are legibly orchestrator-only — a fresh-session agent reading the skill catalog sees "review the engineer's work" / "review the test writer's work" / "review the doc writer's work" and can tell at a glance the skill assumes an orchestrating workflow.
*Alternative rejected:* rename to "wrapped-review" / "orchestrated-review" / similar. Considered but rejected: the wrapping is a skill-author concept, not a name a fresh agent would reach for; the role-based names map directly to the workflow vocabulary the rest of the skill set already uses.
*Alternative rejected:* drop the `bees-` prefix. Considered but rejected: the prefix conveys package origin (per `CONTRIBUTING.md`'s prior `code-review` → `bees-code-review` rename rationale) and lets a project with multiple skill providers tell at a glance what comes from this repo. The new names retain the prefix.

**Decision: Do NOT add a new `/bees-review` skill for ad-hoc review.**
Rationale: would re-introduce the exact substring-match problem this PRD eliminates. `/review` (Claude Code built-in) is the right ad-hoc tool, and a one-line addendum captures whatever cross-call-site / CLAUDE.md-aware extension a contributor needs at standalone time.
*Alternative rejected:* add `/bees-review` as a thin wrapper over `/review`. Rejected indefinitely; no implementation date.

**Decision: Rewrite the `CONTRIBUTING.md` "Status / type renames history" entry that documents the prior `code-review` → `bees-code-review` rename.**
Rationale: the current entry says "standalone invocation remains supported via the existing dual-mode prose" — this is the load-bearing precedent that future reviewers cite to revert standalone-removal. Replacing it with the new decision and reasoning means the new state is the precedent, not the old one.
*Alternative rejected:* append a new entry beside the old one, leaving the original intact. Considered but rejected: leaves contradictory precedents in the same document, which is exactly the failure mode we're trying to avoid. The history section's purpose is "what is the current state and why" — a stale-and-superseded entry undermines that purpose.

**Decision: Sweep open/in-progress bees tickets, leave done/closed tickets untouched.**
Rationale: open tickets reference old names in still-actionable bodies; updating them keeps the workflow coherent. Done/closed tickets are historical snapshots; rewriting them rewrites history without value, and the editorial cost (noise in `git log`, broken links from PR descriptions) outweighs the consistency benefit.
*Alternative rejected:* sweep all tickets including closed ones. Rejected on the editorial-cost argument.
*Alternative rejected:* skip the ticket sweep entirely. Rejected because in-flight tickets that an Engineer or PM agent reads mid-execution would surface old skill names and trigger confusion or wrong-tool invocations during the work.

**Decision: Decompose the work into four Epics rather than one monolithic Epic.**
Rationale: the four boundaries (skill internals → callers → top-level docs → ticket sweep) are natural review-and-pause points where a contributor would inspect the change before continuing. Each Epic ends with a `grep`-based verification check that produces a clean success signal, and the sequence supports clean per-Epic commit messages.
*Alternative rejected:* one monolithic "rename everything" Epic. Considered but rejected: too much surface in one review pass, and the `grep` verification checks become harder to scope when everything changes at once.
