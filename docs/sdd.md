# quorum ŌĆö Software Design

## Tech stack

- **Skill prose**: Markdown (`SKILL.md`) with YAML frontmatter (`name`, `description`). The body is the instructions Claude follows when the skill is invoked.
- **Helper scripts**: Python 3 (cross-platform). Two exist today ŌĆö `quo-setup/scripts/detect_fast_path.py` (new-machine fast-path detection) and `quo-breakdown-epic/scripts/scoped_marker_resolver.py` (Scoped-marker parser/scoper, sibling-resolved by `quo-execute` and `quo-fix-issue`).
- **External CLI**: [bees](https://github.com/gabemahoney/bees) (`bees-md` on pipx, Python 3.10+) for ticket management.
- **Runtime host**: [Claude Code](https://claude.com/claude-code) ŌĆö skills are invoked via `/<skill>` slash commands. The execution skills (`quo-execute`, `quo-fix-issue`, `quo-breakdown-epic`) orchestrate parallel work by dispatching ephemeral background subagents via the stable `Agent` tool (`run_in_background=true`). Role contracts ship as seven custom-subagent definition files at `agents/<role>.md` in the repo root (`engineer.md`, `test-writer.md`, `doc-writer.md`, `pm.md`, `code-reviewer.md`, `doc-reviewer.md`, `test-reviewer.md`), installed alongside skills at `~/.claude/agents/` (global) or `<repo>/.claude/agents/` (per-project). No experimental feature flags or display-backend configuration are required.

## Architecture overview

The repo ships 14 portable-core skills under `skills/<name>/`, each self-contained as a `SKILL.md` plus optional `scripts/`. Skills are loaded by Claude Code from either `~/.claude/skills/` (global install) or `<repo>/.claude/skills/` (per-project install). When a skill needs a bundled helper script (its own or a sibling's), it resolves the absolute path at runtime from the skill's own base directory ŌĆö which Claude Code provides in the skill invocation header. No per-machine paths are persisted to CLAUDE.md or any other tracked file.

The workflow chain is linear with two entry points:

- `/quo-setup` ŌĆö one-time per repo (idempotent re-runs)
- `/quo-plan` *or* `/quo-plan-from-specs` ŌĆö produces a Plan Bee with Epic children
- `/quo-breakdown-epic` ŌĆö decomposes one Epic into Tasks/Subtasks, commits the new ticket files at end-of-skill
- `/quo-execute` ŌĆö walks every Epic, dispatches subagents per Task, commits
- `/quo-file-issue` *and* `/quo-fix-issue` ŌĆö anytime, for bugs/follow-ups

Four review skills exist. Three of them (`quo-engineer-review`, `quo-doc-writer-review`, `quo-test-writer-review`) are orchestrator-only ŌĆö invoked by `/quo-execute` and `/quo-fix-issue` during their review cycles, with no standalone-invocation path. The fourth, `quo-spec-review`, is dual-mode ŌĆö primarily invoked by orchestrating skills (`/quo-write-prd`, `/quo-write-sdd`, and `/quo-plan`) during their review cycles, with standalone invocation also supported for ad-hoc reviews outside the workflow.

The workflow uses three hives in the target repo: **Plans** (top-level, with t1/t2/t3 = Epic/Task/Subtask, statuses `drafted`/`ready`/`in_progress`/`done`), **Issues** (no children, statuses `open`/`done`), and **Specs** (top-level Spec Bees with `t1=Doc/Docs` children, statuses `drafted`/`ready`). The Specs hive is colonized by `/quo-setup`; the `t1=Doc` children (PRD and SDD) are authored by `/quo-write-prd` and `/quo-write-sdd`, typically invoked inline from `/quo-plan` during planning. `/quo-setup` does not author Spec content itself. Plan Bees and Issues may carry one or more spec sources in their `reference_materials` field, resolved per-item by one of several resolvers the bees CLI supports: the **`file-path` resolver** (the default) treats each entry as a path to an on-disk source document (PRD, SDD, etc.) ŌĆö used by `/quo-plan-from-specs`; the **`bees` resolver** treats each entry as a ticket ID ŌĆö used by `/quo-plan` to point at a Spec Bee whose body and `t1=Doc` children (PRD + SDD, differentiated by exact title-match) hold the authoritative spec content; the **external-URL resolvers** (`github-issue`, `linear-issue`, `url`, etc.) treat each entry's `value` as a URL pointing at an external bug report (GitHub Issue, Linear ticket, internal bug tracker page, Slack archive link, etc.) ŌĆö used by `/quo-file-issue --reference` (and its `--from-github` alias) on Issues, and consumed by `/quo-fix-issue`'s Engineer and PM via `WebFetch` since the bees CLI does not yet ship concrete resolver implementations for these names. When a Plan Bee or Issue has null/empty `reference_materials`, the **ticket body itself becomes the authoritative spec** ŌĆö downstream skills explicitly substitute the body for the PRD/SDD in that mode. The full resolver-contract phrasing lives in `docs/doc-writing-guide.md` `## Project terminology`.

## Key components

- **`skills/quo-setup/`** ŌĆö one-time configuration: colonizes three hives (**Plans** ŌĆö top-level, t1/t2/t3 = Epic/Task/Subtask, statuses `drafted`/`ready`/`in_progress`/`done`; **Issues** ŌĆö no children, statuses `open`/`done`; **Specs** ŌĆö top-level Spec Bees with `t1=Doc/Docs` children, statuses `drafted`/`ready`, colonized by `/quo-setup` for use by future skills, no content is authored here at setup time), writes the two required CLAUDE.md sections (`Documentation Locations`, `Build Commands`), optional PRD/SDD bootstrap from existing codebase. Detects the new-machine case (on-disk hive markers present, the repo's scope not registered in `~/.bees/config.json`, CLAUDE.md already populated) via the bundled `detect_fast_path.py` helper and offers a fast path that re-registers hives from canonical defaults without touching CLAUDE.md.
- **`skills/quo-plan/`** ŌĆö interactive scope discovery for an idea or feature without finalized specs. Produces a Plan Bee with Epic children.
- **`skills/quo-plan-from-specs/`** ŌĆö express path for a finalized PRD+SDD on disk. Same Plan Bee output as `/quo-plan`. Default (single-feature) mode hard-fails on PRDs/SDDs that contain more than one `### Feature: <title>` subsection (the cumulative-PRD pattern produced by repeated `/quo-plan` invocations) and routes the user back to `/quo-plan` to avoid re-planning previously-planned features. The optional `--feature "<title>"` flag bypasses the multi-feature guard and scopes the planning run to a single `### Feature: <title>` subsection extracted from each of the PRD and SDD (heading must exist in both docs); `reference_materials` still points at the canonical full PRD/SDD paths, and the Plan Bee body carries a `Scoped to ...` marker so downstream skills can tell the Bee covers a sub-region of a cumulative spec. **Marker producer.**
- **`skills/quo-write-prd/`** ŌĆö authors or revises a `t1=Doc` child titled `PRD` under a Spec Bee in the Specs hive. Composable as a solo skill for revisions (`/quo-write-prd <spec-bee-id>`) or via inline Skill-tool delegation from `/quo-plan` when initial specs are being authored.
- **`skills/quo-write-sdd/`** ŌĆö authors or revises a `t1=Doc` child titled `SDD` under a Spec Bee in the Specs hive. Composable as a solo skill for revisions (`/quo-write-sdd <spec-bee-id>`) or via inline Skill-tool delegation from `/quo-plan` when initial specs are being authored.
- **`skills/quo-breakdown-epic/`** ŌĆö decompose one Epic into Tasks and Subtasks. The only skill where dispatched subagents run in `mode: "plan"` (read-only researchers). At end-of-skill, stages and commits the new ticket files (Tasks, Subtasks, Epic status update) when the Plans hive lives inside the current git repo ŌĆö resolved via `bees list-hives` in the same pattern `quo-file-issue` uses; when the hive lives outside the repo, the commit step is skipped and the next-steps menu carries a one-line note. The end-of-skill next-steps menu carries per-option "best when ŌĆ”" rationales and moves the Recommended badge across three cases keyed on (a) whether any drafted sibling Epics remain under the parent Bee and (b) whether breaking down the next Epic right now risks rework: when no drafted siblings remain, the Recommended option is "execute the whole Bee"; when drafted siblings remain and the orchestrator judges that the just-broken-down Epic's implementation will reshape contracts consumed by them (foundation-then-rewrites pattern), the Recommended option is "execute this Epic first; defer downstream breakdown"; when drafted siblings remain with pure-ordering dependencies only (no reshape risk), the Recommended option is "break down the next Epic". The case-specific rationale is surfaced only as the Recommended option's `Best when ŌĆ”` subtitle ŌĆö a freestanding rationale paragraph above the menu is forbidden because the Claude Code UI has been observed truncating long header prose mid-sentence. **Marker consumer** ŌĆö Step 1 (read parent Bee) detects the `Scoped to ...` marker and restricts the resolved doc content to the matching `### Feature: <title>` subsection before Task decomposition and Spec Traceability Review. Bundles the shared parser/scoper helper at `scripts/scoped_marker_resolver.py` (sibling-resolved by `quo-execute` and `quo-fix-issue`).
- **`skills/quo-execute/`** ŌĆö execute a Plan Bee end-to-end. Walks Epics in dependency order, dispatches ephemeral background subagents per Task, commits per Task, reviews at Bee close. **Marker consumer** ŌĆö Step 4 PM section detects the `Scoped to ...` marker on the Grandparent Bee and compares per-Task work against the scoped subsection only (sibling-resolves `quo-breakdown-epic`'s `scoped_marker_resolver.py`).
- **`skills/quo-fix-issue/`** ŌĆö fix one or more issue tickets. Single, list, or `all` modes. Same orchestration shape as `quo-execute` but at issue scope. **Marker consumer (via up_dependencies-link)** ŌĆö Issues live in the `issues` hive and have no canonical parent-Plan-Bee field in the bees ticket schema, so the PM discovers a scope-context Plan Bee opportunistically by iterating the Issue's `up_dependencies` array (a deliberate dual-use of that field ŌĆö blocker AND optional scope-context source) and applying the marker from any entry that resolves to a Plan Bee. Best-effort: a missing marker, a non-`plans`-hive `up_dependencies` entry, or a parser hard-fail falls back to full-doc spec content; if multiple Plan Bees in `up_dependencies` carry markers, the first in `up_dependencies` iteration order wins. **External-reference consumer** ŌĆö when an Issue carries `reference_materials` produced by `/quo-file-issue --reference` (resolver name `github-issue`, `linear-issue`, or `url`), the orchestrator embeds the `reference_materials` JSON value alongside the (thin) Issue body in each Agent dispatch prompt; the Engineer and PM fetch the upstream content via `WebFetch` and treat it as the spec source, since concrete resolver implementations for these names do not exist in the bees CLI yet.
- **`skills/quo-file-issue/`** ŌĆö file a new issue ticket (bug, follow-up, small feature, tech debt). Two invocation modes: (a) **in-conversation capture** ŌĆö body-as-spec, populated from a free-text description or distilled from a prior conversation, with optional `## Background and rationale`, `## Decisions and rejected alternatives`, and `## Doc divergence noted` sections; (b) **external-reference mode** ŌĆö invoked as `/quo-file-issue --reference <url>` or `/quo-file-issue --from-github <url>`, produces a thin 2-3 sentence body and writes `reference_materials` as `[{"value":"<url>","resolver":"<name>"}]` where `<name>` is selected by URL pattern (`github-issue` for GitHub Issue URLs, `linear-issue` for Linear ticket URLs, `url` otherwise). The bees CLI does not yet have concrete resolver implementations for these names ŌĆö `/quo-fix-issue`'s Engineer and PM fall back to fetching the upstream content via `WebFetch` until a real resolver lands. Symmetric with `/quo-plan-from-specs` on the planning side.
- **`skills/quo-status/`** ŌĆö show workflow stages and current progress across all hives.
- **`skills/quo-engineer-review/`**, **`skills/quo-doc-writer-review/`**, **`skills/quo-test-writer-review/`** ŌĆö orchestrator-only reviewers invoked by `/quo-execute` and `/quo-fix-issue` during their review cycles, with bees-specific loop-bounding prose for that path.
- **`skills/quo-spec-review/`** ŌĆö dual-mode reviewer for Spec Bee content (PRD and SDD `t1=Doc` children). Primary use: gates a Spec Bee's (or PRD/SDD child's) `drafted ŌåÆ ready` transition; invoked automatically by `/quo-plan` (after both writer skills complete, before the Spec Bee parent's promotion, no `--doc` flag), `/quo-write-prd` (after solo Step 6's user-approval gate, before the PRD child's promotion, with `--doc PRD`), and `/quo-write-sdd` (after solo Step 7's user-approval gate, before the SDD child's promotion, with `--doc SDD`); the per-writer review is skipped when the writer is invoked inline from `/quo-plan` since Site 1 covers both children end-to-end. Secondary use: standalone ad-hoc spec review (`/quo-spec-review <spec-bee-id> [--doc PRD|SDD]`). Returns a list of improvement work items in the same severity-tagged shape as the other three reviewers; does not mutate any ticket. Conceptual analog of upstream apiary's `/req-review`.
- **`agents/`** ŌĆö top-level directory of seven custom-subagent definition files, one per role, installed alongside skills at `~/.claude/agents/` (global) or `<repo>/.claude/agents/` (per-project). Each file carries YAML frontmatter (`name`, `description`, `model`, `tools` allowlist) plus a markdown body capturing role-specific Instructions (what to read, what to do, what to return). The execution skills reference these by `subagent_type: "<role>"` rather than carrying inline role prose, which substantially reduces SKILL.md size and makes role customization a first-class operation.

  | File | Model | Purpose |
  |---|---|---|
  | `agents/engineer.md` | Opus (always) | Implements code changes for a Subtask or set of Subtasks |
  | `agents/test-writer.md` | Opus (always) | Writes/updates tests for completed Engineer work |
  | `agents/doc-writer.md` | User's choice (Opus or Sonnet) | Authors or updates documentation |
  | `agents/pm.md` | User's choice | Per-Task PM review (spec drift, scope creep, in-flight code/doc review invocations) |
  | `agents/code-reviewer.md` | Opus (always) | Wraps `/quo-engineer-review` skill invocation |
  | `agents/doc-reviewer.md` | User's choice | Wraps `/quo-doc-writer-review` skill invocation |
  | `agents/test-reviewer.md` | Opus (always) | Wraps `/quo-test-writer-review` skill invocation |

- **`skills/quo-setup/scripts/detect_fast_path.py`** ŌĆö detect the new-machine fast-path scenario. Emits a JSON status payload (hive markers found, scope-already-registered check, CLAUDE.md sections populated, `fast_path_eligible` boolean) consumed by `/quo-setup`.
- **`skills/quo-breakdown-epic/scripts/scoped_marker_resolver.py`** ŌĆö shared parser/scoper for the `Scoped to ...` marker emitted by `/quo-plan-from-specs --feature`. Takes one positional argument (path to a file containing the parent Bee body); prints `{"scoped": false}` when no marker is present, prints `{"scoped": true, "title": "...", "docs": [{"path": "...", "content": "..."}, ...]}` when the marker is present and well-formed, and exits 2 with a single human-readable line on stderr otherwise. Resolved as a sibling script by `quo-execute` and `quo-fix-issue`. Grammar and hard-fail rules documented in `docs/doc-writing-guide.md` `## The Scoped-marker contract`.

## Contract keys

The target repo's CLAUDE.md carries two sections that act as a string contract between skills. `quo-setup` writes them; every other skill reads them.

- `## Documentation Locations` ŌĆö `Project requirements doc (PRD)`, `Internal architecture docs (SDD)`, `Customer-facing docs`, `Engineering best practices`, `Test writing guide`, `Test review guide`, `Doc writing guide`
- `## Build Commands` ŌĆö `Compile/type-check`, `Format`, `Lint`, `Narrow test`, `Full test`

`quo-execute`, `quo-fix-issue`, and `quo-breakdown-epic` hard-fail with `Run /quo-setup first.` (with a trailing `ŌĆö <reason>` clause naming the specific gap, e.g., `Run /quo-setup first. ŌĆö Specs hive is not colonized for this repo.`) if either of the two required CLAUDE.md sections, any required key inside them, or any of the three required hives (Plans, Issues, Specs) is missing.

Bundled helper script paths (e.g., `scoped_marker_resolver.py`) are *not* part of the CLAUDE.md contract ŌĆö each skill resolves its own bundled scripts at runtime from its own base directory, which Claude Code provides in the skill invocation header. This keeps per-machine paths out of tracked files.

## Orchestration in execution skills

`quo-execute`, `quo-fix-issue`, and `quo-breakdown-epic` orchestrate work by dispatching ephemeral background subagents via the stable `Agent` tool (`run_in_background=true`). The main Claude Code session running the skill is the orchestrator ŌĆö mechanically a reconciliation-loop driver, not a chat hub.

- **Reconciliation loop (read state / reconcile / yield).** Each tick is purely event-driven ŌĆö the harness wakes the orchestrator on Agent completion notifications, user input, and tool results. There is no clock primitive: no `/loop`, no `ScheduleWakeup`, no `CronCreate`, no polling loop. A tick reads ticket state (`bees execute-freeform-query`, `bees show-ticket`), reads the TaskList for active Agent invocations, reconciles the gap (dispatch a fresh Agent for any Subtask whose preconditions are met but no Agent is on it; persist results for any Subtask whose Agent has just returned; trigger PM review for completed Tasks; run inter-Epic checkpoints; dispatch the post-Bee fresh-eyes reviewers when all Epics are done), then yields. This is a Kubernetes-controller shape applied to skill orchestration: declarative state goal (bees ticket statuses), continuous reconciliation against actual state, no hand-managed message queues.
- **Hub-and-spoke as a structural property of the substrate.** Workers do not communicate with each other. All routing is orchestrator ŌåÆ Agent. The artifact-based handoff (Engineer's commits become Test Writer's input; Test Writer's tests become PM's review material) replaces the message-based handoff. This is structurally identical to the prior hub-and-spoke prescription, but achieved through a substrate that doesn't permit peer comms rather than through a "do not message peers" rule the model can drop.
- **Flat orchestration.** Per the [Claude Code sub-agents docs](https://docs.claude.com/en/docs/claude-code/sub-agents), subagents cannot dispatch other subagents ŌĆö only the top-level orchestrator may dispatch Agents. The orchestrator's working-set growth is bounded by the Epic-boundary context-clear discipline (defined in `quo-execute/SKILL.md` Section 4.2).
- **Cold dispatch for all roles.** Engineer, Test Writer, and Doc Writer are dispatched fresh per Subtask; PM, Code Reviewer, Doc Reviewer, and Test Reviewer are always fresh by design (fresh-eyes review). Coherence across dependent Subtasks is preserved through the diff: each later Engineer reads the prior Engineer's commit, matching the hub-and-spoke "diff is the handoff" pattern. Concurrent specialist work is preserved: Engineer-Subtask-N+1 runs in parallel with Test-Writer-Subtask-N once the orchestrator's reconciliation loop ticks past Engineer-Subtask-N's completion.
- **State-aware dispatch.** Before dispatching any Agent, the orchestrator re-reads the target ticket's current state via `bees execute-freeform-query` and skips the dispatch when the ticket is already `done`. This prevents redundant work caused by acting on stale memory.
- **Verbatim ticket-body quoting.** Agent dispatch prompts embed the ticket body exactly as `bees show-ticket` returns it ŌĆö no paraphrasing. Identifier names (function/flag/type names) are preserved literally so workers do not invent or mis-spell them downstream. Framing prose stays outside the quoted block.
- **TaskList as progress UI.** The orchestrator creates one TaskList task per concurrent Agent invocation (e.g., `engineer-qf-subtask-1`, `test-writer-qf-subtask-1`, `pm-qf-subtask-1-review`). Status transitions (`pending ŌåÆ in_progress ŌåÆ completed`) are updated as Agents start and finish; `metadata.activity` carries finer-grained progress (e.g., `"running /quo-engineer-review (~5 min)"`). Claude Code's native TaskList UI renders these live, providing visual parallelism without an Agent Teams display backend.
- **Time-bounded review iteration.** In `quo-execute`, the PM short-circuits `/quo-engineer-review` and `/quo-doc-writer-review` when a single invocation returns more than ~10 items or runs more than ~5 turns: triage to blocker-severity items only and defer the rest to the Task summary.

## Model assignment in execution skills

Hardcoded in `quo-execute` and `quo-fix-issue`:

- **Engineer, Test Writer, Code Reviewer, Test Reviewer**: always Opus. Not user-configurable.
- **Doc Writer, Product Manager, Doc Reviewer**: user picks Opus or Sonnet at the start of the run.

## External dependencies

- **bees CLI** (`bees-md` via pipx) ŌĆö ticket management. Required.
- **Claude Code** ŌĆö runtime host. Required.
- **Python 3** ŌĆö helper scripts and JSON edits in skill prose. Required (already a dependency of bees).
- **tmux** ŌĆö *only* required for the optional later-install skills (`bees-fleet`, `bees-worktree-add`, `bees-worktree-rm`). The portable core does not need tmux.

## Deployment

- **Global install** (recommended for single-user machines): clone the repo, copy `skills/*` into `~/.claude/skills/`. POSIX `cp -r` or PowerShell `Copy-Item -Recurse`.
- **Per-project install**: copy `skills/*` into a single repo's `.claude/skills/`. Useful for trying the workflow on one repo without affecting others.
- **Live-edit symlink** (contributor pattern): symlink each `~/.claude/skills/<skill>` at `~/code/quorum/skills/<skill>` so edits in the clone are immediately picked up by Claude Code. See CONTRIBUTING.md for the per-OS commands.

## Per-feature design

### Feature: Test strategy for the skills repo

**Status: paused as of 2026-05-03.** This feature remains paused; the Ephemeral-Agent Orchestration rewrite (Plan Bee `b.5tm`) shipping on `main` does not auto-unpause it. `b.gar`'s Plan Bee body has now been refreshed to reflect the post-orchestration, bees-only architecture (the "Optional beads backend" feature `b.9xr` and its `ticket_backend.py` dispatcher seam were abandoned, not gating Test strategy any longer), and Layer 2.5 ŌĆö the backend-equivalence harness ŌĆö is explicitly deferred there. The architecture below still describes the originally-planned dual-backend test strategy and will be re-scoped against the refreshed `b.gar` body when this feature resumes.

**Architecture.** Three independent test layers, each with its own entrypoint, unified under a top-level `make test` target. Test code lives in two locations: per-helper unit tests sit beside the helper they cover (`skills/<skill>/scripts/test_<helper>.py`); cross-cutting test infrastructure lives at `tests/` at the repo root.

**Layer 1 ŌĆö Helper unit tests.** Per-helper pytest files at `skills/<skill>/scripts/test_<helper>.py` (and `skills/_shared/scripts/test_ticket_backend.py` for the dispatcher introduced in the previous feature). Tests use pytest's `tmp_path` fixture for filesystem isolation and `monkeypatch` for environment overrides. Coverage targets: every public function, every error path, every JSON contract field. The dispatcher's tests exercise both backend branches when the relevant CLI is present and skip cleanly when it isn't. Entrypoint: `pytest skills/`.

**Layer 2 ŌĆö Structural SKILL.md linter.** A `tools/lint_skills.py` script (Python 3, no third-party deps) walks every `skills/*/SKILL.md`, parses YAML frontmatter, and asserts the design rules baked into CLAUDE.md:

- Frontmatter has `name` and `description`.
- Every fenced code block tagged `bash` (or labeled "POSIX") has a sibling block tagged `powershell` (or labeled "Windows") in the same section, and vice versa.
- No path starting with `/Users/`, `/home/`, or `C:\Users\`.
- No raw `cargo`, `npm`, `pip`, `pipx` commands outside sections explicitly marked as install instructions.
- All bundled-script references use the `<bees-setup-base-dir>` literal placeholder rather than absolute paths.
- All `bees ...` and `bd ...` subcommands in skill prose match an allow-list (catches typos and out-of-vocabulary subcommands).
- Backend-conditional sections come in matching pairs (every "if backend=bees" block has an "if backend=beads" sibling, where applicable).

Output is human-readable: `<file>:<line>: <rule>: <message>`. Exits non-zero on any rule violation. Entrypoint: `python tools/lint_skills.py`.

**Layer 2.5 ŌĆö Backend-equivalence harness.** A pytest test suite at `tests/equivalence/test_dispatcher_equivalence.py`. Each test:

1. Spins up two temp directories ŌĆö one initialized with `ticket_backend.py setup-spaces --backend bees`, the other with `--backend beads`.
2. Runs the same sequence of dispatcher verb calls against each (e.g., create Plan Bee ŌåÆ create Epic ŌåÆ set `up_dependencies` ŌåÆ query unblocked tickets ŌåÆ show ticket).
3. Captures resulting state via the dispatcher's read verbs (`query`, `show`).
4. Normalizes the responses ŌĆö strips `id` (different formats per backend), `created_at` (timestamps), `guid`, and any other backend-specific noise ŌĆö and asserts deep equality on the remaining structure.

The harness skips gracefully if either CLI is missing, with a `pytest.skip` message naming what's absent. Entrypoint: `pytest tests/equivalence/`.

**Top-level entrypoint.** A `Makefile` at the repo root:

```make
.PHONY: test test-helpers test-lint test-equivalence

test: test-lint test-helpers test-equivalence

test-helpers:
	pytest skills/

test-lint:
	python tools/lint_skills.py

test-equivalence:
	pytest tests/equivalence/
```

For Windows contributors without `make`, a `tools/run_tests.py` script (Python; invokes the same commands via subprocess) provides a portable fallback. Both are documented in CLAUDE.md `## Test Commands`.

**CLAUDE.md `## Test Commands`.** A new contract-style section, contributor-facing only (skills don't read it):

```markdown
## Test Commands

- **Run all tests**: `make test`
- **Helper unit tests only**: `pytest skills/`
- **SKILL.md linter only**: `python tools/lint_skills.py`
- **Backend-equivalence harness only**: `pytest tests/equivalence/`
- **Windows (without make)**: `python tools/run_tests.py`
```

**README Contributing section update.** A short paragraph immediately after the existing Contributing principles, naming the three layers and pointing at `make test` and CLAUDE.md `## Test Commands`.

**CI integration.** A `.github/workflows/test.yml` (or equivalent) runs `make test` on every push and PR. Both backend CLIs installed in the workflow image so Layer 2.5 doesn't skip. The workflow file lives in the repo so contributors can see what's gating their PR.

**Sequencing.** This feature blocks on the "Optional beads backend" feature (Plan Bee `b.9xr`) reaching `done`. Layer 1 covers `ticket_backend.py` (only exists post-b.9xr Epic A); Layer 2 needs to know about backend-conditional blocks (only exist post-b.9xr Epic B); Layer 2.5 explicitly requires both backends to be present.

**Decomposition.** Three Epics:

- Epic A ŌĆö Layer 1: pytest infrastructure and unit tests for every bundled helper.
- Epic B ŌĆö Layer 2: structural SKILL.md linter (independent of A; can be worked in parallel).
- Epic C ŌĆö Layer 2.5 + integration: backend-equivalence harness, top-level `make test`, CLAUDE.md `## Test Commands`, README Contributing paragraph, CI workflow. Blocks on Epic A (uses pytest infrastructure) and Epic B (linter must already be wired to the make target).

### Feature: Side-effect-free /quo-plan and /quo-file-issue with preserved context

**Architecture.** The redesign is structured around four architectural moves: a new **Specs hive**, a polymorphic `reference_materials` field with a `bees` resolver, two new composable spec-authoring skills, and an expansion of the `doc-writer` agent's responsibility to own post-implementation cumulative-doc updates. Together these decouple per-feature spec authoring from cumulative project-doc mutation: per-feature PRD/SDD content lives in queryable bees tickets and is folded into the cumulative on-disk docs only after implementation lands.

**Specs hive.** A third top-level hive joins Plans and Issues. Spec Bees are top-level Bees in the Specs hive with `t1=Doc/Docs` children ŌĆö PRD and SDD as separate `t1=Doc` tickets, differentiated by exact title-match (`PRD` vs `SDD`), not by tier. The hive's allowed resolver is `bees`, so a Plan Bee's `reference_materials` can point at a Spec Bee. Status vocabulary is two-state ŌĆö `drafted` (being written) and `ready` (referenceable) ŌĆö distinct from the four-state Plans ladder and the two-state Issues ladder; `worker`/`finished` semantics are conformance-driven for spec docs and do not add meaning, so the shorter ladder is intentional. `/quo-setup` colonizes the Specs hive at first-run and detect-and-adds it on existing repos that pre-date the hive; `/quo-execute`, `/quo-fix-issue`, and `/quo-breakdown-epic` hard-fail with `Run /quo-setup first. ŌĆö Specs hive is not colonized for this repo.` when the hive is missing.

**Reference materials with `bees` resolver.** The bees CLI's `reference_materials` field accepts arbitrary resolvers per entry. Two are now in use across the workflow: the **`file-path` resolver** (the default) treats each entry as a path to an on-disk source document ŌĆö used by `/quo-plan-from-specs`; the **`bees` resolver** treats each entry as a ticket ID of the shape `[{"value":"<bee-id>","resolver":"bees"}]` ŌĆö used by `/quo-plan` to point at a Spec Bee whose body and `t1=Doc` children hold the authoritative spec content. When `reference_materials` is null/empty (e.g., the bootstrap-mode case), the **Plan Bee body itself becomes the authoritative spec** and downstream skills explicitly substitute the Bee body for the PRD/SDD. This three-source-mode design (path on disk / referenced Bee + children / own Bee body) preserves polymorphism for future GitHub-issue, Linear, or external-doc resolvers without skill changes.

**Two-hop lookup.** `agents/pm.md` and `skills/quo-breakdown-epic/SKILL.md` resolve `bees`-resolver entries via a two-hop pattern: hop 1 reads the entry's `value` from the parent Bee's `reference_materials`; hop 2 runs `bees show-ticket --ids <bee-id>` and treats the resolved Bee's body (or, for Spec Bees, its `t1=Doc` children enumerated by exact title-match) as the spec source. The `file-path` resolver path and the body-as-spec fallback remain functional and untouched. Canonical recipes for the two-hop pattern live in `docs/doc-writing-guide.md` `## Project terminology` under the `Reference materials` entry.

**`/quo-write-prd` and `/quo-write-sdd` skills.** Spec-authoring logic factors out of `/quo-plan` into two new composable sub-skills, each owning the quality-bar prose for its respective doc type. Default user experience is unchanged ŌĆö typing `/quo-plan` still produces a Plan Bee with PRD and SDD content authored ŌĆö because `/quo-plan` invokes them inline via the Skill tool. They are also solo-invokable for revisions (`/quo-write-prd <spec-bee-id>`, `/quo-write-sdd <spec-bee-id>`), which is a real use case (revising a PRD after learning something during execution). The factoring shrinks `/quo-plan` substantially and gives mental-model parity with apiary's `/write-prd` + `/write-srd`. PRD and SDD child-ticket bodies include explicit sections for decisions, rejected alternatives, and rationale ŌĆö not just requirements.

**`doc-writer` agent expanded responsibility.** The `agents/doc-writer.md` contract now owns appending or updating `### Feature: <title>` subsections in the cumulative project PRD (`## Per-feature scope` header) and SDD (`## Per-feature design` header) post-implementation, reflecting what was actually built. The redundancy that previously had `/quo-plan` writing speculative SDD content AND `doc-writer` updating SDD from the actual diff is resolved: the speculative write is gone; the post-implementation write (anchored to what shipped) remains. The doc-writer's categorization heuristic (user-facing feature vs architecture-only vs deployment vs pure-refactor) governs which cumulative doc(s) get an entry on a given run; the diff is the primary signal, with the dispatched Task/Subtask body as a secondary signal. Idempotency is enforced by exact-match on the verbatim Plan Bee title ŌĆö `### Feature: <title>` heading text is matched case-sensitively, and an existing subsection is replaced in place rather than duplicated.

**`/quo-file-issue` redesign.** The former filing-time doc-mutation step (now folded into the body-authoring step's optional `## Doc divergence noted` capture) no longer mutates project docs when an Issue surfaces doc divergence. Instead, divergence observations are captured in a `## Doc divergence noted` section in the Issue body for `/quo-fix-issue`'s doc-writer to act on at fix time. The Issue body template gains optional `## Background and rationale` and `## Decisions and rejected alternatives` sections so that the analytical depth of the originating discussion survives into the Issue (the Issue body is the spec source for `/quo-fix-issue` ŌĆö there is no `reference_materials`, no children, no PRD/SDD pointer). The skill is mid-conversation aware: when invoked from a substantive preceding discussion, it does not re-ask discovery questions that the conversation already answered.

**Scoped-marker machinery retained.** The `Scoped to ...` marker, the `skills/quo-breakdown-epic/scripts/scoped_marker_resolver.py` parser, and the asymmetric Path A vs Path B handling in `agents/pm.md` are all retained. `/quo-plan-from-specs --feature "<title>"` remains the marker producer; `/quo-breakdown-epic`, `/quo-execute`, and `/quo-fix-issue` remain marker consumers (the `quo-fix-issue` consumer iterates the Issue's `up_dependencies` array opportunistically, since Issues have no canonical parent-Plan-Bee field). What changed is only that `/quo-plan` no longer emits markers ŌĆö because it no longer co-mingles per-feature content into shared cumulative docs, there is nothing to scope inside.

**Decomposition.** The feature shipped as eight Epics under Plan Bee `b.31f`: Epic 1 (Specs hive setup in `/quo-setup`); Epic 2 (`/quo-write-prd` skill); Epic 3 (`/quo-write-sdd` skill); Epic 4 (`/quo-plan` redesign with Spec Bee + `reference_materials` + `bees` resolver, including mid-conversation context awareness in Steps 0-2); Epic 5 (PM/breakdown two-hop lookup updates); Epic 6 (`doc-writer` cumulative-doc responsibility expansion); Epic 7 (`/quo-plan-from-specs` regression check + Overview prose positioning vs `/quo-plan`); Epic 8 (`/quo-file-issue` redesign ŌĆö no doc mutation, `## Doc divergence noted` capture, optional discussion-derived body sections, mid-conversation awareness). Epic 6 was self-referential (it edited the doc-writer that the cumulative-doc write was meant to use) so the cumulative `### Feature: <title>` entries for `b.31f` itself were backfilled manually under Issue `b.2w1` after the Bee was `done`. ### Feature: Add /quo-spec-review skill (apiary /req-review analog)

**Architecture.** A new dual-mode review skill that mirrors the work-item-list shape of `/quo-engineer-review`, `/quo-doc-writer-review`, and `/quo-test-writer-review` (which are themselves orchestrator-only post the b.mak rename). Lives at `skills/quo-spec-review/SKILL.md` (no bundled scripts ŌĆö the skill is prose-only, dispatching through `bees execute-freeform-query` and `bees show-ticket` for ticket I/O). No new contract keys; no new agents; no changes to the seven role files in `agents/`.

**Inputs and resolution.** One positional argument `<spec-bee-id>` (the Spec Bee whose PRD and/or SDD `t1=Doc` children are under review) and one optional `--doc PRD|SDD` flag scoping to a single child. Doc resolution uses a freeform query stage `[hive=specs, parent=<spec-bee-id>, title~^(PRD|SDD)$]` (or the narrowed single-title regex when `--doc` is passed); the title regex is anchored on both ends so the match is exact and case-sensitive against the canonical `PRD` / `SDD` titles produced by `/quo-write-prd` and `/quo-write-sdd`. Each resolved child's body is fetched via `bees show-ticket --ids <doc-ticket-id>` and treated as the source of truth ŌĆö no on-disk file is read as a substitute, since downstream skills (`/quo-breakdown-epic`, `/quo-execute`'s Engineer / PM, `/quo-fix-issue`) consume the bees ticket body, not any artifact on disk.

**Checklists.** The skill carries three checklist passes inline in its prose:

1. **PRD checklist** ŌĆö eight categories tied to the twelve required sections imposed by `/quo-write-prd` Step 4: section completeness, problem-statement clarity, acceptance-criteria measurability, scope clarity (`## Non-Goals / Out of Scope`), implementation-detail leakage (PRD covers what/why, not how), vague-language detection, rationale-and-decisions substance, and open-questions discipline.
2. **SDD checklist** ŌĆö ten categories tied to the seven required sections imposed by `/quo-write-sdd` Step 5: section completeness, codebase-grounding (real module/file/function names, `RESEARCH NEEDED:` tag surfacing), requirements structure (`SR-` prefix, domain grouping, observable-behavior framing), architecture coverage (top-level framing plus component-by-component coverage), existing-behavior preservation (greenfield placeholder vs explicit contracts), test-fixture conventions, documentation coverage (canonical CLAUDE.md `## Documentation Locations` paths), decomposition signal for downstream Epic breakdown, data-model and contract-key impact callouts, and rationale-and-decisions substance.
3. **Cross-document consistency pass** ŌĆö five categories run only when both PRD and SDD are in scope: goal-to-requirement coverage (every PRD `## Goals` entry maps to at least one SDD `SR-` requirement), acceptance-criterion-to-design coverage (every PRD `## Acceptance Criteria` entry has SDD coverage), out-of-scope alignment (no SDD design element implements anything PRD `## Non-Goals / Out of Scope` excludes), open-question alignment (no SDD section confidently decides a question PRD `## Open Questions` lists as still open), and rationale alignment (PRD and SDD `## Decisions and rejected alternatives` sections agree on rejected alternatives when both have substantive content).

**Output contract.** The skill returns a numbered markdown list under a `## Spec Review Work Items` heading. Each item carries a severity tag (`blocker`, `suggestion`, `nit`) matching the ladder used by the other three reviewers, cites the document (`PRD` or `SDD`) and section heading, and gives a one-line description of the finding. When there are no important findings, the skill says so explicitly rather than padding the list with trivia. The skill does not mutate any ticket ŌĆö output is text-only, returned to the caller (human or orchestrating skill).

**Dual-mode invocation.** Standalone use is `/quo-spec-review <spec-bee-id> [--doc PRD|SDD]` from the prompt ŌĆö the caller is a human or another standalone tool, output the work-item list and stop. Orchestrator-invoked use (when `/quo-write-prd`, `/quo-write-sdd`, or `/quo-plan` later wires this skill in as a post-write gate on the Spec Bee's `drafted ŌåÆ ready` transition) applies the same loop-bounding discipline the other three reviewers carry: be selective, do not report trivial-but-not-important items each pass, say "no important findings" when applicable to avoid infinite review-fix-review loops.

**Preconditions.** Hard-fail with `Run /quo-setup first.` (plus a one-line note naming the missing piece) if the Specs hive is not colonized for the host repo (`bees list-hives` must include a hive whose `normalized_name` is `specs`). The skill does not require CLAUDE.md `## Build Commands` since it produces text findings, not build/test/lint output.

**Catalog and prose updates.** README.md skill table gets a new row parallel to the existing review-skill rows; the surrounding "three reviewers" sentence becomes "four reviewers" and lists `/quo-spec-review` alongside the existing three. The portable-core skill count (referenced in README.md's "Coming soon" preamble, this SDD's `## Architecture overview`, CLAUDE.md's tmux-dependency rule, and `docs/doc-writing-guide.md`'s contributing principles) bumps from 13 to 14. The SDD `## Architecture overview` "Three review skills" sentence becomes "Four review skills" and adds `quo-spec-review` to the parenthetical list; the SDD `## Key components` review-skill entry stays as-is and a new `quo-spec-review/` entry follows it.

**Decomposition.** Shipped as a single Issue (`b.uxa`) rather than a multi-Epic Plan Bee, on the reasoning that the skill is small (one new `SKILL.md`, no new helpers, no contract changes, no agent changes) and the analogs already exist for the Engineer to model from. The `up_dependencies: [b.31f]` on the Issue is a blocker dep, not a scope-context Plan Bee ŌĆö the new skill stands on its own and is not scoped to any subsection of `b.31f`'s spec content.

### Feature: /quo-file-issue: add --reference URL mode for external sources (GitHub, Linear, etc.)

**Architecture.** The redesign threads three architectural moves through the existing `/quo-file-issue` and `/quo-fix-issue` skills plus the Engineer and PM subagents: a **mode-fork branch** in `/quo-file-issue` that routes external-reference invocations away from the body-template authoring path, a **URL-pattern resolver-name heuristic** that picks a canonical resolver name (`github-issue` / `linear-issue` / `url`) without requiring a concrete resolver implementation in the bees CLI, and a **`WebFetch`-fallback fetch path** in `/quo-fix-issue`'s Engineer and PM subagents that consumes external-URL `reference_materials` entries when picking up the Issue. No new skills, no new agents, no new bundled helpers, no new contract keys.

**Mode fork in `/quo-file-issue`.** `skills/quo-file-issue/SKILL.md` gains a "Mode fork" sub-section before Step 0 that parses `--reference <url>` / `--from-github <url>` from the argument string and routes accordingly. When an external-reference flag is present, the skill skips Step 0 (mid-conversation context detection) and Step 1 (distill-vs-restart fork) and Step 3's full body-template path entirely; instead it routes to a new "External-reference branch" at the end of the Steps section with three sub-steps: (A) author a thin Issue body (2-3 sentences, optionally distilled from conversation context, fetched via `WebFetch`, or asked from the user in prose); (B) pick a resolver name from the URL-pattern heuristic; (C) run `bees create-ticket` with the inline `--body` plus `--reference-materials '[{"value":"<url>","resolver":"<name>"}]'`. Steps 4 (commit) and 5 (report back) are shared with the in-conversation capture flow. The `argument-hint` is updated to `[<description> | --reference <url> | --from-github <url>]`.

**URL-pattern resolver-name heuristic.** The resolver name is selected by URL pattern matching on host first, then path-shape: GitHub Issue URLs (`https://github.com/<owner>/<repo>/issues/<n>` and similar) yield `github-issue`; Linear ticket URLs (`https://linear.app/<workspace>/issue/<id>` and similar) yield `linear-issue`; anything else (Slack archive link, internal bug tracker, generic web page) falls back to `url`. The match is best-effort and informational ŌĆö it primarily helps human readers and future resolver implementations identify the source class. Writing the canonical resolver name now (rather than always writing `url`) future-proofs existing Issue tickets so they do not need to be migrated when concrete resolvers ship.

**Concrete-resolver gap and `WebFetch` fallback.** No concrete `github-issue`, `linear-issue`, or `url` resolver exists in the bees CLI today ŌĆö that is intentional and out of scope for this skill set; concrete resolvers are tracked separately as their owners materialize. Until those land, the workflow falls back to fetching the upstream content via `WebFetch` whenever it is needed. `agents/engineer.md` and `agents/pm.md` both gain `WebFetch` in their tool allowlists and a new instruction bullet describing the fetch convention: when the Issue's `reference_materials` is non-empty and points at an external URL, the worker fetches the upstream content via `WebFetch` and treats it as the spec source for the implementation (Engineer) or the spec review (PM); the Issue body itself is intentionally thin on this path. `WebFetch` failures (network policy, auth-gated source, etc.) are surfaced to the orchestrator rather than silently fallen back to the thin body alone ŌĆö the dispatch prompt's embedded body is not enough on this path.

**`/quo-fix-issue` orchestrator wiring.** `skills/quo-fix-issue/SKILL.md` gains a single sub-bullet in the "Dispatch prompt: quote the issue body verbatim" sub-section: when the Issue's `reference_materials` is non-empty, embed the `reference_materials` JSON value alongside the (thin) body in each dispatch prompt so the worker can read the resolver name and URL. The orchestrator does not pre-fetch the URL ŌĆö fetch is the worker's responsibility per the role contracts. The PM dispatch description's resolver-branching list is updated to enumerate `github-issue` / `linear-issue` / `url` alongside the existing `file-path` and `bees` resolver paths; `agents/pm.md` is the authoritative spec for all three resolver-driven paths (`file-path`, `bees`, and the external-URL resolvers like `github-issue` / `linear-issue` / `url`); the body-as-spec path is the null/empty fallback. Scoped-marker resolution is skipped on the external-URL path (Scoped markers apply to file-on-disk PRD/SDD content, not to external URLs).

**Body-shape divergence on the external-reference path.** The thin body shape on this path is intentionally **flat** ŌĆö no `## Description` / `## Current behavior` / `## Expected behavior` / `## Impact` / `## Suggested fix` headings. Those sections exist to extract structure from in-conversation capture; on the external-reference path the structure lives at the URL, and forcing the headings into a thin body would either duplicate URL content or render mostly-empty stubs. The optional `## Doc divergence noted` section may still be appended on this path when the user (or surrounding conversation) explicitly flags a doc-divergence observation that the external source does not already capture; in routine external-reference filings the section is omitted. A single convention line near the top of the body ŌĆö `External reference: <source-class> <url>` ŌĆö names the referenced source so a human reader of the Issue body alone can tell where to look; the line is a convention, not a contract ŌĆö `reference_materials` is the load-bearing signal that downstream `/quo-fix-issue` reads.

**Inline-body vs temp-file fallback in step C.** Unlike the in-conversation capture path (which always writes the body to a temp file under `<tempdir>/.quorum/` to avoid Claude Code's command-injection guard on multi-line bodies and shell-special characters), the external-reference branch passes the thin body inline via `--body` because most thin bodies fit cleanly on a single line. The temp-file convention from Step 3c is the safety valve: if the thin body authored in (A) contains anything that would trip the injection guard ŌĆö a newline followed by a `#` heading, backticks, or shell-special characters ŌĆö fall back to the temp-file path (write under `<tempdir>/.quorum/bees-body-<short-suffix>.md` with the `mkdir -p` / `New-Item -ItemType Directory -Force` create-if-absent step) and replace `--body "<...>"` with `--body-file <path>`, keeping `--reference-materials` unchanged.

**Customer-facing and architecture doc updates.** README.md skill table row for `/quo-file-issue` is rewritten to document both invocation modes (in-conversation capture vs external-reference) and explicitly names the external-URL resolvers and the `WebFetch` fallback. SDD `## Architecture overview` resolver list is extended from two resolver classes (`file-path`, `bees`) to three (adding the external-URL resolver class). SDD `## Key components` entries for `quo-file-issue` and `quo-fix-issue` are updated ŌĆö the `quo-file-issue` entry now describes both modes; the `quo-fix-issue` entry gains an "**External-reference consumer**" rider naming the dispatch-prompt embedding and the `WebFetch` fallback. `docs/doc-writing-guide.md` `## Project terminology` `Reference materials` entry adds an "External-URL resolvers" sub-bullet alongside the existing `file-path` and `bees` resolver sub-bullets.

**Decomposition.** Shipped as a single Issue (`b.qs8`) rather than a multi-Epic Plan Bee. The work is small and surgical (one mode-fork branch in one `SKILL.md`, two tool-list and instruction additions in agents, one orchestrator-side dispatch-prompt sub-bullet, and parallel doc updates), and the analogs already exist for the Engineer to model from (`/quo-plan-from-specs` is the symmetric planning-side skill). The Issue's `up_dependencies: []` is empty ŌĆö the work is independent of `b.31f` (the prior `/quo-plan` and `/quo-file-issue` redesign) per `b.31f`'s explicit deferral rationale.

### Feature: Recommend gh issue close at end of /quo-fix-issue for github-issue resolver

**Architecture.** Add a new Section 8 ("Recommend upstream GitHub close commands") to `skills/quo-fix-issue/SKILL.md`, executed after Section 7's post-completion review either reports clean or its per-finding follow-ups close out, and **before** the orchestrator yields the turn. The section is pure prose-driven ŌĆö no helper script, no agent dispatch, no contract-key change. It iterates the run's fixed-issue list, reads each Issue's `reference_materials` (re-using the value the orchestrator captured during Section 2 / Section 3 dispatch reads when still in context, otherwise re-reading via `bees show-ticket --ids <issue-id>`), filters for entries whose `resolver == "github-issue"`, parses the canonical GitHub Issue URL pattern (`https://github.com/<owner>/<repo>/issues/<n>`) for `<owner>`, `<repo>`, and `<n>`, captures the per-issue commit SHA from Section 6 step 2.4 (preferred path: track at commit time, keyed by issue ID; fallback: re-derive post-hoc by walking back from `HEAD` over the session's commits using `git log --format=%H <pre-session-sha>..HEAD`), abbreviates via `git rev-parse --short`, and emits one bullet of the shape `gh issue close <n> --repo <owner>/<repo> -c "Fixed in <sha>."` per matching entry. The entire block is suppressed when zero bullets are collected. The skill never invokes `gh issue close` itself.

**Why this shape.** Three design decisions shape Section 8:

- **Scope to `github-issue` resolver only.** `linear-issue` close requires Linear's own CLI/API which isn't bundled with the workflow; the generic `url` resolver has no close concept. The recommendation block intentionally does not generalize to those resolvers ŌĆö adding either is a separate Issue once the corresponding integration is decided. This keeps the prose simple (one URL pattern to parse, one command shape to emit) and the surface minimal.
- **Pure recommendation, never executes.** The workflow does not assume the user is authenticated against `gh` on the same machine where `/quo-fix-issue` runs (a real concern: many users develop on one machine and authenticate `gh` only on a build / CI machine, or against multiple accounts). Surfacing the command for copy-paste keeps the auth surface owned by the user. This mirrors the broader "skills must work on Rust, Node, Python, Go" portability principle: no assumption baked into the workflow about the user's external tooling state.
- **Position after Section 7, before yield.** Emitting earlier (e.g., at Section 6's per-issue close-out) would interleave the recommendation lines with per-issue summary blocks and dilute the signal. Emitting after Section 7's post-completion review consolidates the recommendation into a single end-of-run block the user can copy in one gesture. The "before yield" constraint is load-bearing ŌĆö once the orchestrator yields the turn, the user's next action may begin and the recommendation becomes harder to spot.

**SHA capture.** Two equivalent paths are documented to obtain the per-issue commit SHA: (a) **track at commit time** ŌĆö record the SHA the moment Section 6 step 2.4's `git commit` returns, keyed by issue ID, and re-use it in Section 8 (abbreviating via `git rev-parse --short` if the captured form is the full 40-char SHA); (b) **re-derive post-hoc** ŌĆö resolve the commit per Issue by `--grep`-filtering the session's commits against the per-issue commit-message convention from Section 6 step 2.4 (every per-issue commit's subject begins with `Fix issue: <title> (<issue-id>)`), running once per fixed Issue: `git log --reverse --format=%h --grep='Fix issue:.*(<issue-id>)' <pre-session-sha>..HEAD` (substituting the literal Issue ID for `<issue-id>` and the session-start SHA captured at Section 7 step 1 for `<pre-session-sha>`). Each invocation returns at most one match ŌĆö the per-issue commit for that Issue ŌĆö and `%h` emits the abbreviated SHA directly so no follow-up `git rev-parse --short` step is needed on the fallback path. Path (a) is preferred for clarity but path (b) is sound when the orchestrator did not track at commit time. Two design choices on the fallback command are load-bearing: (i) **scoping each lookup by `(<issue-id>)`** filters out any extra commits Section 7 step 6's "Fix in this session" branch may have landed in the same `<pre-session-sha>..HEAD` window ŌĆö a positional bulk-log pairing without `--grep` would mis-pair those follow-up commits against the fixed-issue list; (ii) **`--reverse`** flips `git log`'s default newest-first ordering to oldest-first, which only matters in the (rare but legal) case where multiple commits in the window match the per-issue grep ŌĆö oldest-first then puts the original per-issue fix-commit at position 1 of the output, ahead of any later amendments that share the same Issue ID in their subject. Both ordering decisions defend against silent SHA / Issue mis-pairing.

**Suppression and ordering.** The block is suppressed entirely when zero bullets are collected (every fixed Issue's `reference_materials` is null/empty or carries only non-`github-issue` resolvers). This keeps the trace clean for users who never used the URL path. Bullet ordering matches the run's fixed-issue iteration order (single-issue mode: one bullet; list / `all` mode: bullets in fixed-list order). When a single fixed Issue carries multiple `github-issue` entries in `reference_materials` (uncommon but legal ŌĆö the resolver list is an array), one bullet is emitted per matching entry, in array order. Issues that were skipped (per Section 2's blocked-issue handling in batch mode) or never reached (e.g., upfront validation soft-failed in Section 1) do not contribute bullets ŌĆö only Issues that this run actually marked `done` via Section 6 are iterated.

**Informational, not a gate.** The recommendation block is plain console output ŌĆö NOT an `AskUserQuestion` gate. The user is not asked to confirm; the orchestrator emits the block and yields. Adding a confirmation gate would defeat the purpose: users who want to act on the recommendation copy the line and run it; users who don't want to act on it ignore the block.

**No agent / contract changes.** Section 8 lives entirely in `skills/quo-fix-issue/SKILL.md` orchestrator prose. No agent role files (`agents/engineer.md`, `agents/pm.md`, `agents/doc-writer.md`, etc.) need updates ŌĆö the recommendation is built by the orchestrator from data already captured during Sections 2-7. No `## Build Commands` / `## Documentation Locations` contract keys are added or renamed. Existing Issue tickets do not require migration; the recommendation is computed at run time from `reference_materials` data already on each ticket.

**Decomposition.** Shipped as a single Issue (`b.h3s`) ŌĆö a small, prose-only orchestrator extension at the tail of `skills/quo-fix-issue/SKILL.md`, plus the cumulative-doc subsections in PRD and SDD and a one-line README skill-table addendum. ### Feature: Spec-review loop-back UX gates dropped under orchestrator load

**Architecture.** Move the load-bearing routing prescription for review-skill findings out of nested orchestrator prose and into each review skill's own output as a `**Next action for the orchestrator:**` trailer line. The trailer names the precise gate or follow-up the orchestrator must run after consuming the findings ŌĆö `AskUserQuestion` with the recommended option set on a findings-present return, immediate promote-and-continue on a clean return ŌĆö so the routing rule travels with the artifact a fresh orchestrator agent actually reads, rather than living three levels deep in a long orchestrator-skill body that the agent is reproducibly missing under load. Affects four review skills (`/quo-spec-review`, `/quo-engineer-review`, `/quo-test-writer-review`, `/quo-doc-writer-review`) and five orchestrator sites (`/quo-plan` Step 4c, `/quo-write-prd` Step 6a, `/quo-write-sdd` Step 7a, `/quo-execute` Section 5, `/quo-fix-issue` Section 4). No new skills, no new agents, no new bundled helpers, no new contract keys, no changes to the seven role files in `agents/`. Existing review-skill output consumers that already followed the orchestrator-prose rules continue to work without modification ŌĆö the trailer is additive, not a contract evolution.

**Trailer shape ŌĆö three variants for `/quo-spec-review`.** The spec reviewer's output is severity-tagged (`blocker` / `suggestion` / `nit`), so the trailer collapses to three shapes keyed on what the work-item list above it contains: (1) **Blockers present** ŌĆö trailer prescribes `AskUserQuestion` with finite choices `Revise` (recommended) / `Proceed anyway (override blockers)` before the Spec Bee's `drafted ŌåÆ ready` promotion; (2) **Suggestions / nits only** (no blockers) ŌĆö trailer prescribes `AskUserQuestion` with finite choices `Proceed (acknowledge findings)` / `Revise` before promotion; (3) **No findings** ŌĆö trailer prescribes immediate promotion with no user prompt at this step. The phrasings are emitted verbatim by `/quo-spec-review` Step 4 alongside the work-item list itself.

**Trailer shape ŌĆö two variants for the three code/test/doc reviewers.** `/quo-engineer-review`, `/quo-test-writer-review`, and `/quo-doc-writer-review` do not emit severity tags, so each collapses to two shapes: (1) **Findings present** ŌĆö trailer prescribes the orchestrator-side review-loop discipline (judge whether the items must be addressed; if yes, dispatch a fresh implementer Agent to address them and re-invoke the review skill on the updated diff; if no, carry the ignored items into the final/Bee-level summary so they remain visible); (2) **No findings** ŌĆö trailer prescribes proceeding to the next review lane (or to Task / Issue close-out if this was the last lane), with no re-dispatch needed for the reviewed role on this iteration. The phrasings are subject-specific per skill (the Engineer reviewer's trailer names "Engineer Agent"; the Test Writer reviewer's names "Test Writer Agent"; the Doc Writer reviewer's names "Doc Writer Agent") but share the same skeletal structure.

**Orchestrator-side prose downgrade.** At each of the five orchestrator sites, the prose that previously carried the load-bearing routing prescription is downgraded to reference context with explicit language naming the trailer as authoritative ŌĆö phrasings of the form "follow the **Next action for the orchestrator:** line in the review output" plus a "follow the trailer literally; the prose below is reference context, not a load-bearing rule the orchestrator must recall from memory" callout. The reference prose is kept (it documents the routing rules for human readers and for maintainers who diff the orchestrator skills), but the orchestrator agent is explicitly told that the trailer wins when the two ever diverge (and that divergence is a bug in the review skill to file). `/quo-plan` Step 4c additionally carries a quick-reference summary table of the three trailer shapes (a belt-and-suspenders measure at the orchestrator site that surfaces the routing prescription closer to where the orchestrator's attention lands ŌĆö the table is reference context only; the trailer in the review output remains canonical).

**Standalone-invocation handling.** When a review skill is invoked standalone by a user (not by an orchestrating skill), the "orchestrator" the trailer addresses is the assistant turn presenting the findings to the user, and the user is the human in the gate prompt. The trailer still emits in all output shapes on the standalone path ŌĆö it is not suppressed ŌĆö both so the standalone caller benefits from the same routing prescription and so its presence acts as a self-check that the skill considered the routing question. This is consistent with the dual-mode design of `/quo-spec-review` and the orchestrator-only design of the three code/test/doc reviewers (which on the rare standalone invocation path are read by a human directly).

**Why this shape.** Three design decisions shape the fix:

- **Routing rule lives with the artifact.** Orchestrator agents reproducibly miss routing rules buried in nested orchestrator prose under load ŌĆö the agent reads the rule once at skill-invocation time and then re-reads only the review-skill output when consuming findings. Moving the routing rule into the review-skill output itself means the agent re-reads the rule every time it consumes the findings, which is exactly when the rule is needed. This is structurally similar to the cross-anchored prose pattern shipped in `b.v4g` for the `/quo-file-issue` Ōåö `/quo-fix-issue` hand-off contract: both fixes move a load-bearing prose anchor to the place the consuming agent actually re-reads, rather than relying on prose-elsewhere-in-this-skill-body recall.
- **Verbatim emit, not paraphrase.** The trailer phrasings are emitted verbatim by the review skills (literal `**Next action for the orchestrator:**` prefix; literal `AskUserQuestion`-with-finite-choices language; literal "Do not yield without doing this" enforcement clause where applicable). Paraphrased trailers would invite per-invocation drift in the load-bearing routing prescription, defeating the move-to-artifact purpose. The orchestrator prose downgrade explicitly calls out that the trailer is the authoritative routing prescription and that divergence between trailer and prose is a bug in the review skill.
- **No control-flow change.** The reconciliation loop's tick semantics, the time-budget short-circuit, the AskUserQuestion option-ordering rules, the end-of-skill report bucketing for acknowledged/overridden findings, and the inline-skip contract (`/quo-write-prd` Step 6a and `/quo-write-sdd` Step 7a skip when invoked inline from `/quo-plan` Step 4b) are all unchanged. The fix is purely a relocation of where the routing prescription is authored so the orchestrator agent reproducibly reads it at the right moment.

**No agent / contract changes.** No agent role files (`agents/engineer.md`, `agents/test-writer.md`, `agents/doc-writer.md`, `agents/pm.md`, `agents/code-reviewer.md`, `agents/test-reviewer.md`, `agents/doc-reviewer.md`) are touched ŌĆö the trailer convention is a skill-to-skill prose contract between the four review skills and the five orchestrator sites that consume their output, not an agent-side instruction. No `## Build Commands` / `## Documentation Locations` contract keys are added or renamed. No bundled helper scripts are added or modified. The bees CLI and the ticket shapes are all unchanged. Existing tickets and existing review-loop iteration outputs do not require migration; the trailer is invisible from the bees-state perspective.

**No README change.** The user-typed slash-command invocation surfaces for all four review skills and all five orchestrator skills are byte-for-byte unchanged, and the README skill-table rows describe what each skill does at a user-visible level that this fix does not shift. The trailer convention is an internal contract between review skills and their orchestrators ŌĆö not user-visible ŌĆö and is not surfaced in the README skill catalog.

**Decomposition.** Shipped as a single Issue (`b.sfy`) ŌĆö a prose-only hardening across nine `SKILL.md` files (the four review skills and the five orchestrator skills) plus this cumulative-doc subsection. No multi-Task decomposition was warranted; the trailer-emit prose in the four review skills and the prose-downgrade-plus-quick-reference at each of the five orchestrator sites are tightly coupled and ship together as a single atomic change. No PRD subsection is added because the user-facing behavior is unchanged ŌĆö the categorization heuristic in `agents/doc-writer.md` places this fix in the architecture-only row (orchestrator-orchestrated contract refinement with no user-visible surface change), which warrants an SDD entry but not a PRD entry.

### Feature: PM dispatch in /quo-fix-issue: drop Complex gate, Section 4 timing, short-circuit

**Architecture.** Three coupled changes to the per-Issue PM dispatch flow that disentangle three concerns the previous design conflated ŌĆö *whether* to dispatch PM, *when* to dispatch PM, and *what* PM does on an Issue with no spec surface to verify. The PM goes from "dispatched only when the orchestrator pre-classifies the Issue as Complex, in parallel with the implementers in Section 3" to "always dispatched per Issue, alongside the three reviewers in Section 4 after the diff exists, with a content-based self-short-circuit inside `agents/pm.md` for the no-spec-surface case." The orchestrator's role on this question collapses to "always dispatch"; the substance judgement moves into the PM Agent where the spec content is actually readable.

- **Drop the orchestrator-direct Simple/Complex gate.** The "Assess complexity" sub-section in `skills/quo-fix-issue/SKILL.md` Section 3 is deleted in its entirety. The two bullet lists it carried (Simple = categories of work; Complex = risk surfaces) were structurally non-symmetric ŌĆö a 2-file source+test fix matched neither side cleanly, and the catch-all "could have non-obvious side effects" bullet pulled almost any code-touching change toward Complex. The gate also fired *before* the Engineer ran, but several of its bullets ("changes span 3+ source files") are properties of the post-implementation diff, which the orchestrator was guessing about from the Issue body. More fundamentally, the spec-drift-risk judgement requires reading the spec content; that content lives where the PM has access (the Issue body, the linked Plan Bee body, the cited PRD/SDD docs, the upstream URL content via `WebFetch`) and where the orchestrator does not (the orchestrator only sees the Issue body verbatim). Moving the judgement to the actor that has the context, rather than tuning the gate's thresholds, addresses the root cause of the false-positive Complex classifications the user observed as noise on routine fixes.
- **Relocate PM dispatch from Section 3 (implementer phase) to Section 4 (reviewer phase).** PM joins the three reviewer Agents (Code Reviewer / Test Reviewer / Doc Reviewer) in the Section 4 dispatch block, after the Engineer / Test Writer / Doc Writer have produced their diff. The implementer-phase Section 3 "Per-issue cold dispatch" sub-section drops PM from its role list and the "Per-issue PM dispatch" sub-section relocates to Section 4. Two consequences: (i) the internal contradiction at the former Section 3 reconciliation-loop step 2 (which simultaneously said "PM dispatched **alongside** the implementer Agents" and "the PM's report **joins the implementer outputs as input** to Section 4" ŌĆö parallel-and-sequential at once) is eliminated; (ii) the PM's role contract in `agents/pm.md` ŌĆö already correctly framed as the after-implementers gate ŌĆö is no longer in tension with the SKILL.md prose. The previous design had drift: maintainers updating the SKILL.md were one step away from "fixing" `agents/pm.md` to match the wrong side. The Section 4 move makes the SKILL.md the side that aligns with the role contract.
- **PM is dispatched outside the conditional-spawn rules for reviewers.** The Section 4 dispatch block has historically carried a rule that a reviewer is dispatched only when its corresponding implementer was used (Test Reviewer only when Test Writer ran; Doc Reviewer only when Doc Writer ran). That rule does not apply to PM: spec-alignment review is meaningful on any diff regardless of which implementers contributed, so PM is dispatched per Issue unconditionally. The Section 4 prose names PM as the exception to the conditional-spawn rule and as a fourth Agent alongside the three reviewers, not instead of any of them. The Section 4 review-iteration logic also drops its former "if the original fix was simple (no PM dispatched), do NOT dispatch the PM on iteration" branch ŌĆö PM is always available for re-dispatch on review-loop iterations.
- **Section 5 collapses to a single unified path.** The previous Section 5 branched on the Complex/Simple classification ŌĆö Complex deferred spec-alignment to the PM dispatched in Section 3, Simple did an orchestrator-direct spec-vs-code check. In the new design the PM was always dispatched in Section 4 and has either produced a spec-alignment verdict or short-circuited with "no spec drift surface to review for this Issue" ŌĆö either way the verdict lands in the per-issue summary, so Section 5 is always informational confirmation of what the PM already produced, never load-bearing orchestrator-direct work. The unified path keeps the `## Doc divergence noted` confirmation logic unchanged.
- **Content-based self-short-circuit in `agents/pm.md`.** The PM's flow gains a no-spec-surface short-circuit at the top: read every available spec source first (the Issue body itself, the Plan Bee body if `up_dependencies` links one via the existing Scoped-marker Path B, the PRD/SDD-equivalent docs from CLAUDE.md `## Documentation Locations` if the body cites them, the upstream content fetched via `WebFetch` if `reference_materials` is set), then assess whether those sources *collectively* carry substantive spec content ŌĆö behavior described, architecture constrained, requirements specified ŌĆö beyond a title plus a one-sentence problem statement that merely identifies what to fix without specifying how. When substantive content is present in any source, the PM proceeds with the existing deep spec-vs-diff review against that substantive source. When substance is absent across all available sources, the PM exits with a one-line verdict ("no spec drift surface to review for this Issue") and skips the deep review. The orchestrator reads either outcome as a clean signal and includes it in the per-issue summary.
- **The short-circuit is content-based, not shape-based ŌĆö by construction.** The PM's short-circuit logic explicitly forbids keying off shape signals like "PRD/SDD path mentioned in body," "Plan Bee linked in `up_dependencies`," or "`reference_materials` is non-null." Those signals do not reliably correlate with spec richness: a body can cite a PRD path and still be a one-line rename; a Plan Bee can be linked for a trivial typo fix; a one-sentence GitHub Issue can sit behind a non-null `reference_materials`. Conversely, a self-describing body with no external pointers can carry rich behavioral spec the PM must verify against. A shape-based pre-read check inside the PM would reproduce the exact failure mode of the orchestrator-side gate this Issue eliminates, just relocated; the content-based rule prevents that regression by mandating the PM read content before deciding.

**Why this shape.** Four design decisions shape the fix:

- **Doc Writer precedent for "always dispatched."** The same skill already names Doc Writer as "always dispatched. The Doc Writer decides whether docs actually need updating after reading the diff; the orchestrator does not pre-judge." PM now follows the same pattern, with the spec-surface check standing in for the docs-needed check. This unifies the two roles that both make a "does this diff need review?" judgement that the orchestrator was structurally bad at predicting, and matches the existing precedent rather than introducing a new dispatch pattern.
- **Move the judgement to the actor that has the context.** The orchestrator can only see the Issue body verbatim; the PM Agent can read the Issue body, follow `up_dependencies` to the Plan Bee, walk the Scoped-marker resolver to the PRD/SDD content, and `WebFetch` upstream URLs. Spec-drift risk is a function of how much content is actually in those sources, not a property predictable from the Issue body alone. Putting the judgement where the content is readable addresses the root cause of false-positive Complex classifications rather than tuning thresholds on a structurally under-informed gate.
- **No latency penalty for the always-dispatched move.** PM runs in parallel with Code Reviewer / Test Reviewer / Doc Reviewer in Section 4, and the three reviewers already gate Section 5 entry, so adding PM to that parallel block does not lengthen Section 4. The token-cost delta is one additional PM Agent invocation per Issue that today classifies as Simple (those used to skip PM entirely) ŌĆö but those invocations typically hit the short-circuit and end quickly, so the per-Issue cost on the formerly-Simple path is small rather than zero.
- **No new control-flow primitives.** The reconciliation loop's tick semantics, the per-Issue cold-dispatch shape, the TaskList naming convention (`pm-<issue-id>` survives unchanged; the bullet relocates from the implementer-Agents group to the reviewer-Agents group), and the verbatim ticket-body quoting in dispatch prompts are all unchanged. The fix is a re-routing of when PM is dispatched and a self-short-circuit inside the PM, not a new orchestration primitive. Six alternative designs (two-phase PM, internal-wait PM, gate-only timing fix, drop-gate-keep-Section-3, shape-based short-circuit, and the chosen drop-gate-plus-Section-4-plus-content-short-circuit) were considered; the chosen design is the only one that addresses all three concerns (whether, when, what) without adding orchestration complexity, and is recorded in the Issue's `## Decisions and rejected alternatives` section.

**No README change.** The user-typed `/quo-fix-issue` invocation surface is byte-for-byte unchanged ŌĆö no new flags, no new arguments, no new prompts. The README skill-table row for `/quo-fix-issue` does not name the Complex/Simple distinction (the gate was internal orchestrator prose, never user-visible CLI), and the `## Why this exists` PM bullet at line 31 already describes the PM as "After Engineer, Test Writer, and Doc Writer complete a Task, a PM agent reads ..." ŌĆö which is the post-fix timing, not the pre-fix Section 3 timing the SKILL.md had drifted toward. The README is the side that stayed correct; the SKILL.md is the side that drifted, and only the SKILL.md needs correction.

**No agent / contract changes beyond `agents/pm.md`.** No other agent role files are touched. No `## Build Commands` / `## Documentation Locations` contract keys are added or renamed. No bundled helper scripts are added or modified. The bees CLI, the Issue ticket shape, and `reference_materials` resolver vocabulary are all unchanged. Existing Issue tickets do not require migration; the dispatch-timing and short-circuit changes are invisible from the bees-state perspective. `/quo-execute` is unaffected ŌĆö that skill always dispatches PM per Task and has never carried a complexity gate to remove.

**Decomposition.** Shipped as a single Issue (`b.tmk`) ŌĆö a prose-only fix across one `SKILL.md` file (`skills/quo-fix-issue/SKILL.md`) and one agent role file (`agents/pm.md`) plus this cumulative-doc subsection. The combined fix is cleaner than either fix alone. The user-perceived noise reduction on trivial fixes is an internal quality improvement (a mis-classification was happening invisibly inside the orchestrator and producing extra review-burden lines in the per-issue summary), not a new surface the user interacts with.
